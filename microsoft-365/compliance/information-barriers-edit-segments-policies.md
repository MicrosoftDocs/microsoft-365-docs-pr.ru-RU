# <a name="manage-information-barrier-policies"></a>Управление политиками информационных барьеров

После определения [политик](information-barriers-policies.md)информационных барьеров может потребоваться внести изменения в эти политики или [](information-barriers-troubleshooting.md) в пользовательские сегменты в рамках устранения неполадок или регулярного обслуживания. Используйте эту статью в качестве руководства.

## <a name="what-do-you-want-to-do"></a>Что нужно сделать

|**Действие**|**Описание**|
|:---------|:--------------|
| [Изменение атрибутов учетной записи пользователя](#edit-user-account-attributes) | Заполните атрибуты в Azure Active Directory, которые можно использовать для определения сегментов.<br/>Измените атрибуты учетной записи пользователя, если пользователи не включены в сегменты, которые они должны быть, чтобы изменить сегменты, в которых находятся пользователи, или определить сегменты с использованием различных атрибутов. |
| [Изменение сегмента](#edit-a-segment) | Изменение сегментов при изменении определения сегмента. <br/>Например, вы могли изначально определять сегменты с помощью *Department* и теперь хотите использовать другой атрибут, например *MemberOf.* |
| [Изменение политики](#edit-a-policy) | Измените политику информационных барьеров, если хотите изменить ее работу.<br/>Например, вместо блокирования взаимодействия между двумя сегментами можно разрешить взаимодействие только между определенными сегментами. |
| [Настройка неактивного состояния политики](#set-a-policy-to-inactive-status) |Установите для политики состояние неактивности, если вы хотите внести изменения в политику или если вы не хотите, чтобы политика вступила в силу. |
| [Удаление политики](#remove-a-policy) | Удалите политику информационных барьеров, если вам больше не нужна определенная политика. |
| [Остановка приложения политики](#stop-a-policy-application) | Это действие необходимо, чтобы остановить процесс применения политик информационных барьеров.<br/> Остановка приложения политики не является мгновенной и не отменяет политики, которые уже применены к пользователям. |
| [Определение политик информационных барьеров](information-barriers-policies.md) | Определите политику информационных барьеров, если у вас еще нет таких политик, и необходимо ограничить или ограничить связь между определенными группами пользователей. |
| [Устранение проблем с информационными барьерами](information-barriers-troubleshooting.md) | Если у вас непредвиденное проблемы с информационными барьерами, обратитесь к этой статье. |

## <a name="edit-user-account-attributes"></a>Изменение атрибутов учетной записи пользователя

Используйте эту процедуру для изменения атрибутов, используемых для сегментации пользователей. Например, если вы используете атрибут Department и одна или несколько учетных записей пользователей в настоящее время не имеют значений, указанных для отдела, необходимо изменить эти учетные записи пользователей, чтобы включить сведения о отделе. Атрибуты учетной записи пользователя используются для определения сегментов, чтобы можно было наградить политики информационных барьеров.

1. Для просмотра сведений об определенной учетной записи пользователя, например значений атрибутов и определенных сегментов, используйте с параметрами Identity cmdlet **Get-InformationBarrierRecipientStatus.**

    |**Синтаксис**|**Пример**|
    |:---------|:----------|
    | `Get-InformationBarrierRecipientStatus -Identity <value> -Identity2 <value>` <p> Можно использовать любое уникальное значение, идентифицируя каждого пользователя, например имя, псевдоним, различающейся имя, каноническое доменное имя, адрес электронной почты или ИДЕНТИФИКАТОР GUID. <p> (Вы также можете использовать этот cmdlet для одного пользователя: `Get-InformationBarrierRecipientStatus -Identity <value>` ) |`Get-InformationBarrierRecipientStatus -Identity meganb -Identity2 alexw` <p> В этом примере мы ссылаемся на две учетные записи пользователей в Office 365: *meganb* для Марта *и* *alexw* для *Алекса*. |

2. Определите, какой атрибут нужно изменить для профилей учетных записей пользователей. Дополнительные сведения см. в [атрибутах политик информационных барьеров.](information-barriers-attributes.md) 

3. Отредактируем одну или несколько учетных записей пользователей, чтобы включить значения для атрибута, выбранного на предыдущем шаге. Чтобы воспользоваться этим действием, используйте одну из следующих процедур:

    - Чтобы изменить одну учетную запись, см. статью "Добавление или обновление сведений профиля пользователя [с помощью Azure Active Directory".](https://docs.microsoft.com/azure/active-directory/fundamentals/active-directory-users-profile-azure-portal)

    - Чтобы изменить несколько учетных записей (или изменить одну учетную запись с помощью PowerShell), см. статью "Настройка свойств учетной записи пользователя с помощью [PowerShell в Office 365".](https://docs.microsoft.com/microsoft-365/enterprise/configure-user-account-properties-with-microsoft-365-powershell)

## <a name="edit-a-segment"></a>Изменение сегмента

Используйте эту процедуру для изменения определения пользовательского сегмента. Например, можно изменить имя сегмента или фильтр, используемый для определения, кто входит в этот сегмент.

1. Чтобы просмотреть все существующие сегменты, используйте **cmdlet Get-OrganizationSegment.**

    Синтаксис: `Get-OrganizationSegment`

    Вы увидите список сегментов и сведения для каждого из них, например тип сегмента, его значение UserGroupFilter, кто создал или последним изменил его, GUID и так далее.

    > [!TIP]
    > Распечатайте или сохраните список сегментов для справки позже. Например, если требуется изменить сегмент, необходимо знать его имя или значение идентификации (используется вместе с параметром Identity).

2. Чтобы изменить сегмент,  используйте его с параметром **Identity** и соответствующими сведениями.

    |**Синтаксис**|**Пример**|
    |:---------|:----------|
    | `Set-OrganizationSegment -Identity GUID -UserGroupFilter "attribute -eq 'attributevalue'"` |`Set-OrganizationSegment -Identity c96e0837-c232-4a8a-841e-ef45787d8fcd -UserGroupFilter "Department -eq 'HRDept'"` <p> В этом примере для сегмента с GUID *c96e0837-c232-4a8a-841e-ef45787d8fcd мы* обновили название отдела на HRDept. |

Завершив редактирование сегментов для организации, вы [](information-barriers-policies.md#part-2-define-information-barrier-policies) можете [](#edit-a-policy) определить или изменить политики информационных барьеров.

## <a name="edit-a-policy"></a>Изменение политики

1. Чтобы просмотреть список текущих политик информационных барьеров, используйте cmdlet **Get-InformationBarrierPolicy.**

    Синтаксис: `Get-InformationBarrierPolicy`

    В списке результатов определите политику, которую необходимо изменить. Обратите внимание на GUID и имя политики.

2. Используйте для **параметра Identity параметр Set-InformationBarrierPolicy** и укажите изменения, которые необходимо внести. 

    Пример. Предположим, что определена политика, блокированная для сегмента *Research* связи с сегментами *продаж* и *маркетинга.* Политика была определена с помощью этого cmdlet: `New-InformationBarrierPolicy -Name "Research-SalesMarketing" -AssignedSegment "Research" -SegmentsBlocked "Sales","Marketing"`

    Предположим, что мы хотим изменить его, чтобы люди в сегменте *"Исследования"* могли общаться только с людьми из *сегмента отдела* кадров. Чтобы внести это изменение, мы используем этот cmdlet: `Set-InformationBarrierPolicy -Identity 43c37853-ea10-4b90-a23d-ab8c93772471 -SegmentsAllowed "HR"`

    В этом примере мы изменили "SegmentsBlocked" на "SegmentsAllowed" и указали *сегмент отдела кадров.*

3. Завершив редактирование политики, обязательно примените изменения. [(См. "Применение политик информационных барьеров".)](information-barriers-policies.md#part-3-apply-information-barrier-policies)

## <a name="set-a-policy-to-inactive-status"></a>Настройка неактивного состояния политики

1. Чтобы просмотреть список текущих политик информационных барьеров, используйте cmdlet **Get-InformationBarrierPolicy.**

    Синтаксис: `Get-InformationBarrierPolicy`

    В списке результатов определите политику, которую необходимо изменить (или удалить). Обратите внимание на GUID и имя политики.

2. Чтобы установить состояние политики как неактивной, используйте с параметром Identity с параметром **Set-InformationBarrierPolicy** и параметром State параметр "Неактивный".

    |**Синтаксис**|**Пример**|
    |:---------|:----------|
    | `Set-InformationBarrierPolicy -Identity GUID -State Inactive` | `Set-InformationBarrierPolicy -Identity 43c37853-ea10-4b90-a23d-ab8c9377247 -State Inactive` <p> В этом примере для политики информационных барьеров с GUID *43c37853-ea10-4b90-a23d-ab8c9377247* устанавливается состояние неактивности. |

3. Чтобы применить изменения, используйте cmdlet **Start-InformationBarrierPoliciesApplication.**

    Синтаксис: `Start-InformationBarrierPoliciesApplication`

    Изменения применяются к вашей организации с учетом пользователя. Если организация большая, этот процесс может занять 24 часа (или больше). (Как правило, обработка 5000 учетных записей пользователей занимает около часа.)

На этом этапе одна или несколько политик информационных барьеров находятся в неактивном состоянии. Здесь можно сделать любое из следующих действий:

- Оставить его без изменения (политика с состоянием неактивности не влияет на пользователей)
- [Изменение политики](#edit-a-policy) 
- [Удаление политики](#remove-a-policy)

## <a name="remove-a-policy"></a>Удаление политики

1. Чтобы просмотреть список текущих политик информационных барьеров, используйте cmdlet **Get-InformationBarrierPolicy.**

    Синтаксис: `Get-InformationBarrierPolicy`

    В списке результатов определите политику, которую нужно удалить. Обратите внимание на GUID и имя политики. Убедитесь, что для политики установлено неактивное состояние.

2. Используйте с **параметром Identity параметр Remove-InformationBarrierPolicy.**

    |**Синтаксис**|**Пример**|
    |:---------|:----------|
    | `Remove-InformationBarrierPolicy -Identity GUID` | `Remove-InformationBarrierPolicy -Identity 43c37853-ea10-4b90-a23d-ab8c93772471` <p> В этом примере удаляется политика с GUID *43c37853-ea10-4b90-a23d-ab8c93772471.* |

    При запросе подтвердит изменение.

3. Повторите шаги 1–2 для каждой удаляемой политики.

4. После удаления политик примените изменения. Чтобы сделать это действие, используйте cmdlet **Start-InformationBarrierPoliciesApplication.**

    Синтаксис: `Start-InformationBarrierPoliciesApplication`

    Изменения применяются к вашей организации с учетом пользователя. Если организация большая, этот процесс может занять 24 часа (или больше).

## <a name="stop-a-policy-application"></a>Остановка приложения политики

После начала применения политик информационных барьеров, если вы хотите остановить применение этих политик, используйте следующую процедуру. Процесс начинается примерно через 30–35 минут.

1. Чтобы просмотреть состояние последнего приложения политики информационных барьеров, используйте cmdlet **Get-InformationBarrierPoliciesApplicationStatus.**

    Синтаксис: `Get-InformationBarrierPoliciesApplicationStatus`

    Обратите внимание на GUID приложения.

2. Используйте с **параметром Identity параметр Stop-InformationBarrierPoliciesApplication.**

    |**Синтаксис**|**Пример**|
    |:---------|:----------|
    | `Stop-InformationBarrierPoliciesApplication -Identity GUID` | `Stop-InformationBarrierPoliciesApplication -Identity 46237888-12ca-42e3-a541-3fcb7b5231d1` <p> В этом примере мы останавливаем применять политики информационных барьеров. |
