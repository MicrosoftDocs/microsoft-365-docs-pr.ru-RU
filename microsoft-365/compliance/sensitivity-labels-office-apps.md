# <a name="manage-sensitivity-labels-in-office-apps"></a>Управление метками конфиденциальности в приложениях Office

>*[Руководство по лицензированию Microsoft 365 для обеспечения безопасности и соответствия требованиям](https://aka.ms/ComplianceSD).*

Когда вы [](create-sensitivity-labels.md#publish-sensitivity-labels-by-creating-a-label-policy) опубликовали метки конфиденциальности из центра соответствия требованиям Microsoft 365 или центра эквивалентной маркировки, они начинают появляться в приложениях Office для классификации и защиты данных по мере их создания или редактирования.

Используйте сведения в этой статье, чтобы успешно управлять метами конфиденциальности в приложениях Office. Например, определите минимальные версии приложений, необходимых для поддержки встроенной маркировки, и определите взаимодействие с клиентом единой метки Azure Information Protection и совместимость с другими приложениями и службами.

## <a name="labeling-client-for-desktop-apps"></a>Маркировка клиента для настольных приложений

Чтобы использовать метки конфиденциальности, встроенные в настольные приложения Office для Windows и Mac, необходимо использовать подписку на выпуск Office. Этот клиент с меткой не поддерживает автономные выпуски Office, например Office 2016 или Office 2019.

Чтобы использовать метки конфиденциальности с помощью этих автономных выпусков Office на компьютерах Windows, установите клиент единой маркировки [Azure Information Protection.](https://docs.microsoft.com/azure/information-protection/rms-client/aip-clientv2)

## <a name="office-built-in-labeling-client-and-other-labeling-solutions"></a>Клиент и другие решения для метки Office, встроенные в office

Клиент, встроенный в office, загружает метки чувствительности и параметры политики меток конфиденциальности из следующих центров администрирования:

- Центр соответствия требованиям Microsoft 365
- Центр безопасности Microsoft 365
- Центр безопасности и соответствия требованиям Office 365

Чтобы использовать клиент встроенной маркировки Office, необходимо [](create-sensitivity-labels.md#publish-sensitivity-labels-by-creating-a-label-policy) опубликовать пользователям одну или несколько политик меток из одного из перечисленных центров администрирования и поддерживаемую версию [Office.](#support-for-sensitivity-label-capabilities-in-apps)

Если оба этих условия выполнены, но необходимо отключить клиент встроенной маркировки Office, используйте следующий параметр групповой политики:

1. Перейдите **к настройкам пользователя/административным шаблонам/Microsoft Office 2016/Security Settings**.

2. Set **Use the Sensitivity feature in Office to apply and view sensitivity labels** to **0.** 
 
Развертывание этого параметра с помощью групповой политики или службы [облачной политики Office.](https://docs.microsoft.com/DeployOffice/overview-office-cloud-policy-service) Параметр вступает в силу при перезапуске приложений Office.

### <a name="office-built-in-labeling-client-and-the-azure-information-protection-client"></a>Клиент встроенной маркировки Office и клиент Azure Information Protection

Если у пользователей установлен один из клиентов Azure Information Protection[(клиент](https://docs.microsoft.com/azure/information-protection/rms-client/aip-clientv2) единой маркировки или классический [клиент),](https://docs.microsoft.com/azure/information-protection/rms-client/aip-client)по умолчанию встроенный клиент метки отключен в приложениях Office. 

Чтобы использовать встроенную маркировку, а не клиент Azure Information Protection для приложений Office, используйте инструкции из предыдущего раздела, но установите параметр Групповой политики Используйте функцию "Чувствительность" в **Office** для применения и просмотра меток чувствительности до **1**. 

Кроме того, отключите или удалите надстройку Office, **Azure Information Protection.** Этот метод подходит для одного компьютера и для отдельного тестирования. Инструкции см. в обзоре, управлении и установке [надстройок в программах Office.](https://support.office.com/article/16278816-1948-4028-91e5-76dca5380f8d) 

Когда вы отключите или удалите эту надстройку Office, клиент Azure Information Protection остается установленным, чтобы вы могли продолжать маркировать файлы за пределами приложений Office. Например, с помощью Обозревателя файлов или PowerShell.

Сведения о том, какие функции поддерживаются клиентами Azure Information Protection и клиентом встроенной маркировки Office, см. в документе Выбор клиента метки, который можно использовать для компьютеров [Windows](https://docs.microsoft.com/azure/information-protection/rms-client/use-client#choose-which-labeling-client-to-use-for-windows-computers) из документации по защите информации Azure.

## <a name="office-file-types-supported"></a>Поддерживаемые типы файлов Office

Приложения Office, которые имеют встроенную метку для файлов Word, Excel и PowerPoint, поддерживают формат Open XML (например, .docx и .xlsx Microsoft Office), но не формат 97-2003 (например, форматы .doc и .xls), формат open document (например, форматы .odt и .ods) или другие форматы. Если тип файла не поддерживается для встроенной маркировки, кнопка **Sensitivity** недоступна в приложении Office.

Клиент единой маркировки Azure Information Protection поддерживает формат Open XML Microsoft Office формате 97-2003. Дополнительные сведения см. в материалах [Типов файлов,](https://docs.microsoft.com/azure/information-protection/rms-client/clientv2-admin-guide-file-types) поддерживаемых клиентом единой маркировки Azure Information Protection из руководства по администрированию этого клиента.

Для других решений по маркировке проверьте их документацию на поддерживаемые типы файлов.

## <a name="protection-templates-and-sensitivity-labels"></a>Шаблоны защиты и метки конфиденциальности

Шаблоны защиты, определенные [администратором,](https://docs.microsoft.com/azure/information-protection/configure-policy-templates)например те, которые вы определяете для шифрования сообщений Office 365, не видны в приложениях Office при использовании встроенной маркировки. Это упрощенное впечатление отражает, что нет необходимости выбирать шаблон защиты, так как эти же параметры включены с метки чувствительности, которые имеют шифрование включено.

Если необходимо преобразовать существующие шаблоны защиты в метки, используйте портал Azure и следующие инструкции. Чтобы преобразовать шаблоны [в метки.](https://docs.microsoft.com/azure/information-protection/configure-policy-templates#to-convert-templates-to-labels)

## <a name="information-rights-management-irm-options-and-sensitivity-labels"></a>Параметры управления правами на информацию (IRM) и метки конфиденциальности

Метки конфиденциальности, настроенные для применения шифрования, устраняют сложности с пользователями, чтобы указать собственные параметры шифрования. Во многих приложениях Office эти отдельные параметры шифрования по-прежнему могут настраиваться пользователями вручную с помощью параметров управления правами на информацию (IRM). Например, для приложений Windows:

- Для документа: **Защита информации**  >  **о**  >  **файлах, ограничивающий**  >  **доступ к документу**
- для электронной почты: с **вкладки Параметры** > **Шифровать** 
  
Когда пользователи изначально помечены документом или электронной почтой, они всегда могут переопределять параметры конфигурации меток с помощью собственных параметров шифрования. Например:

- Пользователь применяет метку **Confidential \ All Employees** к документу, и эта метка настроена для применения параметров шифрования для всех пользователей в организации. Затем этот пользователь вручную настраивает параметры IRM, чтобы ограничить доступ к пользователю за пределами организации. Конечным результатом является документ, помеченный как **Конфиденциальный \ Все** сотрудники и зашифрованный, но пользователи в вашей организации не могут открыть его, как ожидалось.

- Пользователь применяет метку **Confidential \ Recipients Only** для электронной почты, и эта электронная почта настроена для применения параметра шифрования Do Not **Forward**. Затем этот пользователь вручную настраивает параметры IRM, чтобы электронная почта была неограниченной. Конечный результат — электронная почта может быть отправлена получателями, несмотря на наличие метки **Confidential \ Recipients Only.**

- Пользователь применяет **общую** метку к документу, и эта метка не настроена для применения шифрования. Затем этот пользователь вручную настраивает параметры IRM, чтобы ограничить доступ к документу. Конечным результатом является документ с меткой **General,** который также применяет шифрование, чтобы некоторые пользователи не могли открыть его так, как ожидалось.

Если документ или электронная почта уже помечены, пользователь может сделать любое из этих действий, если содержимое еще не зашифровано, или у них есть право на экспорт или полный контроль использования. [](https://docs.microsoft.com/azure/information-protection/configure-usage-rights#usage-rights-and-descriptions) 

Чтобы обеспечить более согласованную метку с содержательными отчетами, внесите соответствующие метки и рекомендации, чтобы пользователи применяли только метки для защиты документов. Например:

- Для исключений, когда пользователям необходимо назначить собственные разрешения, укайте метки, которые дают возможность пользователям [назначать собственные разрешения.](encryption-sensitivity-labels.md#let-users-assign-permissions) 

- Вместо того, чтобы пользователи вручную удаляли шифрование после выбора метки, применяемой для шифрования, предостанавлив альтернативу подлабелю, когда пользователям нужна метка с той же классификацией, но без шифрования. Например:
    - **Конфиденциально \ Все сотрудники**
    - **Confidential \ Anyone (без шифрования)**

> [!NOTE]
> Если пользователи вручную удалят шифрование из помеченного документа, хранимого в SharePoint или OneDrive, и вы включили метки конфиденциальности для файлов Office в SharePoint и [OneDrive,](sensitivity-labels-sharepoint-onedrive-files.md)шифрование меток будет автоматически восстановлено при следующем доступе к документу или его загрузке. 


## <a name="apply-sensitivity-labels-to-files-emails-and-attachments"></a>Применение меток конфиденциальности к файлам, электронным письмам и вложениям

Пользователи могут применять только одну метку одновременно для каждого документа или электронной почты.

Когда вы напишите сообщение электронной почты с вложениями, вложения наследуют метку только в том случае, если метка, которая применяется к сообщению электронной почты, применяется шифрование, а вложение — это документ Office, который еще не зашифрован. Так как наследуемая метка применяет шифрование, вложение становится вновь зашифрованным.

Вложение не наследует метки из сообщения электронной почты, когда метка, примененная к сообщению электронной почты, не применяет шифрование или вложение уже зашифровано.

Примеры наследования меток, где метка **Конфиденциально**  применяет шифрование, а общая метка не применяет шифрование:

- Пользователь создает новое сообщение электронной почты и применяет к этому сообщению метку **Confidential.** Затем они добавляют документ Word, который не помечен и не зашифрован. В результате наследования документ вновь помечен как **Конфиденциальный** и теперь имеет шифрование, примененное с этой метки.

- Пользователь создает новое сообщение электронной почты и применяет к этому сообщению метку **Confidential.** Затем они добавляют документ Word, помеченный как **General,** и этот файл не шифруется. В результате наследования документ получает повторное обозначение **Confidential** и теперь используется шифрование с этой метки.

## <a name="sensitivity-label-compatibility"></a>Совместимость меток чувствительности

С помощью просветленных приложений с поддержкой **RMS.** Если вы откроете помеченный и зашифрованный документ или электронную почту в просветленном [RMS-приложении,](https://docs.microsoft.com/azure/information-protection/requirements-applications#rms-enlightened-applications) которое не поддерживает метки конфиденциальности, приложение по-прежнему обеспечивает шифрование и управление правами.

**С клиентом Azure Information Protection**: Вы можете просматривать и изменять метки конфиденциальности, которые применяются к документам и электронным письмам с клиентом встроенной маркировки Office с помощью клиента Azure Information Protection и наоборот.

**С другими версиями Office**: Любой авторизованный пользователь может открывать помеченные документы и электронные письма в других версиях Office. Однако просмотреть или изменить метку можно только в поддерживаемых версиях Office или с помощью клиента Azure Information Protection. Поддерживаемые версии приложений Office перечислены в предыдущем [разделе](#support-for-sensitivity-label-capabilities-in-apps).

## <a name="support-for-sharepoint-and-onedrive-files-protected-by-sensitivity-labels"></a>Поддержка файлов SharePoint и OneDrive, защищенных метами конфиденциальности

Чтобы использовать клиент встроенной маркировки Office с Office в Интернете для документов в SharePoint или OneDrive, убедитесь, что вы включили метки конфиденциальности для файлов Office в SharePoint и [OneDrive.](sensitivity-labels-sharepoint-onedrive-files.md)

## <a name="support-for-external-users-and-labeled-content"></a>Поддержка внешних пользователей и помеченного контента

При метке документа или электронной почты метка хранится в виде метаданных, включаемой клиента и GUID-метки. Когда помеченный документ или электронная почта открываются приложением Office, которое поддерживает метки конфиденциальности, эти метаданные читают, и только если пользователь принадлежит одному и тому же клиенту, метка отображается в приложении. Например, для встроенной маркировки word, PowerPoint и Excel имя метки отображается на панели состояния. 

Это означает, что если вы делитесь документами с другой организацией, использующей разные имена меток, каждая организация может применить и увидеть свою собственную метку, примененную к документу. Однако для пользователей за пределами организации видны следующие элементы из примененной метки:

- Маркировка контента. Когда метка применяет загон, подножку или водяной знак, они добавляются непосредственно в содержимое и остаются видимыми до тех пор, пока кто-то не изменяет или не удаляет их.

- Имя и описание основному шаблону защиты от метки, применяемой шифрованием. Эта информация отображается в панели сообщений в верхней части документа, чтобы предоставить сведения о том, кому разрешено открывать документ, и их права на использование для этого документа.

### <a name="sharing-encrypted-documents-with-external-users"></a>Обмен зашифрованными документами с внешними пользователями

Помимо ограничения доступа к пользователям в вашей организации, вы можете расширить доступ к любому другому пользователю, у которого есть учетная запись в Azure Active Directory. Однако если в организации используются политики условного [](#conditional-access-policies) доступа, дополнительные соображения см. в следующем разделе.

Все приложения Office и другие просветленные приложения с помощью [RMS могут](https://docs.microsoft.com/azure/information-protection/requirements-applications#rms-enlightened-applications) открывать зашифрованные документы после успешной проверки подлинности пользователем. 

Если внешние пользователи не имеют учетной записи в Azure Active Directory, они могут проверить подлинность с помощью учетных записей гостей в клиенте. Эти гостевых учетные записи также можно использовать для доступа к общим документам в SharePoint или OneDrive, если вы включили метки конфиденциальности для файлов Office в [SharePoint и OneDrive:](sensitivity-labels-sharepoint-onedrive-files.md)

- Одним из вариантов является создание этих учетных записей для гостей самостоятельно. Вы можете указать любой адрес электронной почты, который уже используется этими пользователями. Например, их адрес Gmail.
    
    Преимущество этого параметра заключается в том, что вы можете ограничить доступ и права определенным пользователям, указав их адрес электронной почты в параметрах шифрования. Недостатком является накладная часть администрирования для создания учетной записи и координации с конфигурацией меток.

- Другим вариантом является использование интеграции SharePoint и OneDrive с [Azure AD B2B (Preview),](https://docs.microsoft.com/sharepoint/sharepoint-azureb2b-integration-preview) чтобы учетные записи гостей автоматически создавались при совместном использовании пользователями ссылок.
    
    Преимуществом этого параметра является минимальная административная накладная часть, так как учетные записи создаются автоматически, а конфигурация меток проще. Для этого сценария необходимо выбрать вариант шифрования [Добавить](encryption-sensitivity-labels.md#requirements-and-limitations-for-add-any-authenticated-users) любого пользователя с проверкой подлинности, так как вы не будете знать адреса электронной почты заранее. Недостатком является то, что этот параметр не позволяет ограничить доступ и права на использование для определенных пользователей.

Внешние пользователи также могут использовать учетную запись Майкрософт для открытия зашифрованных документов при использовании приложений Windows и Microsoft 365 (ранее[приложения Office 365)](https://docs.microsoft.com/deployoffice/name-change)или автономных выпусков Office 2019. Более недавно поддерживаемые для других платформ учетные записи Майкрософт также поддерживаются для открытия зашифрованных документов на macOS (Microsoft 365 Apps, версия 16.42+), Android (версия 16.0.13029+) и iOS (версия 2.42+). Например, пользователь в организации делится зашифрованным документом с пользователем за пределами организации, а параметры шифрования указывают адрес электронной почты Gmail для внешнего пользователя. Этот внешний пользователь может создать свою учетную запись Майкрософт, которая использует свой адрес электронной почты Gmail. После регистрации в этой учетной записи они могут открыть документ и изменить его в соответствии с указанными для них ограничениями использования. Пример этого сценария см. в примере открытия и [редактирования защищенного документа.](https://docs.microsoft.com/azure/information-protection/secure-collaboration-documents#opening-and-editing-the-protected-document)

> [!NOTE]
> Адрес электронной почты учетной записи Майкрософт должен соответствовать указанному адресу электронной почты, чтобы ограничить доступ к настройкам шифрования.

Если пользователь с учетной записью Майкрософт открывает зашифрованный документ таким образом, он автоматически создает учетную запись гостя для клиента, если гостевой учетной записи с тем же именем еще не существует. Когда учетная запись для гостей существует, ее можно использовать для открытия документов в SharePoint и OneDrive с помощью Office в Интернете, а также для открытия зашифрованных документов из поддерживаемых настольных и мобильных приложений Office.

Однако автоматическая учетная запись для гостей не создается сразу в этом сценарии из-за задержки репликации. Если вы указываете личные адреса электронной почты в качестве части параметров шифрования меток, рекомендуем создать соответствующие учетные записи гостей в Azure Active Directory. Затем дайте этим пользователям знать, что они должны использовать эту учетную запись для открытия зашифрованного документа из вашей организации.

> [!TIP]
> Так как вы не можете быть уверены, что внешние пользователи будут использовать поддерживаемые клиентские приложения Office, обмен ссылками из SharePoint и OneDrive после создания гостевых учетных записей (для определенных пользователей) или при использовании интеграции SharePoint и OneDrive с [Azure AD B2B](https://docs.microsoft.com/sharepoint/sharepoint-azureb2b-integration-preview) (для любого пользователя с проверкой подлинности) является более надежным методом для поддержки безопасного взаимодействия с внешними пользователями.

### <a name="conditional-access-policies"></a>Политики условного доступа

Если ваша организация реализовала политики условного доступа [Azure Active Directory,](https://docs.microsoft.com/azure/active-directory/conditional-access/overview)проверьте конфигурацию этих политик. Если политики включают защиту информации **Microsoft Azure** и политика распространяется на внешних пользователей, эти внешние пользователи должны иметь учетную запись гостя в клиенте, даже если у них есть учетная запись Azure AD в своем клиенте.

Без этой гостевой учетной записи они не могут открыть зашифрованный документ и увидеть сообщение об ошибке. Текст сообщения может информировать их о том, что их учетная запись должна быть добавлена в качестве внешнего пользователя в клиенте, с неправильной инструкцией для этого сценария, чтобы выйти и снова войти с другой учетной записью **Azure Active Directory**.

Если вы не можете создать и настроить учетные записи гостей в клиенте для внешних пользователей, которым необходимо открыть документы, зашифрованные меткой, необходимо либо удалить Azure Information Protection из политик условного доступа, либо исключить внешних пользователей из политик.

Дополнительные сведения о условном доступе и защите информации Azure, службе шифрования, используемой метами конфиденциальности, см. в часто задаемом вопросе. Я вижу, что Azure Information Protection указана как доступное облачное приложение для условного доступа, как это [работает?](https://docs.microsoft.com/azure/information-protection/faqs#i-see-azure-information-protection-is-listed-as-an-available-cloud-app-for-conditional-accesshow-does-this-work)

## <a name="when-office-apps-apply-content-marking-and-encryption"></a>Когда приложения Office применяют маркировку контента и шифрование

Приложения Office применяют маркировку контента и шифрование с меткой конфиденциальности по-разному, в зависимости от используемого приложения.

| Приложение | Маркировка содержимого | Шифрование |
| --- | --- | --- |
| Word, Excel, PowerPoint на всех платформах | Сразу | Сразу |
| Outlook для ПК и Mac | После отправки электронной почты Exchange Online | Сразу |
| Outlook в Интернете, для iOS и Android | После отправки электронной почты Exchange Online | После отправки электронной почты Exchange Online |
|

Решения, применяющие метки конфиденциальности к файлам за пределами приложений Office, применяются путем применения метаданных меток к файлу. В этом сценарии маркировка контента из конфигурации метки не вставляется в файл, но применяется шифрование. 

Когда эти файлы открываются в настольном приложении Office, маркировка контента автоматически применяется клиентом единой маркировки Azure Information Protection. Маркировка контента не применяется автоматически при использовании встроенной маркировки для настольных, мобильных или веб-приложений.

Сценарии, включающие применение метки конфиденциальности за пределами приложений Office, включают следующие:

- Сканер, обозреватель файлов и PowerShell из клиента единой маркировки Azure Information Protection 

- Политики автоматической маркировки для SharePoint и OneDrive

- Экспортировать помеченные и зашифрованные данные из Power BI

- Microsoft Cloud App Security

Для этих сценариев с помощью приложений Office пользователь со встроенной маркировкой может применять маркировку контента метки метки, временно удаляя или заменяя текущую метку, а затем повторно применяя оригинальную метку.

### <a name="dynamic-markings-with-variables"></a>Динамическая маркировка с переменными

> [!IMPORTANT]
> В настоящее время не все приложения на всех платформах поддерживают динамические разметки контента, которые можно указать для загона, подножки и водяные знаки. Для приложений, которые не поддерживают эту возможность, они применяют маркировку в качестве исходного текста, указанного в конфигурации меток, а не для решения переменных.
> 
> Клиент единой маркировки Azure Information Protection поддерживает динамическую маркировку. Для маркировки, встроенной в Office, см. таблицы в разделе [возможности](#support-for-sensitivity-label-capabilities-in-apps) на этой странице.

При настройке метки чувствительности для маркировки контента можно использовать следующие переменные в текстовой строке для загона, подножки или водяного знака:

| Переменная | Описание | Пример при применении метки |
| -------- | ----------- | ------- |
| `${Item.Label}` | Имя отображения метки примененной метки| **Общие**|
| `${Item.Name}` | Имя файла или тема электронной почты помеченного контента | **Sales.docx** |
| `${Item.Location}` | Путь и имя файла помеченного документа или тема электронной почты для метки электронной почты | **\\\Sales\2020\Q3\Report.docx**|
| `${User.Name}` | Отображение имени пользователя, применяемого меткой| **Саймон, Ричард** |
| `${User.PrincipalName}` | Основное имя пользователя Azure AD (UPN) пользователя, применяя метку | **rsimone \@ contoso.com** |
| `${Event.DateTime}` | Дата и время, когда содержимое помечено, в локальном часовом поясе пользователя, применяя метку | **08.10.2020 13.30** |

> [!NOTE]
> Синтаксис для этих переменных является чувствительным к делу.

#### <a name="setting-different-visual-markings-for-word-excel-powerpoint-and-outlook"></a>Настройка различных визуальных разметок для Word, Excel, PowerPoint и Outlook

В качестве дополнительной переменной можно настроить визуальную разметку для каждого типа приложений Office с помощью параметра "If.App" в текстовой строке и определить тип приложения с помощью значений **Word,** **Excel,** **PowerPoint** или **Outlook.** Вы также можете указать эти значения, которые необходимо указать несколько в том же заявлении If.App.

> [!NOTE]
> Для полноты включены инструкции для Outlook, хотя в настоящее время поддерживаются только клиентом единой маркировки Azure Information Protection.

Используйте указанный ниже синтаксис.

```
${If.App.<application type>}<your visual markings text> ${If.End}
```

Как и другие динамические визуальные разметки, синтаксис является чувствительным к делу.

Примеры:

- **Установите текст загона только для документов Word:**

    `${If.App.Word}This Word document is sensitive ${If.End}`

    Только в заглавных главах документов Word метка применяет текст заглавного текста "Этот документ Word является конфиденциальным". Текст заглавного текста не применяется к другим приложениям Office.

- **Установите текст для word, Excel и Outlook и другой текст для footer для PowerPoint:**

    `${If.App.WXO}This content is confidential. ${If.End}${If.App.PowerPoint}This presentation is confidential. ${If.End}`

    В Word, Excel и Outlook метка применяет текст "Этот контент является конфиденциальным". В PowerPoint метка применяет текст "Эта презентация является конфиденциальной".

- **Установите определенный текст водяного знака для Word и PowerPoint, а затем текст водяного знака для Word, Excel и PowerPoint:**

    `${If.App.WP}This content is ${If.End}Confidential`

    В Word и PowerPoint метка применяет текст водяного знака "Этот контент является конфиденциальным". В Excel метка применяет текст водяного знака "Confidential". В Outlook метка не применяет текст водяного знака, так как водяные знаки в качестве визуальной разметки не поддерживаются для Outlook.

- Для клиента единой маркировки Azure Information Protection:
    - Для документов (Word, Excel, PowerPoint): При сэкономлении неозначаемой документации или закрытии документа пользователями.
    - Для сообщений электронной почты (Outlook): в то время, когда пользователи отправляют ненаблюдное сообщение.

- Для маркировки, встроенной в приложения Office:
    - Для документов (Word, Excel, PowerPoint): При вскрытии или сэкономлении неозначаемой документации.
    - Для сообщений электронной почты (Outlook): во время отправки пользователями сообщения электронной почты без побелки.

Дополнительные сведения для встроенной маркировки:

- Когда пользователям предлагается добавить метку конфиденциальности, поскольку они открывают документ без маркировки, они могут добавить метку или открыть документ в режиме только для чтения.

- Когда обязательная маркировка действует, пользователи не могут удалить метки конфиденциальности из документов, но могут изменить существующую метку.

Сведения о параметрах политики см. в инструкции по использованию [этого параметра.](sensitivity-labels.md#what-label-policies-can-do)
