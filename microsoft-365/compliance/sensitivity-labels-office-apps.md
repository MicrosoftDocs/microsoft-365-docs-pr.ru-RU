# <a name="use-sensitivity-labels-in-office-apps"></a>Использование меток конфиденциальности в приложениях Office

>*[Руководство по лицензированию Microsoft 365 для обеспечения безопасности и соответствия требованиям](https://aka.ms/ComplianceSD).*

После публикации меток конфиденциальности из Центра соответствия требованиям Microsoft 365 или эквивалентного центра меток они начинают отображаться в приложениях Office, чтобы пользователи могли классифицировать и защищать данные по мере их создания или редактирования. [](create-sensitivity-labels.md#publish-sensitivity-labels-by-creating-a-label-policy)

Используйте сведения в этой статье, чтобы помочь вам успешно управлять метами конфиденциальности в приложениях Office. Например, определите минимальные версии приложений, необходимых для поддержки встроенных меток, и определите взаимодействие с клиентом унифицированных меток Azure Information Protection и совместимость с другими приложениями и службами.

## <a name="labeling-client-for-desktop-apps"></a>Клиент меток для классических приложений

Чтобы использовать метки конфиденциальности, встроенные в классические приложения Office для Windows и Mac, необходимо использовать выпуск Office по подписке. Этот клиент меток не поддерживает автономные выпуски Office, такие как Office 2016 или Office 2019.

Чтобы использовать метки конфиденциальности с этими автономными выпусками Office на компьютерах с Windows, установите клиент унифицированных меток [Azure Information Protection.](https://docs.microsoft.com/azure/information-protection/rms-client/aip-clientv2)

## <a name="office-built-in-labeling-client-and-other-labeling-solutions"></a>Встроенный клиент меток Office и другие решения для меток

Встроенный клиент меток Office загружает метки конфиденциальности и параметры политики меток конфиденциальности из следующих центров администрирования:

- Центр соответствия требованиям Microsoft 365
- Центр безопасности Microsoft 365
- Центр безопасности и соответствия требованиям Office 365

Чтобы использовать встроенный клиент меток Office, необходимо [](create-sensitivity-labels.md#publish-sensitivity-labels-by-creating-a-label-policy) опубликовать одну или несколько политик меток для пользователей из одного из перечисленных центров администрирования и поддерживаемой версии [Office.](#support-for-sensitivity-label-capabilities-in-apps)

Если оба этих условия выполнены, но необходимо отключить встроенный клиент меток Office, используйте следующий параметр групповой политики:

1. Перейдите **в меню "Конфигурация пользователя/Административные шаблоны/Microsoft Office 2016/Параметры безопасности".**

2. Установите **для применения меток конфиденциальности** и просмотра меток конфиденциальности в Office функцию конфиденциальности **0**. 
 
Развернем этот параметр с помощью групповой политики или службы [политики облака Office.](https://docs.microsoft.com/DeployOffice/overview-office-cloud-policy-service) Этот параметр вступает в силу при перезапуске приложений Office.

### <a name="office-built-in-labeling-client-and-the-azure-information-protection-client"></a>Встроенный клиент меток Office и клиент Azure Information Protection

Если у пользователей установлен один из[](https://docs.microsoft.com/azure/information-protection/rms-client/aip-clientv2) клиентов Azure Information Protection (унифицированный клиент меток или классический [клиент),](https://docs.microsoft.com/azure/information-protection/rms-client/aip-client)по умолчанию встроенный клиент меток отключен в приложениях Office. 

Чтобы использовать встроенные метки, а не клиент Azure Information Protection для приложений Office, воспользуйтесь инструкциями из предыдущего раздела, но установите для параметра групповой политики применение и просмотр меток конфиденциальности **1** с помощью функции конфиденциальности **в Office.** 

Кроме того, отключите или удалите надстройку Office, **Azure Information Protection.** Этот метод подходит для одного компьютера и для проведения неучетного тестирования. Инструкции см. в инструкциях по просмотру, управлению и установке надстройки [в программах Office.](https://support.office.com/article/16278816-1948-4028-91e5-76dca5380f8d) 

Когда вы отключите или удалите эту надстройку Office, клиент Azure Information Protection останется установленным, чтобы вы могли продолжать пометить файлы за пределами приложений Office. Например, с помощью проводника или PowerShell.

Сведения о том, какие функции поддерживаются клиентами Azure Information Protection и встроенным клиентом меток Office, см. в документации azure Information Protection по выбору клиента меток для компьютеров [с Windows.](https://docs.microsoft.com/azure/information-protection/rms-client/use-client#choose-which-labeling-client-to-use-for-windows-computers)

## <a name="office-file-types-supported"></a>Поддерживаемые типы файлов Office

Приложения Office, которые имеют встроенные метки для файлов Word, Excel и PowerPoint, поддерживают формат Open XML (например, DOCX и XLSX), но не формат Microsoft Office 97-2003 (например, DOC и XLS). Если тип файла не поддерживается для встроенных  меток, кнопка "Конфиденциальность" недоступна в приложении Office.

Клиент унифицированных меток Azure Information Protection поддерживает формат Open XML и формат Microsoft Office 97-2003. Дополнительные сведения [см.](https://docs.microsoft.com/azure/information-protection/rms-client/clientv2-admin-guide-file-types) в руководстве администратора по типам файлов, поддерживаемых клиентом унифицированных меток Azure Information Protection.

Для других решений по маркировке проверьте их документацию на поддерживаемые типы файлов.

## <a name="protection-templates-and-sensitivity-labels"></a>Шаблоны защиты и метки конфиденциальности

Шаблоны защиты, определенные [администратором,](https://docs.microsoft.com/azure/information-protection/configure-policy-templates)например шаблоны, определенные для шифрования сообщений Office 365, не видны в приложениях Office при использовании встроенных меток. Это упрощенное решение отражает, что нет необходимости выбирать шаблон защиты, так как те же параметры включены в метки конфиденциальности с включенным шифрованием.

Если вам нужно преобразовать существующие шаблоны защиты в метки, воспользуйтесь порталом Azure и следующими инструкциями: преобразование шаблонов [в метки.](https://docs.microsoft.com/azure/information-protection/configure-policy-templates#to-convert-templates-to-labels)

## <a name="information-rights-management-irm-options-and-sensitivity-labels"></a>Параметры управления правами на данные (IRM) и метки конфиденциальности

Метки конфиденциальности, настроенные для применения шифрования, устраняют сложность настройки шифрования для пользователей. Во многих приложениях Office эти индивидуальные параметры шифрования по-прежнему могут настраиваться пользователями вручную с помощью параметров управления правами на доступ к данным (IRM). Например, для приложений для Windows:

- Для документа: защита  >  **информации**  >  **о файле ограничивает** доступ к  >  **документу**
- for an email: From the **Options** tab > **Encrypt** 
  
Когда пользователи изначально пометить документ или электронную почту, они всегда могут переопределять параметры конфигурации меток с помощью собственных параметров шифрования. Пример:

- Пользователь применяет метку **"Конфиденциально** \ Все сотрудники" к документу, и эта метка настроена на применение параметров шифрования для всех пользователей в организации. Затем этот пользователь вручную настраивает параметры IRM, чтобы ограничить доступ к пользователю за пределами организации. Конечный результат — документ с меткой **"Конфиденциально\** Все сотрудники" и зашифрованный, но пользователи в вашей организации не могут открыть его ожидаемым образом.

- Пользователь применяет метку **"Конфиденциально\ Только** получатели" к сообщению электронной почты, и это сообщение настроено для применения параметра шифрования **"Не переад вперед".** Затем этот пользователь вручную настраивает параметры IRM, чтобы электронная почта была неограниченной. В результате получатели могут переададантировать сообщение, несмотря на метку **"Конфиденциально\ Только получатели".**

- Пользователь применяет  к документу метку "Общие", и эта метка не настроена для применения шифрования. Затем этот пользователь вручную настраивает параметры IRM, чтобы ограничить доступ к документу. В результате получается документ с  меткой "Общие", который также применяет шифрование, чтобы некоторые пользователи не могли открыть его ожидаемым образом.

Если документ или сообщение электронной почты уже помечены, пользователь может сделать любое из этих [](https://docs.microsoft.com/azure/information-protection/configure-usage-rights#usage-rights-and-descriptions) действий, если содержимое еще не зашифровано или у пользователя есть право на экспорт или полный доступ. 

Для более согласованной работы с меткой с содержательными отчетами предо внесите соответствующие метки и рекомендации, чтобы пользователи применяли только метки для защиты документов. Пример:

- В случаях исключений, когда пользователям необходимо назначить собственные разрешения, предодайте метки, которые дают пользователям возможность назначать [собственные разрешения.](encryption-sensitivity-labels.md#let-users-assign-permissions) 

- Вместо того чтобы вручную удалять шифрование после выбора метки, применяемой шифрованием, следует предоставить альтернативу подчиненной метки, если пользователям нужна метка с такой же классификацией, но без шифрования. Например:
    - **Конфиденциально \ все сотрудники**
    - **Конфиденциально \ Любой (без шифрования)**

> [!NOTE]
> Если пользователи вручную удаляют шифрование из документа с меткой, который хранится в SharePoint или OneDrive, и вы включили метки конфиденциальности для файлов Office в SharePoint и [OneDrive,](sensitivity-labels-sharepoint-onedrive-files.md)шифрование меток будет автоматически восстановлено при следующем доступе или загрузке документа. 


## <a name="apply-sensitivity-labels-to-files-emails-and-attachments"></a>Применение меток конфиденциальности к файлам, электронным письмам и вложениям

Пользователи могут применять только одну метку за раз для каждого документа или сообщения электронной почты.

Когда вы пометите сообщение электронной почты с вложениями, вложения не наследуют метку за одним исключением:

- Вложение — это документ Office с меткой, которая не применяет шифрование, а метка, применяемая к сообщению электронной почты, применяет шифрование. В этом случае документ Office, отправленный по электронной почте, наследует метку электронного письма с его настройками шифрования.

В противном случае: 

- Если вложения имеют метку, они сохраняют свою изначально примененную метку.
- Если вложения зашифрованы без метки, шифрование остается, но они не помечены.
- Если вложения не имеют метки, они остаются без меток.

## <a name="sensitivity-label-compatibility"></a>Совместимость меток конфиденциальности

С помощью приложений с поддержкой **RMS:** если вы открываете помеченный и зашифрованный документ или электронную почту в приложении с поддержкой [RMS,](https://docs.microsoft.com/azure/information-protection/requirements-applications#rms-enlightened-applications) которое не поддерживает метки конфиденциальности, приложение по-прежнему принудительно применяет шифрование и управление правами.

С помощью клиента **Azure Information Protection:** вы можете просматривать и изменять метки конфиденциальности, которые применяются к документам и электронным письмам, с помощью встроенного клиента меток Office с помощью клиента Azure Information Protection и наоборот.

**В других версиях Office** любой авторизованный пользователь может открывать документы и сообщения электронной почты с меткой в других версиях Office. Однако вы можете просматривать или изменять метку только в поддерживаемых версиях Office или с помощью клиента Azure Information Protection. Поддерживаемые версии приложений Office перечислены в предыдущем [разделе.](#support-for-sensitivity-label-capabilities-in-apps)

## <a name="support-for-sharepoint-and-onedrive-files-protected-by-sensitivity-labels"></a>Поддержка файлов SharePoint и OneDrive, защищенных с помощью меток конфиденциальности

Чтобы использовать встроенный клиент меток Office с Office в Интернете для документов в SharePoint или OneDrive, убедитесь, что вы включили метки конфиденциальности для файлов Office в SharePoint и [OneDrive.](sensitivity-labels-sharepoint-onedrive-files.md)

## <a name="support-for-external-users-and-labeled-content"></a>Поддержка внешних пользователей и содержимого с меткой

При пометке документа или сообщения электронной почты метка сохраняется в виде метаданных, включаемой в себя ваш клиент и GUID метки. Когда документ с меткой или сообщение электронной почты открывается приложением Office, которое поддерживает метки конфиденциальности, эти метаданные считыются, и только если пользователь принадлежит к одному и тому же арендатору, метка отображается в приложении. Например, для встроенных меток для Word, PowerPoint и Excel имя метки отображается на панели состояния. 

Это означает, что при совместном применении документов с другой организацией, использующей разные имена меток, каждая организация может применять и видеть свою собственную метку, примененную к документу. Тем не менее, следующие элементы из примененной метки видны пользователям за пределами организации:

- Маркировка содержимого. Когда метка применяет загон, footer или водяной знак, они добавляются непосредственно к содержимому и остаются видимыми, пока кто-то не изменяет или не удаляет их.

- Имя и описание шаблона защиты из метки, применяемой к шифрованию. Эти сведения отображаются в панели сообщений в верхней части документа для предоставления сведений о том, кому разрешено открывать документ, и об их правах на использование для этого документа.

### <a name="sharing-encrypted-documents-with-external-users"></a>Совместное использование зашифрованных документов с внешними пользователями

Помимо ограничения доступа для пользователей в вашей организации, вы можете расширить доступ к любому другому пользователю, у которого есть учетная запись в Azure Active Directory. Все приложения Office и другие приложения [с RMS могут](https://docs.microsoft.com/azure/information-protection/requirements-applications#rms-enlightened-applications) открывать зашифрованные документы после успешной проверки подлинности пользователя.

Если у внешних пользователей нет учетной записи в Azure Active Directory, они могут проверить подлинность с помощью гостевых учетных записей в клиенте. Эти гостевых учетные записи также можно использовать для доступа к общим документам в SharePoint или OneDrive, если вы включили метки конфиденциальности для файлов Office в [SharePoint и OneDrive:](sensitivity-labels-sharepoint-onedrive-files.md)

- Один из вариантов — создать эти гостевых учетные записи самостоятельно. Вы можете указать любой адрес электронной почты, который эти пользователи уже используют. Например, адрес Gmail.
    
    Преимущество этого параметра заключается в том, что вы можете ограничить доступ и права определенным пользователям, указав их адрес электронной почты в параметрах шифрования. Недостатком является накладные расходы на администрирование для создания учетной записи и координации с конфигурацией меток.

- Другой вариант — использовать интеграцию SharePoint и OneDrive с [Azure AD B2B (предварительная версия),](https://docs.microsoft.com/sharepoint/sharepoint-azureb2b-integration-preview) чтобы гостовые учетные записи автоматически создавались при совместном использовании ссылок пользователями.
    
    Преимуществом этого параметра является минимальная административная нагрузка, так как учетные записи создаются автоматически и упро внимание к настройке меток. В этом сценарии необходимо выбрать параметр шифрования ["Добавить](encryption-sensitivity-labels.md#requirements-and-limitations-for-add-any-authenticated-users) любого пользователя, подлинность и подлинность пользователя", так как вы не будете знать адреса электронной почты заранее. Недостаток этого параметра заключается в том, что этот параметр не позволяет ограничить доступ и права на использование определенными пользователями.

Внешние пользователи также могут использовать учетную запись Майкрософт для зашифрованных документов, если они используют приложения Microsoft 365 (ранее[приложения Office 365)](https://docs.microsoft.com/deployoffice/name-change)в Windows и только что поддерживаемые в macOS (версия 16.42+), Android (версия 16.0.13029+) и iOS (версия 2.42+). Например, кто-то делится зашифрованным документом, а параметры шифрования указывают адрес электронной почты Gmail. Этот пользователь может создать собственную учетную запись Майкрософт, которая использует свой адрес электронной почты Gmail. Затем после вход с помощью этой учетной записи они могут открыть документ и изменить его в соответствии с ограничениями использования, указанными для этого пользователя. По walkthrough example of this scenario, see [Opening and editing the protected document](https://docs.microsoft.com/azure/information-protection/secure-collaboration-documents#opening-and-editing-the-protected-document).

> [!NOTE]
> Адрес электронной почты для учетной записи Майкрософт должен совпадать с адресом электронной почты, указанным для ограничения доступа к параметрам шифрования.

Когда пользователь с учетной записью Майкрософт открывает зашифрованный документ таким образом, он автоматически создает гостевую учетную запись для клиента, если гостевая учетная запись с таким же именем еще не существует. Если гостевая учетная запись существует, ее можно использовать для открытия документов в SharePoint и OneDrive с помощью браузера (Office в Интернете), а также для открытия зашифрованных документов из классических приложений для Windows. 

Однако автоматическая гостевая учетная запись не создается сразу в этом сценарии из-за задержки репликации. При указании личных адресов электронной почты в параметрах шифрования меток рекомендуется создать соответствующие гостевых учетные записи в Azure Active Directory. Затем дайте этим пользователям знать, что они должны использовать эту учетную запись для открытия зашифрованного документа из вашей организации.

> [!TIP]
> Так как вы не можете быть уверены, что внешние пользователи будут использовать поддерживаемые клиентские приложения Office, обмен ссылками из SharePoint и OneDrive после создания гостевых учетных записей (для определенных пользователей) или при использовании интеграции SharePoint и OneDrive с [Azure AD B2B](https://docs.microsoft.com/sharepoint/sharepoint-azureb2b-integration-preview) (для любого пользователя, подлинность и т. д.) является более надежным методом для поддержки безопасной совместной работы с внешними пользователями.

## <a name="when-office-apps-apply-content-marking-and-encryption"></a>Когда приложения Office применяют маркировку и шифрование содержимого

Приложения Office применяют маркировку содержимого и шифрование с меткой конфиденциальности по-разному в зависимости от используемого приложения.

| Приложение | Маркировка содержимого | Шифрование |
| --- | --- | --- |
| Word, Excel, PowerPoint на всех платформах | Сразу | Сразу |
| Outlook для ПК и Mac | После того как Exchange Online отправит сообщение электронной почты | Сразу |
| Outlook в Интернете, для iOS и Android | После того как Exchange Online отправит сообщение электронной почты | После того как Exchange Online отправит сообщение электронной почты |
|

Решения, которые применяют метки конфиденциальности к файлам за пределами приложений Office, применяют к файлу метаданные меток. В этом сценарии маркировка содержимого из конфигурации метки не вставляется в файл, но применяется шифрование. 

Когда эти файлы открываются в настольном приложении Office, метки содержимого автоматически применяются клиентом унифицированных меток Azure Information Protection. Маркировка содержимого не применяется автоматически при использовании встроенных меток для классических, мобильных или веб-приложений.

Сценарии, включающие применение метки конфиденциальности за пределами приложений Office, включают следующие:

- Сканер, проводник и PowerShell из клиента унифицированных меток Azure Information Protection 

- Политики автоматического маркировки для SharePoint и OneDrive

- Экспортируются помеченные и зашифрованные данные из Power BI

- Microsoft Cloud App Security

В этих сценариях, используя свои приложения Office, пользователь со встроенными метами может применять метки содержимого метки, временно удаляя или заменяя текущую метку, а затем повторно применяя исходную метку.

### <a name="dynamic-markings-with-variables"></a>Динамическая маркировка с переменными

> [!IMPORTANT]
> В настоящее время не все приложения на всех платформах поддерживают динамическую маркировку содержимого, которую можно указать для ваших headers, footers и watermarks. Для приложений, которые не поддерживают эту возможность, они применяют маркировку как исходный текст, указанный в конфигурации меток, вместо разрешения переменных.
> 
> Клиент унифицированных меток Azure Information Protection поддерживает динамическую маркировку. Для меток, встроенных в Office, см. таблицы в [разделе возможностей](#support-for-sensitivity-label-capabilities-in-apps) на этой странице.

При настройке метки конфиденциальности для разметки содержимого можно использовать следующие переменные в текстовой строке для своего главного, footer или водяного знака:

| Переменная | Описание | Пример при применении метки |
| -------- | ----------- | ------- |
| `${Item.Label}` | Отображаемого имени метки примененной метки| **Общие**|
| `${Item.Name}` | Имя файла или тема электронной почты для пометки содержимого | **Sales.docx** |
| `${Item.Location}` | Путь и имя файла пометки документа или тема электронной почты для пометки электронной почты | **\\\Sales\2020\Q3\Report.docx**|
| `${User.Name}` | Отображаемого имени пользователя, применяющих метку| **Сергей Ерме** |
| `${User.PrincipalName}` | Имя основного пользователя (UPN) Azure AD пользователя, применяющий метку | **rsimone \@ contoso.com** |
| `${Event.DateTime}` | Дата и время применения метки к содержимому в локальном часовом поясе пользователя, применяет метку | **10.08.2020 13:30** |

> [!NOTE]
> Синтаксис этих переменных с чувствительностью к делу.

#### <a name="setting-different-visual-markings-for-word-excel-powerpoint-and-outlook"></a>Настройка различных визуальных меток для Word, Excel, PowerPoint и Outlook

В качестве дополнительной переменной можно настроить визуальную маркировку для каждого типа приложения Office с помощью переменной "If.App" в текстовой строке и определить тип приложения с помощью значений **Word,** **Excel,** **PowerPoint** или **Outlook.** Вы также можете указать сокращение этих значений, что необходимо, если вы хотите указать несколько значений в одном и том же заявлении If.App.

> [!NOTE]
> Для полноты включены инструкции для Outlook, хотя в настоящее время поддерживаются только клиентом унифицированных меток Azure Information Protection.

Используйте указанный ниже синтаксис.

```
${If.App.<application type>}<your visual markings text> ${If.End}
```

Как и в случае с другими динамическими визуальными маркировками, синтаксис чувствителен к случаю.

Примеры:

- **За set header text for Word documents only:**

    `${If.App.Word}This Word document is sensitive ${If.End}`

    Только в headers документа Word метка применяет текст "Этот документ Word является конфиденциальным". Текст загона не применяется к другим приложениям Office.

- **Установите текст для footer для Word, Excel и Outlook, а также другой текст для footer для PowerPoint:**

    `${If.App.WXO}This content is confidential. ${If.End}${If.App.PowerPoint}This presentation is confidential. ${If.End}`

    В Word, Excel и Outlook метка применяет текст "Это содержимое является конфиденциальным". В PowerPoint метка применяет текст "Эта презентация является конфиденциальной".

- **Установите определенный текст водяного знака для Word и PowerPoint, а затем текст водяного знака для Word, Excel и PowerPoint:**

    `${If.App.WP}This content is ${If.End}Confidential`

    В Word и PowerPoint метка применяет текст водяного знака "Это содержимое является конфиденциальным". В Excel метка применяет текст водяного знака "Конфиденциально". В Outlook метка не применяет текст водяного знака, так как водяные знаки в виде визуальной маркировки не поддерживаются в Outlook.

## <a name="require-users-to-apply-a-label-to-their-email-and-documents"></a>Требовать от пользователей применения метки к электронной почте и документам

> [!IMPORTANT]
> Также называется обязательной маркировкой, но не все приложения на всех платформах в настоящее время поддерживают параметр политики "Требовать от пользователей применения метки к электронной почте **и документам".**
> 
> Клиент унифицированных меток [Azure Information Protection](https://docs.microsoft.com/azure/information-protection/rms-client/install-unifiedlabelingclient-app) поддерживает обязательное маркировку, а для меток, встроенных в приложения Office, см. таблицы в разделе "Возможности" этой страницы. [](#support-for-sensitivity-label-capabilities-in-apps)

При выборе этого параметра политики пользователи, для пользователей, на которые распространяется политика, должны выбрать и применить метку конфиденциальности в следующих сценариях:

- Для клиента унифицированных меток Azure Information Protection:
    - Для документов (Word, Excel, PowerPoint): при сэкономленном документе или при закрытии документа пользователями.
    - Для сообщений электронной почты (Outlook): в то время, когда пользователи отправляют сообщение без поверки.

- Для меток, встроенных в приложения Office:
    - Для документов (Word, Excel, PowerPoint): при открываемом или сохраненном документе без поверки.
    - Для сообщений электронной почты (Outlook): в момент отправки пользователями сообщения электронной почты без поверки.

Дополнительные сведения о встроенной маркировке:

- Когда пользователям предлагается добавить метку конфиденциальности, так как они открывают документ без метки, они могут добавить метку или открыть документ в режиме только для чтения.

- Если действует обязательная маркировка, пользователи не могут удалять метки конфиденциальности из документов, но могут изменять существующую метку.

Рекомендации по использованию этого параметра см. в сведениях о [параметрах политики.](sensitivity-labels.md#what-label-policies-can-do)
