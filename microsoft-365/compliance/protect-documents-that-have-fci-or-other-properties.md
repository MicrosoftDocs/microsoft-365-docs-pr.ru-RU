# <a name="create-a-dlp-policy-to-protect-documents-with-fci-or-other-properties"></a>Создание политики защиты от потери данных для защиты документов с помощью FCI или других свойств

Политики защиты от потери данных (DLP) Microsoft 365 могут использовать свойства классификации или свойства элементов для идентификации конфиденциальных элементов. Например, можно использовать:

- Свойства инфраструктуры классификации файлов Windows Server (FCI)
- Свойства документов SharePoint
- свойства сторонних системных документов

![Схема, в которой указана служба Office 365 и внешняя система классификации](../media/59ad0ac1-4146-4919-abd1-c74d8508d25e.png)

Например, ваша организация может использовать Windows Server FCI для идентификации элементов с персональными данными, такими как номера социального страхования, а затем классифицировать документ, установив свойство Личной идентификации в **high,** **Moderate,** **Low, Public** **или** **Not PII** в зависимости от типа и количества случаев появления персональных данных, найденных в документе. 

В Microsoft 365 можно создать политику DLP, которая идентифицирует документы с определенными значениями, такими как **High** и **Medium,** и затем принимает меры, такие как блокировка доступа к этим файлам. У той же политики может быть другое правило, которое принимает другое действие, если свойство настроено на **низкий** уровень , например отправку уведомления по электронной почте. Таким образом, DLP интегрируется с Windows Server FCI и может защитить документы Office, загруженные или общие для Microsoft 365 с файлового сервера Windows Server.

Политика DLP просто ищет определенное имя свойства и пару значений. Любое свойство документа можно использовать, если свойство имеет соответствующее управляемое свойство для поиска SharePoint. Например, в коллекции сайтов SharePoint можно использовать тип контента с именем **Trip Report** с требуемой областью с именем **Клиент.** Всякий раз, когда человек создает отчет о поездке, он должен ввести имя клиента. Эта пара имени свойства и значения также может использоваться в политике DLP, например, если вы хотите  правило, которое блокирует доступ к документу для гостей, когда поле клиента содержит **Contoso.**

Если вы хотите применить политику DLP к контенту с определенными метями Microsoft 365, не следует следовать этим шагам. Вместо этого узнайте, как [использовать метку хранения в качестве условия политики DLP.](data-loss-prevention-policies.md#using-a-retention-label-as-a-condition-in-a-dlp-policy)

## <a name="before-you-create-the-dlp-policy"></a>Перед созданием политики DLP

Прежде чем использовать свойство FCI Windows Server или другое свойство в политике DLP, необходимо создать управляемое свойство в центре администрирования SharePoint. Вот почему.

В SharePoint Online и OneDrive для бизнеса индекс поиска создан путем обхода контента на сайтах. Программа-обходчик извлекает контент и метаданные из документов в форме свойств для обхода. Схема поиска помогает обходительу решать, какое содержимое и метаданные выбрать. Примерами метаданных являются автор и название документа. Однако, чтобы получить содержимое и метаданные из документов в индекс поиска, свойства обхода должны быть соедино с управляемыми свойствами. Индекс содержит только управляемые свойства. Например, свойство обхода, связанное с автором, относясь к управляемому свойству, связанному с автором.

> [!NOTE]
> Не забудьте использовать имя управляемого свойства, а не имя обхода свойств при создании правил DLP с помощью `ContentPropertyContainsWords` условия.

Это важно, так как DLP использует обходной обходной запрос для определения и классификации конфиденциальных сведений на сайтах, а затем хранения этих конфиденциальных сведений в безопасной части индекса поиска. При отправке документа в Office 365 SharePoint автоматически создает свойства обхода на основе свойств документа. Но для использования FCI или другого свойства в политике DLP для обхода свойства необходимо соедествовать с управляемым свойством, чтобы содержимое этого свойства было сохранено в индексе.

Дополнительные сведения о свойствах поиска и управляемых свойств см. в раздел Управление схемой поиска [в SharePoint Online.](https://go.microsoft.com/fwlink/p/?LinkID=627454)

### <a name="step-1-upload-a-document-with-the-needed-property-to-office-365"></a>Шаг 1. Отправка документа с необходимым свойством в Office 365

Сначала необходимо загрузить документ с свойством, на которое вы хотите ссылаться в политике DLP. Microsoft 365 обнаруживает свойство и автоматически создает свойство обхода из него. На следующем шаге вы создайте управляемое свойство, а затем соберет управляемое свойство с этим свойством обхода.

### <a name="step-2-create-a-managed-property"></a>Шаг 2. Создание управляемого свойства

1. Войдите в Центр администрирования Microsoft 365.

2. В левой навигации выберите **центры администрирования** \> **SharePoint.** Теперь вы в Центре администрирования SharePoint.

3. В левой навигации выберите **поиск на** \> странице **администрирования** поиска \> **Управление схемой поиска.**

   ![Страница администрирования поиска в Центре администрирования SharePoint](../media/6bcd3aec-d11a-4f8c-9987-8f35da14d80b.png)

4. На странице **Управляемые свойства** \> **Новое управляемое свойство**.

   ![Страница "Управляемые свойства" с выделенной кнопкой "Создать управляемое свойство"](../media/b161c764-414c-4037-83ed-503a49fb4410.png)

5. Введите имя и описание свойства. Это имя будет отображаться в политиках DLP.

6. Для **type** выберите **Текст**.

7. В **соответствии с основными характеристиками** выберите **Запросимый** и **запросимый**.

8. В **соответствии с сопоставлениями для обхода свойств** Добавьте \> **сопоставление.**

9. В **диалоговом** окне выбора свойств обхода найдите и выберите свойство обхода, соответствующее свойству FCI Windows Server или другому свойству, которое вы будете использовать в политике \> DLP \> **ОК.**

   ![Диалоговое окно "Выбор свойств для обхода"](../media/aeda1dce-1342-48bf-9594-a8e4f230e8aa.png)

10. В нижней части страницы \> **ОК**.

## <a name="create-a-dlp-policy-that-uses-an-fci-property-or-other-property"></a>Создание политики DLP, использующей свойство FCI или другое свойство

В этом примере организация использует FCI на своих файловом сервере Windows Server; В частности, они используют свойство классификации FCI с именем **Personally Identifiable Information** с возможными значениями **High,** **Moderate,** **Low,** **Public** и **Not PII**. Теперь они хотят использовать существующую классификацию FCI в своих политиках DLP в Office 365.

Сначала они следуют выше, чтобы создать управляемое свойство в SharePoint Online, которое сопопоает с свойством обхода, созданным автоматически из свойства FCI.

Далее создается политика DLP с двумя правилами, которые используют свойства документа условия, содержащие любое **из этих значений:**

- **FCI PII контента - высокая, умеренная** Первое правило ограничивает доступ к документу, если свойство классификации FCI  **personally identifiable Information** равно высокому или умеренному и документ передается людям, не входя в организацию. 

- **Контент PII FCI — низкий** Второе правило отправляет уведомление владельцу документа, если свойство классификации FCI **personally identifiable Information** равно **Low** и документ передается людям, не вступаемым в организацию.

### <a name="create-the-dlp-policy-by-using-powershell"></a>Создание политики DLP с помощью PowerShell

Свойства **документа** условия, содержащие любое из этих значений, временно недоступны в пользовательском интерфейсе Центра соответствия требованиям безопасности, но это условие можно использовать с &amp; помощью PowerShell. Вы можете использовать cmdlets для работы с политикой DLP, а также использовать  `New\Set\Get-DlpCompliancePolicy` cmdlets с параметром, чтобы добавить свойства документа условия содержат любое из  `New\Set\Get-DlpComplianceRule`  `ContentPropertyContainsWords` этих **значений**.

Дополнительные сведения об этих cmdlets см. в [ &amp; cmdlets Центра](https://docs.microsoft.com/powershell/exchange/exchange-online-powershell)соответствия требованиям безопасности.

1. [Подключение к Центру соответствия &amp; требованиям безопасности с помощью удаленной PowerShell](https://docs.microsoft.com/powershell/exchange/connect-to-scc-powershell)

2. Создайте политику с помощью  `New-DlpCompliancePolicy` .

В этом PowerShell создается политика DLP, применяемая ко всем расположениям.

   ```powershell
   New-DlpCompliancePolicy -Name FCI_PII_policy -ExchangeLocation All -SharePointLocation All -OneDriveLocation All -Mode Enable
   ```

3. Создайте два правила, описанные выше, с помощью , где одно правило для низкого значения, а другое правило для `New-DlpComplianceRule` **высоких** и **умеренных**  значений.

   Вот пример PowerShell, который создает эти два правила. Пары имен и значений свойств заключены в кавычках, и имя свойства может указывать несколько значений, разделенных запятой без пробелов, например  `"<Property1>:<Value1>,<Value2>","<Property2>:<Value3>,<Value4>"....`

   ```powershell
   New-DlpComplianceRule -Name FCI_PII_content-High,Moderate -Policy FCI_PII_policy -AccessScope NotInOrganization -BlockAccess $true -ContentPropertyContainsWords "Personally Identifiable Information:High,Moderate" -Disabled $falseNew-DlpComplianceRule -Name FCI_PII_content-Low -Policy FCI_PII_policy -AccessScope NotInOrganization -BlockAccess $false -ContentPropertyContainsWords "Personally Identifiable Information:Low" -Disabled $false -NotifyUser Owner
   ```

   Windows Server FCI включает множество встроенных свойств, в том числе личные сведения, используемые в этом примере.  Возможные значения для каждого свойства могут быть разными для каждой организации. Используемые здесь значения **High,** **Moderate** и **Low** являются только примером. В организации можно просмотреть свойства классификации Windows Server FCI с их возможными значениями в файле Server Resource Manager на файловом сервере windows Server. Дополнительные сведения см. в [свойство Create a classification property.](https://go.microsoft.com/fwlink/p/?LinkID=627456)

По завершению политики должны быть два новых правила, в которых оба свойства документа содержат любое **из этих значений.** Это условие не будет отображаться в пользовательском интерфейсе, хотя будут отображаться другие условия, действия и параметры.

Одно правило блокирует доступ к содержимому, в котором свойство **личной** информации равно **высокому** или **умеренному.** Второе правило отправляет уведомление о содержимом, в котором свойство **личной** информации равно **Low**.

![Диалоговое окно "Создание политики защиты от потери данных", в котором показаны два новых правила](../media/5c56c13b-62a5-4f25-8eb7-ce83a844bb12.png)

## <a name="after-you-create-the-dlp-policy"></a>После создания политики DLP

При этом в предыдущих разделах будет создаваться политика DLP, которая будет быстро обнаруживать содержимое с этим свойством, но только в том случае, если это содержимое будет недавно загружено (чтобы проиндексировать контент), или если это содержимое старое, но только что измененное (чтобы повторно проиндексировать контент).

Для обнаружения контента с этим свойством во всем мире может потребоваться вручную переиндексировать библиотеку, сайт или коллекцию сайтов, чтобы политика DLP знала о всем содержимом этого свойства. В SharePoint Online обход контента выполняется автоматически по определенному расписанию. Программа-обходчик выбирает контент, который изменился с момента последнего обхода, и обновляет индекс. Если вам нужна политика DLP для защиты контента перед следующим запланированным обходом, вы можете предпринять эти действия.

> [!CAUTION]
> [!Внимание!]  Переиндексация сайта может создать большую нагрузку на систему поиска. Не переиндексировать свой сайт, если сценарий не требует его.

Дополнительные сведения см. в статье [Ручной запрос обхода контента и переиндексации сайта, библиотеки или списка](https://go.microsoft.com/fwlink/p/?LinkID=627457).

### <a name="reindex-a-site-optional"></a>Reindex сайт (необязательный)

1. На сайте выберите **Параметры** (значок шестеренки в верхнем правом) \> **Параметры сайта**.

2. В **режиме Поиска** выберите **сайт Поиска и автономной доступности** \> **Reindex.**
