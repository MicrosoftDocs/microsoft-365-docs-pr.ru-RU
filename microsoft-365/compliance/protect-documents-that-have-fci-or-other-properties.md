# <a name="create-a-dlp-policy-to-protect-documents-with-fci-or-other-properties"></a>Создание политики защиты от потери данных для защиты документов с помощью FCI или других свойств

Политики защиты от потери данных (DLP) Microsoft 365 могут использовать свойства классификации или элемента для идентификации конфиденциальных элементов. Например, можно использовать:

- Свойства инфраструктуры классификации файлов Windows Server (FCI)
- Свойства документов SharePoint
- свойства сторонних системных документов

![Схема, в которой указана служба Office 365 и внешняя система классификации](../media/59ad0ac1-4146-4919-abd1-c74d8508d25e.png)

Например, ваша организация может использовать Windows Server FCI для идентификации элементов с персональными данными, например номерами социального страхования, а затем классифицировать документ, установив для свойства **"Персональные** данные" значения **High,** **Moderate,** **Low,** **Public** или **Not PII** на основе типа и количества найденных в документе персональных данных.

В Microsoft 365 вы можете создать политику DLP, которая определяет документы, для свойства для них за установлены определенные значения, такие как **High** и **Medium,** а затем принимает действие, например блокирование доступа к этим файлам. Эта же политика может иметь другое правило, которое принимает другое действие, если для свойства установлено низкое (например, отправка уведомления по электронной почте). Таким образом, DLP интегрируется с Windows Server FCI и помогает защитить документы Office, загруженные или общие для Microsoft 365, с файлового сервера Windows Server.

Политика DLP просто ищет определенную пару "имя-значение свойства". Можно использовать любое свойство документа, если свойство имеет соответствующее управляемое свойство для поиска SharePoint. Например, в коллекции веб-сайтов SharePoint может быть тип контента с именем **Trip Report** с нужным полем **Customer**. Каждый раз, когда человек создает отчет о поездке, он должен ввести имя клиента. Эта пара "имя-значение свойства" также может использоваться в политике DLP, например, если вы хотите,  чтобы правило блокировало доступ к документу для гостей, когда поле "Клиент" содержит **Contoso.**

Если вы хотите применить политику DLP к содержимому с определенными метами Microsoft 365, не следуйте этим шагам. Вместо этого узнайте, как [использовать метку хранения в качестве условия в политике DLP.](data-loss-prevention-policies.md#using-a-retention-label-as-a-condition-in-a-dlp-policy)

## <a name="before-you-create-the-dlp-policy"></a>Перед созданием политики DLP

Прежде чем использовать свойство Windows Server FCI или другое свойство в политике DLP, необходимо создать управляемое свойство в Центре администрирования SharePoint. Вот почему.

В SharePoint Online и OneDrive для бизнеса индекс поиска составляется путем обхода контента сайтов. Программа-обходчик извлекает контент и метаданные из документов в форме свойств для обхода. Схема поиска помогает обходщику решить, какой контент и метаданные необходимо получить. Примерами метаданных являются автор и заголовок документа. Однако для получения контента и метаданных из документов в индекс поиска свойства для обхода должны быть соедиана с управляемыми свойствами. Индекс содержит только управляемые свойства. Например, свойство для обхода, связанное с автором, со картой управляемого свойства, связанного с автором.

Это важно, так как DLP использует обходник поиска для идентификации и классификации конфиденциальной информации на сайтах, а затем сохраняет эти конфиденциальные данные в безопасной части индекса поиска. При отправке документа в Office 365 SharePoint автоматически создает свойства для обхода на основе свойств документа. Но чтобы использовать FCI или другое свойство в политике DLP, это свойство для обхода необходимо соедеть с управляемым свойством, чтобы контент с этим свойством хранился в индексе.

Дополнительные сведения о свойствах поиска и управляемых свойствах см. в под управлением схемы [поиска в SharePoint Online.](https://go.microsoft.com/fwlink/p/?LinkID=627454)

### <a name="step-1-upload-a-document-with-the-needed-property-to-office-365"></a>Шаг 1. Отправка документа с необходимым свойством в Office 365

Сначала необходимо отправить документ со свойством, на которое вы хотите ссылаться в политике DLP. Microsoft 365 обнаружит свойство и автоматически создаст из него свойство для обхода. На следующем этапе вы создадим управляемое свойство, а затем соберем управляемое свойство с этим свойством для обхода.

### <a name="step-2-create-a-managed-property"></a>Шаг 2. Создание управляемого свойства

1. Войдите в Центр администрирования Microsoft 365.

2. В области навигации слева выберите **"Центры администрирования** \> **SharePoint".** Теперь вы в Центре администрирования SharePoint.

3. В области навигации слева выберите **поиск на** странице администрирования \> поиска  \> **"Управление схемой поиска".**

   ![Страница администрирования поиска в Центре администрирования SharePoint](../media/6bcd3aec-d11a-4f8c-9987-8f35da14d80b.png)

4. На странице **"Управляемые свойства"** \> **новое управляемое свойство.**

   ![Страница "Управляемые свойства" с выделенной кнопкой "Создать управляемое свойство"](../media/b161c764-414c-4037-83ed-503a49fb4410.png)

5. Введите имя и описание свойства. Это имя будет отображаться в политиках DLP.

6. For **Type**, choose **Text**.

7. Under **Main characteristics,** select **Queryable** and **Retrievable**.

8. В **области сопоставлений с свойствами для обхода** \> **добавьте сопоставление.**

9. В **диалоговом** окне выбора свойств для обхода найдите и выберите свойство для обхода, соответствующее свойству FCI Windows Server или другому свойству, которое будет применяться в политике \> DLP \> **OK.**

   ![Диалоговое окно "Выбор свойств для обхода"](../media/aeda1dce-1342-48bf-9594-a8e4f230e8aa.png)

10. В нижней части страницы \> **ОК**.

## <a name="create-a-dlp-policy-that-uses-an-fci-property-or-other-property"></a>Создание политики DLP, использующей свойство FCI или другое свойство

В этом примере организация использует FCI на своих файловом сервере Windows Server; В частности, они используют свойство классификации FCI с именем **Personally Identifiable Information** с возможными значениями **High,** **Moderate,** **Low,** **Public** и **Not PII.** Теперь им нужно использовать существующую классификацию FCI в своих политиках DLP в Office 365.

Сначала они следуют шагам выше, чтобы создать управляемое свойство в SharePoint Online, которое сопоируется со свойством для обхода, автоматически созданным из свойства FCI.

Затем они создают политику DLP с двумя правилами, которые используют условие Свойства документа, содержащие любое **из следующих значений:**

- **Контент pii FCI — высокий, средний** Первое правило ограничивает доступ к документу, если свойство классификации FCI  **личных** сведений имеет высокий или средний уровень, а документ передается людям за пределами организации. 

- **Контент pii FCI — низкий** Второе правило отправляет уведомление владельцу документа, если свойство классификации FCI **личных**  сведений имеет низкий уровень и документ передается людям за пределами организации.

### <a name="create-the-dlp-policy-by-using-powershell"></a>Создание политики DLP с помощью PowerShell

Условие **Свойства документа** содержат любые из этих значений временно недоступны в пользовательском интерфейсе Центра соответствия требованиям безопасности, но это условие по-прежнему можно использовать с &amp; помощью PowerShell. Вы можете использовать эти данные для работы с политикой DLP и использовать их вместе с параметром, чтобы добавить условие, при котором свойства document содержат любое из этих `New\Set\Get-DlpCompliancePolicy` `New\Set\Get-DlpComplianceRule` `ContentPropertyContainsWords` **значений.**

Дополнительные сведения об этих cmdlets см. в центре безопасности и [ &amp; соответствия требованиям.](https://go.microsoft.com/fwlink/?LinkID=799772&amp;clcid=0x409)

1. [Подключение к Центру соответствия &amp; требованиям безопасности с помощью удаленной службы PowerShell](https://go.microsoft.com/fwlink/?LinkID=799771&amp;clcid=0x409)

2. Создайте политику с помощью  `New-DlpCompliancePolicy` .

В этой powerShell создается политика DLP, которая применяется ко всем расположениям.

   ```powershell
   New-DlpCompliancePolicy -Name FCI_PII_policy -ExchangeLocation All -SharePointLocation All -OneDriveLocation All -Mode Enable
   ```

3. Создайте два правила, описанные выше, с помощью которых одно правило имеет значение Low, а другое — для значений `New-DlpComplianceRule` **High** и **Moderate.** 

   Вот пример PowerShell, который создает эти два правила. Пары имя-значение свойства заключены в кавычках, а имя свойства может указывать несколько значений, разделенных запятой без пробелов, например  `"<Property1>:<Value1>,<Value2>","<Property2>:<Value3>,<Value4>"....`

   ```powershell
   New-DlpComplianceRule -Name FCI_PII_content-High,Moderate -Policy FCI_PII_policy -AccessScope NotInOrganization -BlockAccess $true -ContentPropertyContainsWords "Personally Identifiable Information:High,Moderate" -Disabled $falseNew-DlpComplianceRule -Name FCI_PII_content-Low -Policy FCI_PII_policy -AccessScope NotInOrganization -BlockAccess $false -ContentPropertyContainsWords "Personally Identifiable Information:Low" -Disabled $false -NotifyUser Owner
   ```

   Windows Server FCI включает множество встроенных свойств, в том числе личные сведения, **используемые** в этом примере. Возможные значения для каждого свойства могут быть разными для каждой организации. В этом **примере используются** значения **High,** Moderate и **Low.** В организации можно просмотреть свойства классификации FCI Windows Server с их возможными значениями в диспетчере ресурсов файлового сервера на файловом сервере Windows Server. Дополнительные сведения см. в [теме "Создание свойства классификации".](https://go.microsoft.com/fwlink/p/?LinkID=627456)

После завершения в политике должно быть два новых правила, которые оба используют свойства Document, содержат любое **из этих условий значений.** Это условие не будет отображаться в пользовательском интерфейсе, хотя другие условия, действия и параметры будут отображаться.

Одно правило блокирует доступ к содержимому, у которых свойство **"Личные** сведения" равно **"Высокий"** или "Средний".  Второе правило отправляет уведомление о содержимом, где свойство **"Личные** сведения" равно **Low.**

![Диалоговое окно "Создание политики защиты от потери данных", в котором показаны два новых правила](../media/5c56c13b-62a5-4f25-8eb7-ce83a844bb12.png)

## <a name="after-you-create-the-dlp-policy"></a>После создания политики DLP

В предыдущих разделах будет создана политика DLP, которая будет быстро обнаруживать содержимое с этим свойством, но только в том случае, если содержимое было загружено недавно (для индексации содержимого) или если это содержимое старое, но только что измененное (для повторной индексации содержимого).

Чтобы обнаруживать контент с этим свойством везде, может потребоваться вручную запросить переиндексацию библиотеки, сайта или коллекции сайтов, чтобы политика DLP понимала весь контент с этим свойством. В SharePoint Online обход контента выполняется автоматически по определенному расписанию. Программа-обходчик выбирает контент, который изменился с момента последнего обхода, и обновляет индекс. Если для защиты контента перед следующим запланированным обходом необходимо использовать политику защиты от содержимого, вы можете сделать следующее.

> [!CAUTION]
> [!Внимание!]  Переиндексация сайта может создать большую нагрузку на систему поиска. Не переиндексировать сайт, если он не требуется вашему сценарию.

Дополнительные сведения см. в статье [Ручной запрос обхода контента и переиндексации сайта, библиотеки или списка](https://go.microsoft.com/fwlink/p/?LinkID=627457).

### <a name="reindex-a-site-optional"></a>Переиндекска сайта (необязательно)

1. На сайте выберите **"Параметры"** (значок шестеренки в правом верхнем) \> **"Параметры сайта".**

2. В **области поиска** выберите **"Поиск" и** переиндексный сайт с автономной \> **доступностью.**
