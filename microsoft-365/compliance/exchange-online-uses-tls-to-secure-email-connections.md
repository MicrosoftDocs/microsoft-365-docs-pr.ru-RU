# <a name="how-exchange-online-uses-tls-to-secure-email-connections"></a>Использование протокола TLS службой Exchange Online для защиты электронной почты

Узнайте, как Exchange Online и Microsoft 365 используют безопасность транспортных уровней (TLS) и секретность вперед (FS) для защиты сообщений электронной почты. Здесь также приведена информация о сертификате, выданном корпорацией Майкрософт для Exchange Online.
  
## <a name="tls-basics-for-microsoft-365-and-exchange-online"></a>Основы TLS для Microsoft 365 и Exchange Online

Протокол TLS и его предшественник SSL защищают данные при передаче в сети, шифруя соединения между компьютерами с помощью сертификатов безопасности. TLS заменяет протокол SSL и часто называется SSL 3.1. Для Exchange Online мы используем TLS для шифрования подключений между нашими серверами Exchange и соединений между нашими серверами Exchange и другими серверами, такими как локальное серверы Exchange или почтовые серверы получателей. Если соединение зашифровано, все соответствующие данные пересылаются по зашифрованному каналу. Но когда вы переадресуете сообщение, полученное по зашифрованному при помощи TLS соединению, это сообщение шифруется не всегда. Это потому, что, простыми словами, TLS не шифрует сообщение, только подключение.
  
Чтобы зашифровать сообщение, необходимо использовать технологию, которая шифрует содержимое сообщения, например шифрование сообщений Office. Сведения о вариантах шифрования в Office 365 см. в статьях [Email encryption in Office 365](email-encryption.md) и [Office 365 Message Encryption (OME)](ome.md). 
  
Рекомендуется использовать TLS в ситуациях, когда необходимо настроить безопасный канал корреспонденции между Корпорацией Майкрософт и локальной организацией или другой организацией, например партнером. Exchange Online всегда пытается использовать TLS сначала для защиты электронной почты, но не всегда может сделать это, если другая сторона не обеспечивает безопасность TLS. Продолжайте чтение, чтобы узнать, как обеспечить безопасность всей почты на локальном сервере или важных партнеров с помощью *соединителок.* 

Чтобы обеспечить наилучшее шифрование в своем классе для наших клиентов, Корпорация Майкрософт имеет обесценив транспортный уровень безопасности (TLS) версии 1.0 и 1.1 в [Office 365](tls-1.0-and-1.1-deprecation-for-office-365.md) и [Office 365 GCC](tls-1-2-in-office-365-gcc.md). Тем не менее, вы можете продолжать использовать незашифрованное подключение SMTP без TLS. Мы не рекомендуем отправлять сообщения электронной почты без шифрования.  
  
## <a name="how-exchange-online-uses-tls-between-exchange-online-customers"></a>Использование протокола TLS службой Exchange Online для защиты данных, передаваемых между пользователями этой службы

Серверы Exchange Online всегда шифруют соединения с другими серверами Exchange Online в наших центрах обработки данных с помощью протокола TLS 1.2. При отправке почты получателю, который находится в вашей организации, эта электронная почта автоматически отправляется по подключению, зашифрованному с помощью TLS. Кроме того, вся электронная почта, которую вы отправляете другим клиентам, отправляется через соединения, зашифрованные с помощью TLS и защищенные с помощью forward secrecy.
  
## <a name="how-microsoft-365-uses-tls-between-microsoft-365-and-external-trusted-partners"></a>Как Microsoft 365 использует TLS между Microsoft 365 и внешними доверенными партнерами

По умолчанию Exchange Online всегда использует оппортунистические TLS. Это означает, что Exchange Online всегда пытается сначала шифровать подключения с самой безопасной версией TLS, а затем прорабатывать список шифров TLS до тех пор, пока не найдет один, с которым обе стороны могут договориться. Если вы не настроили Exchange Online для обеспечения того, чтобы сообщения этому получателю отправляли только с помощью безопасных подключений, то по умолчанию сообщение будет отправлено незашифрованным, если организация-получатель не поддерживает шифрование TLS. Opportunistic TLS является достаточным для большинства предприятий. Однако для бизнеса с требованиями соответствия требованиям, таким как медицинские, банковские или государственные организации, вы можете настроить Exchange Online для требования или принудительного применения TLS. Инструкции см. в [сообщении Configure mail flow using connectors in Office 365.](https://technet.microsoft.com/library/ms.exch.eac.connectorselection%28v=exchg.150%29.aspx)
  
Exchange Online может использовать TLS принудительно для создания надежных каналов связи между вашей организацией и организацией доверенного партнера. В этом случае, чтобы стала возможной отправка почты из организации партнера, обязательна проверка подлинности этой организации в Exchange Online с помощью сертификата безопасности. Для этого вашему партнеру потребуется управлять собственными сертификатами. В Exchange Online мы используем соединители для защиты сообщений, которые вы отправляете из несанкционированного доступа, прежде чем они прибудут к поставщику электронной почты получателя. Сведения об использовании соединители для настройки потока почты см. в сообщении [Configure mail flow using connectors in Office 365.](https://technet.microsoft.com/library/ms.exch.eac.connectorselection%28v=exchg.150%29.aspx)
  
## <a name="tls-and-hybrid-exchange-server-deployments"></a>Протокол TLS и гибридные развертывания Exchange Server

Если вы управляете гибридным развертыванием Exchange, локальному серверу Exchange необходимо проверить подлинность в Microsoft 365 с помощью сертификата безопасности для отправки почты получателям, почтовые ящики которых находятся только в Office 365. В результате необходимо управлять собственными сертификатами безопасности для локального сервера Exchange. Эти сертификаты также необходимо надежно хранить и обслуживать. Дополнительные сведения об управлении сертификатами в гибридных развертываниях см. в справке о требованиях сертификата [к гибридным развертываниям.](https://technet.microsoft.com/library/hh563848%28v=exchg.150%29.aspx)
  
## <a name="how-to-set-up-forced-tls-for-exchange-online-in-office-365"></a>Как настроить принудительное использование TLS для Exchange Online в Office 365

Чтобы обеспечить защиту электронной почты для пользователей Exchange Online с помощью принудительного использования TLS, необходимо настроить несколько соединителей, для которых оно обязательно. Вам понадобится один соединитель для входящих сообщений и один для исходящих. Эти соединители создаются в Центре администрирования Exchange в Office 365. Инструкции см. в [сообщении Configure mail flow using connectors in Office 365.](https://technet.microsoft.com/library/ms.exch.eac.connectorselection%28v=exchg.150%29.aspx)

## <a name="prepare-for-the-new-exchange-online-certificate"></a>Подготовка к новому сертификату Exchange Online

Новый сертификат выдан другим органом сертификации (CA) по сравнению с предыдущим сертификатом, используемым Exchange Online. В результате для использования нового сертификата может потребоваться выполнить некоторые действия.

Новый сертификат требует подключения к конечным точкам нового ЦС в рамках проверки сертификата. Невыполнение этого может отрицательно повлиять на поток почты. Если вы защищаете почтовые серверы брандмауэрами, которые только разрешат почтовым серверам подключаться к определенным пунктам назначения, необходимо проверить, сможет ли ваш сервер проверить новый сертификат. Чтобы подтвердить, что сервер может использовать новый сертификат, выполните следующие действия:

1. Подключись к локальному Exchange Server с Windows PowerShell и запустите следующую команду:  
  `certutil -URL https://crl.globalsign.com/gsorganizationvalsha2g3.crl`

1. В окне, которое появится, выберите **Retrieve**.

1. После завершения проверки утилита возвращает состояние. Если состояние отображается **нормально,** почтовый сервер может успешно проверить новый сертификат. Если нет, необходимо определить, что вызывает сбой подключений. Скорее всего, необходимо обновить параметры брандмауэра. Полный список конечных точек, к которые необходимо получить доступ, включает следующие:
    - ocsp.globalsign.com
    - crl.globalsign.com
    - secure.globalsign.com   

Обычно вы получаете обновления корневых сертификатов автоматически с помощью Обновления Windows. Однако в некоторых развертываниях есть дополнительная безопасность, которая не позволяет автоматически происходить этим обновлениям. В этих заблокированных развертываниях, где обновление Windows не может автоматически обновлять корневые сертификаты, необходимо убедиться, что правильный корневой сертификат ЦС установлен, завершив следующие действия:
1.  Подключись к локальному Exchange Server с Windows PowerShell и запустите следующую команду:  
  `certmgr.msc`

2. В **доверенного корневого органа сертификации /Сертификаты**, подтвердить, что новый сертификат указан.
