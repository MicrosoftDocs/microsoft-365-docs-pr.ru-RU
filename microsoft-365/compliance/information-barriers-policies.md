# <a name="define-information-barrier-policies"></a>Определение политик информационных барьеров

С помощью информационных барьеров можно определить политики, предназначенные для предотвращения связи определенных сегментов пользователей друг с другом, или разрешить определенным сегментам взаимодействовать только с определенными другими сегментами. Политики информационных барьеров могут помочь вашей организации обеспечить соответствие соответствующим отраслевым стандартам и нормативным требованиям и избежать потенциальных конфликтов интересов. Дополнительные сведения см. в [информационных барьерах.](information-barriers.md)

В этой статье описывается планирование, определение, реализация и управление политиками информационных барьеров. Необходимо предпринять несколько действий, а рабочий поток разделен на несколько частей. Обязательно ознакомьтесь с [](#prerequisites) предварительными условиями и всем процессом, прежде чем приступить к определению (или редактированию) политик информационных барьеров.

> [!TIP]
> В этой статье содержится [пример](#example-contosos-departments-segments-and-policies) сценария и загружаемая книга [Excel,](https://github.com/MicrosoftDocs/OfficeDocs-O365SecComp/raw/public/SecurityCompliance/media/InfoBarriers-PowerShellGenerator.xlsx) которая поможет вам спланировать и определить политики информационных барьеров.

## <a name="concepts-of-information-barrier-policies"></a>Понятия политик информационных барьеров

При определении политик информационных барьеров вы будете работать с атрибутами учетной записи пользователя, сегментами, политиками блокировки и/или "разрешить" и приложениями политики.

- Атрибуты учетной записи пользователя определены в Azure Active Directory (или Exchange Online). Эти атрибуты могут включать сведения о отделе, должности, расположении, имени группы и других профилях. 
- Сегменты — это наборы пользователей, определенные в Центре безопасности & соответствия требованиям с помощью выбранного атрибута **учетной записи пользователя.** (См. [список поддерживаемых атрибутов.)](information-barriers-attributes.md)
- Политики информационных барьеров определяют ограничения связи. При выборе политик информационных барьеров можно выбрать один из двух типов политик:
    - "Блокировка" политик не позволяет одному сегменту связываться с другим сегментом.
    - Политики "Разрешить" позволяют одному сегменту взаимодействовать только с определенными другими сегментами.
- Приложение политики делается после определения всех политик информационных барьеров и готовности к их применению в организации.

## <a name="part-1-segment-users"></a>Часть 1. Сегментация пользователей

На этом этапе вы определяете, какие политики информационных барьеров необходимы, со списком сегментов для определения, а затем определяете сегменты.

### <a name="determine-what-policies-are-needed"></a>Определение необходимых политик

Учитывая юридические и отраслевые нормы, кто является группами в организации, которым нужны политики информационных барьеров? Со списком. Существуют ли группы, которым следует запретить общение с другой группой? Существуют ли группы, которые должны иметь возможность взаимодействовать только с одной или двумя другими группами? Подумайте о политиках, которые вам нужны как принадлежащие одной из двух групп:

- "Блокировка" политик не позволяет одной группе связываться с другой группой.
- Политики "Разрешить" позволяют группе взаимодействовать только с определенными, определенными группами.

Когда у вас будет начальный список групп и политик, определите нужные сегменты.

### <a name="identify-segments"></a>Определение сегментов

В дополнение к первоначальному списку политик соделайте список сегментов для организации. Пользователи, которые будут включены в политики информационных барьеров, должны принадлежать к сегменту. Тщательно спланируйте сегменты, так как пользователь может быть только в одном сегменте. К каждому сегменту может применяться только одна политика информационных барьеров.

> [!IMPORTANT]
> Пользователь может быть только в одном сегменте.

Определите, какие атрибуты в данных каталога организации будут использовать для определения сегментов. Вы можете использовать *Department,* *MemberOf* или любой из поддерживаемых атрибутов. Убедитесь, что в атрибуте, выбранном для пользователей, есть значения. [См. список поддерживаемых атрибутов для информационных барьеров.](information-barriers-attributes.md)

> [!IMPORTANT]
> **Перед тем как перейти к следующему разделу, убедитесь,** что в данных каталога есть значения атрибутов, которые можно использовать для определения сегментов. Если данные каталога не имеют значений для атрибутов, которые вы хотите использовать, учетные записи пользователей должны быть обновлены, чтобы включить эти сведения, прежде чем перейти к информационным барьерам. Чтобы получить помощь по этому, см. следующие ресурсы:<br/>- [Настройка свойств учетной записи пользователя с помощью PowerShell в Office 365](/microsoft-365/enterprise/configure-user-account-properties-with-microsoft-365-powershell)<br/>- [Добавление или обновление сведений профиля пользователя с помощью Azure Active Directory](/azure/active-directory/fundamentals/active-directory-users-profile-azure-portal)

### <a name="define-segments-using-powershell"></a>Определение сегментов с помощью PowerShell

Определение сегментов не влияет на пользователей; Он просто задает стадию определения политик информационных барьеров, а затем применяет их.

1. Используйте **cmdlet New-OrganizationSegment** с параметром **UserGroupFilter,** который соответствует [атрибуту,](information-barriers-attributes.md) который вы хотите использовать.

    |**Синтаксис**|**Пример**|
    |:---------|:----------|
    | `New-OrganizationSegment -Name "segmentname" -UserGroupFilter "attribute -eq 'attributevalue'"` |`New-OrganizationSegment -Name "HR" -UserGroupFilter "Department -eq 'HR'"` <p>В этом примере сегмент hr *определяется* с помощью *hr*, значение в *атрибуте Department.* Часть **-eq** в этом случае относится к "равно". (Кроме того, можно использовать **-ne,** чтобы означать "не равно". См. ["Использование "равно" и "не равно" в определениях сегментов.)](#using-equals-and-not-equals-in-segment-definitions) |

    После запуска каждого из них должен появиться список сведений о новом сегменте. Сведения включают тип сегмента, кто создал или изменил его в последний раз, и так далее. 

2. Повторите этот процесс для каждого сегмента, который необходимо определить.

    > [!IMPORTANT]
    > **Убедитесь, что ваши сегменты не перекрываются.** Каждый пользователь, на которого влияют информационные барьеры, должен принадлежать к одному (и только одному) сегменту. Ни один пользователь не должен принадлежать к двум или более сегментам. (См. пример: определенные сегменты [Contoso](#contosos-defined-segments) в этой статье.)

После определения сегментов переходите к определению [политик информационных барьеров.](#part-2-define-information-barrier-policies)

### <a name="using-equals-and-not-equals-in-segment-definitions"></a>Использование "равно" и "не равно" в определениях сегментов

В следующем примере мы определяем сегмент таким образом, что "Отдел равен персоналу". 

|**Пример**|**Примечание**|
|:----------|:-------|
|`New-OrganizationSegment -Name "HR" -UserGroupFilter "Department -eq 'HR'"` | Обратите внимание, что в этом примере определение сегмента содержит параметр "equals", который **обозначается как -eq**. |

Вы также можете определить сегменты с помощью параметра "not equals", который обозначается **как -ne,** как показано в следующей таблице:

|**Синтаксис**|**Пример**|
|:---------|:----------|
| `New-OrganizationSegment -Name "NotSales" -UserGroupFilter "Department -ne 'Sales'"` | В этом примере мы определили сегмент *NotSales,* включавший всех, кто не входит в отдел *продаж.* Часть **-ne** в этом случае относится к термину "не равно". |

Помимо определения сегментов с использованием "равно" или "не равно", можно определить сегмент с помощью параметров "равно" и "не равно". Вы также можете определить сложные групповые фильтры с помощью логических операторов *AND* и *OR.*

|**Синтаксис**|**Пример**|
|:---------|:----------|
| `New-OrganizationSegment -Name "LocalFTE" -UserGroupFilter "Location -eq 'Local'" -and "Position -ne 'Temporary'"` | В этом примере мы определили сегмент *LocalFTE,* включавший людей, которые находятся локально и чьи позиции не указаны как *временные.* |
| `New-OrganizationSegment -Name "Segment1" -UserGroupFilter "MemberOf -eq 'group1@contoso.com'' -and MemberOf -ne 'group3@contoso.com'"`| В этом примере мы определили сегмент *Segment1,* включавший в себя group1@contoso.com и не включавших group3@contoso.com. |
| `New-OrganizationSegment -Name "Segment2" -UserGroupFilter "MemberOf -eq 'group2@contoso.com' -or MemberOf -ne 'group3@contoso.com'"` | В этом примере мы определили сегмент *Segment2,* включавший в себя group2@contoso.com и не включавших group3@contoso.com. |
| `New-OrganizationSegment -Name "Segment1and2" -UserGroupFilter "(MemberOf -eq 'group1@contoso.com' -or MemberOf -eq 'group2@contoso.com') -and MemberOf -ne 'group3@contoso.com'"`| В этом примере мы определили сегмент *Segment1and2,* который включает участников group1@contoso.com и group2@contoso.com, а не членов group3@contoso.com. |

> [!TIP]
> По возможности используйте определения сегментов, которые включают в себя "-eq" или "-ne". Старайтесь не определять сложные определения сегментов.

## <a name="part-2-define-information-barrier-policies"></a>Часть 2. Определение политик информационных барьеров

Определите, следует ли запретить связь между определенными сегментами или ограничить связь определенными сегментами. В идеале вы будете использовать минимальное количество политик, чтобы убедиться, что ваша организация соответствует юридическим и отраслевым требованиям.

С помощью списка сегментов пользователей и политик информационных барьеров, которые вы хотите определить, выберите сценарий, а затем выполните действия.

- [Сценарий 1. Блокировка взаимодействия между сегментами](#scenario-1-block-communications-between-segments)
- [Сценарий 2. Разрешить сегменту взаимодействовать только с одним другим сегментом](#scenario-2-allow-a-segment-to-communicate-only-with-one-other-segment)

> [!IMPORTANT]
> **Убедитесь, что при** определении политик сегменту не назначается несколько политик. Например, если вы определяете одну политику для сегмента *"Продажи",* не определите дополнительную политику для *отдела продаж.*<p> Кроме того, определяя политики информационных барьеров, не забудьте сделать эти политики неактивными, пока не будете готовы к их применении. Определение (или изменение) политик не влияет на пользователей, пока эти политики не будут настроены в активном состоянии и не применены.

(См. пример: политики информационных барьеров компании [Contoso](#contosos-information-barrier-policies) в этой статье.)

### <a name="scenario-1-block-communications-between-segments"></a>Сценарий 1. Блокировка взаимодействия между сегментами

Если необходимо заблокировать взаимодействие сегментов друг с другом, необходимо определить две политики: по одной для каждого направления. Каждая политика блокирует взаимодействие только в одном случае.

Например, предположим, что вы хотите заблокировать связь между сегментами A и B. В этом случае вы определяете одну политику, предотвращая связь сегмента A с сегментом B, а затем определяете вторую политику, чтобы запретить сегменту B связываться с сегментом A.

1. Чтобы определить первую политику блокировки, используйте с параметром **SegmentsBlocked** cmdlet **New-InformationBarrierPolicy.**

    |**Синтаксис** | **Пример**| |**--------|:----------| |`New-InformationBarrierPolicy -Name "policyname" -AssignedSegment "segment1name" -SegmentsBlocked "segment2name"` | `New-InformationBarrierPolicy -Name "Sales-Research" -AssignedSegment "Sales" -SegmentsBlocked "Research" -State Inactive` <p> В этом примере мы определили политику *Sales-Research* для сегмента *Sales*. Когда эта политика активна и применяется, она не позволяет людям из отдела продаж общаться с людьми в сегменте *Research.*  |

2. Чтобы определить второй блокирующий сегмент, снова воспользуйтесь с параметром **SegmentsBlocked** с реверсом сегментов с помощью cmdlet **New-InformationBarrierPolicy.**

    |**Пример**|**Примечание**|
    |:----------|:-------|
    |`New-InformationBarrierPolicy -Name "Research-Sales" -AssignedSegment "Research" -SegmentsBlocked "Sales" -State Inactive` | В этом примере мы определили политику *Research-Sales,* чтобы запретить *Research* связываться с *отделом продаж.* |

3. Переходите к одному из следующих действий:

   - (При необходимости) [Определение политики, разрешаемой сегменту взаимодействовать только с одним другим сегментом](#scenario-2-allow-a-segment-to-communicate-only-with-one-other-segment) 
   - (После определения всех политик) [Применение политик информационных барьеров](#part-3-apply-information-barrier-policies)

### <a name="scenario-2-allow-a-segment-to-communicate-only-with-one-other-segment"></a>Сценарий 2. Разрешить сегменту взаимодействовать только с одним другим сегментом

1. Чтобы разрешить одному сегменту взаимодействовать только с одним другим сегментом, используйте с параметром **SegmentsAllowed** cmdlet **New-InformationBarrierPolicy.**

    |**Синтаксис**|**Пример**|
    |:----------|:----------|
    | `New-InformationBarrierPolicy -Name "policyname" -AssignedSegment "segment1name" -SegmentsAllowed "segment2name","segment1name"` | `New-InformationBarrierPolicy -Name "Manufacturing-HR" -AssignedSegment "Manufacturing" -SegmentsAllowed "HR","Manufacturing" -State Inactive` <p> В этом примере мы определили политику *Manufacturing-HR* для сегмента Под названием *Manufacturing*. Когда эта политика активна и применяется, она позволяет пользователям в производственной сфере общаться только с людьми в сегменте hr *.*  (В этом случае производство *не* может взаимодействовать с пользователями, которые не являются частью *отдела кадров.)* |

    **При необходимости можно указать несколько сегментов с помощью этого cmdlet, как показано в следующем примере.**

    |**Синтаксис**|**Пример**|
    |:---------|:----------|
    | `New-InformationBarrierPolicy -Name "policyname" -AssignedSegment "segment1name" -SegmentsAllowed "segment2name", "segment3name","segment1name"` | `New-InformationBarrierPolicy -Name "Research-HRManufacturing" -AssignedSegment "Research" -SegmentsAllowed "HR","Manufacturing","Research" -State Inactive` <p> В этом примере мы определили политику, которая позволяет сегменту *Research* взаимодействовать только с *отделом* кадров и *производством.* |

    Повторите этот шаг для каждой политики, которая будет определена, чтобы разрешить определенным сегментам взаимодействовать только с определенными определенными сегментами.

2. Переходите к одному из следующих действий:

   - (При необходимости) [Определение политики блокировки связи между сегментами](#scenario-1-block-communications-between-segments) 
   - (После определения всех политик) [Применение политик информационных барьеров](#part-3-apply-information-barrier-policies)

## <a name="part-3-apply-information-barrier-policies"></a>Часть 3. Применение политик информационных барьеров

Политики информационных барьеров не действуют до тех пор, пока вы не установите для них активное состояние, а затем не примените политики.

1. Используйте для получения списка определенных политик с помощью cmdlet **Get-InformationBarrierPolicy.** Обратите внимание на состояние и идентификатор (GUID) каждой политики.

    Синтаксис: `Get-InformationBarrierPolicy`

2. Чтобы установить для политики активное состояние, используйте с параметром **Identity** с параметром **Set-InformationBarrierPolicy** и **параметром State** параметр **Active.** 

    |**Синтаксис**|**Пример**|
    |:---------|:----------|
    | `Set-InformationBarrierPolicy -Identity GUID -State Active` | `Set-InformationBarrierPolicy -Identity 43c37853-ea10-4b90-a23d-ab8c93772471 -State Active` <p> В этом примере мы устанавливаем активную политику информационных барьеров с guID *43c37853-ea10-4b90-a23d-ab8c93772471.* |

    Повторите этот шаг соответствующим образом для каждой политики.

3. Завершив настройку активного состояния для политик информационных барьеров, используйте в Центре безопасности и соответствия требованиям & **start-InformationBarrierPoliciesApplication.**

    Синтаксис: `Start-InformationBarrierPoliciesApplication`

    После запуска разрешите системе применять политики в течение `Start-InformationBarrierPoliciesApplication` 30 минут. Система применяет политики от пользователя к пользователю. Система обрабатывает около 5000 учетных записей пользователей в час.

## <a name="view-status-of-user-accounts-segments-policies-or-policy-application"></a>Просмотр состояния учетных записей пользователей, сегментов, политик или приложений политик

|:---------------|:----------|
| Учетные записи пользователей | Используйте для **параметра Identity параметр Get-InformationBarrierRecipientStatus.** <p> Синтаксис: `Get-InformationBarrierRecipientStatus -Identity <value> -Identity2 <value>` <p> Можно использовать любое уникальное значение, идентифицируя каждого пользователя, например имя, псевдоним, различающейся имя, каноническое доменное имя, адрес электронной почты или ИДЕНТИФИКАТОР GUID. <p> Пример: `Get-InformationBarrierRecipientStatus -Identity meganb -Identity2 alexw` <p> В этом примере мы ссылаемся на две учетные записи пользователей в Office 365: *meganb* для Марта и *alexw* для  *Алекса*. <p> (Вы также можете использовать этот cmdlet для одного пользователя: `Get-InformationBarrierRecipientStatus -Identity <value>` ) <p> Этот cmdlet возвращает сведения о пользователях, такие как значения атрибутов и применяемые политики информационных барьеров.|
| Сегменты | Используйте **cmdlet Get-OrganizationSegment.**<p> Синтаксис: `Get-OrganizationSegment` <p> Этот cmdlet отображает список всех сегментов, определенных для организации. |
| Политики информационных барьеров | Используйте **cmdlet Get-InformationBarrierPolicy.** <p> Синтаксис: `Get-InformationBarrierPolicy` <p> Этот cmdlet отобразит список определенных политик информационных барьеров и их состояние. |
| Последнее приложение политики информационных барьеров | Используйте **cmdlet Get-InformationBarrierPoliciesApplicationStatus.** <p> Синтаксис: `Get-InformationBarrierPoliciesApplicationStatus`<p> Этот cmdlet отображает сведения о том, завершено ли приложение политики, завершилось ли оно сбой или находится в процессе выполнения. |
| Все приложения политики информационных барьеров|Воспользуйтесь `Get-InformationBarrierPoliciesApplicationStatus -All`<p> Этот cmdlet отображает сведения о том, завершено ли приложение политики, завершилось ли оно сбой или находится в процессе выполнения.|

## <a name="example-contosos-departments-segments-and-policies"></a>Пример: отделы, сегменты и политики contoso

Чтобы узнать, как организация может определить сегменты и политики, рассмотрим следующий пример.

### <a name="contosos-departments-and-plan"></a>Отделы и план компании Contoso

Contoso имеет пять отделов: отдел кадров, отдел продаж, маркетинг, исследования и производство. Чтобы обеспечить соответствие отраслевым нормативным требованиям, некоторые отделы не должны взаимодействовать с другими отделами, как по указано в следующей таблице:

|**Сегмент**|**Может общаться с**|**Не удается поговорить с**|
|:----------|:--------------|:-----------------|
| HR | Все | (без ограничений) |
| Sales | Отдел кадров, маркетинг, производство | Справочные материалы |
| Маркетинг | Все | (без ограничений) |
| Справочные материалы | Отдел кадров, маркетинг, производство | Sales |
| Производство | Отдел кадров, маркетинг | Любой, кроме отдела кадров или маркетинга |

Для этой структуры план Contoso включает три политики информационных барьеров:

1. Политика, предназначенная для предотвращения связи отдела продаж с research (и другая политика, чтобы запретить research связываться с отделом продаж).
2. Политика, предназначенная для того, чтобы разрешить производство взаимодействовать только с отделом кадров и маркетингом.

В этом сценарии не требуется определять политики для отдела кадров или маркетинга.

### <a name="contosos-defined-segments"></a>Определенные сегменты Contoso

Contoso будет использовать атрибут Department в Azure Active Directory для определения сегментов следующим образом:

|**Department**|**Определение сегмента**|
|:-------------|:---------------------|
| HR | `New-OrganizationSegment -Name "HR" -UserGroupFilter "Department -eq 'HR'"` |
| Sales | `New-OrganizationSegment -Name "Sales" -UserGroupFilter "Department -eq 'Sales'"` |
| Маркетинг | `New-OrganizationSegment -Name "Marketing" -UserGroupFilter "Department -eq 'Marketing'"` |
| Справочные материалы | `New-OrganizationSegment -Name "Research" -UserGroupFilter "Department -eq 'Research'"` |
| Производство | `New-OrganizationSegment -Name "Manufacturing" -UserGroupFilter "Department -eq 'Manufacturing'"` |

После определения сегментов Contoso переходит к определению политик.

### <a name="contosos-information-barrier-policies"></a>Политики информационных барьеров компании Contoso

Contoso определяет три политики, как описано в следующей таблице:

|**Политика**|**Определение политики**|
|:---------|:--------------------|
| **Политика 1. Запретить связь отдела продаж с исследовательскими исследованиями** | `New-InformationBarrierPolicy -Name "Sales-Research" -AssignedSegment "Sales" -SegmentsBlocked "Research" -State Inactive` <p> В этом примере политика информационных барьеров называется *Sales-Research.* Если эта политика активна и применена, она поможет запретить пользователям из сегмента "Продажи" общаться с пользователями в сегменте Research. Эта политика является политикой в одну сторону; Это не помешает Research общаться с отделом продаж. Для этого требуется политика 2. |
| **Политика 2. Запретить исследованиям связываться с отделом продаж** | `New-InformationBarrierPolicy -Name "Research-Sales" -AssignedSegment "Research" -SegmentsBlocked "Sales" -State Inactive` <p> В этом примере политика информационных барьеров называется *Research-Sales*. Если эта политика активна и применена, она поможет запретить пользователям, которые находятся в сегменте Research, общаться с пользователями в сегменте продаж. |
| **Политика 3. Разрешить производство взаимодействовать только с отделом кадров и маркетингом** | `New-InformationBarrierPolicy -Name "Manufacturing-HRMarketing" -AssignedSegment "Manufacturing" -SegmentsAllowed "HR","Marketing","Manufacturing" -State Inactive` <p> В этом случае политика информационных барьеров называется *Manufacturing-HRMarketing.* Если эта политика активна и применена, производство может взаимодействовать только с отделом кадров и маркетингом. Отдел кадров и маркетинг не ограничиваются общением с другими сегментами. |

Задав сегменты и политики, contoso применяет политики, выполнив для этого с помощью cmdlet **Start-InformationBarrierPoliciesApplication.**

По истечении этого cmdlet Contoso соответствует юридическим и отраслевым требованиям.
