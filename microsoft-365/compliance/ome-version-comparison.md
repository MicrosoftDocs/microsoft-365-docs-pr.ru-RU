# <a name="compare-versions-of-ome"></a>Сравнение версий OME

> [!IMPORTANT]
> 28 февраля 2021 г. корпорация Майкрософт покиает поддержку AD RMS в Exchange Online. Если вы развернули гибридную среду, в которой ваши почтовые ящики Exchange находятся в сети и используете IRM с active Directory RMS в локальной среде, необходимо перейти на Azure. Это также затрагивает организации, развернутые в среде среднего развертывания GCC. В этой статье см. статью "Обзор амортизации AD RMS в Exchange Online".

В остальной части этой статьи сравнивает устаревшее шифрование сообщений Office 365 (OME) с новыми возможностями OME и расширенным шифрованием сообщений Office 365. Новые возможности — это слияние и более новая версия OME и управления правами на данные (IRM). Также описаны уникальные характеристики развертывания в GCC High. Они могут сосуществовать в вашей организации. Сведения о работе новых возможностей см. в сообщении [шифрования сообщений Office 365 (OME).](ome.md)

## <a name="overview-of-ad-rms-deprecation-in-exchange-online"></a>Обзор амортизации AD RMS в Exchange Online

Exchange Online включает функции управления правами на доступ к данным (IRM), которые обеспечивают защиту сообщений электронной почты и вложений в сети и автономном режиме. По умолчанию Exchange Online использует Azure Information Protection. Однако в вашей организации может быть настроена служба управления правами на доступ к данным Exchange Online для использования локальной службы управления правами Active Directory (AD RMS). Поддержка AD RMS в Exchange Online больше не поддерживается. Вместо этого Azure Information Protection полностью заменит AD RMS.

Прежде чем начать, просмотрите и оцените влияние на вашу организацию. Если ваша организация уже использует Azure Information Protection для шифрования электронной почты в Exchange Online, вам не нужно ничего делать. Если вы шифруете электронную почту с помощью правил потока почты Exchange, например шифрования сообщений Office 365, вам не придется изменять защищенную электронную почту. В противном случае вам потребуется подготовиться к отключению службы AD RMS, переключившись на Azure Information Protection.

### <a name="prepare-for-ad-rms-deprecation"></a>Подготовка к амортизации AD RMS

Если вы уже настроили Azure Information Protection, но не используете ее, в включить службу с помощью Exchange Online PowerShell. На локальном компьютере, используя учетную запись для работы или учебного заведения с разрешениями глобального администратора в организации, подключите к [Exchange Online PowerShell](https://docs.microsoft.com/powershell/exchange/connect-to-exchange-online-powershell) в Windows PowerShell окна.

Чтобы включить Azure Information Protection, используйте командлет Set-IrmConfiguration, введя следующую команду.

```powershell
Set-IrmConfiguration -AzureRMSLicensingEnabled $true
```

Если в вашей организации еще не настроена служба Azure Information Protection, необходимо перейти с AD RMS на Azure Information Protection. Инструкции см. в инструкциях по переходу с [AD RMS на Azure Information Protection.](https://docs.microsoft.com/azure/information-protection/migrate-from-ad-rms-to-azure-rms)

## <a name="advantages-of-the-new-ome-capabilities-over-legacy-ome"></a>Преимущества новых возможностей OME по сравнению с устаревшим OME

Новые возможности предоставляют следующие преимущества:

- Возможность использовать Encrypt-Only (что обеспечивает безопасную совместную работу), "Не переад вперед" и настраиваемые ограничения.
- Отправители могут отправлять почту, зашифрованную с помощью новых возможностей, вручную из классических клиентов Outlook, Outlook для Mac и Outlook в Интернете.
- Получатели Microsoft 365 получают возможность использовать ветвь в поддерживаемых клиентах Outlook. Кроме того, администраторы могут показать получателям Microsoft 365 фирменный интерфейс.
- Учетные записи за пределами Microsoft 365, такие как учетные записи Gmail, Yahoo и Microsoft, совмещаются с порталом OME, что обеспечивает лучшее пользовательское интерфейс для этих получателей. Все остальные удостоверения используют одноветный код для доступа к зашифрованным сообщениям.
- Администраторы могут настраивать фирменной настройки и создавать несколько шаблонов фирменной настройки.
- Администраторы могут отоскить сообщения электронной почты, зашифрованные с помощью новых возможностей.
- Новые возможности предоставляют подробные отчеты об использовании через Центр соответствия &amp; требованиям безопасности.

## <a name="office-365-advanced-message-encryption-capabilities"></a>Расширенные возможности шифрования сообщений Office 365

Office 365 Advanced Message Encryption предоставляет дополнительные возможности поверх новых возможностей OME. Чтобы использовать расширенные возможности шифрования сообщений, в организации должны быть настроены новые возможности шифрования сообщений Office 365. Кроме того, для использования этих возможностей получатели должны просматривать безопасную почту на портале OME и отвечать на них. К расширенным возможностям относятся:

- Отзыв сообщений

- Срок действия сообщения

- Несколько шаблонов фирменной оформления

Office 365 Advanced Message Encryption не поддерживается в GCC High.

Дополнительные сведения об использовании advanced Message Encryption см. в office [365 Advanced Message Encryption.](ome-advanced-message-encryption.md)

## <a name="unique-characteristics-of-office-365-message-encryption-in-a-gcc-high-deployment"></a>Уникальные характеристики шифрования сообщений Office 365 в развертывании GCC High

Если вы планируете использовать шифрование сообщений Office 365 в среде GCC High, существуют некоторые уникальные характеристики, касающиеся работы получателей.

### <a name="encrypted-email-between-gcc-high-and-gcc-high-recipients"></a>Зашифрованная электронная почта между получателями GCC High и GCC High

Отправители могут вручную шифровать сообщения электронной почты в Outlook для компьютеров с Mac и Outlook в Интернете, либо организации могут настроить политику шифрования сообщений электронной почты с помощью правил потока обработки почты Exchange.

Получатели в GCC High получают такой же внутренний опыт чтения в Outlook для ПК, Mac и Outlook в Интернете, что и все остальные пользователи.

### <a name="encrypted-email-between-gcc-high-and-non-gcc-high-recipients"></a>Зашифрованная электронная почта между получателями GCC High и Non-GCC High

Отправители в GCC High могут отправлять зашифрованную электронную почту за пределами границы GCC High и наоборот.

Все получатели за пределами GCC High, включая коммерческих пользователей Microsoft 365, Outlook.com пользователей и других пользователей других поставщиков электронной почты, таких как Gmail и Yahoo, получают сообщение-оболочку. Эта почта-оболочка перенаправляет получателя на портал OME, где получатель может читать и отвечать на сообщение. Это также относится к отправителям за пределами GCC High, отправляющих зашифрованную почту OME в GCC High.

## <a name="coexistence-of-legacy-ome-and-the-new-capabilities-in-the-same-tenant"></a>Сосуществование устаревших OME и новых возможностей в том же клиенте

Вы можете использовать как устаревшие возможности OME, так и новые возможности в одном клиенте. Администратор должен выбрать версию OME, которую вы хотите использовать при создании правил потока почты.

- Чтобы указать устаревшую версию OME, используйте действие правила потока почты Exchange **Apply the previous version of OME**.

- Чтобы указать новые возможности, используйте действие правила потока почты Exchange **Apply Office 365 Message Encryption and rights protection.**

Пользователи могут вручную отправлять почту, зашифрованную с помощью новых возможностей, из классических версий Outlook, Outlook для Mac и Outlook в Интернете.

## <a name="migrate-from-legacy-ome-to-the-new-capabilities"></a>Переход с устаревшей OME на новые возможности

Хотя обе версии OME могут сосуществовать, мы настоятельно рекомендуем изменить старые правила потока обработки почты, которые используют действие правила "Применить предыдущую версию **OME",** чтобы использовать новые возможности. Обновите эти правила, чтобы использовать действие правила потока почты "Применить шифрование сообщений **Office 365" и защиту прав.** Инструкции см. в инструкциях по определению правил потока почты для шифрования сообщений электронной почты [в Office 365.](define-mail-flow-rules-to-encrypt-email.md)

## <a name="get-started-with-ome"></a>Начало работы с OME

Как правило, новые возможности OME автоматически включены для вашей организации. Дополнительные сведения о новых возможностях OME в организации см. в настройках новых возможностей шифрования [сообщений Office 365.](set-up-new-message-encryption-capabilities.md)

Устаревшая версия OME автоматически включена для вашей организации, если вы включили Azure Information Protection. В прошлом устаревший OME работал, даже если служба Azure Information Protection не была включена. Теперь это не так.

Чтобы начать использовать устаревшую версию OME, если вы включили Azure Information Protection, настройте правила потока почты, использующие действие правила "Применить предыдущую **версию OME".** Инструкции см. в инструкциях [по определению правил потока почты для шифрования сообщений электронной почты.](define-mail-flow-rules-to-encrypt-email.md)
