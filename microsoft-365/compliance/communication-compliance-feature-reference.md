# <a name="communication-compliance-feature-reference"></a>Справочные материалы по функциям обеспечения соответствия требованиям

## <a name="policies"></a>Политики

>[!Important]
>Использование PowerShell для создания политик соответствия требованиям и управления ими не поддерживается. Для создания и управления этими политиками необходимо использовать элементы управления политикой управления политиками в [решении "соответствие требованиям Microsoft 365](https://compliance.microsoft.com/supervisoryreview)".

Вы создадите политики соответствия требованиям для организаций Microsoft 365 в центре соответствия требованиям Microsoft 365. Политики соответствия требованиям определяют, какие связи и пользователи подлежат проверке в вашей организации, определяют, какие настраиваемые условия должны отвечать на связь, и укажите, кто должен выполнять проверку. Пользователи, которым назначена роль *"Администратор соответствия требованиям"* , могут настраивать политики, а все пользователи, которым назначена эта роль, могут получить доступ к странице " **соответствие требованиям** " и глобальным параметрам в центре соответствия требованиям Microsoft 365. При необходимости вы можете экспортировать историю изменений политики в CSV-файл, который также включает состояние "оповещения, ожидающие проверки", "укрупненные элементы" и "разрешенные элементы". Политики невозможно переименовать, и их можно удалить, если они больше не нужны.

>[!NOTE]
>Политики контроля, созданные в центре безопасности & соответствия требованиям для подписок на Office 365, не могут переноситься в Microsoft 365. Если вы выполняете миграцию из подписки на Office 365 в подписку на Microsoft 365, вам потребуется создать новые политики обеспечения соответствия требованиям к связи, чтобы заменить существующие политики контроля.

## <a name="policy-templates"></a>Шаблоны политики

Шаблоны политик — это предопределенные параметры политики, которые можно использовать для быстрого создания политик для решения распространенных сценариев соответствия требованиям. Каждый из этих шаблонов имеет различия в условиях и области, а все шаблоны используют одни и те же типы сигналов сканирования. Можно выбрать один из следующих шаблонов политики:

|**Область**|**Шаблон политики**|**Details**|
|:-----|:-----|:-----|
| **Оскорбительный язык и борьба за защиту** | Отслеживание коммуникаций для оскорбительного языка | -Locations: Exchange Online, Microsoft Teams, Yammer, Skype для бизнеса <br> — Направление: входящий, исходящий, внутренний <br> — Анализ в процентах: 100% <br> Условия: оскорбительный классификатор языка |
| **Конфиденциальная информация** | Мониторинг связи для конфиденциальной информации | -Locations: Exchange Online, Microsoft Teams, Yammer, Skype для бизнеса <br> — Направление: входящий, исходящий, внутренний <br> — Анализ в процентах: 10% <br> Условия: конфиденциальные сведения, готовые шаблоны контента и типы, параметр вспомогательного словаря, вложения больше 1 МБ |
| **Соответствие нормативным требованиям** | Отслеживание коммуникаций для информации, связанной с соответствием финансовой нормативной информации | -Locations: Exchange Online, Microsoft Teams, Yammer, Skype для бизнеса <br> — Направление: входящий, исходящий <br> — Анализ в процентах: 10% <br> Условия: параметр вспомогательного словаря, вложения, размер которых превышает 1 МБ |

## <a name="permissions-preview"></a>Разрешения (Предварительная версия)

>[!Important]
>По умолчанию глобальные администраторы не имеют доступа к функциям обеспечения соответствия требованиям. Роли, назначенные на этом этапе, необходимы до того, как будут доступны все функции обеспечения соответствия для взаимодействия.

Существует пять групп ролей, которые используются для настройки разрешений на управление функциями обеспечения соответствия в связи. Чтобы обеспечить **соответствие требованиям** в качестве варианта меню в центре соответствия требованиям Microsoft 365 и продолжить выполнение этих действий, необходимо назначить группы ролей " *соответствие требованиям* *" или "Управление соответствием требованиям"* . Чтобы получить доступ к функциям обеспечения соответствия связи и управлять ими после начальной настройки, пользователи должны быть членами по крайней мере одной группы ролей обеспечения соответствия для взаимодействия.

В зависимости от того, как вы хотите управлять политиками связи и оповещениями, необходимо назначить пользователей определенным группам ролей. Вы можете назначить пользователям различные обязанности по обеспечению соответствия требованиям определенным группам ролей, чтобы управлять различными областями функций обеспечения соответствия в общении. Или вы можете назначить все учетные записи пользователей для указанных администраторов, аналитиков, наблюдателей и зрителей для группы ролей " *соответствие требованиям* ". Используйте одну группу ролей или несколько групп ролей, чтобы обеспечить соответствие требованиям к управлению соответствием требованиям.

При настройке обеспечения соответствия требованиям выберите эти параметры группы ролей.

|**Role**|**Разрешения роли**|
|:-----|:-----|
| **Соответствие требованиям** | Используйте эту группу ролей для управления соответствием связью в Организации в одной группе. Добавляя все учетные записи пользователей для назначенных администраторов, аналитиков, наблюдателей и средств просмотра, вы можете настроить разрешения на соответствие требованиям в одной группе. Эта группа ролей содержит все роли разрешений на соответствие требованиям. Эта конфигурация является самым простым способом для быстрого начала работы с обеспечением соответствия требованиям и подходит для организаций, которым не требуются отдельные разрешения, определенные для отдельных групп пользователей. |
| **Администратор соответствия требованиям** | Используйте эту группу ролей для первоначальной настройки соответствия требованиям к связи и более поздних версий, чтобы отделить администраторов по обеспечению связи с определенной группой. Пользователи, которым назначена эта группа ролей, могут создавать, читать, обновлять и удалять политики соответствия требованиям к связи, глобальные параметры и назначения групп ролей. Пользователи, которым назначена эта группа ролей, не могут просматривать оповещения о сообщениях. |
| **Аналитика соответствия требованиям к связи** | Используйте эту группу для назначения разрешений пользователям, которые будут использоваться в качестве аналитик обеспечения соответствия требованиям. Пользователи, которым назначена эта группа ролей, могут просматривать политики, в которых они назначены рецензентами, просматривать метаданные сообщений (а не содержимое сообщений), повышать их до дополнительных рецензентов или отправлять уведомления пользователям. Аналитики не могут разрешать ожидающие оповещения. |
| **Соответствие требованиям Инвестигатор** | Используйте эту группу для назначения разрешений пользователям, которые будут использоваться в качестве средств анализа соответствия требованиям. Пользователи, которым назначена эта группа ролей, могут просматривать метаданные и содержимое сообщений, повышать их до дополнительных проверяющих, расширять до расширенного случая обнаружения электронных данных, отправлять уведомления пользователям и разрешать оповещение. |
| **Средство просмотра соответствия требованиям** | Используйте эту группу для назначения разрешений пользователям, которые будут управлять отчетами о связи. Пользователи, которым назначена эта группа ролей, могут получать доступ ко всем графическим пользователям отчетов на домашней странице соответствия требованиям к связи и просматривать отчеты о соответствии требованиям. |

### <a name="for-organizations-using-the-original-permissions-and-role-groups"></a>Для организаций, использующих исходные разрешения и группы ролей

Новая структура группы ролей заменяет исходную структуру группы ролей для обеспечения соответствия требованиям. Для организаций, уже использующих соответствие требованиям, вам необходимо назначить роль администратора экспертной проверки, чтобы приступить к обеспечению соответствия требованиям в центре соответствия требованиям Microsoft 365. Кроме того, вам пришлось создать новую группу ролей для проверяющих с администратором экспертной проверки, управлением делами, администратором соответствия требованиям и просмотром ролей, чтобы проверить и исправить сообщения, соответствующие политике. По сути, все администраторы и рецензенты были в одной группе ролей, а все имеют одинаковые разрешения на доступ и управление. С учетом последних обновлений для обеспечения соответствия требованиям необходимо запланировать миграцию из предыдущей структуры группы ролей в новую структуру группы ролей. Поддержка предыдущей структуры группы ролей будет по поэтапу.

Чтобы помочь вам в планировании миграции, рассмотрим следующий пример. В настоящее время у вас есть три типа пользователей в вашей организации, ИТ, администраторы, рассмотрения и рецензентов. Эти три типа пользователей находятся в предыдущей структуре группы ролей, а все участники одной группы ролей — с назначенными им следующими ролями:

- Администратор супервизорной проверки
- Управление обращениями
- Администратор соответствия
- Проверка

Чтобы изменить роли этих пользователей для новой структуры группы ролей, а также для разделения разрешений на доступ и управление для пользователей, можно рассмотреть три новые группы и соответствующие новые назначения группы ролей:

- **ИТ — администраторы**: назначены новой группе ролей *"Администратор соответствия требованиям"* .
- **Рассмотрение**: назначено группе ролей *аналитика соответствия требованиям к связи* .
- **Проверяющие**: присвоено новой группе ролей *Инвестигатор для обеспечения соответствия связи* .

## <a name="supervised-users"></a>Контролируемые пользователи

Прежде чем приступить к работе с обеспечением соответствия требованиям, необходимо определить, кому нужно проверить связь. В политике пользовательские адреса электронной почты идентифицируют отдельных пользователей или группы людей, которым необходимо поднадзоро. Некоторые примеры этих групп: Microsoft 365, списки рассылки на основе Exchange, сообщества Yammer и каналы Microsoft Teams. Кроме того, вы можете исключить определенных пользователей или группы из проверки с определенной группой исключений или списком групп.

>[!IMPORTANT]
>Для пользователей, охваченных политиками соответствия требованиям, должна быть лицензия на соответствие требованиям Microsoft 365 о корпоративном формате, лицензия на Office 365 корпоративный E3 с дополнительным соответствием требованиям или включение в подписку Office 365 корпоративный Если у вас нет плана корпоративного выпуска и вы хотите попробовать обеспечить соответствие требованиям, вы можете [зарегистрироваться в пробной версии Office 365 Enterprise](https://go.microsoft.com/fwlink/p/?LinkID=698279)с ошибкой.

## <a name="reviewers"></a>Рецензенты

При создании политики соответствия требованиям к общению необходимо определить, кто просматривает сообщения от пользователей, которым вы управляете. В политике пользовательские адреса электронной почты идентифицируют пользователей или группы людей для просмотра контролируемых коммуникаций. Все рецензенты должны иметь почтовые ящики, размещенные в Exchange Online, и должны быть назначены ролям *анализа* или проверки соответствия требованиям *на связь.* В проверяющих (аналитиках или исследованиях) также должна быть назначена роль *управления делами соответствия требованиям для связи* . Когда проверяющие добавляются в политику, они автоматически получают сообщение электронной почты с уведомлением о назначении на политику и предоставляет ссылки на сведения о процессе проверки.

## <a name="groups-for-supervised-users-and-reviewers"></a>Группы для контролируемых пользователей и рецензентов

Чтобы упростить установку, создайте группы для пользователей, которым необходимы их проанализировать и группы для пользователей, которые просматривают эти сообщения. Если вы используете группы, может потребоваться несколько. Например, если необходимо сканировать связь между двумя отдельными группами людей или вы хотите указать группу, которая не защищена.

При назначении группы рассылки в политике она следит за всеми сообщениями электронной почты от каждого пользователя в группе рассылки. При назначении группы Microsoft 365 в политике политика отслеживает все сообщения электронной почты, отправляемые в эту группу, а не отдельные сообщения, полученные каждым участником группы.

Добавление групп и списков рассылки в политики соответствия требованиям связи входят в состав общих условий и правил, поэтому максимальное количество групп и списков рассылки, поддерживаемых политикой, зависит от количества условий, также добавленных в политику. Каждая политика должна поддерживать около 20 групп или списков рассылки в зависимости от количества дополнительных условий, присутствующих в политике.

## <a name="supported-communication-types"></a>Поддерживаемые типы связи

С помощью политик соответствия требованиям можно выбрать сканирование сообщений на одной или нескольких коммуникационных платформах в качестве группы или автономных источников. Связи, полученные на этих платформах, хранятся в течение семи лет для каждой политики по умолчанию, даже если пользователи оставляют свою организацию и их почтовые ящики удаляются.

- **Microsoft Teams**: беседы в чате можно просканировать как в общедоступных, так и в частных каналах Microsoft Teams. Когда пользователям назначается политика соответствия требованиям для обеспечения связи с выбранным покрытием Microsoft Teams, взаимодействие с другими пользователями автоматически отслеживается во всех Microsoft Teams, в которых пользователи являются участниками. Покрытие Microsoft Teams автоматически включается для предварительно определенных шаблонов политик и по умолчанию выбирается в шаблоне настраиваемой политики. В командные разговоры, удовлетворяющие условиям политики соответствия требованиям, может потребоваться до 24 часов. Используйте следующие конфигурации группового управления для управления обсуждениями отдельных пользователей и общения каналов в teams:

    - **Для общения в Microsoft teams:** Назначьте отдельных пользователей или назначьте [группу рассылки](https://support.office.com/article/Distribution-groups-E8BA58A8-FAB2-4AAF-8AA1-2A304052D2DE) политике соответствия для связи. Этот параметр предназначен для отношений "один к одному" или "один-ко-многим" для пользователей и чата.
    - **Для обмена данными по каналам teams:** Назначьте каждому каналу Microsoft Teams или группе Microsoft 365, которую нужно проверить, которая содержит определенного пользователя в политике соответствия требованиям. Если вы добавите одного и того же пользователя в другие каналы Microsoft Teams или в группы Microsoft 365, обязательно добавьте эти новые каналы и группы в политику соответствия требованиям к связи.
    - **Для общения в Teams с гибридными средами электронной почты**: соответствие требованиям может отслеживать сообщения чата для пользователей в организациях, использующих локальное развертывание Exchange или внешний поставщик услуг электронной почты с включенной поддержкой Microsoft Teams. Для отслеживания необходимо создать группу рассылки для пользователей с локальными или внешними почтовыми ящиками. При создании политики обеспечения соответствия требованиям вы назначаете эту группу рассылки в качестве выбранных в мастере политики **пользователей и групп** .

    >[!IMPORTANT]
    >Вы должны подать запрос в службу поддержки Майкрософт, чтобы в вашей организации для поиска данных чата Teams локальных пользователей мог использоваться графический пользовательский интерфейс в Центре безопасности и соответствия требованиям. Дополнительные сведения см в разделе [Поиск в облачных почтовых ящиках для локальных пользователей](search-cloud-based-mailboxes-for-on-premises-users.md).

Вы должны подать запрос в службу поддержки Microsoft, чтобы ваша организация могла использовать графический интерфейс пользователя в Центре безопасности и соответствия требованиям для поиска данных чата команд в облачных почтовых ящиках для локальных пользователей.

- **Электронная почта Exchange**: почтовые ящики, размещенные в Exchange Online в составе подписки на Microsoft 365 или Office 365, подходят для сканирования сообщений. Для обработки сообщений электронной почты и вложений, соответствующих условиям политики соответствия требованиям, могут потребоваться до 24 часов. Поддерживаемые типы вложений для обеспечения соответствия условиям связи совпадают с [типами файлов, которые поддерживаются для проверки содержимого правила обработки почты Exchange](https://docs.microsoft.com/exchange/security-and-compliance/mail-flow-rules/inspect-message-attachments#supported-file-types-for-mail-flow-rule-content-inspection).

- **Yammer**: можно просканировать частные сообщения и беседы и связанные с ними вложения в сообществах Yammer. Когда пользователь добавляется в политику соответствия требованиям к общению, которая включает Yammer в качестве определенного канала, взаимодействие между всеми сообществами Yammer, участником которых является пользователь, включается в процесс сканирования. Для обработки обсуждений и вложений Yammer, соответствующих условиям политики соответствия требованиям, может потребоваться до 24 часов. Для отслеживания подключений и вложений Yammer в Yammer должен быть установлен [собственный режим](https://docs.microsoft.com/yammer/configure-your-yammer-network/overview-native-mode) для политик соответствия требованиям в Организации. В собственном режиме все пользователи Yammer находятся в Azure Active Directory (AAD), все группы представляют собой группы Office 365, а все файлы хранятся в SharePoint Online.

- **Skype для бизнеса Online**: возможности общения и связанные вложения в Skype для бизнеса Online можно заносить в Skype для бизнеса Online. Для обработки сеансов Skype для бизнеса Online, соответствующих условиям политики соответствия требованиям, может потребоваться до 24 часов.     - **Для обмена данными в Skype для бизнеса Online**: назначьте отдельных пользователей или назначьте [группу рассылки](https://support.office.com/article/Distribution-groups-E8BA58A8-FAB2-4AAF-8AA1-2A304052D2DE) политике соответствия. Этот параметр предназначен для отношений "один к одному" или "один-ко-многим" для пользователей и чата.

- **Сторонние источники**: вы можете сканировать связи для данных, импортированных в почтовые ящики в организации Microsoft 365, из сторонних источников, таких как [мгновенный Bloomberg](archive-instant-bloomberg-data.md), [резервный резерв](archive-slack-data.md), [масштабирование](archive-zoommeetings-data.md), SMS и многие другие. Полный список соединителей, поддерживаемых для обеспечения соответствия требованиям, приведен в разделе [Archive Data третьих лиц](archiving-third-party-data.md).

    Перед тем как присвоить соединитель политике соответствия требованиям к связи, необходимо настроить сторонний соединитель для организации Microsoft 365. В разделе **сторонние источники** мастера политики соответствия требованиям к связи отображаются только настроенные соединители сторонних поставщиков.

## <a name="transitioning-from-supervision-in-office-365"></a>Переход с надзора в Office 365

В организациях, использующих политики контроля в Office 365 и планировании перехода к политикам соответствия требованиям в Microsoft 365, необходимо понимать следующие важные моменты:

- Оба решения могут параллельно использоваться в Организации, но политики, используемые в каждом решении, должны иметь уникальные имена политик. Группы и пользовательские словари ключевых слов могут совместно использоваться разными решениями в течение периода перехода.
- Сообщения, сохраненные в надзоре в Office 365, не могут быть перемещены или предоставлены в соответствие условиям взаимодействия в Microsoft 365.
- Решение по надзору в Office 365 будет полностью заменено на решение для обеспечения соответствия требованиям в Microsoft 365. Мы рекомендуем создавать новые политики в соответствии с требованиями к связи, которые имеют те же параметры, что и существующие политики контроля, чтобы использовать новые улучшения расследования и исправлений. При переходе на соответствие требованиям в Microsoft 365 необходимо запланировать экспорт данных отчетов из контроля в Office 365, если у вас есть требования к политике хранения соответствия требованиям.

Сведения о выбытиях для контроля в Office 365 вы найдете в [статье Путеводитель по Microsoft 365](https://www.microsoft.com/microsoft-365/roadmap) для получения дополнительных сведений.

## <a name="policy-settings"></a>Параметры политики

### <a name="users"></a>Пользователи

Вы можете выбрать **всех пользователей** или определить определенных пользователей в политике соответствия требованиям к общению. При выборе параметра **все пользователи** политика применяется ко всем пользователям и группам, в которые входит любой пользователь. Определение определенных пользователей применяет политику к определенным пользователям и группам, в которые они включены в качестве участника.

### <a name="direction"></a>Direction

По умолчанию **это** условие отображается и не может быть удалено. Параметры направления связи в политике выбираются по отдельности или вместе:

- **Входящее**: вы можете выбрать пункт **входящий трафик** для просмотра сообщений **, отправленных выбранным** пользователям.
- **Исходящие**: вы можете выбрать параметр **исходящий трафик** , если вы хотите просмотреть сообщения **, отправленные** пользователями, для которых вы выбрали контроль.
- **Internal**: можно выбрать параметр **internal (внутренний** ) для просмотра сообщений, передаваемых **между** пользователями, указанными в политике.

### <a name="sensitive-information-types"></a>Типы конфиденциальной информации

Вы можете включить типы конфиденциальной информации в соответствии с политикой соответствия требованиям к общению. Типы конфиденциальной информации являются предопределенными или пользовательскими типами данных, которые могут помочь определить и защитить номера кредитных карт, номера банковских счетов, номера паспортов и многое другое. В рамках защиты от [потери данных (DLP)](data-loss-prevention-policies.md)конфигурация конфиденциальной информации может использовать шаблоны, сходство символов, уровни вероятности и даже настраиваемые типы данных для определения и отметки содержимого, которое может быть конфиденциальным. По умолчанию типы конфиденциальной информации:

- Финансы
- Медицинское и исправная
- Конфиденциальность
- Настраиваемый тип данных

Для получения дополнительных сведений о конфиденциальной информации и шаблонах, включенных в типы по умолчанию, ознакомьтесь с разделами [тип конфиденциальной информации](sensitive-information-type-entity-definitions.md).

### <a name="custom-keyword-dictionaries"></a>Настраиваемые словари ключевых слов

Настройте словари ключевых слов (или лексикона), чтобы обеспечить простое управление ключевыми словами, характерными для вашей организации или отрасли. Словари ключевых слов поддерживают до 100 KБ терминов (после сжатия) в словаре и поддерживают любой язык. Предельное число клиентов также 100 КБ после сжатия. При необходимости можно применить несколько настраиваемых словарей ключевых слов к одной политике или иметь один словарь ключевых слов для каждой политики. Эти словари назначаются в политике соответствия требованиям к общению и могут быть источником из файла (например, из файла CSV или txt) или из списка, который можно [импортировать в центр соответствия требованиям](create-a-keyword-dictionary.md). Используйте настраиваемые словари, чтобы обеспечить поддержку терминов или языков, относящихся к вашей организации и политикам.

### <a name="classifiers"></a>Классификаторов

Встроенные и глобальные классификаторы проверяют отправленные или принимаемые сообщения по всем каналам связи в Организации для различных типов проблем с соответствием. Классификаторы используют сочетание искусственного интеллекта и ключевых слов, чтобы определить язык в сообщениях, которые скорее всего нарушают политики защиты от преследований. Встроенные классификаторы в настоящее время поддерживают только ключевые слова английского языка в сообщениях.

Встроенные и глобальные классификаторы для обеспечения соответствия требованиям, которые проверяют связь для терминов, изображений и мнений для следующих типов языков и контента:

- **Угроза**: сканирование на наличие угроз для фиксации насилия или физического ущерба для пользователя или свойства.
- **Нацеленное преследование**: сканирование на предмет нежелательных дел для людей, касающихся состязания, цвета, религиона и национального происхождения.
- **Ненормативность**: сканирование на наличие ненормативных выражений, которые считают большинство людей.
- **Нежелательные изображения**: сканирование изображений, которые в природе сексуальны.
- **Изображения раци**: сканирование изображений, которые являются сексуальными сугжестиве в природе, но содержат менее четкий контент, чем изображения, сопризнанные взрослым.
- **Изображения Гори**: сканирование изображений, которые изображают сцены насилия и горе.

Классификаторы изображений для *взрослых*, *раци*и *Гори* проверяют файлы в. JPEG,. PNG,. GIF и. Форматы BMP. Размер файлов изображений должен быть меньше 4 мегабайт (МБ), а размеры изображений должны быть больше 50x50 пикселов и превышать 50 килобайт (КБ), чтобы изображение было выполнить для оценки. Идентификация изображений поддерживается в сообщениях электронной почты Exchange Online и каналах и беседах Microsoft Teams.

Встроенные и глобальные классификаторы не предоставляют исчерпывающий список терминов или изображений в этих областях. Кроме того, язык и региональные стандарты постоянно меняются, и в свете этих реалитиес Корпорация Майкрософт оставляет за собой право на обновление классификаторов по собственному усмотрению. В то время как классификаторы могут помочь вашей организации в наблюдении за этими областями, классификаторы не предназначены для того, чтобы предоставить единственный способ отслеживания или адресации такого языка или изображений. Ваша организация, а не Майкрософт, несет ответственность за все решения, связанные с сканированием и блокировкой языка и изображений в этих областях.

Сведения о выученных классификаторах в Microsoft 365 приведены в разделе [Начало работы с помощью классификаторов для обучения](classifier-get-started-with.md).

#### <a name="matching-words-and-phrases-to-emails-or-attachments"></a>Поиск слов и фраз в сообщениях или вложениях
<a name="Matchwords"> </a>

Каждое слово, которое вводится и отделяется запятой, применяется отдельно (для применения к сообщению электронной почты или вложению следует применить только одно слово). Например, используйте условие, **сообщение содержит любое из этих слов**с ключевыми словами "Банк", "конфиденциальный" и "Торговый отдел", разделенными запятой (банком, конфиденциальным сообщением "Торговый торговый"). Политика применяется ко всем сообщениям, включающим в себя слово "Банк", "конфиденциальный" или фразу "торговля для предварительной оценки". Для применения этого условия политики достаточно наличия одного из этих слов или фраз. Слова в сообщении или вложении должны точно совпадать с введенными.

>[!IMPORTANT]
>При импорте файла настраиваемого словаря каждое слово или фраза должно быть разделено символом возврата каретки и в отдельной строке. <br> Например, <br><br>
>*Банк* <br>
>*конфиденциальный* <br>
>*торговля для участников программы предварительной оценки*

Чтобы сканировать как сообщения электронной почты, так и вложения для тех же ключевых слов, создайте [политику защиты от потери данных](create-test-tune-dlp-policy.md) со [словарем пользовательских ключевых слов](create-a-keyword-dictionary.md) для терминов, которые вы хотите сканировать в сообщениях. Эта конфигурация политики определяет определенные ключевые слова, которые отображаются как в сообщении электронной почты, **так и** во вложении электронной почты. Использование стандартных параметров условной политики (*сообщение содержит любое из этих слов* и *Вложение содержит любое из этих*слов) чтобы определить термины в сообщениях и вложениях, необходимо, чтобы они присутствовали как в сообщении, **так** и во вложении.
  
#### <a name="enter-multiple-conditions"></a>Ввод нескольких условий

Если вы вводите несколько условий, Microsoft 365 использует все условия вместе, чтобы определить, когда применять политику соответствия для связи для обмена сообщениями. При настройке нескольких условий для применения политики должны выполняться все условия, если не указать исключение. Например, необходима политика, которая применяется, если сообщение содержит слово "торговля", размер которого превышает 2 МБ. Однако если в сообщении также содержатся слова "утверждено Contoso Contoso", политика не должна применяться. В этом примере три условия определяются следующим образом:
  
- **Сообщение содержит любое из этих слов**с ключевым словом "торговля"
- **Размер сообщения**превышает значение 2 МБ
- **Сообщение не содержит ни одного из этих слов**с ключевыми словами "утверждено компанией Contoso Financial Team"

### <a name="review-percentage"></a>Процент проверки

Если необходимо уменьшить объем контента для проверки, можно указать процент от всех подключений, управляемых политикой соответствия требованиям. В режиме реального времени случайный выбор контента выбирается из общего процента контента, который соответствует выбранным условиям политики. Если вы хотите, чтобы проверяющие могли просматривать все элементы, можно настроить **100%** в политике соответствия требованиям.

## <a name="privacy-preview"></a>Конфиденциальность (Предварительная версия)

Защита конфиденциальности пользователей, имеющих соответствия политике, важна и помогает повысить обжективити при расследовании и анализе данных для оповещений о соответствии. Этот параметр применяется только к именам пользователей, которые представляют собой решение для обеспечения соответствия требованиям. Он не влияет на отображение имен в других решениях для обеспечения соответствия требованиям или в центре администрирования.

Для пользователей с соответствием соответствия требованиям к связи можно выбрать один из следующих параметров в разделе **параметры обеспечения соответствия требованиям**:

- **Показывать анонимные версии**имен пользователей: имена пользователей анонимны, чтобы запретить администраторам, аналитикам, исследованиям данных и проверяющим видеть, какие пользователи связаны с оповещениями политики. Например, пользователь "льготный Тайлор" будет отображаться с случайным псеудоним, таким как "AnonIS8 988" во всех областях взаимодействия с соответствием. При выборе этого параметра анонимизес все пользователи с текущей и предыдущей политикой соответствия и применяются ко всем политикам. При выборе этого параметра сведения о профиле пользователя в подробных сведениях оповещений о соответствии условиям связи будут недоступны. Однако имена пользователей отображаются при добавлении новых пользователей в существующие политики или при назначении пользователей новым политикам. Если отключить этот параметр, имена пользователей будут отображаться для всех пользователей, которым соответствуют текущие или прошлые политики.
- **Не показывать анонимные версии**имен пользователей: отображаются имена всех текущих и прошлых соответствий политик для оповещений о соответствии в связи. Сведения о профиле пользователя (имя, должность, псевдоним, организация или отдел) отображаются для пользователя обо всех оповещениях о соответствии в связи.

## <a name="notice-templates"></a>Шаблоны зазамечаний

Вы можете создавать шаблоны примечаний, если вы хотите отправлять пользователям уведомление о соответствии политики по электронной почте в рамках процесса разрешения проблем. Уведомления можно отправлять только на адрес электронной почты пользователя, связанный с соответствующей политикой, которая создала конкретное оповещение об исправлении. При выборе шаблона уведомления, применяемого к нарушению политики в рамках рабочего процесса исправления, можно выбрать, принимать ли значения полей, определенные в шаблоне, или заменять поля нужным образом.

Уведомления шаблоны — это настраиваемые шаблоны электронной почты, в которых можно определить следующие поля сообщений в области **Параметры соответствия для взаимодействия** :

|**Field**|**Required**| **Details** |
|:-----|:-----|:-----|
|**Название шаблона** | Да | Понятное имя для шаблона уведомления, которое вы выбираете в рабочем процессе при исправлении, поддерживает текстовые символы. |
| **Адрес отправителя** | Да | Адрес одного или нескольких пользователей или групп, которые отправляют сообщение пользователю с совпадающим параметром, выбранным в службе каталогов Active Directory для вашей подписки. |
| **Адреса получателей копий и получателей СКРЫТой копии** | Нет | Необязательные пользователи или группы, уведомляющие о соотношении с политикой, выбранные из Active Directory для вашей подписки. |
| **Тема** | Да | Информация, которая отображается в строке темы сообщения, поддерживает текстовые символы. |
| **Тело сообщения**. | Да | Сведения, отображаемые в теле сообщения, поддерживают текстовые значения или значения HTML. |

### <a name="html-for-notices"></a>HTML-код уведомлений

Если вы хотите создать больше простого текстового сообщения электронной почты для уведомлений, можно создать более подробное сообщение с помощью HTML в поле текст сообщения шаблона уведомления. В следующем примере представлен формат текста сообщения для базового шаблона уведомления электронной почты на основе HTML:

```HTML
<!DOCTYPE html>
<html>
    <body>
        <h2>Action Required: Contoso Employee Code of Conduct Policy Training</h2>
        <p>A recent message you've sent has generated a policy alert for the Contoso Employee <a href='https://www.contoso.com'>Code of Conduct Policy</a>.</p>
        <p>You are required to attend the Contoso Employee Code of Conduct <a href='https://www.contoso.com'>training</a> within the next 14 days. Please contact <a href='mailto:hr@contoso.com'>Human Resources</a> with any questions about this training request.</p>
        <p>Thank you,</p>
        <p><em>Human Resources</em></p>
    </body>
</html>
```

>[!NOTE]
>Реализация атрибута href в HTML в шаблонах уведомлений о соответствии требованиям к связи в настоящее время поддерживает только одиночные кавычки, а не двойные кавычки для URL-ссылок.

## <a name="filters"></a>Фильтры

Фильтры соответствия требованиям на связь позволяют фильтровать и сортировать сообщения оповещения для быстрого исследования и устранения действий. Фильтрация доступна на вкладках **ожидающих** и **разрешенных** для каждой политики. Чтобы сохранить фильтр или набор фильтров как сохраненный запрос фильтра, необходимо настроить одно или несколько значений фильтра. В приведенной ниже таблице представлены сведения о фильтре.

|**Фильтр**|**Details**|
|:-----|:-----|
| **Date** | Дата, когда сообщение было отправлено или получено пользователем в Организации. Чтобы отфильтровать за один день, выберите диапазон дат, начиная с дня, в который вы хотите получить результаты, и заканчивайте его на следующий день. Например, если вы хотите отфильтровать результаты для 9/20/2020, выберите диапазон дат фильтра 9/20/2020-9/21/2020.|
| **Класс File** | Класс сообщения на основе типа сообщения: *сообщение* или *вложение*. |
| **Имеет вложение** | Присутствие вложения в сообщении. |
| **Класс элемента** | Источник сообщения на основе типа сообщения, электронной почты, Microsoft Team Chat, Bloomberg и т. д. Дополнительные сведения об общих типах элементов и классах сообщений можно узнать в статье [типы элементов и классы сообщений](https://docs.microsoft.com/office/vba/outlook/concepts/forms/item-types-and-message-classes). |
| **Домены получателей** | Домен, в который было отправлено сообщение. По умолчанию этот домен обычно является доменом подписки на Microsoft 365. |
| **Получатель** | Пользователь, которому было отправлено сообщение. |
| **Sender** | Пользователь, который отправил сообщение. |
| **Домен отправителя** | Домен, который отправил сообщение. |
| **Размер** | Размер сообщения в КИЛОБАЙТах. |
| **Тема или название** | Тема сообщения или название чата. |
| **Tags** | Теги *,* назначенные сообщению, *совместимым, совместимым*или *несоответствующим требованиям*. |
| **Расширено до** | Имя пользователя, включенное в состав действия по распространению сообщений. |
| **Классификаторов** | Имя встроенных и настраиваемых классификаторов, которые применяются к сообщению. Некоторые примеры: *оскорбительный язык*, *целевое преследование*, *ненормативная лексика, угроза ненормативности*, *угроза*и многое другое.

## <a name="alert-policies"></a>Политики оповещений

После настройки политики автоматически создается соответствующая политика оповещений, а для сообщений, которые соответствуют условиям, определенным в политике, создаются оповещения. По умолчанию для всех политик триггеров оповещений назначен уровень серьезности Medium в соответствующей политике оповещений. Оповещения для политики соответствия требованиям к общению создаются, когда уровень порогового значения триггера агрегирования достигается в соответствующей политике оповещений.

Для политик соответствия требованиям к связи следующие значения политики оповещений настраиваются по умолчанию.

|**Триггер политики оповещений**|**Значение, используемое по умолчанию**|
|:-----|:-----|
| Объединен | Простая агрегация |
| Порог | 4 действия |
| Окно | 60 минут |

>[!Note]
>Параметры триггера пороговых значений политики оповещений для действий поддерживают минимальное значение 3 или выше для политик соответствия требованиям.

Вы можете изменить параметры по умолчанию для триггеров на количество действий, период для действий, а также для определенных пользователей в политиках оповещений на странице **политики оповещений** в центре безопасности & соответствия требованиям.

### <a name="change-the-severity-level-for-an-alert-policy"></a>Изменение уровня серьезности для политики оповещений

Если вы хотите изменить уровень серьезности, назначенный в политике оповещений для определенной политики соответствия требованиям на связь, выполните следующие действия:

1. Войдите в систему [https://compliance.microsoft.com](https://compliance.microsoft.com) , используя учетные данные для учетной записи администратора в организации Microsoft 365.

2. В центре соответствия требованиям Microsoft 365 перейдите к разделу **политики**.

3. На странице " **политики** " выберите пункт " **оповещение о пакете Office 365** ", чтобы открыть страницу **политики оповещений** в **центре безопасности & соответствия требованиям Office 365**.

4. Установите флажок для политики соответствия, которую необходимо обновить, а затем выберите **изменить политику**.

5. На вкладке **Описание** выберите раскрывающийся список **серьезность** , чтобы настроить уровень оповещения политики.

6. Нажмите кнопку **сохранить** , чтобы применить новый уровень серьезности к политике.

7. Нажмите кнопку **Закрыть** , чтобы выйти со страницы сведений о политике оповещений.

## <a name="power-automate-flows-preview"></a>Автоматизированные потоки управления питанием (ознакомительная версия)

[Microsoft Power автоматизиру](https://docs.microsoft.com/power-automate/getting-started) — это служба рабочих процессов, автоматизирующая действия между приложениями и службами. Используя потоки из шаблонов или созданные вручную, вы можете автоматизировать распространенные задачи, связанные с этими приложениями и службами. При включении автоматизации Power Автоматизация для обеспечения соответствия требованиям к связи можно автоматизировать важные задачи для оповещений и пользователей. Вы можете настроить автоматический поток управления питанием, чтобы уведомлять руководителей об оповещениях о соответствии пользователей и других приложениях.

Клиентам с подписками Microsoft 365, которые включают в себя обеспечение соответствия требованиям, не требуется дополнительных лицензий Power автоматизирования для использования рекомендуемого шаблона автоматизированного управления по умолчанию для обеспечения соответствия требованиям. Шаблон по умолчанию можно изменить, чтобы обеспечить поддержку Организации и основные сценарии обеспечения соответствия требованиям. Если вы решили использовать функции автоматизированного управления питанием в этих шаблонах, создайте настраиваемый шаблон с помощью соединителя соответствия требованиям Microsoft 365 или используйте шаблоны Power автоматизировать для других областей соответствия в Microsoft 365, возможно, вам потребуются дополнительные лицензии Power автоматизирующие.

>[!IMPORTANT]
>Приходят приглашения на дополнительные проверки лицензий при тестировании автоматизации энергопотребления? Возможно, ваша организация еще не получала обновления службы для этой функции предварительного просмотра. Выполняется развертывание обновлений, а все организации с подпиской Microsoft 365, которые включают соответствие условиям связи, должны иметь поддержку лицензий для потоков, созданных на основе рекомендуемых автоматизированных шаблонов, на 30 октября 2020.

![Автоматизация управления питанием для обеспечения соответствия требованиям](../media/communication-compliance-power-automate.png)

Для поддержки автоматизации процессов для оповещений о соответствии в связи предоставляется следующий шаблон Power автоматизированный шаблон.

- **Уведомить руководителя, когда у пользователя есть оповещение о соответствии условиям связи**: некоторым организациям может потребоваться уведомление о немедленном управлении, когда пользователь имеет оповещение о соответствии требованиям. Когда этот процесс настроен и выбран, руководителю пользователя в случае отправляется сообщение электронной почты со следующими сведениями о всех оповещениях:
    - Соответствующая политика для оповещения
    - Дата и время оповещения
    - Уровень серьезности оповещения

### <a name="create-a-power-automate-flow"></a>Создание автоматизированного процесса управления питанием

Чтобы создать поток автоматизации управления питанием из рекомендуемого шаблона по умолчанию, вы будете использовать параметр **Управление потоками автоматизации управления питанием** из **автоматизированного** управления при непосредственной работе с оповещением. Чтобы создать поток автоматизации управления питанием с помощью **управления потоками автоматизации управления питанием**, необходимо быть членом по крайней мере одной группы ролей обеспечения соответствия для взаимодействия.

Выполните следующие действия, чтобы создать потоки автоматизации энергопотребления из шаблона по умолчанию:

1. В центре соответствия требованиям Microsoft 365 перейдите к разделу политики **обеспечения соответствия требованиям**  >  **Policies** и выберите политику с оповещением, которое нужно просмотреть.
2. В политике выберите вкладку **ожидающие** и выберите ожидающее оповещение.
3. Выберите **Power Автоматизация** в меню Действие оповещения.
4. На странице " **Автоматизация управления питанием** " выберите шаблон по умолчанию в разделе **шаблоны обеспечения соответствия, которые могут быть выбраны** на странице.
5. В этом случае будет передаваться список внедренных подключений, необходимых для этого процесса, и отображается, если состояние подключения доступно. При необходимости обновите все подключения, которые не отображаются как доступные. Нажмите кнопку **продолжить**.
6. По умолчанию Рекомендуемые потоки предварительно настроены с использованием рекомендуемого соответствия для связи и полей данных службы Microsoft 365, необходимых для выполнения назначенной задачи для потока. При необходимости настройте компоненты Flow с помощью элемента управления **Показать дополнительные параметры** и настройте доступные свойства для компонента Flow.
7. При необходимости добавьте дополнительные действия в потоки, нажав кнопку **создать шаг** . В большинстве случаев это изменение не требуется для рекомендуемых шаблонов по умолчанию.
8. Выберите **Сохранить черновик** , чтобы сохранить данные для дальнейшей настройки позже, или нажмите кнопку **сохранить** , чтобы завершить настройку этого процесса.
9. Нажмите кнопку **Закрыть** , чтобы вернуться на страницу Power автоматизированного управления. Новый шаблон будет отображаться как поток на вкладке **мои потоки** и автоматически доступен из элемента управления Power автоматизиру для пользователя, создавшего поток при работе с оповещениями о соответствии требованиям.

### <a name="share-a-power-automate-flow"></a>Совместное использование автоматизированного управления питанием

По умолчанию автоматизированные потоки управления питанием, созданные пользователем, доступны только этому пользователю. Чтобы другие пользователи могли получить доступ к такому каналу и использовать потоки, этот канал должен совместно использоваться создателем поточнее. Для совместного использования рабочего процесса вы будете использовать элемент управления **Power автоматизировать** при работе непосредственно в оповещении.

Для совместного использования автоматизированного управления питанием необходимо быть членом по крайней мере одной группы ролей соответствия для взаимодействия.
Выполните следующие действия, чтобы предоставить общий доступ к автоматизации энергопотребления:

1. В центре соответствия требованиям Microsoft 365 перейдите к разделу политики **обеспечения соответствия требованиям**  >  **Policies** и выберите политику с оповещением, которое нужно просмотреть.
2. В политике выберите вкладку **ожидающие** и выберите ожидающее оповещение.
3. Выберите **Power Автоматизация** в меню Действие оповещения.
4. На странице **Power автоматизация потоков** перейдите на вкладку **мои потоки** или **потоки команд** .
5. Выберите способ передачи, а затем выберите пункт **общий доступ** в меню Параметры процесса.
6. На странице "общий доступ к данным" выберите имя пользователя или группы, которые требуется добавить в качестве владельца для этого процесса.
7. В диалоговом окне **используемое подключение** нажмите кнопку **ОК** , чтобы подтвердить, что добавленный пользователь или группа будут иметь полный доступ к данному блоку.

### <a name="edit-a-power-automate-flow"></a>Изменение автоматизированного процесса управления питанием

Если вам нужно изменить последовательность, вы будете использовать элемент управления **Power автоматизировать** при работе непосредственно в оповещении. Чтобы изменить потоки автоматизации энергопотребления, необходимо быть членом по крайней мере одной группы ролей соответствия для взаимодействия.

Выполните следующие действия, чтобы изменить процесс автоматизации Power Автоматизация:

1. В центре соответствия требованиям Microsoft 365 перейдите к разделу политики **обеспечения соответствия требованиям**  >  **Policies** и выберите политику с оповещением, которое нужно просмотреть.
2. В политике выберите вкладку **ожидающие** и выберите ожидающее оповещение.
3. Выберите **Power Автоматизация** в меню Действие оповещения.
4. На странице **Power автоматизирующие потоки** выберите поток для редактирования. В меню Управление выполнением выберите команду **изменить** .
5. Нажмите **кнопку**"  >  **Параметры** ", чтобы изменить параметр компонента или **кнопку "многоточие**"  >  **Delete** , чтобы удалить компонент процесса.
6. Нажмите кнопку **сохранить** , а затем **закройте** , чтобы завершить изменение процесса.

### <a name="delete-a-power-automate-flow"></a>Удаление автоматизированного процесса управления питанием

Если вам нужно удалить рабочий процесс, вы будете использовать элемент управления **Power автоматизировать** при работе непосредственно в оповещении. Для удаления автоматизированного процесса управления питанием необходимо быть членом по крайней мере одной группы ролей соответствия для взаимодействия.

Выполните следующие действия, чтобы удалить процесс автоматизации Power:

1. В центре соответствия требованиям Microsoft 365 перейдите к разделу политики **обеспечения соответствия требованиям**  >  **Policies** и выберите политику с оповещением, которое нужно просмотреть.
2. В политике выберите вкладку **ожидающие** и выберите ожидающее оповещение.
3. Выберите **Power Автоматизация** в меню Действие оповещения.
4. На странице **Power автоматизация потоков** выберите поток, который требуется удалить. В меню Управление выполнением выберите команду **Удалить** .
5. В диалоговом окне подтверждения удаления нажмите кнопку **Удалить** , чтобы удалить этот процесс, или кнопку **Отмена** , чтобы выйти из действия удаления.

## <a name="reports-preview"></a>Отчеты (Предварительная версия)

Новая панель мониторинга **отчетов** является центральным расположением для просмотра всех отчетов о соответствии требованиям. Графические элементы отчетов предоставляют краткий обзор ценных сведений, наиболее часто необходимых для общей оценки состояния действий обеспечения соответствия требованиям. Информация, включенная в графические элементы отчетов, недоступна для экспорта. Подробные отчеты предоставляют подробные сведения, связанные с определенными областями обеспечения соответствия требованиям, и предлагают возможность фильтровать, группировать, сортировать и экспортировать данные во время рецензирования.

**Панель мониторинга отчетов** содержит следующие графические элементы отчетов и ссылки на подробные отчеты:

- **Последнее соответствие политике** Widget: отображает количество совпадений по активной политике с течением времени.
- **Разрешенные элементы с помощью** мини-приложения политики: отображает количество оповещений о соответствии политики, разрешенных политикой с течением времени.
- **Пользователи с большинством** мини-приложения "соответствие политик": Отображает пользователей (или анонимных имен) и число совпадений политик для данного периода.
- **Политика с наиболее сопоставленным** графическим элементом: отображает политики и количество совпадений для определенного периода, с наивысшим приоритетом для совпадений.
- "Мини-приложение **политики** ": отображает количество укрупнений для каждой политики за заданное время.
- **Параметры политик и подробные сведения о состоянии** : подробное описание настройки и параметров политики, а также общего состояния каждой политики (соответствия и действий) для сообщений. Используйте параметр *Export* , чтобы создать. CSV-файл, содержащий сведения отчета.
- Подробный отчет **элементов и действий для каждой политики** : Проверка и экспорт сопоставленных элементов и действий исправления для каждой политики. Используйте параметр *Export* , чтобы создать. CSV-файл, содержащий сведения отчета.
- **Элемент и действия в указанном местоположении** подробный отчет: Проверка и экспорт совпадений элементов и действий по исправлению для Microsoft 365. Используйте параметр *Export* , чтобы создать. CSV-файл, содержащий сведения отчета.

## <a name="audit"></a>Аудит

В некоторых случаях необходимо предоставить сведения для аудиторов или аудиторов соответствия требованиям, чтобы доказать, что действия пользователей и общения. Эта информация может представлять собой сводку по всем действиям, связанным с определенной организационной политикой или в момент изменения политики соответствия. Политики обеспечения соответствия требованиям имеют встроенные журналы аудита для полной готовности к внутренним или внешним аудитам. Подробные журналы аудита каждого действия создания, изменения и удаления фиксируются в политиках взаимодействия для предоставления подтверждения процедур супервизора.

>[!Important]
>Прежде чем записывать события соответствия, необходимо включить аудит для вашей организации. Чтобы включить аудит, обратитесь к разделу [Включение журнала аудита](communication-compliance-configure.md#step-2-required-enable-the-audit-log).

Чтобы просмотреть действия по обновлению политики соответствия требованиям, на главной странице для любой политики выберите элемент Управление **обновлениями политики** . Для экспорта действий обновления необходимо назначить роли *глобального администратора* или *администратора соответствия требованиям к связи* . Это действие создает файл аудита в формате CSV, который содержит следующие сведения:

|**Field**|**Details**|
|:-----|:-----|
| **CreationDate** | Дата выполнения действия обновления в политике. |
| **UserIds** | Пользователь, который выполнил действие обновления в политике. |
| **Operations** | Операции обновления, выполняемые для политики. |
| **аудитдата** | Это поле является основным источником данных для всех действий по обновлению политики. Все действия по обновлению записываются и разделяются разделителями запятыми. |

Чтобы просмотреть действия проверки соответствия требованиям для политики, выберите элемент Управление **действиями проверки** на странице **Обзор** для определенной политики. Для экспорта действий проверки необходимо назначить роли *глобального администратора* или *администратора соответствия требованиям* . Это действие создает файл аудита в формате CSV, который содержит следующие сведения:

|**Field**|**Details**|
|:-----|:-----|
| **CreationDate** | Дата выполнения действия проверки в политике. |
| **UserIds** | Пользователь, который выполнил действие проверки в политике. |
| **Operations** | Операции проверки, выполняемые для политики. |
| **аудитдата** | Это поле является основным источником данных для всех действий по проверке политики. Все действия проверки записываются и разделяются разделителями запятыми. |

Вы также можете просматривать действия аудита в едином журнале аудита или с помощью командлета PowerShell [Search — UnifiedAuditLog](https://docs.microsoft.com/powershell/module/exchange/search-unifiedauditlog) .

Например, в приведенном ниже примере возвращаются действия для всех действий супервизора (политик и правил):

```PowerShell
Search-UnifiedAuditLog -StartDate $startDate -EndDate $endDate -RecordType AeD -Operations SupervisoryReviewTag
```

В этом примере возвращаются действия по обновлению для политик соответствия требованиям.

```PowerShell
Search-UnifiedAuditLog -StartDate $startDate -EndDate $endDate -RecordType Discovery -Operations SupervisionPolicyCreated,SupervisionPolicyUpdated,SupervisionPolicyDeleted
```
