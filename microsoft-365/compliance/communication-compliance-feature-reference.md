# <a name="communication-compliance-feature-reference"></a>Ссылка на функцию соответствия требованиям связи

## <a name="policies"></a>Политики

> [!IMPORTANT]
> Использование PowerShell для создания политик соответствия требованиям к обмену данными и управления ими не поддерживается. Для создания и управления этими политиками необходимо использовать элементы управления политиками в решении Microsoft 365 соответствия [требованиям.](https://compliance.microsoft.com/supervisoryreview)

Политики соответствия требованиям к обмену данными для организаций Microsoft 365 создаются в Центре соответствия требованиям Microsoft 365. Политики соответствия требованиям к коммуникациям определяют, какие сообщения и пользователи подлежат проверке в вашей организации, определяют настраиваемые условия, которые должны соответствовать коммуникациям, и определяют, кому следует выполнять обзоры. Пользователи, которым  назначена роль администратора соответствия требованиям связи, могут настроить политики, а все, кому назначена эта роль, могут получить доступ к странице соответствия требованиям связи и глобальным настройкам в Центр соответствия требованиям Microsoft 365.  При необходимости можно экспортировать историю изменений в политику в файл .csv (разделенные запятой значения), который также включает состояние оповещений, ожидающих рассмотрения, эскалации элементов и разрешенных элементов. Политики не могут быть переименованы и могут быть удалены, если они больше не нужны.

> [!NOTE]
> Политики надзора, созданные в Центре & безопасности для Office 365 подписки, не могут перейти в Microsoft 365. Если вы мигрируете из подписки Office 365 подписку на подписку Microsoft 365, вам потребуется создать новые полиции соответствия требованиям связи, чтобы заменить существующие политики надзора.

## <a name="policy-templates"></a>Шаблоны политик

Шаблоны политики — это заранее определенные параметры политики, которые можно использовать для быстрого создания политик для устранения распространенных сценариев соответствия требованиям. Каждый из этих шаблонов имеет различия в условиях и области, и все шаблоны используют одинаковые типы сигналов сканирования. Вы можете выбрать из следующих шаблонов политики:

|**Область**|**Шаблон политики**|**Сведения**|
|:-----|:-----|:-----|
| **Оскорбительные выражения и противодействие домогательствам** | Мониторинг сообщений для оскорбительных языков | - Расположения: Exchange Online, Microsoft Teams, Yammer, Skype для бизнеса <br> - Направление: входящие, исходящие, внутренние <br> - Процент отзывов: 100% <br> - Условия: классификатор оскорбительных языков |
| **Конфиденциальные данные.** | Мониторинг сообщений для конфиденциальной информации | - Расположения: Exchange Online, Microsoft Teams, Yammer, Skype для бизнеса <br> - Направление: входящие, исходящие, внутренние <br> - Процент отзывов: 10% <br> - Условия: конфиденциальную информацию, шаблоны и типы контента, настраиваемый параметр словаря, вложения размером более 1 МБ |
| **Соответствие нормативным требованиям** | Мониторинг сообщений для информации, связанной с соблюдением финансового законодательства | - Расположения: Exchange Online, Microsoft Teams, Yammer, Skype для бизнеса <br> - Направление: входящие, исходящие <br> - Процент отзывов: 10% <br> - Условия: настраиваемый параметр словаря, вложения размером более 1 МБ |
| **Конфликт интересов** | Мониторинг сообщений между двумя группами или двумя пользователями, чтобы избежать конфликтов интересов | - Расположения: Exchange Online, Microsoft Teams, Yammer, Skype для бизнеса <br> - Направление: внутренняя <br> - Процент отзывов: 100% <br> - Условия: Нет |

Сообщения сканируют каждые 24 часа после создания политик времени. Например, если в 11:00 создается оскорбительная языковая политика, политика будет собирать сигналы соответствия требованиям связи каждые 24 часа в 11:00 ежедневно. На этот раз изменение политики не изменится. Чтобы просмотреть последнюю дату и время проверки политики, перейдите в столбец *Последняя* проверка политики на странице **Политика.** После создания новой политики может потребоваться до 24 часов для просмотра первой даты и времени проверки политики. Дата и время последнего сканирования преобразуются в часовой пояс локальной системы.

## <a name="pausing-a-policy-preview"></a>Прио пауза политики (предварительный просмотр)

После создания политики соответствия требованиям связи политика может быть временно приостановлена при необходимости. Прио пауза в политике может использоваться для тестирования или устранения неполадок, а также для оптимизации условий политики. Вместо удаления политики в этих условиях прио время действия политики также сохраняется существующее оповещение о политике и сообщения для текущих расследований и обзоров. Приостановка политики предотвращает проверку и генерацию оповещений для всех условий сообщений пользователей, определенных в политике на время приостановки политики. Чтобы приостановить или перезапустить политику, пользователи должны быть членами группы ролей администратора соответствия *требованиям* связи.

Чтобы приостановить политику, перейдите на страницу **Политика,** выберите политику и выберите политику **Пауза** из панели инструментов действий. На области **политики паузы** подтвердим, что вы хотите приостановить политику, выбрав **паузу.** В некоторых случаях для приостановки политики может потребоваться до 24 часов. После приостановки политики оповещения о сообщениях, совпадающих с политикой, не создаются. Тем не менее сообщения, связанные с оповещениями, созданными до приоэзии политики, остаются доступными для расследования, проверки и устранения.

Состояние политики для приостановленных политик может указывать на несколько штатов:

- **Active.** Политика активна
- **Пауза.** Политика полностью приостановлена.
- **Приостановка.** Политика находится в процессе приостановки.
- **Возобновление.** Политика в процессе возобновления.
- **Ошибка при повторном** веяние. Для отслеживания стека ошибок  наведите курсор мыши над ошибкой при повторном состоянии в столбце Состояние на странице Политика.
- **Ошибка при прио паузе.** При прио паузе политики была допущена ошибка. Для отслеживания стека ошибок наведите курсор мыши над ошибкой при *прио* паузе в столбце Состояние на странице Политика.

Чтобы возобновить политику, перейдите на страницу **Политика,** выберите политику и выберите политику **Resume** из панели инструментов действий. На области **политики Резюме** подтвердим, что вы хотите возобновить политику, выбрав **резюме.** В некоторых случаях для возобновления политики может потребоваться до 24 часов. После возобновления политики будут созданы оповещения о сообщениях, совпадающих с политикой, которые будут доступны для расследования, проверки и восстановления.

## <a name="permissions"></a>Разрешения

> [!IMPORTANT]
> По умолчанию глобальные администраторы не имеют доступа к функциям соответствия требованиям связи. Роли, задамые на этом шаге, необходимы для того, чтобы все функции соответствия требованиям связи были доступны.

Существует пять групп ролей, используемых для настройки разрешений для управления функциями соответствия требованиям связи. Чтобы **сделать** соответствие коммуникации доступным в качестве параметра меню в Центр соответствия требованиям Microsoft 365 и продолжить выполнение  этих  действий по настройке, необходимо быть назначены группам ролей администратора соответствия требованиям к связи или соответствия требованиям. Чтобы получить доступ и управлять функциями соответствия требованиям связи после начальной конфигурации, пользователи должны быть членами по крайней мере одной группы ролей соответствия требованиям.

В зависимости от того, как управлять политиками и оповещениями в области связи, необходимо назначить пользователей определенным группам ролей. Вы можете назначить пользователей с различными обязанностями по обеспечению соответствия определенным группам ролей для управления различными областями функций соответствия требованиям. Или вы можете назначить все учетные записи пользователей назначенным администраторам, аналитикам, следователям и зрителям в группу ролей *соответствия* требованиям связи. Используйте одну группу ролей или несколько групп ролей, чтобы наилучшим образом соответствовать вашим требованиям по управлению соответствием требованиям.

Выберите из этих параметров групп ролей при настройке соответствия требованиям связи:

|**Группа ролей**|**Разрешения групп ролей**|
|:-----|:-----|
| **Соответствие требованиям связи** | Используйте эту группу ролей для управления соответствием требованиям к общению для организации в одной группе. Добавляя все учетные записи пользователей для назначенных администраторов, аналитиков, следователей и зрителей, можно настроить разрешения на соответствие требованиям связи в одной группе. Эта группа ролей содержит все роли разрешений на связь. Эта конфигурация является самым простым способом быстрого начала работы с соответствием требованиям к связи и подходит для организаций, которые не нуждаются в отдельных разрешениях, определенных для отдельных групп пользователей. |
| **Администратор соответствия требованиям к обмену данными** | Используйте эту группу ролей для первоначальной настройки соответствия требованиям к коммуникациям, а затем для разделения администраторов соответствия требованиям связи в определенную группу. Пользователи, назначенные этой группе ролей, могут создавать, читать, обновлять и удалять политики соответствия требованиям к связи, глобальные параметры и назначения групп ролей. Пользователи, назначенные этой группе ролей, не могут просматривать оповещения о сообщениях. |
| **Аналитик соответствия требованиям связи** | Используйте эту группу для назначения разрешений пользователям, которые будут выступать в качестве аналитиков соответствия требованиям связи. Пользователи, назначенные этой группе ролей, могут просматривать политики, в которых они назначены в качестве рецензентов, просматривать метаданные сообщений (а не содержимое сообщения), переадрастать другим рецензентам или отправлять уведомления пользователям. Аналитики не могут разрешить ожидающих оповещений. |
| **Следователь соответствия требованиям связи** | Используйте эту группу для назначения разрешений пользователям, которые будут выступать в качестве исследователей соответствия требованиям связи. Пользователи, направляемые в эту группу ролей, могут просматривать метаданные и содержимое сообщений, перенаправляться к другим рецензентам, перенаправляться в Advanced eDiscovery, отправлять уведомления пользователям и устранять предупреждение. |
| **Зритель соответствия требованиям к обмену данными** | Используйте эту группу для назначения разрешений пользователям, которые будут управлять отчетами о связи. Пользователи, заметив эту группу ролей, могут получать доступ ко всем виджетам отчетности на домашней странице соответствия требованиям связи и просматривать все отчеты о соответствии требованиям связи. |

### <a name="for-organizations-using-the-original-permissions-and-role-groups"></a>Для организаций, использующих исходные разрешения и группы ролей

Новая структура групп ролей заменяет начальную групповую структуру ролей для соответствия требованиям к связи. Для организаций, которые уже используют соответствие требованиям связи, необходимо на должное место администратора надзорного надзора, чтобы начать работу с соблюдением требований к Центр соответствия требованиям Microsoft 365. Кроме того, необходимо создать новую группу ролей для рецензентов с помощью администратора надзорных обзоров, управления случаями, администратора соответствия требованиям и ролей Review для изучения и исправлений сообщений со совпадениями политик. По сути, все администраторы и рецензенты были в одной группе ролей и у всех были одинаковые разрешения на доступ и управление. С помощью последних обновлений для соответствия требованиям связи следует планировать перенос из предыдущей групповой структуры ролей в новую групповую структуру ролей. Поддержка предыдущей групповой структуры ролей будет поэтапной.

Чтобы помочь в планировании миграции, рассмотрим следующий пример. В настоящее время в организации есть три типа пользователей: ИТ-администраторы, триадж и рецензенты. Эти три типа пользователей находятся в предыдущей групповой структуре ролей и являются членами одной группы ролей со следующими ролями:

- Администратор проверки надзора
- Управление случаем
- Администратор соответствия требованиям
- Проверка

Чтобы обновить роли для этих пользователей для новой структуры групп ролей и отделить разрешения доступа и управления для пользователей, можно рассмотреть три новые группы и связанные с ними назначения групп ролей:

- **ИТ-администраторы:** назначены новой группе *ролей администратора* соответствия требованиям связи.
- **Triage**: назначен группе *аналитика соответствия требованиям* связи.
- **Рецензенты.** Назначена новая группа *исследователей* соответствия требованиям связи.

## <a name="supervised-users"></a>Контролируемые пользователи

Прежде чем начинать контроль соответствия требованиям к обмену данным, необходимо определить, кому требуется этот контроль. Для идентификации пользователей или групп, подлежащих контролю, в политике используются адреса электронной почты пользователей. Некоторые примеры из этих групп Microsoft 365, Exchange на основе списков рассылки, Yammer сообществ и Microsoft Teams каналов. Также можно исключить из сканирования определенных пользователей или группы, задав группу исключения или список групп. Дополнительные сведения о типах групп, поддерживаемых в политиках соответствия требованиям связи, см. в руб. [Начало работы с соответствием требованиям связи.](communication-compliance-configure.md#step-3-optional-set-up-groups-for-communication-compliance)

> [!IMPORTANT]
> Пользователи, охваченные политиками соответствия требованиям связи, должны иметь лицензию Соответствие требованиям Microsoft 365 E5, лицензию Office 365 корпоративный E3 с надстройки Advanced Compliance или быть включены в подписку Office 365 корпоративный E5. Если у вас нет существующего плана Enterprise E5 и вы хотите попробовать соответствие требованиям связи, вы можете зарегистрироваться для пробного Office 365 корпоративный [E5](https://go.microsoft.com/fwlink/p/?LinkID=698279).

## <a name="reviewers"></a>Рецензенты

При создании политики соответствия требованиям связи необходимо определить, кто просматривает сообщения контролируемых пользователей. Для идентификации пользователей или групп, подлежащих проверке контролируемых сообщений, в политике используются адреса электронной почты пользователей. Все рецензенты должны иметь почтовые ящики, Exchange Online на них должны  быть назначены роли анализ соответствия требованиям связи или расследования соответствия *требованиям.* Рецензентам (аналитикам или следователям) также должна быть назначена роль управления случаями соответствия требованиям.  Когда рецензенты добавляются в политику, они автоматически получают сообщение электронной почты, которое сообщает им о назначении политики и предоставляет ссылки на сведения о процессе проверки.

## <a name="groups-for-supervised-users-and-reviewers"></a>Группы для контролируемых пользователей и рецензентов

Чтобы упростить настройку, создайте группы для людей, которые нуждаются в просмотре их сообщений, и группы для людей, которые просматривают эти сообщения. Таких групп может потребоваться несколько. Например, если вы хотите контролировать общение двух разных групп пользователей или указать группу, которую не нужно контролировать.

При назначении группы рассылки в политике политика отслеживает все сообщения электронной почты от каждого пользователя группы рассылки. При назначении Microsoft 365 группы политики политика отслеживает все сообщения, отправленные этой группе, а не отдельные сообщения, полученные каждым участником группы.

Добавление групп и списков рассылки в политики соответствия требованиям к связи является частью общего набора условий и правил, поэтому максимальное число групп и списков рассылки, поддерживаемых политикой, зависит от количества условий, добавленных в политику. Каждая политика должна поддерживать около 20 групп или списков рассылки в зависимости от количества дополнительных условий, присутствующих в политике.

## <a name="supported-communication-types"></a>Поддерживаемые типы сообщений

В соответствии с политиками соответствия требованиям связи можно сканировать сообщения на одной или более следующих платформах связи как групповой, так и в качестве автономных источников. Сообщения, захваченные на этих платформах, сохраняются в течение семи лет для каждой политики по умолчанию, даже если пользователи покидают организацию и их почтовые ящики удаляются.

- **Microsoft Teams.** Можно отсканировать сообщения чата в общедоступных и частных каналах Microsoft Teams и отдельных чатах. Когда пользователям назначена политика соответствия коммуникации с выбранным Microsoft Teams, сообщения чата для пользователей автоматически отслеживаются во всех Microsoft Teams, где пользователи являются членами. Microsoft Teams автоматически включается для заранее определенных шаблонов политики и выбирается по умолчанию в настраиваемом шаблоне политики. Teams чаты, совпадающие с условиями политики соответствия требованиям связи, могут обрабатываться до 48 часов. Используйте следующие конфигурации управления группой для контроля за отдельными пользовательскими чатами и каналами связи в Teams:

    - **Для Teams чата:** Назначение отдельных пользователей или назначение группы [рассылки](https://support.office.com/article/Distribution-groups-E8BA58A8-FAB2-4AAF-8AA1-2A304052D2DE) политике соответствия требованиям к коммуникации. Этот параметр задается для отношений между пользователями и чатами "один к одному" или "один ко многим".
    - **Для Teams каналов связи:** Назначить каждый Microsoft Teams или Microsoft 365 группу, которую необходимо просмотреть, которая содержит определенного пользователя, политике соответствия требованиям к коммуникации. Если вы включаете одного и того же пользователя в другие каналы Microsoft Teams или группы Microsoft 365, обязательно добавьте эти новые каналы и группы в политику соответствия требованиям обмена данными. Если какой-либо член канала является подконтрольным  пользователем в политике, а направление входящие настроено в политике, все сообщения, отправленные в канале, подлежат проверке и потенциальным совпадениям политики (даже для пользователей канала, которые явно не контролируются). Например, пользователь A является владельцем или членом канала. Пользователи B и User C являются членами одного канала и используют язык, который совпадает с языковой политикой оскорбительного языка, которая контролирует только пользователя A. Пользователи B и User C создают политики для бесед в канале, даже если они непосредственно не контролируются в оскорбительной языковой политике. Teams между пользователем B и пользователем C, которые находятся за пределами канала, включаемого пользователя A, не будут подвергаться оскорбительной языковой политике, включающей пользователя A. Чтобы исключить из наблюдения членов канала, если другие члены канала  находятся под явным наблюдением, отключите параметр направления входящего сообщения в применимой политике соответствия требованиям к коммуникации.
    - **Для Teams** чата с гибридными средами электронной почты: соответствие требованиям связи может отслеживать сообщения чата для пользователей для организаций с локальной развертыванием Exchange или внешнего поставщика электронной почты, который включил Microsoft Teams. Необходимо создать группу рассылки для пользователей с локальной или внешними почтовыми ящиками для мониторинга. При создании политики соответствия требованиям связи эту группу  рассылки назначит в качестве выбора подконтрольных пользователей и групп в мастере политики. Дополнительные сведения о требованиях и ограничениях для включения облачного хранилища и Teams локальной поддержки см. в Teams данных чата для локального [пользователя.](search-cloud-based-mailboxes-for-on-premises-users.md)

- **Exchange** электронной почты: почтовые ящики, которые Exchange Online в Microsoft 365 или Office 365, имеют право на сканирование сообщений. Exchange сообщений электронной почты и вложений, совпадающих с условиями политики соответствия требованиям, может занять до 24 часов. Для контроля соответствия требованиям к обмену данными поддерживаются те же типы вложений, что и для [проверки содержимого согласно правилам потока почты Exchange](/exchange/security-and-compliance/mail-flow-rules/inspect-message-attachments#supported-file-types-for-mail-flow-rule-content-inspection).

- **Yammer:** можно отсканировать частные сообщения, публичные беседы и связанные с ними вложения в Yammer сообществах. При добавлении пользователя в политику соответствия требованиям связи, которая включает Yammer как определенный канал, в процесс сканирования включаются Yammer всех Yammer, в которые входит пользователь. Yammer чаты и вложения, совпадающие с условиями политики соответствия требованиям связи, могут обрабатываться в течение 24 часов. Yammer должен быть в [режиме native для](/yammer/configure-your-yammer-network/overview-native-mode) политик соответствия требованиям к коммуникациям для мониторинга Yammer сообщений и вложений. В режиме Native все Yammer находятся в Azure Active Directory (AAD), все группы Office 365 группы, а все файлы хранятся в SharePoint Online.

- **Skype для бизнеса Online**: можно контролировать сообщения чата и вложения в них в Skype для бизнеса Online. Обработка чатов Skype для бизнеса Online, которые удовлетворяют условиям политики соответствия требованиям к обмену данными, может занять до 24 часов. Контролируемые беседы в чате являются источником из предыдущих [бесед, сохраненных в Skype для бизнеса Online](https://support.office.com/article/Find-a-previous-Skype-for-Business-conversation-18892eba-5f18-4281-8c87-fd48bd72e6a2).  Используйте следующую конфигурацию управления группой, чтобы контролировать общение пользователей в чате Skype для бизнеса Online:

    - **Для Skype для бизнеса сетевых чатов:** назначьте [](https://support.office.com/article/Distribution-groups-E8BA58A8-FAB2-4AAF-8AA1-2A304052D2DE) отдельных пользователей или назначьте группу рассылки политике соответствия требованиям. Этот параметр задается для отношений между пользователями и чатами "один к одному" или "один ко многим".

- **Сторонние** источники. Вы можете сканировать сообщения для данных, импортируемых в почтовые ящики в вашей Microsoft 365 организации из сторонних источников, таких как [Instant Bloomberg,](archive-instant-bloomberg-data.md) [Slack,](archive-slack-data.md) [Zoom,](archive-zoommeetings-data.md)SMS и многих других. Полный список соединители, поддерживаемые в соответствии с данными связи, см. в [архиве сторонних данных.](archiving-third-party-data.md)

    Необходимо настроить сторонний соединители для Microsoft 365 организации, прежде чем назначить соединители политике соответствия требованиям. В **разделе Сторонние источники** мастера политики соответствия требованиям связи отображаются только настроенные в настоящее время сторонние соединители.

## <a name="policy-settings"></a>Параметры политики

### <a name="users"></a>Пользователи

Вы можете выбрать выбор всех **пользователей** или определить конкретных пользователей в политике соответствия требованиям к коммуникациям. Если выбрать **Все пользователи**, политика будет применена ко всем пользователям и ко всем группам, в которые входит каждый пользователь. Если задать определенных пользователей, политика будет применена к этим пользователям и к любым группам, в которые входят эти пользователи.

### <a name="direction"></a>Направление

По умолчанию отображается **условие Direction,** которое не может быть удалено. Параметры направления коммуникации в политике выбираются по отдельности или вместе:

- **Входящий**: обнаруживает сообщения, отправляемые контролируемым пользователям от внешних и внутренних отправителей, в том числе других контролируемых пользователей в политике. 
- **Исходящие:** обнаруживает сообщения,  отосланные от контролируемых пользователей внешним и внутренним получателям, в том числе другим контролируемым пользователям в политике.
- **Внутренняя:** обнаруживает связь **между контролируемыми** пользователями или группами в политике.

### <a name="sensitive-information-types"></a>Типы конфиденциальной информации

Вы можете включить типы конфиденциальной информации в качестве части политики соответствия требованиям к коммуникациям. Типы конфиденциальной информации — это заранее определенные или настраиваемые типы данных, которые могут помочь определить и защитить номера кредитных карт, номера банковских счетов, номера паспортов и другие. В рамках [](dlp-learn-about-dlp.md)"Сведения о предотвращении потери данных" конфигурация конфиденциальной информации может использовать шаблоны, близость символов, уровни уверенности и даже настраиваемые типы данных, чтобы помочь определить и пометить содержимое, которое может быть конфиденциальным. Типы конфиденциальной информации по умолчанию:

- Финансы
- Медицина
- Конфиденциальность
- Тип настраиваемой информации

Дополнительные сведения о конфиденциальных сведениях и шаблонах, включенных в типы по умолчанию, см. в материале [Sensitive information type entity definitions.](sensitive-information-type-entity-definitions.md)

### <a name="custom-keyword-dictionaries"></a>Настраиваемые словари ключевых слов

Настройте настраиваемые словари ключевых слов (или словари), чтобы обеспечить простое управление ключевыми словами, определенными для вашей организации или отрасли. Словари ключевых слов поддерживают до 100 КБ терминов (после сжатия) в словаре и поддерживают любой язык. Ограничение клиента также составляет 100 КБ после сжатия. При необходимости можно применить несколько пользовательских словарей ключевых слов к одной политике или иметь один словарь ключевых слов для политики. Эти словари назначены в политике соответствия требованиям связи и могут быть источником из файла (например, списка .csv или .txt), или из списка, который можно импортировать в центре соответствия [требованиям.](create-a-keyword-dictionary.md) Используйте настраиваемые словари для поддержки терминов или языков, определенных для организации и политик.

### <a name="classifiers"></a>Классификаторы

Проверка встроенных обучаемых и глобальных классификаторов, отправленных или полученных сообщений по всем каналам связи в вашей организации, для различных типов проблем соответствия требованиям. Классификаторы используют сочетание искусственного интеллекта и ключевых слов для выявления в сообщениях лексики, которая может нарушать политику борьбы с оскорблениями. Встроенные классификаторы в настоящее время поддерживают идентификацию ключевого слова сообщения на нескольких языках:

- Китайский (упрощенное письмо)
- Английский
- французский;
- немецкий;
- итальянский;
- японский;
- Португальский
- Испанский

Встроенные в обучаемые и глобальные классификаторы соответствия требованиям связи сканируют коммуникации для терминов, изображений и настроений для следующих типов языка и контента:

- **Threat**: Scans for threats to commit violence or physical harm to a person or property.
- **Целевое домогательство:** сканирование для оскорбительных действий, нацеленных на людей, касающееся расы, цвета, религии, национального происхождения.
- **Ненормативная** лексика: сканирует ненормативную лексику, которая смущает большинство людей.
- **Изображения для взрослых.** Сканирует изображения, которые имеют сексуальный характер.
- **Racy images:** Scans for images that are sexually suggestive in nature, but contain less explicit content than images deemed Adult.
- **Gory images:** Scans for images that depict violence and gore.

*Классификаторы* изображений Adult, *Racy* и *Gory* сканируют файлы в форматах .jpeg, .png, .gif и .bmp. Размер файлов изображений должен быть не более 4 мегабайт(МБ), а размеры изображений должны быть больше 50x50 пикселей и более 50 килобайт (КБ), чтобы изображение можно было получить для оценки. Идентификация изображений поддерживается для Exchange Online сообщений электронной почты и Microsoft Teams каналов и чатов.

Встроенные классификаторы, которые можно обучить, и глобальные классификаторы не предоставляют исчерпывающий список терминов или изображений в этих областях. Кроме того, языковые и культурные стандарты постоянно изменяются, и в свете этих реалий Корпорация Майкрософт оставляет за собой право обновлять классификаторы по своему усмотрению. Хотя классификаторы могут помочь вашей организации в мониторинге этих областей, классификаторы не предназначены для предоставления единственным средствам мониторинга организации или обращения к таким языкам или изображениям. Ваша организация, а не Корпорация Майкрософт, по-прежнему несет ответственность за все решения, связанные с мониторингом, сканированием и блокировкой языка и изображений в этих областях, включая соблюдение местных законов о конфиденциальности и других применимых законов. Корпорация Майкрософт рекомендует консультироваться с адвокатом перед развертыванием и использованием.

> [!NOTE]
> Политики, использующие классификаторы, проверяют и оценивают сообщения с количеством слов шесть или больше. Сообщения, содержащие менее шести слов, не оцениваются в политиках с помощью классификаторов. Чтобы определить и принять меры по более коротким сообщениям, содержащим ненадлежащий контент, рекомендуется включить настраиваемый словарь ключевых слов в мониторинг политик соответствия требованиям связи для этого типа контента.

Сведения об обучаемых классификаторах в Microsoft 365 см. в руб. Начало работы с обучаемыми [классификаторами.](classifier-get-started-with.md)

### <a name="optical-character-recognition-ocr"></a>Распознавание текста (OCR)

Настройте встроенные или настраиваемые политики соответствия требованиям к коммуникациям, чтобы сканировать и идентифицировать печатный или рукописный текст из изображений, которые могут оказаться неуместными в организации. Интегрированные познавательные службы [Azure](/azure/cognitive-services/computer-vision/overview-ocr) и поддержка оптического сканирования для идентификации текста на изображениях помогают аналитикам и следователям обнаруживать и действовать в случаях, когда недопустимое поведение может быть пропущено в сообщениях, которые в основном не являются текстовыми.

Вы можете включить оптическое распознавание символов (OCR) в новых политиках из шаблонов, пользовательских политик или обновить существующие политики, чтобы расширить поддержку обработки встроенных изображений и вложений. При включенной политике, созданной из шаблона политики, автоматическое сканирование поддерживается для встроенных или присоединенных изображений в сообщениях электронной почты и Microsoft Teams чате. Для настраиваемых политик в политике необходимо настроить один или несколько условных параметров, связанных с ключевыми словами, встроенными классификаторами или конфиденциальными типами информации, чтобы включить выбор сканирования OCR.

Изображения от 50 до 4 МБ в следующих форматах изображений отсканированы и обрабатываются:

- .jpg/.jpeg (совместная группа экспертов по фотографии)
- .png (портативная сеть графики)
- .bmp (bitmap)
- .tiff (формат файла изображений тегов)
- .pdf (переносной формат документа)

> [!NOTE]
> Сканирование и извлечение встроенных и присоединенных .pdf в настоящее время поддерживается только для сообщений электронной почты.

При просмотре ожидающих оповещений для политик с включенной поддержкой OCR изображения, идентифицированные и совпадающие с условиями политики, отображаются в качестве детских элементов для связанных оповещений. Вы можете просмотреть исходное изображение для оценки идентифицированного текста в контексте исходного сообщения. Обнаружение обнаруженных изображений с оповещением может занять до 48 часов.

### <a name="conditional-settings"></a>Условные параметры
<a name="ConditionalSettings"> </a>

Условия, которые вы выбираете для политики, применяются к сообщениям как из электронной почты, так и из сторонних источников в вашей организации (например, из Instant Bloomberg).

В следующей таблице подробно рассказывается о каждом условии.

|**Condition**|**Как использовать это условие**|
|:-----|:-----|
| **Содержимое соответствует любому из этих классификаторов** | Применяются к политике, если в сообщение включены или исключены любые классификаторы. Некоторые классификаторы предварительно определены в клиенте, и настраиваемые классификаторы должны быть настроены отдельно, прежде чем они будут доступны для этого условия. Только один классификатор можно определить как условие в политике. Дополнительные сведения о настройке классификаторов см. в дополнительных сведениях об обучаемых [классификаторах (предварительный просмотр).](classifier-learn-about.md) |

Каждый тип конфиденциальной информации, который вы выбираете, применяется отдельно, и только один из этих типов конфиденциальной информации должен применяться для применения политики к сообщению. Дополнительные сведения о настраиваемом типе конфиденциальной информации см. в дополнительных сведениях [о типах конфиденциальной информации.](sensitive-information-type-learn-about.md) |
| **Сообщение получается из любого из этих доменов**  <br><br> **Сообщение не получается ни из одного из этих доменов** | Применить политику, чтобы включить или исключить определенные домены или адреса электронной почты в полученных сообщениях. Введите каждый домен или адрес электронной почты и разделять несколько доменов или адресов электронной почты с запятой. Каждый вложенный домен или электронный адрес применяется отдельно, для применения политики к сообщению должен применяться только один домен или адрес электронной почты. <br><br> Если вы хотите сканировать все сообщения электронной почты из определенного домена, но хотите исключить сообщения, которые  не нуждаются в просмотре (бюллетени, объявления и т. п.), необходимо настроить сообщение, не полученное из любого из этих доменов, исключаемое адрес электронной почты (пример "newsletter@contoso.com"). |
| **Сообщение отправляется в любой из этих доменов**  <br><br> **Сообщение не отправляется ни в один из этих доменов** | Применить политику, чтобы включить или исключить определенные домены или адреса электронной почты в отправленных сообщениях. Введите каждый домен или адрес электронной почты и разделять несколько доменов или адресов электронной почты с запятой. Каждый домен или адрес электронной почты применяется отдельно, для применения политики к сообщению должен применяться только один домен или адрес электронной почты. <br><br> Если вы хотите отсканировать всю электронную почту, отправленную в определенный домен, но хотите исключить отправленные сообщения, которые не требуют проверки, необходимо настроить два условия: <br> — **Сообщение отправляется в любое из этих** доменов, определяя домен ("contoso.com"), И <br> - **Сообщение не отправляется ни** в одно из этих доменов, которое исключает адрес электронной почты ("subscriptions@contoso.com"). |
| **Сообщение классифицируется с любой из этих меток**  <br><br> **Сообщение не классифицируется ни с одной из этих меток** | Применение политики при включении или исключении определенных меток хранения в сообщении. Метки хранения должны быть настроены отдельно, и в рамках этого условия выбраны метки. Каждая метка, которая вы выбираете, применяется отдельно (только одна из этих меток должна применяться для применения политики к сообщению). Дополнительные сведения о метки хранения см. в дополнительных сведениях о политиках [хранения и метки хранения.](retention.md)|
| **Сообщение содержит любое из этих слов**  <br><br> **Сообщение не содержит ни одного из этих слов** | Чтобы применить политику, если в сообщение включены или исключены определенные слова или фразы, введите каждое слово, разделенное запятой. Для фраз из двух слов или более используйте кавычка вокруг этой фразы. Каждое введите слово или фраза применяются отдельно (для применения политики к сообщению должно применяться только одно слово). Дополнительные сведения о вводе слов или фраз см. в следующем разделе [Поиск слов и фраз в сообщениях или вложениях](communication-compliance-feature-reference.md#Matchwords).|
| **Вложение содержит любое из этих слов**  <br><br> **Вложение не содержит ни одного из этих слов** | Чтобы применить политику, если определенные слова или фразы включены или исключены в вложение сообщения (например, документ Word), введите каждое слово, разделенное запятой. Для фраз из двух слов или более используйте кавычка вокруг этой фразы. Каждое введите слово или фраза применяются отдельно (для применения политики к вложению должно применяться только одно слово). Дополнительные сведения о вводе слов или фраз см. в следующем разделе [Поиск слов и фраз в сообщениях или вложениях](communication-compliance-feature-reference.md#Matchwords).|
| **Вложение — это любой из этих типов файлов**  <br><br> **Вложение не является одним из этих типов файлов** | Чтобы контролировать сообщения, которые включают или исключают определенные типы вложений, введите расширения файлов (например, .exe или .pdf). Если вы хотите включить или исключить несколько расширений файлов, введите их в отдельные строки. Для применения политики должно соответствовать только одно расширение вложений.|

#### <a name="matching-words-and-phrases-to-emails-or-attachments"></a>Поиск слов и фраз в сообщениях или вложениях
<a name="Matchwords"> </a>

Каждое слово, которое вы вводите и разделяете с запятой, применяется отдельно (только одно слово должно применяться для применения условия политики к электронной почте или вложению). Например, давайте использовать **условие,** сообщение содержит любое из этих слов, с ключевыми словами "банкир", "конфиденциальной" и "инсайдерской торговли", разделенных запятой (банкир, конфиденциально, "инсайдерской торговли"). Политика применяется к любым сообщениям, которые включают слово "банкир", "конфиденциальный" или фразу "инсайдерской торговли". Для применения этого условия политики достаточно наличия одного из этих слов или фраз. Слова в сообщении или вложении должны точно соответствовать вводу.

> [!IMPORTANT]
>
> При импорте настраиваемого файла словаря каждое слово или фраза должно быть разделено с возвратом вагона и отдельной строкой. Например:
>
> *банкир* <br>
> *конфиденциальность* <br>
> *инсайдерской торговли*

Чтобы сканировать сообщения электронной почты и вложения для тех [](create-a-keyword-dictionary.md) же ключевых слов, создайте политику предотвращения потери данных с помощью настраиваемого словаря ключевых слов для терминов, которые необходимо сканировать в сообщениях. [](create-test-tune-dlp-policy.md) Эта конфигурация политики определяет определенные ключевые слова, которые отображаются в сообщении электронной почты **или в** вложении электронной почты. Использование стандартных условных параметров *политики* (Сообщение содержит любое из этих слов, а вложение содержит любое из этих *слов)* для определения терминов в сообщениях и вложениях требует, чтобы термины присутствовали как в сообщении, так и в вложении. 

#### <a name="enter-multiple-conditions"></a>Ввод нескольких условий

Если вы вводите несколько условий, Microsoft 365 использует все условия вместе, чтобы определить, когда применять политику соответствия требованиям связи к средствам связи. При настрое нескольких условий необходимо соблюсти все условия для применения политики, если вы не введите исключение. Например, вам нужна политика, применяемая, если сообщение содержит слово "торговля" и превышает 2 МБ. Однако если в сообщении также содержатся слова "Approved by Contoso financial", политика не должна применяться. В этом примере три условия будут определены следующим образом:

- **Сообщение содержит любое из этих слов,** с ключевым словом "торговля"
- **Размер сообщения больше , со** значением 2 МБ
- **Сообщение не содержит ни одного из этих слов** с ключевыми словами "Утвержденный финансовой командой Contoso"

### <a name="review-percentage"></a>Процент отзывов

Если необходимо уменьшить количество контента для просмотра, можно указать процент от всех сообщений, управляющих политикой соответствия требованиям связи. Случайная выборка производится в реальном времени из общего процента содержимого, которое удовлетворяет выбранным условиям политики. Если проверяющие должны проверять все элементы, можно задать в политике соответствия требованиям к обмену данных процент **100%**.

## <a name="privacy"></a>Конфиденциальность

Защита конфиденциальности пользователей, которые имеют совпадения с политикой, имеет важное значение и может содействовать объективности при проверке данных и анализе для оповещений о соответствии требованиям. Этот параметр применяется только к именам пользователей, отображаемой в решении соответствия требованиям связи. Это не влияет на отображение имен в других решениях или центре администрирования.

Для пользователей с соответствием требованиям связи можно выбрать один из следующих параметров в параметрах соответствия **требованиям связи:**

- **Показать анонимизированные** версии имен пользователей. Имена пользователей  анонимизируются, чтобы пользователи в группе ролей аналитика соответствия требованиям связи не видели, кто связан с оповещениями о политике. Пользователи из группы *исследователей* соответствия требованиям связи всегда будут видеть имена пользователей, а не анонимные версии. Например, пользователь "Grace Taylor" появится со рандомизированным псевдонимом, таким как "AnonIS8-988" во всех областях взаимодействия. При выборе этого параметра все пользователи с текущими и прошлыми совпадениями политик будут анонимизированы и применяются ко всем политикам. Сведения о профиле пользователя в сведениях оповещений о соответствии требованиям связи не будут доступны, когда будет выбран этот параметр. Однако имена пользователей отображаются при добавлении новых пользователей к существующим политикам или при назначении пользователей новым политикам. Если вы решите отключить этот параметр, имена пользователей отображаются для всех пользователей, которые имеют текущие или прошлые совпадения политик.
- **Не показывать анонимизированные** версии имен пользователей: имена пользователей отображаются для всех текущих и прошлых совпадений политик для оповещений о соответствии требованиям связи. Сведения о профиле пользователя (имя, название, псевдоним и организация или отдел) отображаются для пользователя для всех оповещений о соответствии требованиям связи.

## <a name="notice-templates"></a>Шаблоны уведомлений

Вы можете создать шаблоны уведомлений, если вы хотите отправить пользователям уведомление о напоминаниях электронной почты для совпадений политик в рамках процесса разрешения проблем. Уведомления можно отправлять только на адрес электронной почты пользователя, связанный с соответствием политики, который вызвал определенное оповещение о исправлении. При выборе шаблона уведомлений для применения к нарушению политики в рамках процесса восстановления можно принять значения поля, определенные в шаблоне, или переписать поля по мере необходимости.

Шаблоны уведомлений — это настраиваемые шаблоны электронной почты, в которых можно определить следующие поля сообщений в области параметров **соответствия требованиям связи:**

|**Field**|**Required**| **Сведения** |
|:-----|:-----|:-----|
|**Название шаблона** | Да | Удобное имя шаблона уведомлений, который вы выберете в процессе уведомления во время восстановления, поддерживает текстовые символы. |
| **Адрес отправителя** | Да | Адрес одного или более пользователей или групп, отправив сообщение пользователю с совпадением политики, выбранный из Active Directory для подписки. |
| **Адреса CC и BCC** | Нет | Необязательные пользователи или группы, которые будут уведомлены о совпадении политик, выбранных из Active Directory для вашей подписки. |
| **Subject** | Да | Сведения, которые появляются в строке темы сообщения, поддерживают текстовые символы. |
| **Тело сообщения**. | Да | Сведения, которые появляются в теле сообщения, поддерживают текстовые или HTML-значения. |

### <a name="html-for-notices"></a>HTML для уведомлений

Если вы хотите создать не просто текстовое сообщение электронной почты для уведомлений, вы можете создать более подробное сообщение с помощью HTML в поле тела сообщений шаблона уведомлений. В следующем примере содержится формат тела сообщения для базового шаблона уведомлений электронной почты на основе HTML:

```HTML
<!DOCTYPE html>
<html>
    <body>
        <h2>Action Required: Contoso Employee Code of Conduct Policy Training</h2>
        <p>A recent message you've sent has generated a policy alert for the Contoso Employee <a href='https://www.contoso.com'>Code of Conduct Policy</a>.</p>
        <p>You are required to attend the Contoso Employee Code of Conduct <a href='https://www.contoso.com'>training</a> within the next 14 days. Please contact <a href='mailto:hr@contoso.com'>Human Resources</a> with any questions about this training request.</p>
        <p>Thank you,</p>
        <p><em>Human Resources</em></p>
    </body>
</html>
```

> [!NOTE]
> Реализация атрибута HTML href в шаблонах уведомлений о соответствии требованиям связи в настоящее время поддерживает только отдельные кавычка, а не двойные кавычка для ссылок на URL-адрес.

## <a name="filters"></a>Фильтры

Фильтры соответствия требованиям связи позволяют фильтровать и сортировать сообщения оповещения для более быстрого расследования и устранения последствий. Фильтрация доступна на вкладке **"Ожидание"** и **"Разрешено"** для каждой политики. Чтобы сохранить набор фильтра или фильтра в качестве сохраненного запроса фильтра, необходимо настроить одно или несколько значений в качестве выбора фильтра. В следующей таблице описаны сведения о фильтре:

|**Filter**|**Сведения**|
|:-----|:-----|
| **Date** | Дата, когда сообщение было отправлено или получено пользователем в организации. Чтобы фильтровать в течение одного дня, выберите диапазон дат, который начинается со дня, для чего нужны результаты, и завершится на следующий день. Например, если вы хотите фильтровать результаты для 9/20/2020, необходимо выбрать диапазон дат фильтрации 9/20/2020-9/21/2020.|
| **Класс File** | Класс сообщения, основанный на типе сообщения, *сообщении или* *вложении.* |
| **Имеет вложение** | Присутствие вложения в сообщении. |
| **Класс Item** | Источник сообщения на основе типа сообщения, электронной почты, чата Microsoft Team, Bloomberg и т.д. Дополнительные сведения об общих типах элементов и классах сообщений см. в статье [Типы элементов и классы сообщений.](/office/vba/outlook/concepts/forms/item-types-and-message-classes) |
| **Домены получателей** | Домен, в который было отправлено сообщение. Этот домен обычно является вашим Microsoft 365 по умолчанию. |
| **Получатель** | Пользователь, которому было отправлено сообщение. |
| **Sender** | Отправивший сообщение. |
| **Домен отправителя** | Домен, отправив сообщение. |
| **Size** | Размер сообщения в КБ. |
| **Subject/Title** | Тема сообщения или название чата. |
| **Tags** | Теги, присвоенные сообщению, сомнительный, *совместимый* или *не соответствующий требованиям.* |
| **Language** | Обнаруженный язык текста в сообщении. Сообщение классифицируется в соответствии с языком большинства текста сообщения. Например, для сообщения, содержащего немецкий и итальянский тексты, но большинство текстов — немецкий, сообщение классифицируется как немецкое (DE). Поддерживаются следующие языки: китайский (упрощенный - ZH), английский (EN), французский (FR), немецкий (DE), итальянский (ИТ), японский (JP), португальский (PT) и испанский (ES). Например, чтобы фильтровать сообщения, классифицируются как немецкие и итальянские, введите "DE,IT" (двухзначные языковые коды) в поле поиска языкового фильтра. Чтобы просмотреть обнаруженную языковую классификацию сообщения, выберите сообщение, выберите сведения о сообщении Просмотра и прокрутите в поле EmailDetectedLanguage. |
| **Escalated To** | Имя пользователя, включенного в действие эскалации сообщений. |
| **Классификаторы** | Имя встроенных и настраиваемые классификаторы, применимые к сообщению. Некоторые примеры включают *оскорбительный язык,* *целевые домогательства,* *ненормативную лексику,* *угрозы* и другие.

## <a name="alert-policies"></a>Политики оповещений

После настройки политики автоматически создается соответствующая политика оповещения и создаются оповещения для сообщений, соответствующих условиям, определенным в политике. По умолчанию все триггеры оповещения, соответствующие политике, назначены уровню серьезности среды в связанной политике оповещения. Оповещения создаются для политики соответствия требованиям к связи после выполнения порогового уровня триггера агрегации в связанной политике оповещения.

Для политик соответствия требованиям к коммуникациям по умолчанию настраиваются следующие значения политики оповещения:

|**Триггер политики оповещения**|**Значение, используемое по умолчанию**|
|:-----|:-----|
| Агрегация | Простое агрегирование |
| Порог | 4 действия |
| Окно | 60 минут |

> [!NOTE]

Вы можете изменить параметры по умолчанию для триггеров по числу действий, периоду  для действий и для определенных пользователей в политиках оповещения на странице Политики оповещения в Центре соответствия требованиям & безопасности.

### <a name="change-the-severity-level-for-an-alert-policy"></a>Изменение уровня серьезности для политики оповещения

Если вы хотите изменить уровень серьезности, замещенный политикой оповещения для определенной политики соответствия требованиям связи, выполните следующие действия:

2. В Центр соответствия требованиям Microsoft 365 перейдите к **политикам**.

3. Выберите **Office 365** оповещение на странице **Политики,** чтобы открыть страницу политики оповещения в центре Office 365 безопасности **& соответствия** требованиям. 

4. Выберите почтовый ящик для политики соответствия требованиям к коммуникациям, которая необходимо обновить, а затем выберите **политику Редактирование.**

## <a name="power-automate-flows"></a>Power Automate потоков

[Microsoft Power Automate](/power-automate/getting-started) — это служба рабочего процесса, которая автоматизирует действия в приложениях и службах. Используя потоки из шаблонов или созданных вручную, можно автоматизировать общие задачи, связанные с этими приложениями и службами. Когда вы включаете Power Automate для соответствия требованиям к связи, вы можете автоматизировать важные задачи для оповещений и пользователей. Вы можете настроить потоки Power Automate, чтобы уведомить руководителей о том, что у пользователей есть оповещение о соответствии требованиям связи и другие приложения.

Клиентам с Microsoft 365 подписками, которые включают соответствие требованиям к Power Automate связи, не требуются дополнительные Power Automate для использования рекомендуемых шаблонов Power Automate связи по умолчанию. Шаблон по умолчанию можно настроить, чтобы поддерживать организацию и охватывать основные сценарии соответствия требованиям. Если вы решите использовать функции Power Automate премиум-класса в этих шаблонах, создайте настраиваемый шаблон с помощью соединиттеля соответствия требованиям Microsoft 365 или используйте шаблоны Power Automate для других областей соответствия требованиям Microsoft 365, вам могут потребоваться дополнительные Power Automate лицензии.

> [!IMPORTANT]
> Вы получаете запросы на дополнительную проверку лицензии при тестировании Power Automate потоков? Возможно, ваша организация еще не получала обновлений служб для этой функции предварительного просмотра. Развертываются обновления, и все организации с Microsoft 365 подписками, которые включают соответствие требованиям связи, должны иметь поддержку лицензий для потоков, созданных из рекомендуемых шаблонов Power Automate до 30 октября 2020 г.

![Соответствие требованиям Power Automate](../media/communication-compliance-power-automate.png)

Ниже представлен Power Automate для поддержки автоматизации процессов для оповещения о соответствии требованиям к коммуникациям:

- **Уведомлять диспетчера о** том, что у пользователя есть оповещение о соответствии требованиям связи. Некоторые организации могут потребовать немедленного уведомления об управлении, если у пользователя есть оповещение о соответствии требованиям. Когда этот поток настроен и выбран, менеджеру для пользователя дела отправляется сообщение электронной почты со следующей информацией обо всех оповещениях:
  - Применимая политика для оповещений
  - Дата и время оповещения
  - Уровень серьезности оповещений

### <a name="create-a-power-automate-flow"></a>Создание потока Power Automate

Чтобы создать поток Power Automate из рекомендуемого шаблона по умолчанию, при работе непосредственно в оповещении используется параметр **Manage Power Automate потоки** из управления **Automate.** Чтобы создать поток Power Automate **управление** потоками Power Automate, необходимо быть членом по крайней мере одной группы ролей соответствия требованиям.

Выполните следующие действия, чтобы создать поток Power Automate из шаблона по умолчанию:

1. В Центр соответствия требованиям Microsoft 365 перейдите к **политике** соответствия требованиям связи и выберите политику с оповещением,  >   необходимое для проверки.
2. Из политики выберите вкладку **"Ожидание"** и выберите ожидающих оповещений.
3. Выберите **Power Automate** из меню действий оповещения.
4. На **Power Automate** выберите шаблон по умолчанию из **шаблонов** соответствия требованиям связи, которые могут понравиться в разделе на странице.
5. Поток будет перечислять встроенные подключения, необходимые для потока, и будет отображаться, если доступны состояния подключения. При необходимости обнови все подключения, которые не отображаются как доступные. Нажмите **Продолжить**.
6. По умолчанию рекомендуемые потоки предварительно настроены с рекомендуемой комплаенс-связью и Microsoft 365 полей данных службы, необходимых для выполнения назначенной задачи для потока. При необходимости настройте компоненты потока с помощью управления расширенными опциями **Show** и настроив доступные свойства для компонента потока.
7. При необходимости добавьте дополнительные действия в поток, выбрав кнопку **New step.** В большинстве случаев это изменение не требуется для рекомендуемых шаблонов по умолчанию.
8. Выберите **сохранить черновик,** чтобы сохранить поток для дальнейшей конфигурации позже, или выберите **Сохранить** для завершения конфигурации для потока.
9. Выберите **Close,** чтобы вернуться на страницу Power Automate потока. Новый шаблон будет указан в качестве  потока на вкладке Мои потоки и автоматически доступен из Power Automate для пользователя, создав поток при работе с оповещениями о соответствии требованиям связи.

### <a name="share-a-power-automate-flow"></a>Совместное Power Automate потока

По умолчанию Power Automate, созданные пользователем, доступны только этому пользователю. Чтобы другие пользователи связи получили доступ и использовали поток, поток должен быть общим для создателя потока. Чтобы поделиться потоком, вы будете использовать Power Automate **при** работе непосредственно в оповещении.

Чтобы поделиться потоком Power Automate, необходимо быть членом по крайней мере одной группы ролей соответствия требованиям связи.
Выполните следующие действия, чтобы поделиться потоком Power Automate:

1. В Центр соответствия требованиям Microsoft 365 перейдите к **политике** соответствия требованиям связи и выберите политику с оповещением,  >   необходимое для проверки.
2. Из политики выберите вкладку **"Ожидание"** и выберите ожидающих оповещений.
3. Выберите **Power Automate** из меню действий оповещения.
4. На странице **Power Automate потоки** выберите **вкладку Мои потоки** или **потоки команды.**
5. Выберите поток для обмена, а затем **выберите Share** из меню параметры потока.
6. На странице общего доступа к потоку введите имя пользователя или группы, которые необходимо добавить в качестве владельца потока.
7. В **диалоговом** окте подключения выберите **ОК,** чтобы подтвердить, что добавленный пользователь или группа будут иметь полный доступ к потоку.

Если требуется изменить поток, вы будете  использовать Power Automate при работе непосредственно в оповещении. Чтобы изменить Power Automate потока, необходимо быть членом по крайней мере одной группы ролей соответствия требованиям.

3. Выберите **Power Automate** из меню действий оповещения.
4. На странице **Power Automate потоков** выберите поток для редактирования. Выберите **Изменение** из меню управления потоком.
5. Выберите параметр **ellipsis** Параметры для изменения параметра компонента потока или удаления  >   **эллипсиса** для  >   удаления компонента потока.
6. Выберите **Сохранить и** **закрыть** для завершения редактирования потока.

## <a name="reports"></a>Отчеты

Новая панель **мониторинга отчетов** является центральным расположением для просмотра всех отчетов о соответствии требованиям связи. Виджеты отчетов предоставляют краткий обзор наиболее часто необходимых для общей оценки состояния действий по обеспечению соответствия требованиям связи. Сведения, содержащиеся в виджетах отчета, не экспортируются. Подробные отчеты предоставляют подробные сведения, связанные с определенными областями соответствия требованиям связи, и предоставляют возможность фильтрации, групповой сортировки и экспорта информации при просмотре.

## <a name="audit"></a>Аудит

В некоторых случаях для проверки надзора за действиями пользователей и коммуникациями необходимо предоставить сведения аудиторам нормативных органов или аудиторам по соблюдению требований. Эти сведения могут быть сводными сведениями обо всех действиях, связанных с определенной организационной политикой или в любое время изменения политики соответствия требованиям. Политики соответствия требованиям к коммуникациям имеют встроенные маршруты аудита для полной готовности к внутренним или внешним аудитам. Подробные истории аудита всех действий по созданию, редактированию и удалите захватываются политиками связи для подтверждения процедур надзора.

> [!IMPORTANT]
> Аудит должен быть включен для организации до записи событий соответствия требованиям связи. Чтобы включить аудит, см. [в журнале Включить журнал аудита.](communication-compliance-configure.md#step-2-required-enable-the-audit-log) Если действия вызывают события, которые Microsoft 365 в журнале аудита, может потребоваться до 48 часов, прежде чем эти события можно будет просмотреть в политиках соответствия требованиям к связи.

Чтобы просмотреть действия по обновлению политики соответствия требованиям, **выберите** управление обновлениями экспортной политики на главной странице для любой политики. Для экспорта действий по  обновлению вам должны быть назначены роли *администратора* глобального администрирования или администратора соответствия требованиям связи. Это действие создает файл аудита в формате .csv, содержащий следующие сведения:

|**Field**|**Сведения**|
|:-----|:-----|
| **CreationDate** | Дата выполнения действия обновления в политике. |
| **UserIds** | Пользователь, который выполнял действия обновления в политике. |
| **Operations** | Операции обновления, выполняемые в политике. |
| **AuditData** | Это поле является основным источником данных для всех действий по обновлению политики. Все действия обновления записывают и разделяют запятые. |

Чтобы просмотреть действия по проверке  соответствия требованиям для политики, выберите управление действиями экспортной проверки на странице **Обзор** для определенной политики. Для экспорта действий по  обзору необходимо на назначены роли администратора глобального *администрирования* или администратора соответствия требованиям связи. Это действие создает файл аудита в формате .csv, содержащий следующие сведения:

|**Field**|**Сведения**|
|:-----|:-----|
| **CreationDate** | Дата выполнения действия проверки в политике. |
| **UserIds** | Пользователь, который выполнял действия проверки в политике. |
| **Operations** | Операции проверки, выполняемые в политике. |
| **AuditData** | Это поле является основным источником данных для всех действий по обзору политики. Все действия по проверке записывают и разделяют запятые. |

Вы также можете просматривать действия аудита в едином журнале аудита или с помощью [cmdlet Search-UnifiedAuditLog](/powershell/module/exchange/search-unifiedauditlog) PowerShell. Дополнительные данные о политиках хранения журналов аудита см. в журнале Управление политиками хранения [журналов аудита.](audit-log-retention-policies.md)

Например, в следующем примере возвращаются действия для всех действий надзорного надзора (политики и правила):

```PowerShell
Search-UnifiedAuditLog -StartDate $startDate -EndDate $endDate -RecordType AeD -Operations SupervisoryReviewTag
```

В этом примере возвращаются действия по обновлению для политик соответствия требованиям к коммуникациям:

```PowerShell
Search-UnifiedAuditLog -StartDate $startDate -EndDate $endDate -RecordType Discovery -Operations SupervisionPolicyCreated,SupervisionPolicyUpdated,SupervisionPolicyDeleted
```

В этом примере возвращаются действия, которые соответствуют текущим политикам соответствия требованиям связи:

```PowerShell
Search-UnifiedAuditLog -StartDate $startDate -EndDate $endDate -Operations SupervisionRuleMatch
```

Совпадения политик соответствия требованиям связи хранятся в почтовом ящике надзора для каждой политики. В некоторых случаях может потребоваться проверить размер почтового ящика надзора для политики, чтобы убедиться, что вы не приближались к текущему лимиту в 50 ГБ. Если предел почтовых ящиков достигнут, совпадения политик не будут захвачены, и вам потребуется создать новую политику (с одинаковыми настройками), чтобы продолжить захват совпадений для тех же действий.

Чтобы проверить размер почтового ящика надзора для политики, выполните следующие действия:

1. Используйте [командлет Подключение ExchangeOnline](/powershell/module/exchange/connect-exchangeonline) в модуле Exchange Online PowerShell V2 для подключения к Exchange Online PowerShell с помощью современной проверки подлинности.
2. Запустите следующую команду в PowerShell:

    ```PowerShell
    ForEach ($p in Get-SupervisoryReviewPolicyV2 | Sort-Object Name)
    {
       "<Name of your communication compliance policy>: " + $p.Name
       Get-MailboxStatistics $p.ReviewMailbox | ft ItemCount,TotalItemSize
    }
    ```

## <a name="transitioning-from-supervision-in-office-365"></a>Переход от надзора в Office 365

Организации, использующие политики надзора в Office 365, должны немедленно запланировать переход к политикам соответствия требованиям связи в Microsoft 365 и должны понимать эти важные моменты:

- Решение надзора в Office 365 полностью заменено решением соответствия требованиям связи в Microsoft 365. Мы рекомендуем создать новые политики соответствия требованиям связи, которые имеют те же параметры, что и существующие политики надзора, для использования новых улучшений в области расследования и исправлений.
- Сообщения, сохраненные в надзоре Office 365 соответствия политикам, не могут быть перемещены или разделены в соответствии с Microsoft 365.
- Для организаций с обоими решениями, используемыми бок о бок в процессе перехода, политики, используемые в каждом решении, должны иметь уникальные имена политик. В течение переходного периода группы и настраиваемые словари ключевых слов можно использовать в обоих решениях.

Сведения об отставке для наблюдения в Office 365 см. в [Microsoft 365 roadmap.](https://www.microsoft.com/microsoft-365/roadmap)
