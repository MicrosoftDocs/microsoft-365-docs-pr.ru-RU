# <a name="azure-active-directory-classification-and-sensitivity-labels-for-microsoft-365-groups"></a>Метки классификации и чувствительности Azure Active Directory для групп Microsoft 365

В этой статье обсуждаются классические классификации Azure Active Directory и метки чувствительности.

Метки конфиденциальности поддерживаются [этими службами.](./sensitivity-labels-teams-groups-sites.md#support-for-the-sensitivity-labels)

Полную информацию о метки конфиденциальности см. в см. [в руберсе Сведения о метки конфиденциальности.](sensitivity-labels.md)

Дополнительные материалы о метках конфиденциальности и их поведении для сайтов и групп Microsoft 365 см. в разделЕ Использование меток конфиденциальности для защиты контента в [microsoft Teams, группах Microsoft 365 и сайтах SharePoint.](sensitivity-labels-teams-groups-sites.md)

См. следующие сценарии для наилучшей практики при переходе из классической классификации AAD в метки чувствительности.

## <a name="scenario-1-tenant-never-used-classic-aad-classifications-or-sensitivity-labels-for-documents-and-emails"></a>Сценарий 1. Клиент никогда не использовал классические классификации AAD или метки конфиденциальности для документов и электронных писем

- Администратор клиента включает метки конфиденциальности для групп, установив флаг клиента "EnableMIPLabels" с помощью комлета AAD powershell.
- Администратор клиента создает метки конфиденциальности в центре соответствия [требованиям Microsoft 365.](https://compliance.microsoft.com)
    - Администратор клиента может выбирать действия, связанные с файлами и электронной почтой, такие как шифрование и водяные знаки.
    - Администратор клиента может выбрать действия Microsoft 365 Groups и SharePoint Online, связанные с сайтом, для меток конфиденциальности.
- Администратор клиента публикует политику.
- **Совместимые рабочие нагрузки показывают** метки чувствительности. Используйте метки конфиденциальности для создания групп. Совместимые рабочие нагрузки — это службы, поддерживают метки конфиденциальности.
- **Не совместимые рабочие нагрузки** — это службы, которые еще не поддерживают метки конфиденциальности. Группы могут быть созданы, однако они не могут быть связаны с меткой конфиденциальности из-за не совместимых рабочих нагрузок. Чтобы связать такие группы с метами конфиденциальности, администраторы клиентов могут запускать группы PowerShell.

> [!NOTE]
> В случае настольного клиента Outlook (Win 32) после того, как администратор включает метки конфиденциальности для клиента, а их пользователь находится в более старой версии настольного клиента Outlook (Win 32):
>
> - Пользователь видит, как метки чувствительности отображаются в старой версии настольного клиента Outlook.
> - Однако, когда пользователь редактирует группу и сохраняет группу с меткой конфиденциальности, выбранный параметр конфиденциальности переопределяется параметром конфиденциальности примененной метки конфиденциальности.
>
> Рекомендуется пользователям старой версии обновления клиента Outlook до более новой версии.

## <a name="scenario-2-tenant-is-already-using-classic-aad-classifications"></a>Сценарий 2. Клиент уже использует классические классификации [AAD](../enterprise/manage-microsoft-365-groups-with-powershell.md)

### <a name="case-a-tenant-never-used-sensitivity-labels-for-documents-and-emails"></a>Случай A. Клиент никогда не использовал метки конфиденциальности для документов и электронных писем

1. В центре [соответствия требованиям Microsoft 365](https://compliance.microsoft.com)рекомендуется создавать метки конфиденциальности с тем же именем, что и существующие классические метки Azure AD.
2. Чтобы применить эти метки чувствительности к существующим группам Microsoft 365 и сайтам SharePoint с помощью сопоставления имен, используйте для этого групп PowerShell.
3. Администратор может удалить классические метки Azure AD:
    - Совместимые рабочие нагрузки показывают, что с ними создаются метки и группы конфиденциальности.
    - Не совместимые рабочие нагрузки работают при создании групп, но к ним не прилагается метка конфиденциальности.
4. Администраторы могут запускать комлеты PowerShell для применения меток чувствительности к этим группам без меток.
    - Кроме того, администратор может сохранить классические метки Azure AD:
        - Совместимые рабочие нагрузки показывают эти метки чувствительности, и группы создаются вместе с ними. Совместимые рабочие нагрузки — это службы, поддерживают метки конфиденциальности.
        - Не совместимые рабочие нагрузки работают при создании групп и показывают классические метки Azure AD. Эти классические метки Azure AD присоединены к этим группам, созданным с не совместимыми рабочими нагрузками.
5. Мы настоятельно рекомендуем администраторам запускать группы PowerShell для применения меток чувствительности к этим группам с классическими меточками Azure AD.

### <a name="case-b-tenant-used-sensitivity-labels-for-documents-and-emails"></a>Case B. Клиент использовал метки конфиденциальности для документов и электронных писем

1. Как только администратор включает функцию метки конфиденциальности на клиенте, установив флаг клиента "EnableMIPLabels", появляются метки конфиденциальности документов и электронной почты в диалоговом окнах group/site/team, создав и редактировать диалоговое окно.
2. Администратор может использовать одинаковые метки конфиденциальности документов и электронной почты для обеспечения конфиденциальности и внешнего доступа пользователей к группе/сайту/группе, указав соответствующие параметры группы:
    1. В центре [соответствия требованиям Microsoft 365](https://compliance.microsoft.com)выберите вкладку **"Сайты и группы".**
    2. Изменение метки конфиденциальности документа или электронной почты.
