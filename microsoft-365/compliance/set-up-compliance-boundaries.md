# <a name="set-up-compliance-boundaries-for-ediscovery-investigations"></a>Настройка границ соответствия требованиям для расследований eDiscovery

Инструкции в этой статье можно применять при использовании core eDiscovery или Advanced eDiscovery для управления расследованиями.

Границы соответствия требованиям создают логические границы в организации, управляющие расположениями контента пользователей (например, почтовыми ящиками, учетной записью OneDrive и сайтами SharePoint), которые менеджеры по обнаружению электронных данных могут искать. Кроме того, границы соответствия управляют доступом к делам eDiscovery, используемым для управления юридическими, кадровым или другими расследованиями в организации. Необходимость в границах соответствия часто необходима для транснациональных корпораций, которые должны соблюдать географические границы и нормы, а также для государственных организаций, которые часто делятся на различные агентства. В Microsoft 365 границы соответствия требованиям помогают выполнять эти требования при выполнении поиска контента и управлении расследованиями по делам eDiscovery.
  
Мы используем пример на следующем рисунке, чтобы объяснить, как работают границы соответствия.
  
![Границы соответствия требованиям состоят из фильтров разрешений поиска, которые контролируют доступ к агентствам и группам ролей администраторов, которые контролируют доступ к делам eDiscovery](../media/M365_ComplianceBoundary_OrgChart_v2.png)
  
В этом примере Contoso LTD — это организация, состоящая из двух дочерних компаний, Fourth Coffee и Coho Winery. Компании требуется, чтобы агенты и следователи, которые могут искать только почтовые ящики Exchange, учетные записи OneDrive и сайты SharePoint в своем агентстве. Кроме того, руководители и следователи, управляющие eDiscovery, могут видеть дела eDiscovery только в своих агентствах, и они могут получать доступ только к тем делам, в которые они входит. Вот как границы соответствия требованиям соответствуют этим требованиям.
  
- Функция фильтрации разрешений поиска в поиске контента управляет расположениями контента, которые могут искать руководители и следователи eDiscovery. Это означает, что менеджеры по обнаружению электронных сообщений и следователи в агентстве Fourth Coffee могут искать контент только в филиале Fourth Coffee. Это же ограничение применяется к дочернему предприятию Coho Winery.

    Группы ролей контролируют, кто может видеть дела eDiscovery в Центре & соответствия требованиям. Это означает, что руководители и следователи, управляющие eDiscovery, могут видеть только дела eDiscovery в своем агентстве.

- Группы ролей также контролируют, кто может назначать участников делу eDiscovery. Это означает, что менеджеры по обнаружению электронных данным и следователи могут назначать участников только тем делам, в которые они сами являются.

## <a name="before-you-set-up-compliance-boundaries"></a>Перед настройками границ соответствия требованиям

Для успешной синхронизации атрибута Azure Active Directory (Azure AD) (на шаге 1) с учетной записью Пользователя в OneDrive (на шаге 2) необходимо выполнить следующие предварительные условия.

- Пользователям должна быть назначена лицензия Exchange Online и лицензия SharePoint Online.

- Размер почтовых ящиков пользователей должен быть не менее 10 МБ. Если почтовый ящик пользователя меньше 10 МБ, атрибут, используемый для определения ваших агентств, не будет синхронизироваться с учетной записью OneDrive пользователя.

- Границы соответствия и атрибуты, используемые для создания фильтров разрешений поиска, требуют синхронизации атрибутов Azure Active Directory (Azure AD) с почтовыми ящиками пользователей. Чтобы убедиться, что атрибуты, которые вы хотите использовать, синхронизированы, запустите в Exchange Online PowerShell cmdlet [Get-User.](https://docs.microsoft.com/powershell/module/exchange/get-user) В выходных данных этого cmdlet отображаются атрибуты Azure AD, синхронизированные с Exchange Online.

## <a name="step-1-identify-a-user-attribute-to-define-your-agencies"></a>Шаг 1. Определение атрибута пользователя для определения ваших агентств

Первым шагом является выбор атрибута Azure AD, который будет определять ваши агентства. Этот атрибут используется для создания фильтра разрешений поиска, который ограничивает диспетчер eDiscovery поиском только в расположениях контента пользователей, которым назначено определенное значение для этого атрибута. Например, предположим, что Contoso решает использовать атрибут **Department.** Значением этого атрибута для пользователей дочернего подразделения Fourth Coffee будет, а для пользователей дочернего подразделения  `FourthCoffee`  Coho Winery будет `CohoWinery` . В шаге 4 эта пара  `attribute:value`  (например, *Department:FourthCoffee)* используется для ограничения местоположений пользовательского контента, которые могут искать менеджеры по обнаружению электронных данных. 
  
Вот список атрибутов пользователя Azure AD, которые можно использовать для границ соответствия требованиям:
  
- Company

- CustomAttribute1 — CustomAttribute15

- Отдел

- Кабинет

- C (двух буквный код страны) <sup>*</sup>

  > [!NOTE]
  > <sup>*</sup>Этот атрибут сопоошел со свойством CountryOrRegion, которое возвращается с помощью запуска в Exchange Online PowerShell с помощью cmdlet **Get-User.** Он возвращает локализованное название страны, которое преобразуется из двухбуквового кода страны. Дополнительные сведения см. в описании параметра CountryOrRegion в справочной статье о cmdlet [Set-User.](https://docs.microsoft.com/powershell/module/exchange/set-user)

Несмотря на то что доступны дополнительные атрибуты пользователя, особенно для почтовых ящиков Exchange, перечисленные выше атрибуты являются единственными атрибутами, которые в настоящее время поддерживаются OneDrive.
  
## <a name="step-2-file-a-request-with-microsoft-support-to-synchronize-the-user-attribute-to-onedrive-accounts"></a>Шаг 2. Запрос в службу поддержки Майкрософт для синхронизации атрибута пользователя с учетными записями OneDrive

Далее необходимо подать запрос в службу поддержки Майкрософт для синхронизации атрибута Azure AD, выбранного на шаге 1, со всеми учетными записями OneDrive в организации. После синхронизации атрибут (и его значение), выбранные на шаге 1, будет сописан со скрытым управляемым свойством с именем `ComplianceAttribute` . Этот атрибут используется для создания фильтра разрешений поиска для OneDrive на шаге 4.
  
При отправке запроса в службу поддержки Майкрософт включаем следующие сведения:
  
- Доменное имя организации по умолчанию

- Имя атрибута Azure AD (с шага 1)

- Следующий заголовок или описание цели запроса на поддержку: "Включить синхронизацию OneDrive для бизнеса с Azure AD для фильтров безопасности соответствия требованиям". Это помогает маршрутить запрос команде инженеров по обнаружению электронных электронных данным, которая реализует запрос.

После внося технических изменений и синхронизации атрибута с OneDrive служба поддержки Майкрософт отправит вам номер сборки, в который было внося изменение, и примерную дату развертывания. Процесс развертывания обычно занимает 4–6 недель после отправки запроса в службу поддержки.
  
> [!IMPORTANT]
> Вы можете выполнить шаг 3–5 перед развертыванием этого изменения атрибута. Но поиск контента не возвращает документы из учетных записей OneDrive, указанных в фильтре разрешений поиска, пока синхронизация атрибутов не будет развернута.
  
## <a name="step-3-create-a-role-group-for-each-agency"></a>Шаг 3. Создание группы ролей для каждого агентства

Следующим шагом является создание групп ролей в Центре безопасности & соответствия требованиям, которые будут согласованы с вашими агентствами. Мы рекомендуем создать группу ролей, скопируя встроенную группу "Менеджеры по обнаружению электронных данным", добавив соответствующих членов и удалив роли, которые могут не подстраиться под ваши потребности. Дополнительные сведения о ролях, связанных с eDiscovery, см. в центре безопасности и соответствия требованиям [Office 365 &.](assign-ediscovery-permissions.md)
  
Чтобы создать группы ролей,  перейдите на страницу "Разрешения" в Центре безопасности & соответствия требованиям и создайте группу ролей для каждой группы в каждом агентстве, которая будет использовать границы соответствия и дела eDiscovery для управления расследованиями.
  
С помощью сценария границ соответствия требованиям Contoso необходимо создать четыре группы ролей и добавить в каждую из них соответствующих членов.
  
- Fourth Coffee eDiscovery Managers

- Fourth Coffee Investigators

- Менеджеры coho Winery eDiscovery

- Следователи Coho Winery
  
## <a name="step-4-create-a-search-permissions-filter-to-enforce-the-compliance-boundary"></a>Шаг 4. Создание фильтра разрешений поиска для применения границы соответствия требованиям

После создания групп ролей для каждого агентства необходимо создать фильтры разрешений поиска, которые связывают каждую группу ролей с конкретным агентством и определяют границу соответствия. Необходимо создать один фильтр разрешений поиска для каждого агентства. Дополнительные сведения о создании фильтров разрешений безопасности см. в подстроке "Настройка фильтрации разрешений [для поиска контента".](permissions-filtering-for-content-search.md)
  
Вот синтаксис, используемый для создания фильтра разрешений поиска, используемой для границ соответствия требованиям.

```powershell
New-ComplianceSecurityFilter -FilterName <name of filter> -Users <role groups> -Filters "Mailbox_<ComplianceAttribute>  -eq '<AttributeVale> '", "Site_<ComplianceAttribute>  -eq '<AttributeValue>' -or Site_Path -like '<SharePointURL>*'" -Action <Action >
```

Вот описание каждого параметра в команде:
  
- `FilterName`: указывает имя фильтра. Используйте имя, которое описывает или идентифицирует агентство, в котором используется фильтр.

- `Users`: указывает пользователей или группы, которые получают этот фильтр, применяемый к выполняемым ими действиям поиска контента. Для границ соответствия этому параметру заданы группы ролей (созданные на шаге 3) в агентство, для которое создается фильтр. Обратите внимание, что это параметр с несколькими значениями, поэтому можно включить одну или несколько групп ролей, разделенных запятой.

- `Filters`: указывает условия поиска для фильтра. Для границ соответствия требованиям необходимо определить следующие фильтры. Каждое из них применяется к расположению контента. 

    - `Mailbox`: указывает почтовые ящики, в которые могут искать группы ролей, определенные в  `Users` параметре. Для границ соответствия требованиям  *ComplianceAttribute*  — это тот же атрибут, который вы определили в шаге 1, а  *AttributeValue*  указывает агентство. Этот фильтр позволяет членам группы ролей искать только почтовые ящики в определенном агентстве; например, `"Mailbox_Department -eq 'FourthCoffee'"` . 

    - `Site`: указывает учетные записи OneDrive, которые могут искать группы ролей, определенные в `Users` параметре. Для фильтра OneDrive используйте фактическую  `ComplianceAttribute` строку. Это сопоизируется с тем же атрибутом, который был определен на шаге 1 и синхронизирован с учетными записями OneDrive в результате запроса на поддержку, отправленного на шаге 2; *AttributeValue*  указывает агентство. Этот фильтр позволяет членам группы ролей искать только учетные записи OneDrive в определенном агентстве; например,  `"Site_ComplianceAttribute -eq 'FourthCoffee'"` .

    - `Site_Path`: указывает сайты SharePoint, которые группы ролей, определенные в параметре, могут  `Users` искать. *SharePointURL* указывает сайты в агентстве, которые могут искать участники группы ролей. Пример: `"Site_Path -like 'https://contoso.sharepoint.com/sites/FourthCoffee*'"`. Обратите `Site` внимание, что `Site_Path` фильтры и фильтры соединены **оператором -or.**

     > [!NOTE]
     > Синтаксис `Filters` параметра включает список *фильтров.* Список фильтров — это фильтр, который включает фильтр почтовых ящиков и фильтр сайта, разделенный запятой. В предыдущем примере обратите внимание, что запятая разделяет Mailbox_ComplianceAttribute **и** **Site_ComplianceAttribute**: `-Filters "Mailbox_<ComplianceAttribute>  -eq '<AttributeVale> '", "Site_ComplianceAttribute  -eq '<AttributeValue>' -or Site_Path -like '<SharePointURL>*'"` . При обработке этого фильтра во время поиска контента создаются два фильтра разрешений поиска из списка фильтров: один фильтр почтовых ящиков и один фильтр сайта. Альтернативой использованию списка фильтров может быть создание двух отдельных фильтров разрешений поиска для каждого агентства: один фильтр разрешений поиска для атрибута почтового ящика и один фильтр для атрибутов сайта. В любом случае результаты будут одинаковыми. Использование списка фильтров или создание отдельных фильтров разрешений поиска — это вопрос предпочтения.

- `Action`: указывает тип действия поиска соответствия требованиям, к который применяется фильтр. Например, фильтр применяется только в том случае, если члены группы ролей, определенной в параметре, запускают  `-Action Search` `Users` поиск контента. В этом случае фильтр не будет применен при экспорте результатов поиска. Для границ соответствия требованиям используйте  `-Action All` фильтр, чтобы он применял ко всем действиям поиска. 

    Список действий поиска контента см. в разделе "New-ComplianceSecurityFilter" статьи "Настройка фильтрации разрешений для поиска [контента".](permissions-filtering-for-content-search.md#new-compliancesecurityfilter)

Вот примеры двух фильтров разрешений поиска, которые будут созданы для поддержки сценария границ соответствия требованиям Contoso. Оба этих примера включают список фильтров, разделенных запятой, в котором фильтры почтовых ящиков и сайтов включаются в один фильтр разрешений поиска и разделяются запятой.
  
### <a name="fourth-coffee"></a>Fourth Coffee

```powershell
New-ComplianceSecurityFilter -FilterName "Fourth Coffee Security Filter" -Users "Fourth Coffee eDiscovery Managers", "Fourth Coffee Investigators" -Filters "Mailbox_Department -eq 'FourthCoffee'", "Site_ComplianceAttribute -eq 'FourthCoffee' -or Site_Path -like 'https://contoso.sharepoint.com/sites/FourthCoffee*'" -Action ALL
```

### <a name="coho-winery"></a>Coho Winery

```powershell
New-ComplianceSecurityFilter -FilterName "Coho Winery Security Filter" -Users "Coho Winery eDiscovery Managers", "Coho Winery Investigators" -Filters "Mailbox_Department -eq 'CohoWinery'", "Site_ComplianceAttribute -eq 'CohoWinery' -or Site_Path -like 'https://contoso.sharepoint.com/sites/CohoWinery*'" -Action ALL
```

## <a name="step-5-create-an-ediscovery-case-for-intra-agency-investigations"></a>Шаг 5. Создание дела eDiscovery для расследований внутри агентства

Последний этап — создание основного дела eDiscovery или Advanced eDiscovery в Центре соответствия требованиям Microsoft 365, а затем добавление группы ролей, созданной на шаге 3, в качестве участника дела. Это приводит к двум важным характеристикам использования границ соответствия:
  
- Только члены группы ролей, добавленной в дело, смогут видеть и получать доступ к делу в Центре безопасности & соответствия требованиям. Например, если группа ролей Fourth Coffee Investigators является единственным участником дела, члены группы ролей Fourth Coffee eDiscovery Managers (или члены любой другой группы ролей) не смогут видеть дело и получать к нему доступ.

- Когда участник группы ролей, назначенной делу, выполняет поиск, связанный с делом, он сможет искать только расположения контента в своем агентстве (которое определяется фильтром разрешений поиска, созданным на шаге 4).)

Чтобы создать дело и назначить участников:

1. Перейдите на **страницу Core eDiscovery** или **Advanced eDiscovery** в Центре соответствия требованиям Microsoft 365 и создайте дело.

2. В списке случаев щелкните имя созданного дела.

3. На странице **"Управление этим случаем"** в **группе** ролей "Управление группами ролей" щелкните значок ![ ](../media/8ee52980-254b-440b-99a2-18d068de62d3.gif) "Добавить". 

    ![Добавление группы ролей в качестве участника дела eDiscovery](../media/f8b4b557-01b9-4388-85be-b5b5ab7c5629.png)
  
4. В списке групп ролей выберите одну из групп ролей, созданных на шаге 3, и нажмите кнопку **"Добавить".**

5. Нажмите **кнопку** **"Сохранить" во flyout "Управление этим** делом", чтобы сохранить изменение.

> [!NOTE]
При добавлении группы ролей в дело можно добавить только группы ролей, в которые вы входите.

## <a name="searching-and-exporting-content-in-multi-geo-environments"></a>Поиск и экспорт контента в средах с несколькими географическими средами

Фильтры разрешений поиска также могут управлять тем, куда перенацелилось содержимое для экспорта и в какой центр обработки данных можно искать при поиске в расположениях контента в среде SharePoint с несколькими [регионами.](https://go.microsoft.com/fwlink/?linkid=860840)
  
- **Экспорт результатов поиска:** Вы можете экспортировать результаты поиска из почтовых ящиков Exchange, сайтов SharePoint и учетных записей OneDrive из определенного центра обработки данных. Это означает, что вы можете указать расположение центра обработки данных, из который будут экспортироваться результаты поиска.

    Чтобы создать или изменить центр обработки данных, через который будет маршрутироваться экспорт, используйте параметр **Region** для **new-ComplianceSecurityFilter** или **Set-ComplianceSecurityFilter.**
  
    |**Значение параметра**|**Расположение центра обработки данных**|
    |:-----|:-----|
    |NAM  <br/> |Северная Америка (центр обработки данных находятся в США)  <br/> |
    |EUR  <br/> |Европа  <br/> |
    |APC  <br/> |Азиатско-Тихоокеанский регион  <br/> |
    |CAN <br/> |Канада|
    |||

- **Маршрут поиска контента:** Поиск контента на сайтах SharePoint и в учетных записях OneDrive можно переназначить во второй центр обработки данных. Это означает, что вы можете указать расположение центра обработки данных, в котором будут вестись поиски.

    Используйте одно из следующих значений для параметра **Region,** чтобы управлять расположением центра обработки данных, в который будет вестись поиск при поиске на сайтах SharePoint и в учетных записях OneDrive. 
  
    |**Значение параметра**|**Расположения маршрутов центра обработки данных для SharePoint**|
    |:-----|:-----|
    |NAM  <br/> |Россия  <br/> |
    |EUR  <br/> |Европа  <br/> |
    |APC  <br/> |Азиатско-Тихоокеанский регион  <br/> |
    |CAN  <br/> |Россия  <br/> |
    |AUS  <br/> |Азиатско-Тихоокеанский регион  <br/> |
    |KOR  <br/> |Центр обработки данных организации по умолчанию  <br/> |
    |GBR  <br/> |Европа  <br/> |
    |JPN  <br/> |Азиатско-Тихоокеанский регион  <br/> |
    |IND  <br/> |Азиатско-Тихоокеанский регион  <br/> |
    |LAM  <br/> |Россия  <br/> |
    |NOR  <br/> |Европа |
    |BRA  <br/> |Центр обработки данных в Северной Америке |
    |||

   Если не указать параметр **Region** для фильтра разрешений поиска, будет искаться основной регион SharePoint организации. Результаты поиска экспортируются в ближайший центр обработки данных.

   Чтобы упростить эту концепцию, параметр **Region** управляет центром обработки данных, используемым для поиска контента в SharePoint и OneDrive. Это не относится к поиску контента в Exchange, так как поиск контента Exchange не привязан географическим расположением центра обработки данных. Кроме того, то же значение параметра **Region** также может определять центр обработки данных, через который маршрутзируются экспорты. Часто это необходимо для управления перемещением данных между географическими досками.

> [!NOTE]
> Если используется Advanced eDiscovery, параметр **Region** не управляет областью, из которую экспортируются данные. Данные экспортируются из основного центра обработки данных организации. Кроме того, поиск контента в SharePoint и OneDrive не привязан географическим расположением центра обработки данных. Поиск ведется во всех центрах обработки данных. Дополнительные сведения о Advanced eDiscovery см. в обзоре решения [Advanced eDiscovery в Microsoft 365.](overview-ediscovery-20.md)

Вот примеры использования параметра **Region** при создании фильтров разрешений поиска для границ соответствия требованиям. Предполагается, что дочернее подразделение Fourth Coffee расположено в Северной Америке, а компания Coho Winery — в Европе. 
  
```powershell
New-ComplianceSecurityFilter -FilterName "Fourth Coffee Security Filter" -Users "Fourth Coffee eDiscovery Managers", "Fourth Coffee Investigators" -Filters "Mailbox_Department -eq 'FourthCoffee'", "Site_Department -eq 'FourthCoffee' -or Site_Path -like 'https://contoso.sharepoint.com/sites/FourthCoffee*'" -Action ALL -Region NAM
```

```powershell
New-ComplianceSecurityFilter -FilterName "Coho Winery Security Filter" -Users "Coho Winery eDiscovery Managers", "Coho Winery Investigators" -Filters "Mailbox_Department -eq 'CohoWinery'", "Site_Department -eq 'CohoWinery' -or Site_Path -like 'https://contoso.sharepoint.com/sites/CohoWinery*'" -Action ALL -Region EUR
```

При поиске и экспорте контента в средах с несколькими географическими средами следует иметь в виду следующее.
  
- Параметр **Регион** не управляет поиском в почтовых ящиках Exchange. При поиске в почтовых ящиках поиск будет происходить во всех центрах обработки данных. Чтобы ограничить область поиска в почтовых ящиках Exchange, используйте параметр **Filters** при создании или изменении фильтра разрешений поиска. 

- Если менеджеру по обнаружению электронных данных необходимо искать данные в нескольких регионах SharePoint, необходимо создать другую учетную запись пользователя, которая будет применяться в фильтре разрешений поиска, чтобы указать регион, в котором расположены сайты SharePoint или учетные записи OneDrive. Дополнительные сведения о настройке этого параметра см. в разделе "Поиск контента в среде SharePoint с несколькими географическими настройками" статьи ["Поиск контента".](content-search.md#searching-for-content-in-a-sharepoint-multi-geo-environment)

- При поиске контента в SharePoint и OneDrive параметр **Region** направляет поиск в основное или в конечное расположение, где менеджер по обнаружению электронных данных будет проводить исследования eDiscovery. Если диспетчер eDiscovery выполняет поиск на сайтах SharePoint и OneDrive за пределами области, указанной в фильтре разрешений поиска, результаты поиска не возвращаются.

- При экспорте результатов поиска контент из всех расположений контента (включая Exchange, Skype для бизнеса, SharePoint, OneDrive и другие службы, которые можно искать с помощью средства поиска контента) загружается в хранилище Azure в центр обработки данных, указанный параметром **Region.** Это помогает организациям обеспечить соответствие требованиям, не позволяя экспортировать контент через контролируемые границы. Если в фильтре разрешений поиска не указан регион, содержимое загружается в основной центр обработки данных организации.

- Чтобы добавить или изменить область, можно изменить существующий фильтр разрешений поиска, выменив следующую команду:

    ```powershell
    Set-ComplianceSecurityFilter -FilterName <Filter name>  -Region <Region>
    ```

## <a name="using-compliance-boundaries-for-sharepoint-hub-sites"></a>Использование границ соответствия для концентраторов SharePoint

[Сайты-концентраторы SharePoint](https://docs.microsoft.com/sharepoint/dev/features/hub-site/hub-site-overview) часто соответствуют тем же географическим или агентным границам, которые следуют за границами соответствия для eDiscovery. Это означает, что вы можете использовать свойство ИД сайта центрального сайта для создания границы соответствия. Для этого используйте в SharePoint Online PowerShell с помощью cmdlet [Get-SPOHubSite,](https://docs.microsoft.com/powershell/module/sharepoint-online/get-spohubsite#examples) чтобы получить SiteId для центрального сайта, а затем используйте это значение для свойства department ID для создания фильтра разрешений поиска.

Используйте следующий синтаксис для создания фильтра разрешений поиска для центрального сайта SharePoint:

```powershell
New-ComplianceSecurityFilter -FilterName <Filter Name> -Users <User or Group> -Filters "Site_Departmentid -eq '{SiteId of hub site}'" -Action ALL
```

Ниже приводится пример создания фильтра разрешений поиска для центрального сайта для агентства Coho Winery.

```powershell
New-ComplianceSecurityFilter -FilterName "Coho Winery Hub Site Security Filter" -Users "Coho Winery eDiscovery Managers", "Coho Winery Investigators" -Filters "Site_Departmentid -eq '44252d09-62c4-4913-9eb0-a2a8b8d7f863'" -Action ALL
```

## <a name="compliance-boundary-limitations"></a>Ограничения границ соответствия требованиям

При управлении делами eDiscovery и расследованиями, которые используют границы соответствия, следует помнить о следующих ограничениях.
  
- При создании и запуске поиска можно выбрать расположения контента, которые находятся за пределами вашего агентства. Однако из-за фильтра разрешений поиска контент из этих местоположений не включается в результаты поиска.

- Границы соответствия не применяются к случаям, когда применяется eDiscovery. Это означает, что менеджер по обнаружению электронных данных в одном агентстве может разместить пользователя в другом агентстве на удержание. Однако граница соответствия будет применяться, если диспетчер eDiscovery выполняет поиск в расположениях контента пользователя, помещенного на удержание. Это означает, что менеджер по обнаружению электронных данных не сможет искать в расположениях контента пользователя, даже если он мог разместить пользователя на удержании.

    Кроме того, статистика удержания будет применяться только к расположениям контента в агентстве.

- Фильтры разрешений поиска не применяются к общедоступным папок Exchange.

## <a name="more-information"></a>Дополнительная информация

- Если почтовый ящик не лицензирован или удален с возможностью его удаления, атрибуты Azure AD больше не будут синхронизироваться с почтовым ящиком. Если при удалении почтового ящика было удержание, содержимое, сохраненное в почтовом ящике, по-прежнему подвергается ограничению соответствия требованиям или фильтру разрешений поиска на основе времени последней синхронизации атрибутов Azure AD перед удалением почтового ящика. 

    Кроме того, синхронизация между почтовым ящиком пользователя и учетной записью OneDrive прекращается, если почтовый ящик не лицензирован или удален. Последнее значение атрибута соответствия для учетной записи OneDrive останется в силе.

- Атрибут соответствия синхронизируется из почтового ящика Exchange пользователя с учетной записью OneDrive каждые семь дней. Как было сказано ранее, такая синхронизация происходит, только если пользователю назначена лицензия На Exchange Online и SharePoint Online, а его почтовый ящик не менее 10 МБ.

- Если для почтового ящика пользователя и учетной записи OneDrive реализованы ограничения соответствия требованиям и разрешения на поиск, рекомендуется не удалять почтовый ящик пользователя, а не его учетную запись OneDrive. Другими словами, при удалении почтового ящика пользователя необходимо также удалить его учетную запись OneDrive.

- В ряде ситуаций (например, при возвращении сотрудника) у пользователя может быть две или более учетных записей OneDrive. В таких случаях синхронизируется только основная учетная запись OneDrive, связанная с пользователем в Azure AD.

- Границы соответствия требованиям и фильтры разрешений поиска зависят от атрибутов, которые помеяются на содержимом в Exchange, OneDrive и SharePoint, а также от последующей индексации этого содержимого. 

- Не рекомендуется использовать фильтры исключений (например, в фильтре разрешений поиска) для границы соответствия требованиям на основе `-not()` контента. Использование фильтра исключений может привести к непредвиденным результатам, если содержимое с недавно обновленными атрибутами не было индексироваться. 

## <a name="frequently-asked-questions"></a>Вопросы и ответы

**Кто может создавать фильтры разрешений поиска и управлять ими (New-ComplianceSecurityFilter и Set-ComplianceSecurityFilter)?**
  
Чтобы создавать, просматривать и изменять фильтры разрешений поиска, необходимо быть членом группы ролей "Управление организацией" в Центре безопасности & соответствия требованиям.
  
**Если диспетчеру eDiscovery назначено несколько групп ролей, охватывающих несколько агентств, как они могут искать контент в одном агентстве или другом?**
  
Менеджер по обнаружению электронных данных может добавлять в поисковый запрос параметры, ограничивающие поиск определенным агентством. Например, если в организации указано свойство **CustomAttribute10** для различения агентств, они могут в приложении к поисковому запросу искать почтовые ящики и учетные записи OneDrive в определенном  `CustomAttribute10:<value> AND Site_ComplianceAttribute:<value>` агентстве:
  
**Что произойдет, если значение атрибута, используемого в качестве атрибута соответствия в фильтре разрешений поиска, изменится?**
  
Фильтрация разрешений поиска занимает до трех дней, чтобы применить границу соответствия, если изменено значение атрибута, используемого в фильтре. Например, в сценарии Contoso предположим, что пользователь из агентства Fourth Coffee переведен в агентство Coho Winery. В результате значение атрибута **Department** в объекте пользователя меняется с *FourthCoffee* на *CohoWinery.* В этой ситуации fourth Coffee eDiscovery и инвестиции будут получать результаты поиска для этого пользователя в течение трех дней после изменения атрибута. Аналогичным образом, до того, как менеджеры и следователи Coho Winery eDiscovery получат результаты поиска для пользователя, требуется до трех дней.
  
**Может ли руководитель eDiscovery видеть контент из двух разных границ соответствия требованиям?**
  
Да, это можно сделать при поиске в почтовых ящиках Exchange, добавив диспетчера eDiscovery в группы ролей, которые имеют видимость для обоих агентств. Однако при поиске на сайтах SharePoint и в учетных записях OneDrive менеджер по обнаружению электронных данных может искать контент в разных границах соответствия, только если агентства находятся в одном регионе или географическом расположении. **Примечание.** Это ограничение для сайтов не применяется в Advanced eDiscovery, так как поиск контента в SharePoint и OneDrive не ограничен географическим расположением.
  
**Работают ли фильтры разрешений поиска для удержания дела eDiscovery, политик хранения Microsoft 365 или DLP?**
  
Нет, не в настоящее время.
  
**Если указать регион для управления экспортом контента, но у меня нет организации SharePoint в этом регионе, можно ли по-прежнему искать SharePoint?**
  
Если область, указанная в фильтре разрешений поиска, не существует в организации, будет искаться регион по умолчанию.
  
**Каково максимальное число фильтров разрешений поиска, которые можно создать в организации?**
  
Количество фильтров разрешений поиска, которые можно создать в организации, не ограничивается. Однако производительность поиска будет влиять на более чем 100 фильтров разрешений поиска. Чтобы максимальное количество фильтров разрешений поиска в организации было максимально небольшим, по возможности создавайте фильтры, которые объединяют правила для Exchange, SharePoint и OneDrive в один фильтр разрешений поиска.
