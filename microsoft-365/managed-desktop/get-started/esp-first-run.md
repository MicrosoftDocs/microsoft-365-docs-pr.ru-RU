# <a name="first-run-experience-with-autopilot-and-the-enrollment-status-page"></a>Интерфейс первого запуска с автоматическим пилотным развертыванием и страницей состояния регистрации

На рабочем столе Майкрософт используется как [Windows для автопилота](https://docs.microsoft.com/windows/deployment/windows-autopilot/windows-autopilot) , так и [страница состояния регистрации Microsoft Intune (ESP)](https://docs.microsoft.com/windows/deployment/windows-autopilot/enrollment-status) , чтобы обеспечить максимально возможный первый запуск для пользователей.

Страница "состояние регистрации" в настоящее время находится в общедоступной предварительной версии.

## <a name="initial-deployment"></a>Начальное развертывание

Чтобы обеспечить взаимодействие с ESP, необходимо зарегистрировать устройства в службе управления рабочими столами Майкрософт. Для получения дополнительных сведений о регистрации обратитесь к разделу [Регистрация новых устройств самостоятельно](../get-started/register-devices-self.md) или [по этапам регистрации устройств для партнеров](../get-started/register-devices-partner.md).

Когда ваши устройства будут зарегистрированы в службе, вы можете включить ESP для настольных устройств, управляемых Майкрософт, путем хранения билета поддержки через [портал администрирования](https://portal.azure.com/). Первоначально мы развернут конфигурацию ESP в тестовой группе, когда вы записываете билет. Он развертывается для других последующих групп развертывания (первый, быстрый и общий), каждые 24 часа. Чтобы приостановить развертывание, файл должен содержать операции с запросом на удержание.

Работа страницы состояния регистрации выполняется в три этапа. Дополнительные [сведения об отслеживании страницы состояния регистрации](https://docs.microsoft.com/mem/intune/enrollment/windows-enrollment-status#enrollment-status-page-tracking-information).

Работа осуществляется следующим образом:

1. Запускается автопилот, и пользователь вводит свои учетные данные.
2. Устройство открывает страницу состояния регистрации и проходит этапы подготовки устройств и настройки устройств. Третий шаг (Настройка учетной записи) в *настоящее время пропущен* в конфигурации рабочего стола, управляемой Майкрософт, так как пользователь ESP отключен. Устройство перезапускается.
3. После перезагрузки устройство открывает страницу входа в Windows с **другим пользователем**.
4. Пользователи снова вводят свои учетные данные и открывает Рабочий стол.

> [!NOTE]
> Приложения Win32 разворачиваются только во время ESP, если версия Windows 10 — 1903 или более поздняя.

Компьютер, управляемый Майкрософт, в настоящее время не поддерживает функцию "White Глове" в Windows автопилот.

## <a name="exceptions"></a>Exceptions

Если настройка, используемая настольным компьютером Майкрософт, не соответствует вашим потребностям, вы можете заархивировать запрос на исключение. Для этого просмотрите подробные сведения в разделе [Request a Exception](../service-description/customizing.md#request-an-exception). Ниже приведено несколько примеров типов исключений, которые могут потребоваться.

### <a name="autopilot-exception"></a>Исключение для автопилота

Вам может потребоваться запросить другой шаблон имени устройства. Однако вы не можете изменить режим развертывания, присоединиться к Azure как, параметры конфиденциальности или тип учетной записи пользователя.

### <a name="enrollment-status-page-exception"></a>Исключение страницы состояния регистрации

- Более длительное количество минут для параметра "показывать сообщение об ошибке, когда установка занимает больше указанного количества минут".
- Сообщение об ошибке, отображаемое
- Добавление или удаление приложений в "блочное устройство используется, пока эти приложения не будут установлены, если они назначены пользователю или устройству".

## <a name="required-applications"></a>Необходимые приложения

- Вы должны быть нацелены на приложения, которые проверяют, первыми, быстро и обширны *группы устройств* современного рабочего места. Приложения должны устанавливаться в контексте "System". Обязательно проверяйте с помощью ESP в тестовой группе, прежде чем назначать их всем группам.
- Для перезагрузки устройства не требуется никаких приложений. Рекомендуется установить для приложений значение "ничего не делать" при создании пакета приложения, если потребуется перезагрузка.
- Ограничьте количество требуемых приложений только базовыми приложениями, которые пользователь должен выполнить сразу же после входа на устройство.
- Храните общий размер всех приложений в течение 1 ГБ, чтобы избежать превышения времени ожидания на этапе установки приложения.
- В идеале приложения не должны иметь зависимостей. Если у вас есть приложения, которые *должны* иметь зависимости, убедитесь, что они настроены, проверены и проверены в рамках оценки ESP.
- Приложения, для которых не требуется контекст "пользователь" (например, Teams), не могут быть включены в общедоступную предварительную версию ESP.
