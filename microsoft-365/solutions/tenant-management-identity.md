# <a name="step-3-identity-for-your-microsoft-365-for-enterprise-tenants"></a>Этап 3. Удостоверение для Microsoft 365 для корпоративных клиентов

Ваш Microsoft 365 включает клиента Azure Active Directory Azure AD для управления удостоверениями и проверкой подлинности для входных входов. Правильная настройка инфраструктуры удостоверений имеет жизненно важное значение для управления доступом Microsoft 365 пользователей и разрешениями для вашей организации.

## <a name="cloud-only-vs-hybrid"></a>Только для облака и гибрида

Вот два типа моделей удостоверений, наиболее подходящих и полезных для них.


| Модель | Описание | Проверка Microsoft 365 проверки подлинности учетных данных пользователей | Оптимально для. | Наибольшее преимущество |
|:-------|:-----|:-----|:-----|:-----|
| Только облако | Учетная запись пользователя существует только в клиенте Azure AD для Microsoft 365 клиента. | Клиент Azure AD для Microsoft 365 выполняет проверку подлинности с помощью учетной записи облачного удостоверения. | Организации, которые не имеют или нуждаются в локальной AD DS. | Простой в использовании. Дополнительные средства каталога или серверы не требуются. |
| Гибридная среда |  Учетная запись пользователя существует в локальной службе доменных служб Active Directory (AD DS), а копия также находится в клиенте Azure AD для Microsoft 365 клиента. Azure AD Подключение на локальном сервере для синхронизации изменений AD DS с клиентом Azure AD. Учетная запись пользователя в Azure AD также может включать версию с hashed паролем учетной записи пользователя AD DS. | Клиент Azure AD для Microsoft 365 или обрабатывает процесс проверки подлинности или перенаправляет пользователя другому поставщику удостоверений. | Организации, использующие AD DS или другого поставщика удостоверений. | Пользователи могут использовать те же учетные данные при доступе к локальному или облачному ресурсам. |
||||||

Вот основные компоненты идентификатора только для облака.

На этой иллюстрации локально и удаленные пользователи въехают в учетные записи в клиенте Azure AD Microsoft 365 клиента.

Вот основные компоненты гибридного удостоверения.

![Основные компоненты гибридной идентификации](../media/about-microsoft-365-identity/hybrid-identity.png)

На этой иллюстрации локально и удаленные пользователи впишутся в Microsoft 365 клиента с учетными записями в клиенте Azure AD, скопированные из локального AD DS.

## <a name="synchronizing-your-on-premises-ad-ds"></a>Синхронизация локальной AD DS

В зависимости от бизнес-потребностей и технических требований гибридная модель удостоверений и синхронизация каталогов является наиболее распространенным выбором для корпоративных клиентов, которые принимают Microsoft 365. Синхронизация каталогов позволяет управлять удостоверениями в AD DS, а все обновления учетных записей, групп и контактов пользователей синхронизируются с клиентом Azure AD вашего Microsoft 365 клиента.

>[!Note]
>Когда учетные записи пользователей AD DS синхронизируются впервые, они не получают автоматически Microsoft 365 лицензии и не могут получить доступ к Microsoft 365, например электронной почте. Сначала необходимо назначить им расположение использования. Затем назначьте лицензию этим учетным записям пользователей в индивидуальном или динамическом виде с помощью членства в группе.
>

Вот два типа проверки подлинности при использовании гибридной модели удостоверений.

| Тип проверки подлинности | Описание |
|:-------|:-----|
| Управляемая проверка подлинности. | Azure AD обрабатывает процесс проверки подлинности с помощью локально хранимой версии пароля или отправляет учетные данные локальному программному агенту для проверки подлинности локальной службой AD DS. <br> <br>  Существует два типа управляемой проверки подлинности: синхронизация хеш-кодов паролей (PHS) и сквозная проверка подлинности (PTA). С помощью PHS Azure AD выполняет проверку подлинности самостоятельно. С помощью PTA служба Azure AD имеет AD DS для проверки подлинности. |
| Федеративная проверка подлинности | Azure AD перенаправляет клиентский компьютер, запрашивающий проверку подлинности, другому поставщику удостоверений. |
|  |  |

Подробнее [см. в подборе](/azure/active-directory/hybrid/choose-ad-authn) правильного метода проверки подлинности.

## <a name="results-of-step-3"></a>Результаты шага 3

Для удостоверения Microsoft 365 клиента вы определили:

- Какую модель удостоверений использовать.
- Обеспечение сильного доступа к пользователю и устройству.

Вот пример клиента с выделенными новыми гибридными элементами удостоверений.

![Пример гибридной идентификации для клиента](../media/tenant-management-overview/tenant-management-tenant-build-step3.png)

На этой иллюстрации у клиента есть:

- Лес AD DS, который синхронизируется с клиентом Azure AD с помощью сервера DirSync и azure AD Подключение.
- Копия учетных записей пользователей AD DS и других объектов из леса AD DS.
- Набор политик условного доступа для обеспечения безопасности входов и доступа пользователей на основе учетной записи пользователя. 

## <a name="ongoing-maintenance-for-identity"></a>Текущее обслуживание удостоверений

На постоянной основе может потребоваться:

- Добавление или изменение учетных записей и групп пользователей. Для удостоверений, основанных только на облаке, вы поддерживаете свои облачные пользователи и группы с помощью средств Azure AD, таких как центр администрирования Microsoft 365 Или PowerShell. Для гибридной идентификации поддерживается локальное обслуживание пользователей и групп с помощью средств AD DS.
- Добавьте или измените конфигурацию доступа к удостоверениям и устройствам, чтобы обеспечить соблюдение требований безопасности входных данных.
