# <a name="protect-information-subject-to-data-privacy-regulation"></a>Защита данных в соответствии с нормами конфиденциальности данных

В вашей подписке можно использовать несколько элементов управления защитой информации для удовлетворения потребностей и нормативам, предъявляемым к конфиденциальности данных. К ним относятся общие сведения о регулировании безопасности данных (GDPR), HIPAA-HITECH (ACT о конфиденциальности в США), ACT для защиты потребителей в Калифорнии (ККПА) и ACT защиты данных в Бразилии (ЛГПД).

Эти элементы управления находятся в следующих областях решения:

- Метки конфиденциальности
- Предотвращение потери данных (DLP)
- Шифрование сообщений Office (OME)
- Элементы управления доступом для Teams и сайтов

![Основные службы для защиты персональных данных от соответствия нормам конфиденциальности данных](../media/information-protection-deploy-protect-information/information-protection-deploy-protect-information-grid.png)

>[!Note]
>Это решение описывает функции обеспечения безопасности и соответствия требованиям для защиты данных, подчиняются нормам конфиденциальности данных. Полный список функций безопасности в Microsoft 365 можно найти в [статье документация по безопасности microsoft 365](https://docs.microsoft.com/microsoft-365/security/). Полный список функций обеспечения соответствия требованиям в Microsoft 365 можно найти в [документации по соответствию майкрософт 365](https://docs.microsoft.com/microsoft-365/compliance/).
>

## <a name="data-privacy-regulations-that-impact-information-protection-controls"></a>Требования к конфиденциальности данных, влияющие на элементы управления защитой информации

Ниже приведен пример списка нормативных актов о конфиденциальности данных, которые могут относиться к средствам защиты информации:

- GDPR статья 5 (1) (f))
- Статья GDPR (32) (1) (a)
- ЛГПД статья 46
- HIPAA-HITECH (45 CFR 164.312 (e) (1))
- HIPAA-HITECH (45 К.Ф.Р. 164.312 (e) (2) (II))

Ознакомьтесь со [статьей Оценка конфиденциальности данных и определение конфиденциальных элементов](information-protection-deploy-assess.md) для получения дополнительной информации.

Требования к конфиденциальности данных для защиты информации рекомендуют:

- Защита от потери или несанкционированного доступа, использования и/или передачи.
- Применение защитных механизмов с учетом риска.
- Использование шифрования там, где это необходимо.

В организации также может потребоваться защита содержимого Microsoft 365 для других целей, таких как другие требования к соответствиям или по бизнес-причинам. Установка вашей схемы защиты информации для обеспечения конфиденциальности данных должна выполняться в рамках общего планирования, реализации и управления информационной защитой.

Чтобы начать работу с схемой защиты информации в Microsoft 365, приведенный ниже раздел содержит краткий список связанных возможностей и действий по улучшению для Microsoft 365. Список включает возможности и действия по улучшению, которые применяются к правилам конфиденциальности данных. Тем не менее, список не включает старые технологии, если существует более новая возможность, которая в значительной степени заменяет старую. Например, служба управления правами на доступ к данным (IRM) для SharePoint и OneDrive не включена в список, но включены метки чувствительности.

## <a name="sensitivity-labels"></a>Метки конфиденциальности

Метки конфиденциальности из Microsoft Information Protection Framework позволяют классифицировать и защищать данные вашей организации, не отменяя производительность пользователей и возможность сотрудничества.

![Метки конфиденциальности в Microsoft 365](../media/information-protection-deploy-protect-information/information-protection-deploy-protect-information-labels.png)

### <a name="prerequisites-for-sensitivity-labels"></a>Необходимые условия для меток конфиденциальности

Выполните эти действия перед реализацией любой из возможностей на основе меток конфиденциальности, выделенных ниже:

1. Ознакомьтесь со следующими сведениями:
   - **Бизнес-требования.** Определите бизнес-причины для применения меток конфиденциальности на предприятии. Например, ваши требования к конфиденциальности данных для защиты информации.
   - **Возможности меток конфиденциальности.** Метки чувствительности могут стать сложными, поэтому перед началом работы обязательно ознакомьтесь с [документацией по меткам конфиденциальности](../compliance/sensitivity-labels.md) .
   - **Основные вещи, которые следует запомнить** Метки конфиденциальности управляются в центре администрирования соответствия требованиям Майкрософт, но параметры нацеленности и приложений значительно отличаются.
      - Существуют метки конфиденциальности для сайтов, групп и Teams на уровне контейнера (параметры не применяются к содержимому в контейнере). Они публикуются для пользователей и групп, которые применяют их при подготовке сайта, группы или группы.
      - Для активного контента существуют метки конфиденциальности. Они также публикуются для пользователей или групп, которые либо вручную применяются, либо автоматически применяются в следующих случаях:
        - Файл открывается/редактируется или сохраняется на рабочем столе пользователя или на сайте SharePoint.
        - Черновик и отправка сообщения электронной почты.
      - Для автоматических приложений существуют метки чувствительности для автоматических приложений в SharePoint и OneDrive в дополнение к сообщениям в транзите через Exchange. Они предназначены для всех сайтов или определенных, и автоматически применяются к файлам в других средах.

2. Рационализации текущей метки конфиденциальности с помощью прошлых или альтернативных методов

   - Azure Information Protection

      Для текущей схемы меток чувствительности может потребоваться согласование с любой существующей реализацией с подпиской на [Azure Information Protection](../compliance/sensitivity-labels.md#sensitivity-labels-and-azure-information-protection) .
   - OME

      Если вы планируете использовать современные метки чувствительности для защиты электронной почты и существующие методы шифрования электронной почты, такие как OME, они могут сосуществовать, но вам следует ознакомиться со сценариями, в которых их следует применять. Ознакомьтесь с разметкой Office 365, посвященной [шифрованию сообщений (OME)](#office-365-message-encryption-ome-new-capabilities), которая включает в себя таблицу, которая сравнивает современные средства защиты типов меток с учетом OME.

3. Запланируйте интеграцию в более обширную схему защиты информации. В верхней части сосуществования с OME текущие метки чувствительности могут использоваться вместе со всеми функциями, такими как Microsoft 365 предотвращение потери данных (DLP) и Microsoft Cloud App Security. Сведения о [метках конфиденциальности и Microsoft Cloud App Security](../compliance/sensitivity-labels.md#sensitivity-labels-and-microsoft-cloud-app-security) для достижения целей защиты данных, связанных с конфиденциальностью данных.

4. Разработайте классификацию меток конфиденциальности и управления схемой. Просмотр [таксономии данных и метки чувствительности](https://aka.ms/dataclassificationwhitepaper).

### <a name="general-guidance"></a>Общие рекомендации

1. **Определение схемы.** Прежде чем использовать технические возможности для применения меток и защиты, работайте в Организации, чтобы определить схему классификации. Возможно, у вас уже есть схема классификации, которая упрощает добавление персональных данных. 
2. **Начало работы.** Начните с определения числа и имен подписей, которые необходимо реализовать. Выполните эту операцию, не беспокоясь о том, какую технологию использовать и как будут применяться метки. Применяйте эту схему к глобально во всей Организации, включая данные, которые хранятся локально и в других облачных службах.
3. **Дополнительные рекомендации** При проектировании и внедрении политик, меток и условий учитывайте следующие рекомендации.

   - **Используйте существующую схему классификации (при наличии).** Во многих организациях уже используется классификация данных в некоторой форме. Тщательно оцените существующую схему меток и, если это возможно, используйте ее как есть. Использование знакомых меток, распознаваемых для конечных пользователей, будет выполнять внедрение.
   - **Пуск мал.** Количество меток, которые можно создать, практически не ограничено. Однако большое количество меток и вложенных меток может замедлить процесс принятия.
   - **Использование сценариев и вариантов использования.** Определите наиболее распространенные варианты использования в Организации и используйте сценарии, полученные из нормативов по конфиденциальности данных, на которые вы являетесь субъектом. Проверьте, правильно ли будет работать Указанная метка и конфигурация классификации.
   - **Запрашивать новые метки.** Требуется ли для каждого сценария или варианта использования новые метки или можно ли использовать уже имеющиеся элементы? Сохранение количества меток в минимальном объеме.
   - **Используйте вложенные метки для ключевых отделов.** Некоторые отделы будут иметь определенные потребности, требующие определенных меток. Определите эти метки как вложенные метки для существующей метки и попробуйте использовать политики, назначаемые группам пользователей, а не глобально.
   - **Рассмотрите политики с областью действия.** Политики, предназначенные для подмножества пользователей, не позволят перегружать метки. Политика с областью позволяет назначать конкретные метки ролей или подметок только сотрудникам, которые работают в определенном отделе. 
   - **Используйте осмысленные имена меток.** Старайтесь не использовать жаргон, стандарты или аббревиатуры в качестве имен меток. Попробуйте использовать имена, ресонате с конечным пользователем, чтобы оптимизировать внедрение. Вместо таких меток, как персональные, PCI, HIPAA, ЛБИ, МБИ и HBI, рекомендуется рассмотреть такие имена, как некоммерческие, общедоступные, общие, конфиденциальные и строго конфиденциальные.

### <a name="create-and-deploy-sensitivity-labels-for-sites-groups-and-teams"></a>Создание и развертывание меток конфиденциальности для сайтов, групп и рабочих групп

Когда вы создаете [метки конфиденциальности](../compliance/sensitivity-labels-teams-groups-sites.md) в центре соответствия требованиям Microsoft 365, теперь вы можете применять их к следующим контейнерам:

- Сайты Microsoft Teams
- Группы Microsoft 365 (ранее группы Office 365)
- Сайты SharePoint

Используйте следующие параметры метки, чтобы защитить содержимое этих контейнеров:

- Конфиденциальность (общедоступная или частная) сайты Teams, подключенных к группам Microsoft 365
- Доступ внешних пользователей
- Доступ с неуправляемых устройств

Для обеспечения конфиденциальности данных, чтобы предотвратить внешний общий доступ для контейнеров, которые будут использоваться для хранения контента с конфиденциальными личными данными, пометьте файлы, содержащие данные, как частные, и потребуются управляемые устройства.

### <a name="create-and-deploy-sensitivity-labels-for-content"></a>Создание и развертывание меток конфиденциальности для контента

Метки конфиденциальности, применяемые к файлам, позволяют шифровать их содержимое, подводя содержимое и определять другие элементы управления для контента приложений Office, включая Outlook и Office в Интернете.

Когда вы будете готовы приступить к защите данных Организации с помощью меток конфиденциальности:

1. **Создание меток.** Создайте и назовите метки конфиденциальности согласно таксономии классификации вашей организации для разных уровней конфиденциальности контента. Дополнительные сведения о разработке таксономии классификации приведены в [статье классификация данных и метка конфиденциальности таксономии](https://aka.ms/dataclassificationwhitepaper).
2. **Определение возможностей каждой метки.** Настройте параметры защиты, связанные с каждой меткой. Например, может потребоваться, чтобы содержимое с низким уровнем конфиденциальности (например, "Общая" надпись) было применено только к верхнему или нижнему колонтитулу, а более высокий уровень конфиденциальности (например, метка "конфиденциальная") должен иметь подложку и включить шифрование.
3. **Публикация меток.** После настройки меток конфиденциальности опубликуйте их с помощью политики меток. Определите, какие пользователи и группы должны иметь доступ к меткам и какие параметры политики использовать. Одну метку можно использовать повторно. Вы можете определить его один раз, а затем включить в несколько политик меток, назначенных разным пользователям.

После публикации меток конфиденциальности из центра соответствия требованиям Microsoft 365 они начинают отображаться в [приложениях Office](../compliance/sensitivity-labels-office-apps.md) для классификации и защиты контента, созданного или измененного пользователем.

![Процесс развертывания меток конфиденциальности в Microsoft 365](../media/information-protection-deploy-protect-information/information-protection-deploy-protect-information-label-flow.png)

Для обеспечения конфиденциальности данных вы вручную применяете метку конфиденциальности с шифрованием и другими правилами к электронной почте или содержимому, содержащему конфиденциальную личную информацию.

>[!Note]
>Метки конфиденциальности с включенным шифрованием, примененные к электронной почте, имеют перекрывающиеся функции с OME. [В статье безопасные сценарии электронной почты сравниваются с метками OME и чувствительностью](#secure-email-scenarios-comparison-with-ome-and-sensitivity-labels).

### <a name="client-side-auto-labeling-when-users-edit-documents-or-compose-emails"></a>Автоматическое добавление меток на стороне клиента при редактировании документов или составлении сообщений электронной почты

При создании метки чувствительности можно [автоматически назначать эту метку](../compliance/apply-sensitivity-label-automatically.md) содержимому, включая электронную почту, если она соответствует заданным условиям.

Возможность автоматически применять метки конфиденциальности к содержимому важна, потому что:

- не потребуется обучать пользователей работе с классификациями;
- вам не нужно будет рассчитывать на то, что пользователи правильно классифицируют весь контент;
- пользователям больше не нужно будет знать о ваших политиках — они могут сосредоточиться на своей работе;

Автомаркировка поддерживает добавление меток для пользователей, а также автоматическое применение метки. В обоих случаях пользователь решает, принять или отклонить метку, чтобы добиться правильного применения меток к содержимому.

Применение меток на стороне клиента сопряжено с минимальными задержками для документов, поскольку метку можно применить еще до сохранения документа. Тем не менее, не все клиентские приложения поддерживают автоматическое применение меток. Эта возможность поддерживается клиентом единой метки Azure Information Protection и [некоторыми версиями приложений Office](../compliance/sensitivity-labels-office-apps.md#support-for-sensitivity-label-capabilities-in-apps).

Инструкции по настройке можно узнать [в статье Настройка автоматической метки для приложений Office](../compliance/sensitivity-labels-office-apps.md#support-for-sensitivity-label-capabilities-in-apps).

Для обеспечения конфиденциальности данных метки чувствительности автоматически применяются к контенту, содержащему конфиденциальную личную информацию.

### <a name="service-side-auto-labeling-when-content-is-already-saved"></a>Автоматическая маркировка на стороне службы, когда контент уже сохранен

Этот метод называется автоматической классификацией с метками конфиденциальности. Кроме того, вы также можете услышать, что она называется автоматическим добавлением меток для данных на других (для документов в SharePoint и OneDrive) и данными в транзитном расположении (для сообщений электронной почты, отправляемых или получаемых Exchange). Для Exchange он не включает сообщения электронной почты в почтовые ящики.
 
Так как эта метка применяется самой службой, а не приложением пользователя, нет необходимости беспокоиться о том, что у пользователей есть приложения и какая версия. Поэтому эта функция доступна сразу во всей организации и пригодна для крупномасштабного применения меток. Политики автоматического применения меток не поддерживают рекомендуемые метки, поскольку пользователь не взаимодействует с процессом применения меток. Вместо этого администратор запускает политики в режиме имитации, чтобы убедиться в правильности меток содержимого до фактического применения меток.

Инструкции по настройке: [Настройка политик автоматических меток для SharePoint, OneDrive и Exchange](../compliance/apply-sensitivity-label-automatically.md#how-to-configure-auto-labeling-policies-for-sharepoint-onedrive-and-exchange).

Для обеспечения конфиденциальности данных на сайтах следует отправлять метки чувствительности для автоматического шифрования содержимого, содержащего конфиденциальную личную информацию.

## <a name="data-loss-prevention"></a>Защита от потери данных 

С помощью [функции защиты от потери данных (DLP)](../compliance/data-loss-prevention-policies.md) в Microsoft 365 можно определять, предупреждать и блокировать рискованные, нежелательные или нежелательные возможности совместного доступа, такие как совместное использование данных с персональными и внешними сведениями.

DLP позволяет:

- Определите и следите за рискованными действиями общего доступа.
- Проучите пользователей с помощью контекстного руководства, чтобы принять правильные решения.
- Принудительно применять политики использования данных для контента без снижения производительности.
- Интеграция с классификацией и добавлением меток для обнаружения и защиты данных при совместной работе.

### <a name="supported-workloads-for-dlp"></a>Поддерживаемые рабочие нагрузки для защиты от потери данных

С помощью политики защиты от потери данных в центре соответствия требованиям Microsoft 365 вы можете определять, отслеживать и автоматически защищать конфиденциальные элементы во многих расположениях в Microsoft 365, таких как Exchange Online, SharePoint, OneDrive и Microsoft Teams.

Например, можно определить любой документ, содержащий номер кредитной карты, который хранится на любом сайте OneDrive, или можно отслеживать только сайты OneDrive для определенных пользователей.

Кроме того, вы можете отслеживать и защищать конфиденциальные элементы в локально установленных версиях Excel, PowerPoint и Word, которые включают возможность определять конфиденциальные элементы и применять политики защиты от потери данных. Защита от потери данных обеспечивает непрерывное отслеживание, когда пользователи совместно используют контент из этих приложений Office.

![Поддерживаемые рабочие нагрузки для защиты от потери данных](../media/information-protection-deploy-protect-information/information-protection-deploy-protect-information-supported-workloads.png)

На этом рисунке показан пример защиты персональных данных с использованием защиты от потери данных.

![Пример защиты персональных данных с помощью DLP](../media/information-protection-deploy-protect-information/information-protection-deploy-protect-information-dlp-example-use.png)

DLP используется для идентификации документа или сообщения электронной почты, содержащего запись о работоспособности, а затем автоматически блокирует доступ к этому документу или блокирует отправку сообщения электронной почты. Затем DLP уведомляет получателя с помощью подсказки политики и отправляет оповещение конечному пользователю и администратору.

### <a name="planning-for-dlp"></a>Планирование защиты от потери данных

Запланируйте политики защиты от потери данных для: 

- Бизнес-требований.

- Оценка организации на основе рисков, описанная в [статье Оценка рисков конфиденциальности данных и определение конфиденциальных элементов](information-protection-deploy-assess.md).

- Другие механизмы защиты и управления информацией на месте или при планировании конфиденциальности данных.

- Типы конфиденциальной информации, которые вы определили для персональных данных на основе работ по оценке, описанные в [статье Оценка рисков конфиденциальности данных и определение конфиденциальных элементов](information-protection-deploy-assess.md). Условия политики DLP могут основываться на типах конфиденциальной информации и метках хранения.

- Метки хранения необходимо указать условия защиты от потери данных. Для получения дополнительных сведений просмотрите [сведения, которые подчиняются нормам конфиденциальности данных в вашей организации](information-protection-deploy-govern.md) .

- Постоянное Управление политиками DLP, которое требует от пользователя в организации работы и настройки политик изменений в типах конфиденциальной информации, метках хранения, нормативах и политиках соответствия требованиям.

Несмотря на то, что метки конфиденциальности невозможно использовать в условиях политики DLP, некоторые сценарии защиты для предотвращения доступа могут быть достижимы с помощью меток конфиденциальности, которые можно автоматически применять на основе типов конфиденциальной информации. Если вы размещаете метки надежности, подумайте о том, следует ли использовать DLP для расширения защиты, так как:

  - Защита от потери данных может препятствовать совместному использованию файлов. Метки конфиденциальности могут просто запретить доступ.

  - Защита от потери данных имеет более детализированные уровни управления с точки зрения правил, условий и действий.

  - Политики защиты от потери данных можно применять к сообщениям чата в Teams и каналах. Метки конфиденциальности можно применять только к документам и электронной почте.

## <a name="office-365-message-encryption-ome-new-capabilities"></a>Новые возможности шифрования сообщений Office 365 (OME)

Пользователи часто используют электронную почту для обмена конфиденциальными элементами, например сведения о работоспособности пациента или сведения о клиентах и сотрудниках. Шифрование сообщений электронной почты гарантирует, что только предназначенные для получателей сообщения могут просматривать содержимое сообщения.

С помощью [OME](../compliance/ome.md)вы можете отправлять и получать зашифрованные сообщения между людьми в Организации и за ее пределами. OME работает с Outlook.com, Yahoo!, Gmail и другими почтовыми службами. OME гарантирует, что только предназначенные получатели могут просматривать содержимое сообщений.

Для обеспечения конфиденциальности данных можно использовать OME для защиты внутренних сообщений, содержащих конфиденциальные элементы. Служба шифрования сообщений Office 365 — это веб-служба, созданная на основе Microsoft Azure Rights Management (Azure RMS), которая является частью Azure Information Protection. К ним относятся политики шифрования, удостоверения и авторизации для защиты электронной почты. Сообщения можно шифровать с помощью шаблонов управления правами, параметра "не пересылать" и "только шифровать".

Вы также можете определить правила для обработки почтового процесса, чтобы применить эту защиту. Например, вы можете создать правило, которое требует шифрования всех сообщений, адресованных определенному получателю, или содержащих определенные слова ключевых слов в строке темы, а также указать, что получатели не могут копировать или печатать содержимое сообщения.

Кроме того, [Расширенное шифрование сообщений](../compliance/ome-advanced-message-encryption.md) OME помогает обеспечить соблюдение обязательств по обеспечению, требующих более гибких элементов управления внешним получателям и их доступ к зашифрованным сообщениям электронной почты. С помощью расширенного шифрования сообщений OME в Microsoft 365 вы можете управлять конфиденциальными сообщениями электронной почты, совместно используемыми в Организации с помощью автоматических политик, которые определяют типы конфиденциальной информации. 

Для обеспечения конфиденциальности данных при необходимости совместного использования электронной почты с внешней стороной можно указать дату истечения срока действия и отозвать сообщения. Вы можете отозвать и задать срок действия для сообщений, отправляемых внешним получателям.

| Внешние стороны <br> Безопасная передача данных и взаимодействие с любыми внешними и потребительными пользователями | Да — предварительно определить получателей в метке | Рекомендовать — JIT-защита на основе получателей |
