# <a name="microsoft-365-client-and-services-app-support"></a>Поддержка приложений для клиентов и служб Microsoft 365

Корпорация Майкрософт поддерживает широкий спектр функций безопасности, проверки подлинности и соответствия требованиям для обеспечения безопасности данных клиентов и позволяет ИТ-администраторам настраивать политики в центре администрирования Microsoft 365 для своих пользователей. Следующие функции — это всего лишь подмножество многих корпоративных функций, которые можно настроить в зависимости от подписки Microsoft 365.

## <a name="client-and-service-support"></a>Поддержка клиентов и служб

### <a name="continuous-access-evaluation-preview"></a>Оценка непрерывного доступа (предварительный просмотр)

Оценка непрерывного доступа реализуется путем включения служб, таких как Exchange Online, SharePoint Online и Teams, для подписки на критически важные события в Azure Active Directory, чтобы эти события могли быть оценены и реализованы в режиме реального времени. Оценка критически важных событий не зависит от политик условного доступа, поэтому доступна в любом клиенте.

В настоящее время оцениваются следующие события:

- Учетная запись пользователя удалена или отключена
- Пароль для пользователя изменен или сброшен
- Для пользователя включена многофакторная проверка подлинности
- Администратор явно отменяет все маркеры обновления для пользователя
- Повышенный риск пользователя, обнаруженный службой Azure AD Identity Protection

Дополнительные сведения о непрерывной оценке доступа для поддержки приложений для клиентов и служб см. в обзоре "Оценка непрерывного [доступа".](/azure/active-directory/conditional-access/concept-continuous-access-evaluation)

## <a name="client-support"></a>Поддержка клиента

### <a name="certificate-based-authentication"></a>Проверка подлинности на основе сертификатов

Проверка подлинности на основе сертификатов (CBA) — это использование цифрового сертификата для идентификации пользователя, машины или устройства перед предоставлением доступа к ресурсу, сети, приложению или службе. При проверке подлинности пользователей он часто развертывается в координации с традиционными методами, такими как имена пользователей и пароли.

Некоторые традиционные решения работают только для пользователей, таких как биометрия и одноразовые пароли (OTP). При проверке подлинности на основе сертификатов одно и то же решение можно использовать для всех конечных точек; пользователей, устройств и растущего Интернета вещей (IoT).

Дополнительные сведения о проверке подлинности на основе сертификатов для поддержки приложений для клиентов и служб см. в справке [Microsoft 365 Client App Support: Certificate-based Authentication.](microsoft-365-client-support-certificate-based-authentication.md)

### <a name="conditional-access"></a>Условный доступ

Условный доступ — это средство, используемое Azure Active Directory для слаживки сигналов, принятия решений и обеспечения политики организационного доступа. Условный доступ — это сердце новой модели управления, управляемой удостоверением.

Политики условного доступа — это утверждения if-then для предоставления доступа к ресурсам. Если пользователь хочет получить доступ к ресурсу, он должен выполнить действие. Общие сигналы, которые может использовать условный доступ при принятии решения о доступе к политике:

- Членство пользователя или группы
- Сведения о расположении IP
- Сведения об устройствах.
- Сведения о приложениях
- Обнаружение рисков в режиме реального времени и расчет
- Microsoft Cloud App Security (MCAS)

При принятии этих решений о доступе политики могут принимать различные действия:

- Политика может блокировать доступ: эта конфигурация является самым ограничительным действием и не позволяет пользователю получить доступ к ресурсу.
- Политика может предоставить доступ: эта конфигурация является менее ограничительным решением и может по-прежнему требовать один или несколько следующих вариантов:

    - Многофакторная проверка подлинности
    - Устройство, которое будет помечено как совместимый
    - Устройство является гибридным Azure AD, к нему присоединились
    - Утвержденное клиентские приложения
    - Политика защиты приложений, настроенная (предварительный просмотр)

Дополнительные сведения об условном доступе для поддержки приложений для клиентов и служб см. в этой информации:

- [Поддержка клиентских приложений Microsoft 365: условный доступ на основе устройств](microsoft-365-client-support-conditional-access.md)

### <a name="mobile-application-management"></a>управление мобильными приложениями;

Пользователи часто имеют доступ как к документам организации, так и к личным документам, электронной почте и данным с одного и того же мобильного устройства. Эти устройства часто принадлежат лично и должны быть настроены для защиты данных организации и личной конфиденциальности пользователя.

Когда пользователь получает доступ к данным организации, организация должна быть уверена в том, что политики организации, такие как политики конфигурации и политики защиты, применяются для защиты данных организации на устройстве. Кроме того, личный контент пользователя на устройстве должен оставаться вне контроля организации.

Для контента, управляемого организацией, можно применять политики управления приложениями для управления доступом к данным, общим доступом и использованием с помощью Microsoft Intune. Например, поддерживаются следующие действия:

- Удаленная стирка управляемого контента организации (также ссылаемая на данные орг.
- Предотвращение вклейки содержимого организации в расположения, не внося их в организацию
- Требуется ПИН-код для доступа к контенту организации
- Предотвращение запуска управляемых приложений на устройствах с побегом из тюрьмы или на корневых устройствах
- Предотвращение сэкономленного контента организации для неодобренных поставщиков облачных хранилища
- Предотвращение переноса неодобренного контента в управляемые приложения
- Разрешить доступ к контенту организации только после примененных политик
- Доставка конфигурации приложения для управления поведением и настройками приложения
- Ограничить управляемое приложение определенным удостоверением путем отключения возможностей нескольких удостоверений или личного использования

Дополнительные сведения об управлении мобильными приложениями в Microsoft Intune см. в приложении [Microsoft Intune?](/mem/intune/apps/app-management)

### <a name="multi-factor-authentication"></a>Многофакторная проверка подлинности

[Многофакторная проверка подлинности ( MFA) — это метод управления доступом к компьютеру, при котором пользователю предоставляется доступ только после успешного предоставления нескольких отдельных фрагментов доказательств механизму проверки подлинности. Этот метод обычно использует по крайней мере две из следующих категорий:

- Знания (что-то, что они знают)
- Владение (то, что у них есть)
- Негерентность (то, что они есть)

Дополнительные сведения о многофакторной проверке подлинности для поддержки приложений для клиентов и служб см. в [веб-сайте Microsoft 365 Client App Support: Multi-factor authentication.](microsoft-365-client-support-multi-factor-authentication.md)

### <a name="single-sign-on"></a>Единый вход

Один вход (SSO) добавляет безопасность и удобство при входе пользователей в приложения в Azure Active Directory. С одним входом пользователи один раз вступали с одной учетной записью, чтобы получить доступ к устройствам доменных служб Active Directory Domain Services (AD DS), программному обеспечению в качестве приложений службы (SaaS) и веб-приложениям в вашей организации.

Дополнительные сведения о едином входе для поддержки приложений для клиентов и служб см. в документе [Microsoft 365 Client App Support: Single sign-on.](microsoft-365-client-support-single-sign-on.md)

## <a name="services-support"></a>Поддержка служб

### <a name="modern-authentication"></a>Современная проверка подлинности

Современная проверка подлинности позволяет клиентам выполнять проверку подлинности в отношении Office 365, а администраторам клиентов — применять определенные требования к проверке подлинности в аренде Office 365, например:

- Поддержка многофакторной проверки подлинности для административного взаимодействия с арендой и службами, а также взаимодействия конечных пользователей с приложениями и их данными
- Условный доступ
- Вход сторонних поставщиков удостоверений на основе SAML
- Журнал Smartcard на персональных компьютерах
- Проверка подлинности на основе сертификатов на мобильных устройствах
- Больше не требуется передача учетных данных по базовой проверке подлинности.

Дополнительные сведения о поддержке современных служб проверки подлинности см. в дополнительных сведениях о [проверке подлинности и авторизации.](/azure/active-directory/develop/authentication-vs-authorization)

### <a name="azure-active-directory-conditional-access"></a>Условный доступ Azure Active Directory

Правила условного доступа Azure Active Directory (Azure AD) позволяют клиентам контролировать доступ к онлайн-службам на основе таких атрибутов, как соответствие требованиям устройства или расположение сети. Можно использовать следующие решения:

- Условный доступ на основе многофакторной проверки подлинности Azure AD
- Условный доступ на основе расположения Azure AD
- Условный доступ на основе устройств Azure AD

Правила условного доступа Azure AD применяются для каждого приложения и доступны клиентам для управления доступом в зависимости от различных условий. С помощью управления мобильными устройствами [(MDM) или Intune](/mem/intune/fundamentals/what-is-device-management)клиенты должны иметь возможность ограничивать доступ к Microsoft 365 только тем пользователям, которые используют устройство организации или зарегистрировали свое личное устройство для управления. Например, клиенты могут настроить правила условного доступа, чтобы обеспечить выполнение таких элементов управления, как:

- Разрешить доступ только с устройств, присоединенных к домену или совместимых с доменом.
- Обеспечение многофакторной проверки подлинности для всего доступа к службам Exchange Online

Дополнительные сведения о условном доступе Azure Active Directory см. в дополнительных сведениях [о условном доступе?](/azure/active-directory/conditional-access/overview)

### <a name="tls-12-support"></a>Поддержка TLS 1.2

Чтобы обеспечить наилучшее шифрование в своем классе для наших клиентов, Корпорация Майкрософт планирует прекратить поддержку версий безопасности транспортных слоев (TLS) 1.0 и 1.1 в Office 365 и Office 365 GCC.

Мы понимаем, что безопасность ваших данных важна, и мы стремимся к прозрачности изменений, которые могут повлиять на использование службы TLS. Мы рекомендуем всем комбинациям клиент-сервер и браузер-сервер использовать TLS 1.2 (или более поздний вариант) для поддержания подключения к службам Office 365. Возможно, придется обновить определенные сочетания клиент-сервер и браузер-сервер.

Дополнительные сведения о поддержке и поддержке служб TLS 1.2 см. в рублях Подготовка к [TLS 1.2 в Office 365 и Office 365 GCC.](../compliance/prepare-tls-1.2-in-office-365.md)
