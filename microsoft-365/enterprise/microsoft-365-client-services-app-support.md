# <a name="microsoft-365-client-and-services-app-support"></a>Поддержка клиентов и приложений служб Microsoft 365

Корпорация Майкрософт поддерживает широкий спектр функций безопасности, проверки подлинности и соответствия требованиям для обеспечения безопасности данных клиентов и позволяет ИТ-администраторам настраивать политики в Центре администрирования Microsoft 365 для своих пользователей. Следующие функции — это лишь часть многих корпоративных функций, которые можно настроить в зависимости от подписки на Microsoft 365.

## <a name="client-and-service-support"></a>Поддержка клиентов и служб

### <a name="continuous-access-evaluation-preview"></a>Оценка непрерывного доступа (предварительная версия)

Оценка непрерывного доступа реализуется путем включения служб, таких как Exchange Online, SharePoint Online и Teams, для подписки на критические события в Azure Active Directory, чтобы эти события можно было оценивать и применять практически в режиме реального времени. Оценка критических событий не зависит от политик условного доступа, поэтому доступна в любом клиенте.

В настоящее время оцениваются следующие события:

- Учетная запись пользователя удалена или отключена
- Пароль пользователя изменен или сброшен
- Для пользователя включена многофакторная проверка подлинности
- Администратор явным образом отзывает все маркеры обновления для пользователя
- Повышенный риск для пользователей, обнаруженный защитой идентификации Azure AD

Дополнительные сведения о оценке непрерывного доступа для поддержки клиентских приложений и приложений служб см. в предварительной [версии.](/azure/active-directory/conditional-access/concept-continuous-access-evaluation)

## <a name="client-support"></a>Поддержка клиента

### <a name="certificate-based-authentication"></a>Проверка подлинности на основе сертификатов

Проверка подлинности на основе сертификатов (CBA) — это использование цифрового сертификата для идентификации пользователя, компьютера или устройства перед предоставлением доступа к ресурсу, сети, приложению или службе. При проверке подлинности пользователей она часто развертывается в координации с помощью традиционных методов, таких как имена пользователей и пароли.

Некоторые традиционные решения работают только для пользователей, например биометрические данные и одноразовой пароль (OTP). При проверке подлинности на основе сертификатов одно и то же решение можно использовать для всех конечных точек; пользователи, устройства и растущий Интернет вещей (IoT).

Дополнительные сведения о проверке подлинности на основе сертификатов для поддержки клиентских приложений и приложений служб см. в справке по поддержке клиентских приложений [Microsoft 365:](microsoft-365-client-support-certificate-based-authentication.md)проверка подлинности на основе сертификатов.

### <a name="conditional-access"></a>Условный доступ

Условный доступ — это средство, используемое Azure Active Directory для совместной работы сигналов, принятия решений и применения политик организационного доступа. Условный доступ находится в основе новой модели управления на основе удостоверений.

Политики условного доступа — это утверждения if-then для предоставления доступа к ресурсам. Если пользователь хочет получить доступ к ресурсу, он должен выполнить действие. Распространенные сигналы, которые условный доступ может использовать при принятии решения о доступе к политике:

- Членство пользователей или групп
- Сведения о расположении IP-адреса
- Сведения об устройствах.
- Сведения о приложении
- Обнаружение рисков в режиме реального времени и вычисляемого риска
- Microsoft Cloud App Security (MCAS)

При принятии этих решений о доступе политики могут принимать различные действия:

- Политика может заблокировать доступ: эта конфигурация является наиболее строгим действием и не позволяет пользователю получить доступ к ресурсу.
- Политика может предоставить доступ: эта конфигурация является менее строгим решением и может потребовать одного или более из следующих вариантов:

    - Многофакторная проверка подлинности
    - Устройство, которое должно быть помечено как совместимее
    - Устройство присоединяется к гибридной службе Azure AD
    - Утвержденное клиентские приложения
    - Настроена политика защиты приложений (предварительная версия)

Дополнительные сведения об условном доступе для поддержки приложений для клиентов и служб см. в:

- [Поддержка клиентских приложений Microsoft 365: условный доступ на основе устройств](microsoft-365-client-support-conditional-access.md)

### <a name="mobile-application-management"></a>управление мобильными приложениями;

Пользователи часто имеют доступ как к личным документам организации, так и к электронной почте и данным с одного мобильного устройства. Эти устройства часто являются личными и должны быть настроены для защиты как данных организации, так и личной конфиденциальности пользователя.

Когда пользователь получает доступ к данным организации, организация должна быть уверены, что политики организации, такие как политики конфигурации и политики защиты, применяются для защиты данных организации на устройстве. Кроме того, личное содержимое пользователя на устройстве должно оставаться вне контроля организации.

Для контента, управляемого организацией, можно применять политики управления приложениями для управления доступом к данным, их общим доступом и использованием с помощью Microsoft Intune. Например, поддерживаются следующие действия:

- Удаленная стирка контента управляемой организации (также именуемой данными организации)
- Запретить в pasting organization content into non-organization locations
- Требовать ПИН-код для доступа к содержимому организации
- Запретить запуск управляемых приложений на устройствах со схватами или корневыми устройствами
- Запретить хранение контента организации неодобряемым поставщикам облачных хранилищ
- Предотвращение переноса неподтверченного содержимого в управляемые приложения
- Разрешить доступ к контенту организации только после примененных политик
- Доставка конфигурации приложения для управления его поведением и настройками
- Ограничение управляемого приложения определенным удостоверением путем отключения возможностей использования нескольких удостоверений или личного использования

Дополнительные сведения об управлении мобильными приложениями с помощью Microsoft Intune см. в руководстве по управлению приложениями [Microsoft Intune?](/mem/intune/apps/app-management)

### <a name="multi-factor-authentication"></a>Многофакторная проверка подлинности

[Многофакторная проверка подлинности (MFA) — это метод управления доступом к компьютеру, при котором пользователю предоставляется доступ только после успешного предоставления нескольких отдельных свидетельств механизму проверки подлинности. Этот метод обычно использует по крайней мере две из следующих категорий:

- Знание (что они знают)
- Владение (то, что у них есть)
- Постоясть (то, что они есть)

Дополнительные сведения о многофакторной проверке подлинности для поддержки клиентских приложений и приложений служб см. в подзадаче "Поддержка клиентских приложений [Microsoft 365: многофакторная проверка подлинности".](microsoft-365-client-support-multi-factor-authentication.md)

### <a name="single-sign-on"></a>Единый вход

Единый вход (SSO) добавляет безопасность и удобство при входе пользователей в приложения в Azure Active Directory. При едином входе пользователи вошел один раз с одной учетной записью для доступа к устройствам, присоединенным к домену доменных служб Active Directory (AD DS), приложениям SaaS и веб-приложениям в организации.

Дополнительные сведения об едином входе для поддержки клиентских приложений и приложений служб см. в документе "Поддержка клиентских приложений [Microsoft 365: единый вход".](microsoft-365-client-support-single-sign-on.md)

## <a name="services-support"></a>Поддержка служб

### <a name="modern-authentication"></a>Современная проверка подлинности

Современная проверка подлинности позволяет клиентам выполнять проверку подлинности в Office 365, а администраторам клиентов — применять определенные требования к проверке подлинности в клиенте Office 365, например:

- Поддержка многофакторной проверки подлинности для административного взаимодействия с клиентами и службами, а также взаимодействия конечных пользователей с приложениями и их данными
- условный доступ;
- Вход сторонного поставщика удостоверений на основе SAML
- Журнал смарт-карт на персональных компьютерах
- Проверка подлинности на основе сертификатов на мобильных устройствах
- Более не требуется передача учетных данных через базовую проверку подлинности.

Дополнительные сведения о поддержке современных служб проверки подлинности см. в сведениях о проверке [подлинности и авторизации.](/azure/active-directory/develop/authentication-vs-authorization)

### <a name="azure-active-directory-conditional-access"></a>Условный доступ Azure Active Directory

Правила условного доступа Azure Active Directory (Azure AD) позволяют клиентам управлять доступом к веб-службам на основе таких атрибутов, как соответствие устройств требованиям или сетевое расположение. Можно использовать следующие решения:

- Условный доступ на основе многофакторной проверки подлинности Azure AD
- Условный доступ на основе расположения Azure AD
- Условный доступ на основе устройств Azure AD

Правила условного доступа Azure AD применяются для каждого приложения и доступны клиентам для управления доступом на основе различных условий. С помощью управления мобильными устройствами [(MDM) или Intune](/mem/intune/fundamentals/what-is-device-management)клиенты должны иметь возможность ограничить доступ к Microsoft 365 только тем пользователям, которые используют устройство организации или зарегистрировали свое личное устройство для управления. Например, клиенты могут настроить правила условного доступа для применения таких элементов управления, как:

- Разрешить доступ только с устройств, присоединенных к домену или совместимых с доменом
- Обеспечение многофакторной проверки подлинности для всего доступа к службам Exchange Online

Дополнительные сведения об условном доступе Azure Active Directory см. в подстановок ["Что такое условный доступ"?](/azure/active-directory/conditional-access/overview)

### <a name="tls-12-support"></a>Поддержка TLS 1.2

Чтобы обеспечить наилучшее в своем классе шифрование для наших клиентов, корпорация Майкрософт планирует прекратить поддержку протоколов TLS версий 1.0 и 1.1 в Office 365 и Office 365 GCC.

Мы понимаем, что безопасность ваших данных важна, и мы стремимся к прозрачности изменений, которые могут повлиять на использование службы TLS. Для поддержания подключения к службам Office 365 рекомендуется, чтобы все сочетания клиент-сервер и браузер-сервер использовали TLS 1.2 (или более поздней версии). Возможно, придется обновить определенные сочетания клиент-сервер и браузер-сервер.

Дополнительные сведения о поддержке TLS 1.2 и службах см. в подготовии к [TLS 1.2 в Office 365 и Office 365 GCC.](/microsoft-365/compliance/prepare-tls-1.2-in-office-365)
