# <a name="migration-phases-actions-and-impacts-for-the-migration-from-microsoft-cloud-deutschland"></a>Миграция фазы действий и последствий для миграции из Microsoft Cloud Deutschland

Миграции клиента из Microsoft Cloud Deutschland (MCD) в регион "Германия" глобальных служб Microsoft Office 365 выполняются в наборе этапов и их настроенных действий для каждой рабочей нагрузки. На этом рисунке показаны десять этапов миграции в новые немецкие центра обработки данных.

[![Десять этапов миграции в новые центр обработки данных в Германии ](../media/ms-cloud-germany-migration-opt-in/migration-organization.png)](../media/ms-cloud-germany-migration-opt-in/migration-organization.png#lightbox)

Процесс миграции будет завершен в течение многих недель в зависимости от общего размера и сложности организации. В то время как миграция продолжается, пользователи и администраторы могут продолжать использовать службы с заметными изменениями, подробными в этой документации. График и таблица определяют этапы и действия во время миграции.

Этапы и их действия обеспечивают перенос критически важных данных и опыта в глобальные службы Office 365. После того как клиент будет добавлен в очередь миграции, каждая рабочая нагрузка будет выполнена в качестве набора действий, выполняемых в службе backend. Некоторые рабочие нагрузки могут требовать действий администратора (или пользователя), или миграция может повлиять на использование для этапов, которые выполняются и обсуждаются в How [is the migration organized?](ms-cloud-germany-transition.md#how-is-the-migration-organized)

В следующих разделах содержатся действия и эффекты для рабочих нагрузок по мере их выполнения на различных этапах миграции. Просмотрите таблицы и определите, какие действия или эффекты применимы к вашей организации. Убедитесь, что вы готовы выполнить необходимые действия на соответствующих этапах. Невыполнение необходимых действий может привести к сбою в работе служб и задержке завершения миграции в службы Office 365.

## <a name="phase-2-azure-ad-migration"></a>Этап 2. Миграция Azure AD
На этом этапе Azure Active Directory будет перенесен в новый регион центра обработки данных и активен. Старые конечные точки Azure AD будут по-прежнему доступны.

### <a name="exchange-online-hybrid---modify-authserver-on-premises"></a>Exchange Online Hybrid — изменение локального режима AuthServer
**Применяется к:** Все клиенты, использующие активную гибридную конфигурацию Exchange с локальной серверами Exchange

**При применении:** после окончания 2-й фазы

Локальное управление AuthServer должно указать на глобальную службу маркеров безопасности (STS) для проверки подлинности после завершения миграции Azure AD.
Это обеспечивает проверку подлинности запросов на проверку подлинности для запросов на доступность Exchange от пользователей в состоянии миграции, которые ориентированы на гибридную локальной среду, для доступа к локальной службе. Кроме того, это обеспечит проверку подлинности запросов из локальной точки в конечные точки глобальных служб Office 365. После завершения миграции Azure AD (этап 2) администратор локальной топологии Exchange (гибридная) должен добавить новую конечную точку службы проверки подлинности для глобальных служб Office 365. С помощью этой команды из Exchange PowerShell замените ИД клиента организации, найденный на портале `<TenantID>` Azure в Azure Active Directory.

```powershell
New-AuthServer GlobalMicrosoftSts -AuthMetadataUrl https://accounts.accesscontrol.windows.net/<TenantID>/metadata/json/1
```

Невыполнеть эту задачу может привести к невозможности предоставления сведений о пользователях почтовых ящиков, которые были перенесены из Microsoft Cloud Deutschland в службы Office 365.

## <a name="phase-5-exchange-online"></a>Этап 5: Exchange Online 
Начиная с 5-го этапа почтовые ящики Exchange Online перемещаются из Microsoft Cloud Deutschland в глобальные службы Office 365.

Область глобальных служб Office 365 установлена по умолчанию, что позволяет внутренней службе балансировки нагрузки перераспределять почтовые ящики в соответствующий регион по умолчанию в службах Office 365. В этом переходе пользователи с обеих сторон (MCD или Глобальные службы) находятся в одной организации и могут использовать конечную точку URL-адреса.

В организацию добавляется новый регион "Германия". Конфигурация Exchange Online добавляет новый локальный немецкий регион в организацию перехода.

- Переход пользователей и служб из старых URL-адресов MCD () в новые URL-адреса служб `https://outlook.office.de` Office 365 ( `https://outlook.office365.com` ).
-  На этом этапе будут доступны службы Exchange Online (Outlook Web Access и Exchange Admin Center) для нового немецкого центра обработки данных, которые ранее не будут доступны.
- Пользователи могут продолжать получать доступ к службе с помощью устаревших URL-адресов MCD во время миграции, однако после завершения миграции им необходимо прекратить использование устаревших URL-адресов.
- Пользователям следует перейти на использование всемирного портала Office для функций Office Online (Calendar, Mail, People). Навигация по службам, которые еще не перенесены в службы Office 365, не будет работать, пока они не будут перенесены. 
- Это ограничение распространяется и на фоновые службы, такие как "Моя учетная запись". Моя учетная запись для глобальных служб станет доступной после завершения 9-го этапа. До этого пользователи должны использовать портал MCD для управления настройками учетной записи.
- В Outlook Web App не будет предоставляться доступ к общедоступным папкам во время миграции.

Если вы хотите изменить фотографии пользователей на этапе 5, см. в обзоре [Exchange Online PowerShell — Set-UserPhoto на этапе 5](#exchange-online-powershell).

### <a name="dns-record-for-autodiscover-in-exchange-online"></a>Запись DNS для автообнаружия в Exchange Online
**Применяется к:** Клиенты, использующие Exchange Online с настраиваемой областью

Параметры DNS, управляемые клиентом для автообнаруженения, которые в настоящее время указывают на Microsoft Cloud Deutschland, необходимо обновить для ссылки на глобальную конечную точку Office 365 по завершении этапа Exchange Online (этап 5). <br> Существующие записи DNS с CNAME с autodiscover-outlook.office.de должны быть обновлены, чтобы указать на **autodiscover.outlook.com**.

Клиенты, которые не выполняют эти обновления DNS по завершении этапа **миграции 9,** могут испытывать проблемы с обслуживанием при завершении миграции.

> [!NOTE]

### <a name="exchange-online-powershell"></a>Exchange Online PowerShell
**Применяется к:** Администраторы Exchange Online с помощью Exchange Online PowerShell

На этапе миграции с помощью cmdlets PowerShell **New-MigrationEndpoint,** **Set-MigrationEndpoint** и **Test-MigrationsServerAvailability** могут привести к ошибкам (ошибка в прокси-сервере). Это происходит, когда почтовый ящик арбитража мигрирует во всем мире, но почтовый ящик администратора не имеет или наоборот. Чтобы устранить это, создав сеанс PowerShell клиента, используйте почтовый ящик арбитража в качестве подсказки маршрутивки **в ConnectionUri**. Например:

```powershell
New-PSSession 
    -ConfigurationName Microsoft.Exchange 
    -ConnectionUri "https://outlook.office365.com/powershell-liveid?email=Migration.8f3e7716-2011-43e4-96b1-aba62d229136@<tenant>.onmicrosoft.de"
    -Credential $UserCredential
    -Authentication Basic
    -AllowRedirection
```
Использование cmdlet **Set-UserPhoto** PowerShell приводит к ошибке, если почтовый ящик пользователя был перенесен, но почтовый ящик администратора не был перенесен, или наоборот. В этой ситуации администратор должен передать ID электронной почты пользователя, фотографию которого необходимо изменить при создании сеанса `ConnectionUri` PowerShell клиента: 
```powershell
-ConnectionUri "https://outlook.office.de/powershell-liveid?email=<user_email>" 
```
 где `<user_email>` является местообнамерщиком почтового ящика электронной почты. 

Дополнительные соображения:
- Пользователям Outlook Web App, которые имеют доступ к общему почтовому ящику в другой среде (например, пользователь в среде MCD получает доступ к общему почтовому ящику в глобальной среде), будет предложено проверить подлинность во второй раз. Пользователь должен сначала проверить подлинность и получить доступ к своему почтовому ящику, а затем открыть общий почтовый `outlook.office.de` ящик, который находится `outlook.office365.com` в . При доступе к общим ресурсам, которые находятся в другой службе, им потребуется второй раз проверить подлинность.
- Для существующих клиентов Microsoft Cloud Deutschland или для тех, кто находится в переходном периоде, когда общий почтовый ящик добавляется в Outlook с помощью Файла > **Info > Добавление** учетной записи, разрешения на просмотр календаря могут привести к сбою (клиент Outlook пытается использовать API `https://outlook.office.de/api/v2.0/Me/Calendars` Rest). Клиенты, которые хотят добавить учетную запись для просмотра разрешений календаря, могут добавить ключ реестра, как описано в изменениях пользовательского опыта для общего доступа к календарю в [Outlook,](https://support.microsoft.com/office/user-experience-changes-for-sharing-a-calendar-in-outlook-5978620a-fe6c-422a-93b2-8f80e488fdec) чтобы убедиться, что это действие будет успешным. Этот ключ реестра можно развернуть по всей организации с помощью групповой политики.
- Все клиенты, использующие активную гибридную конфигурацию Exchange, не могут перемещать почтовые ящики из локального Exchange Server в Exchange Online, ни в Microsoft Cloud Deutschland, ни в новый регион центра обработки данных в Германии. Клиенты должны убедиться, что текущие перемещения почтовых ящиков были завершены до этапа 5 и будут возобновлены после завершения этого этапа.
- Убедитесь, что все пользователи, использующие устаревшие протоколы (POP3/IMAP4/SMTP) для своих устройств, будут готовы изменить конечные точки в клиенте после переноса почтового ящика Exchange в новый немецкий центр обработки данных, как описано в действиях перед миграцией [для Exchange Online.](ms-cloud-germany-transition-add-pre-work.md#exchange-online)
- Планирование собраний Skype для бизнеса в Outlook Web App больше недоступна после переноса почтового ящика. При необходимости пользователям необходимо использовать Outlook.

Чтобы узнать больше о различиях для организаций в миграции и после переноса ресурсов Exchange Online, просмотрите сведения в клиентском опыте во время миграции в [службы Office 365](ms-cloud-germany-transition-experience.md)в новых немецких регионах центра обработки данных.

## <a name="phase-6-exchange-online-protection--security-and-compliance"></a>Этап 6. Exchange Online Protection / Security and Compliance

**Применяется к:** Все клиенты, использующие Exchange Online<br>

Функции back-end Exchange Online Protection (EOP) копируется в новом регионе "Германия". Exchange Online позволяет маршрутизать от внешних хостов до Office 365, а исторические сведения клиента переносят, что также включает в себя сторонние службы для функций безопасности и соответствия требованиям.

На данном этапе клиентам, использующим только возможности Exchange Online (Non-Hybrid), не нужно обращать внимание.

### <a name="exchange-online-hybrid-deployments"></a>Гибридные развертывания Exchange Online
**Применяется к:** Все клиенты, использующие активную гибридную конфигурацию Exchange с локальной серверами Exchange

Убедитесь, что [предварительная работа Exchange](ms-cloud-germany-transition-add-pre-work.md#exchange-online-hybrid-customers) была применена до начала этапа **5 этапа миграции.** Гибридные клиенты Exchange Online должны запустить последнюю версию мастера гибридной конфигурации Exchange (HCW) в режиме "Office 365 Germany" для подготовки локальной конфигурации для перехода на глобальные службы Office 365.

**Действия администратора:**
- Между началом этапа миграции 6 и завершением этапа миграции 9 (когда будет опубликовано уведомление Центра сообщений) необходимо снова запустить HCW с помощью параметров Office 365 Worldwide, чтобы указать свои локальной системы на глобальные службы Office 365. Невыполнение этой задачи до этапа 9 [Миграция завершена] может привести к NDRs для почты, на маршруте между локальной развертыванием Exchange и Office 365.
- Остановите или удалите любые перемещения почтовых ящиков с бортовой или оффборда, а именно не перемещайте почтовые ящики между локальной сетью Exchange и Exchange Online.  Это гарантирует, что запросы на перемещение почтовых ящиков не сбой с ошибкой. Невыполнение этого может привести к сбою службы или клиентов Office.
- Дополнительные Send-Connectors, созданные помимо соединителя, созданного HCW и нацеленных на Exchange Online, должны обновляться на этом этапе сразу после выполнения запуска HCW, в противном случае они перестанут работать. Домен TLS должен быть обновлен для этих соединителов отправки. <br> Чтобы обновить домен TLS, используйте следующую команду PowerShell в Exchange Server среде:
```powershell
Set-SendConnector -Identity <SendConnectorName> -TlsDomain "mail.protection.outlook.com"
```
