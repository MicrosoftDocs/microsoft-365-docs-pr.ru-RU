# <a name="microsoft-365-isolation-and-access-control-in-azure-active-directory"></a><span data-ttu-id="0940f-103">Изоляция и управление доступом в Microsoft 365 в Azure Active Directory</span><span class="sxs-lookup"><span data-stu-id="0940f-103">Microsoft 365 Isolation and Access Control in Azure Active Directory</span></span>

<span data-ttu-id="0940f-104">Azure Active Directory (Azure AD) была разработана для более надежного размещения нескольких клиентов с помощью логической изоляции данных.</span><span class="sxs-lookup"><span data-stu-id="0940f-104">Azure Active Directory (Azure AD) was designed to host multiple tenants in a highly secure way through logical data isolation.</span></span> <span data-ttu-id="0940f-105">Доступ к Azure AD осуществляется уровнем авторизации.</span><span class="sxs-lookup"><span data-stu-id="0940f-105">Access to Azure AD is gated by an authorization layer.</span></span> <span data-ttu-id="0940f-106">Azure AD изолирует клиентов, использующих контейнеры клиентов, как границы безопасности, чтобы защитить контент клиента, чтобы получить доступ к содержимому и получить к нему доступ с помощью совместных клиентов.</span><span class="sxs-lookup"><span data-stu-id="0940f-106">Azure AD isolates customers using tenant containers as security boundaries to safeguard a customer's content so that the content cannot be accessed or compromised by co-tenants.</span></span> <span data-ttu-id="0940f-107">Уровень авторизации Azure AD выполняется для трех проверок:</span><span class="sxs-lookup"><span data-stu-id="0940f-107">Three checks are performed by Azure AD's authorization layer:</span></span>

- <span data-ttu-id="0940f-108">Включен участник для доступа к клиенту Azure AD?</span><span class="sxs-lookup"><span data-stu-id="0940f-108">Is the principal enabled for access to Azure AD tenant?</span></span>
- <span data-ttu-id="0940f-109">Включен участник для доступа к данным в этом клиенте?</span><span class="sxs-lookup"><span data-stu-id="0940f-109">Is the principal enabled for access to data in this tenant?</span></span>
- <span data-ttu-id="0940f-110">Роль субъекта в этом клиенте авторизована для типа запрашиваемого доступа к данным?</span><span class="sxs-lookup"><span data-stu-id="0940f-110">Is the principal's role in this tenant authorized for the type of data access requested?</span></span>

<span data-ttu-id="0940f-111">Приложение, пользователь, сервер или служба не могут получить доступ к Azure AD без соответствующей проверки подлинности, маркера или сертификата.</span><span class="sxs-lookup"><span data-stu-id="0940f-111">No application, user, server, or service can access Azure AD without the proper authentication and token or certificate.</span></span> <span data-ttu-id="0940f-112">Запросы отклоняются, если они не сопровождаются правильными учетными данными.</span><span class="sxs-lookup"><span data-stu-id="0940f-112">Requests are rejected if they are not accompanied by proper credentials.</span></span>

<span data-ttu-id="0940f-113">Фактически, Azure AD размещает каждого клиента в собственном защищенном контейнере, с политиками и разрешениями, а также в контейнере исключительно владельцем и управлением клиентом.</span><span class="sxs-lookup"><span data-stu-id="0940f-113">Effectively, Azure AD hosts each tenant in its own protected container, with policies and permissions to and within the container solely owned and managed by the tenant.</span></span>
 
![Контейнер Azure](../media/office-365-isolation-azure-container.png)

<span data-ttu-id="0940f-115">Понятие контейнеров клиентов глубоко недетализировано в службе каталогов на всех уровнях, от всех порталов до постоянного хранилища.</span><span class="sxs-lookup"><span data-stu-id="0940f-115">The concept of tenant containers is deeply ingrained in the directory service at all layers, from portals all the way to persistent storage.</span></span> <span data-ttu-id="0940f-116">Даже если несколько метаданных клиента Azure AD хранятся на одном физическом диске, между контейнерами, отличными от того, что определено службой каталогов, не существует отношения, которое, в свою очередь, зависит администратором клиента.</span><span class="sxs-lookup"><span data-stu-id="0940f-116">Even when multiple Azure AD tenant metadata is stored on the same physical disk, there is no relationship between the containers other than what is defined by the directory service, which in turn is dictated by the tenant administrator.</span></span> <span data-ttu-id="0940f-117">Не существует прямого подключения к хранилищу Azure AD из запрашивающего приложения или службы без предварительного прохода по уровню авторизации.</span><span class="sxs-lookup"><span data-stu-id="0940f-117">There can be no direct connections to Azure AD storage from any requesting application or service without first going through the authorization layer.</span></span>

<span data-ttu-id="0940f-118">В приведенном ниже примере Contoso и Fabrikam имеют отдельные выделенные контейнеры, и даже несмотря на то, что эти контейнеры могут совместно использовать одну и ту же базовую инфраструктуру, например серверы и хранилища, они не отличаются друг от друга и разделяются уровнями авторизации и управления доступом.</span><span class="sxs-lookup"><span data-stu-id="0940f-118">In the example below, Contoso and Fabrikam both have separate, dedicated containers, and even though those containers may share some of the same underlying infrastructure, such as servers and storage, they remain separate and isolated from each other, and gated by layers of authorization and access control.</span></span>
 
![Выделенные контейнеры Azure](../media/office-365-isolation-azure-dedicated-containers.png)

<span data-ttu-id="0940f-120">Кроме того, не существует компонентов приложений, которые можно выполнить из Azure AD, и невозможно принудительно подсчитать целостность другого клиента, получить доступ к ключам шифрования другого клиента или прочитать необработанные данные с сервера.</span><span class="sxs-lookup"><span data-stu-id="0940f-120">In addition, there are no application components that can execute from within Azure AD, and it is not possible for one tenant to forcibly breach the integrity of another tenant, access encryption keys of another tenant, or read raw data from the server.</span></span>

<span data-ttu-id="0940f-121">По умолчанию Azure AD запрещает все операции, выданные удостоверениями в других клиентах.</span><span class="sxs-lookup"><span data-stu-id="0940f-121">By default, Azure AD disallows all operations issued by identities in other tenants.</span></span> <span data-ttu-id="0940f-122">Каждый клиент логически изолирован в Azure AD с помощью управления доступом на основе утверждений.</span><span class="sxs-lookup"><span data-stu-id="0940f-122">Each tenant is logically isolated within Azure AD through claims-based access controls.</span></span> <span data-ttu-id="0940f-123">Операции чтения и записи данных каталога ограничены контейнерами клиента и разделяются внутренним уровнем абстракции и уровнем управления доступом на основе ролей (RBAC), который совместно применяет клиент к границе безопасности.</span><span class="sxs-lookup"><span data-stu-id="0940f-123">Reads and writes of directory data are scoped to tenant containers, and gated by an internal abstraction layer and a role-based access control (RBAC) layer, which together enforce the tenant as the security boundary.</span></span> <span data-ttu-id="0940f-124">Каждый запрос на доступ к данным каталога обрабатывается этими слоями, и каждый запрос на доступ в Microsoft 365 наследует логику выше.</span><span class="sxs-lookup"><span data-stu-id="0940f-124">Every directory data access request is processed by these layers and every access request in Microsoft 365 is policed by the logic above.</span></span>

<span data-ttu-id="0940f-125">Azure AD — это Северная Америка, правительство США, Европейский союз, Европейский союз, Международный раздел.</span><span class="sxs-lookup"><span data-stu-id="0940f-125">Azure AD has North America, U.S. Government, European Union, Germany, and World Wide partitions.</span></span> <span data-ttu-id="0940f-126">Клиент существует в отдельном разделе, а разделы могут содержать несколько клиентов.</span><span class="sxs-lookup"><span data-stu-id="0940f-126">A tenant exists in a single partition, and partitions can contain multiple tenants.</span></span> <span data-ttu-id="0940f-127">Сведения о разделах отменяются от пользователей.</span><span class="sxs-lookup"><span data-stu-id="0940f-127">Partition information is abstracted away from users.</span></span> <span data-ttu-id="0940f-128">Данный раздел (включая все клиенты в нем) реплицируется в несколько центров обработки данных.</span><span class="sxs-lookup"><span data-stu-id="0940f-128">A given partition (including all the tenants within it) is replicated to multiple datacenters.</span></span> <span data-ttu-id="0940f-129">Раздел для клиента выбирается на основе свойств клиента (например, кода страны).</span><span class="sxs-lookup"><span data-stu-id="0940f-129">The partition for a tenant is chosen based on properties of the tenant (e.g., the country code).</span></span> <span data-ttu-id="0940f-130">Секреты и другие конфиденциальные сведения в каждом разделе шифруются с помощью выделенного ключа.</span><span class="sxs-lookup"><span data-stu-id="0940f-130">Secrets and other sensitive information in each partition is encrypted with a dedicated key.</span></span> <span data-ttu-id="0940f-131">Ключи создаются автоматически при создании нового раздела.</span><span class="sxs-lookup"><span data-stu-id="0940f-131">The keys are generated automatically when a new partition is created.</span></span>

<span data-ttu-id="0940f-132">Функциональные возможности системы Azure AD — это уникальный экземпляр для каждого сеанса пользователя.</span><span class="sxs-lookup"><span data-stu-id="0940f-132">Azure AD system functionalities are a unique instance to each user session.</span></span> <span data-ttu-id="0940f-133">Кроме того, Azure AD использует технологии шифрования для обеспечения изоляции общих системных ресурсов на уровне сети, чтобы предотвратить несанкционированные и нежелательные передачи информации.</span><span class="sxs-lookup"><span data-stu-id="0940f-133">In addition, Azure AD uses encryption technologies to provide isolation of shared system resources at the network level to prevent unauthorized and unintended transfer of information.</span></span>
