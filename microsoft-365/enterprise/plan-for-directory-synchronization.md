# <a name="hybrid-identity-and-directory-synchronization-for-microsoft-365"></a>Гибридная синхронизация удостоверений и каталогов для Microsoft 365

*Эта статья относится к Microsoft 365 корпоративный и Office 365 корпоративный.*

В зависимости от бизнес-потребностей и технических требований гибридная модель удостоверений и синхронизация каталогов является наиболее распространенным выбором для корпоративных клиентов, которые принимают Microsoft 365. Синхронизация каталогов позволяет управлять удостоверениями в службах домена Active Directory (AD DS), а все обновления учетных записей пользователей, групп и контактов синхронизируются с клиентом Azure Active Directory Azure AD вашей Microsoft 365 подписки.

>[!Note]
>Когда учетные записи пользователей AD DS синхронизируются впервые, они не получают автоматически Microsoft 365 лицензии и не могут получить доступ к Microsoft 365, например электронной почте. Сначала необходимо назначить им расположение использования. Затем назначьте лицензию этим учетным записям пользователей в индивидуальном или динамическом виде с помощью членства в группе.
>

## <a name="authentication-for-hybrid-identity"></a>Проверка подлинности для гибридного удостоверения

Существует два типа проверки подлинности при использовании гибридной модели удостоверений:

- Управляемая проверка подлинности.

  Azure AD обрабатывает процесс проверки подлинности с помощью локально хранимой версии пароля или отправляет учетные данные локальному программному агенту для проверки подлинности локальной службой AD DS.

- Федеративная проверка подлинности

  Azure AD перенаправляет клиентский компьютер, запрашивающий проверку подлинности, другому поставщику удостоверений.

### <a name="managed-authentication"></a>Управляемая проверка подлинности.

Существует два типа управляемой проверки подлинности:

- Синхронизация хэша паролей (PHS)

  Проверка подлинности выполняется с помощью Azure AD.

- Сквозная проверка подлинности (PTA)

  Проверка подлинности выполняется с помощью доменных служб Aсtive Directory в Azure AD.


#### <a name="password-hash-synchronization-phs"></a>Синхронизация хэша паролей (PHS)

С помощью PHS вы синхронизируете учетные записи пользователей AD DS с Microsoft 365 и управлять пользователями на месте. Хеши паролей пользователей синхронизируются из AD DS в Azure AD, чтобы у пользователей был один и тот же пароль в локальном и облачном помещении. Это самый простой способ включить проверку подлинности удостоверений AD DS в Azure AD. 

![Синхронизация хэша паролей (PHS)](../media/plan-for-directory-synchronization/phs-authentication.png)

При изменении или сбросе паролей на локальном уровне новые хеши паролей синхронизируются с Azure AD, чтобы пользователи всегда могли использовать один и тот же пароль для облачных ресурсов и локального ресурса. Пароли пользователей никогда не отправляются в Azure AD или хранятся в Azure AD в понятном тексте. Некоторые премиум-функции Azure AD, такие как Защита удостоверений, требуют PHS независимо от выбранного метода проверки подлинности.
  
Подробнее [см. в подборе](/azure/active-directory/hybrid/choose-ad-authn) правильного метода проверки подлинности.
  
#### <a name="pass-through-authentication-pta"></a>Сквозная проверка подлинности (PTA)

PTA предоставляет простую проверку пароля для служб проверки подлинности Azure AD с помощью агента программного обеспечения, который работает на одном или несколько локальном сервере для проверки пользователей непосредственно с помощью AD DS. С помощью PTA вы синхронизируете учетные записи пользователей AD DS с Microsoft 365 и управлять пользователями на месте. 

![Сквозная проверка подлинности (PTA)](../media/plan-for-directory-synchronization/pta-authentication.png)

PTA позволяет пользователям войти в локальное и Microsoft 365 и приложения с помощью локальной учетной записи и пароля. Эта конфигурация проверяет пароли пользователей непосредственно к локальной AD DS без хранения хеши паролей в Azure AD. 

PTA также для организаций с требованием безопасности немедленно применять локальное состояния учетных записей пользователей, политики паролей и часы логона. 
  
Подробнее [см. в подборе](/azure/active-directory/hybrid/choose-ad-authn) правильного метода проверки подлинности.
  
### <a name="federated-authentication"></a>Федеративная проверка подлинности

Федерационная проверка подлинности в основном предназначена для крупных корпоративных организаций с более сложными требованиями к проверке подлинности. Удостоверения AD DS синхронизируются с Microsoft 365, а учетные записи пользователей управляются локально. С федерацией проверки подлинности пользователи имеют одинаковый пароль на локальной основе и в облаке, и им не нужно снова входить, чтобы использовать Microsoft 365. 

Федеративная проверка подлинности может поддерживать дополнительные требования к проверке подлинности, например проверку подлинности на основе смарт-карт или сторонней многофакторной проверки подлинности, и обычно требуется, если у организаций есть требование проверки подлинности, не поддерживаемые Azure AD.
 
Подробнее [см. в подборе](/azure/active-directory/hybrid/choose-ad-authn) правильного метода проверки подлинности.
  
#### <a name="third-party-authentication-and-identity-providers"></a>Сторонние поставщики проверки подлинности и удостоверений

Объекты локального каталога могут синхронизироваться с Microsoft 365 и доступ к облачным ресурсам в основном управляется сторонним поставщиком удостоверений (IdP). Если ваша организация использует сторонное решение федерации, вы можете настроить вход с этим решением для Microsoft 365 при условии, что решение федерации сторонних организаций совместимо с Azure AD.
  
Дополнительные дополнительные возможности см. в списке совместимости федерации [Azure AD.](/azure/active-directory/connect/active-directory-aadconnect-federation-compatibility)
  
## <a name="ad-ds-preparation"></a>Подготовка AD DS

Чтобы обеспечить плавный переход к Microsoft 365 с помощью синхронизации, необходимо подготовить лес AD DS, прежде чем приступить к развертыванию синхронизации Microsoft 365 каталогов.
  
Подготовка каталога должна быть сосредоточена на следующих задачах:

- Удалите **дублирующиеся атрибуты proxyAddress** и **userPrincipalName.**
- Обновление пустых и **недействительных атрибутов userPrincipalName с** допустимыми атрибутами **userPrincipalName.**
- Удаление недействительных и сомнительных символов в **givenName**, фамилия ( **sn** ), **sAMAccountName**, **displayName**, **почта**, **proxyAddresses**, **mailNickname** и **userPrincipalName** атрибуты. Подробные сведения о подготовке атрибутов см. в списке атрибутов, синхронизируются с Azure Active Directory [Sync Tool.](https://go.microsoft.com/fwlink/p/?LinkId=396719)

    > [!NOTE]
    > Это те же атрибуты, которые Подключение Azure AD. 
  
## <a name="multi-forest-deployment-considerations"></a>Аспекты развертывания с несколькими лесами

Для нескольких лесов и параметров SSO используйте настраиваемую установку [azure AD Подключение.](/azure/active-directory/hybrid/how-to-connect-install-custom)
  
Если в организации имеется несколько лесов для проверки подлинности (леса с логотипом), настоятельно рекомендуется следующее:
  
- **Рассмотрите возможность консолидации лесов.** Как правило, существует больше накладных расходов, необходимых для поддержания нескольких лесов. Если в организации нет ограничений безопасности, которые диктуют необходимость отдельных лесов, рассмотрите возможность упрощения локальной среды.
- **Используйте только в основном лесу с логотипом.** Развертывание Microsoft 365 только в основном лесу логотипов для первоначального развертывания Microsoft 365. 

Если вы не можете консолидировать развертывание AD DS с несколькими лесами или использовать другие службы каталогов для управления удостоверениями, вы можете синхронизировать их с помощью Microsoft или партнера.
  
Дополнительные [сведения см. в Подключение topologies for Azure AD.](/azure/active-directory/hybrid/plan-connect-topologies)
  
## <a name="features-that-are-dependent-on-directory-synchronization"></a>Функции, зависящие от синхронизации каталогов
  
Синхронизация каталогов необходима для следующих функций и функций:
  
- Azure AD Бесшовный Sign-On (SSO)
- Skype сосуществования
- Exchange гибридного развертывания, в том числе:
  - Полностью общий глобальный список адресов (GAL) между локальной Exchange и Microsoft 365.
  - синхронизацию данных глобального списка адресов из разных почтовых систем;
  - Возможность добавлять пользователей к пользователям и удалять их из Microsoft 365 служб. Для этого необходимо следующее:
  - При настройке синхронизации каталогов необходимо настроить двунастройную синхронизацию. По умолчанию средства синхронизации каталогов записывают сведения о каталогах только в облако. При настройке двунастройной синхронизации вы включаете функции обратного копирования, чтобы ограниченное число атрибутов объектов скопировалось из облака, а затем было записано обратно в локальное AD DS. Записи обратно также называются гибридным Exchange режиме. 
  - Локальное Exchange гибридное развертывание
  - Возможность перемещения некоторых почтовых ящиков пользователей в Microsoft 365, сохраняя при этом другие почтовые ящики пользователей в локальном помещении.
  - Сейф и заблокированные отправители реплицированы в Microsoft 365.
  - базовое делегирование и функциональную возможность отправки электронной почты от имени кого-либо.
  - У вас есть интегрированное локальное решение для проверки подлинности или многофакторная проверка подлинности.
- Синхронизация фотографий, эскизов, конференц-залов и групп безопасности
