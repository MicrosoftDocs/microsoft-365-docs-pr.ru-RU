# <a name="managing-expressroute-for-office-365-connectivity"></a>Управление подключением к Office 365 через ExpressRoute

ExpressRoute для Office 365 предлагает альтернативный маршрут маршрут, чтобы получить доступ ко многим службам Office 365 без необходимости всего трафика для выхода в Интернет. Хотя подключение к Интернету к Office 365 по-прежнему требуется, определенные маршруты, которые Корпорация Майкрософт объявляет через BGP в вашей сети, делают прямую схему ExpressRoute предпочтительной, если в вашей сети нет других конфигураций. К трем общим областям, которые можно настроить для управления этой маршрутизативой, относятся фильтрация префиксов, безопасность и соответствие требованиям.
  
> [!NOTE]
> Корпорация Майкрософт изменила способ проверки домена пиринговой маршрутки Microsoft для Azure ExpressRoute. С 31 июля 2017 г. все клиенты Azure ExpressRoute могут включить Пиринг (Майкрософт) непосредственно из консоли администрирования Azure или с помощью PowerShell. После включения пиринга Microsoft любой клиент может создать фильтры маршрутов для получения объявлений о маршруте BGP для приложений Dynamics 365 Customer Engagement (прежнее название — CRM Online). Клиенты, которые нуждаются в Azure ExpressRoute для Office 365, должны получить отзыв от корпорации Майкрософт, прежде чем создавать фильтры маршрутов для Office 365. Обратитесь в службу поддержки учетных записей Майкрософт, чтобы узнать, как запросить отзыв для включения Office 365 ExpressRoute. Неавторизованные подписки, которые пытаются создать фильтры маршрутов для Office 365, получат сообщение [об ошибке](https://support.microsoft.com/kb/3181709)
  
## <a name="prefix-filtering"></a>Фильтрация префиксов

Корпорация Майкрософт рекомендует клиентам принимать все маршруты BGP, как было заявлено от Корпорации Майкрософт, а предоставленные маршруты проходят тщательную проверку и проверку, удаляя все преимущества для тщательного анализа. ExpressRoute предлагает рекомендуемые элементы управления, такие как владение IP-префиксами, целостность и масштаб, без фильтрации входящие маршруты на стороне клиента.
  
Если требуется дополнительная проверка прав владельца маршрута через общедоступный пиринг ExpressRoute, вы можете проверить объявленные маршруты в списке [](https://www.microsoft.com/download/details.aspx?id=53602)всех IP-префиксов IPv4 и IPv6, которые представляют общедоступные диапазоны IP-адресов Майкрософт. Эти диапазоны охватывают все адресное пространство Майкрософт и изменяются редко, предоставляя надежный набор диапазонов для фильтрации, что также обеспечивает дополнительную защиту для клиентов, которые беспокоясь о утечке в среду маршрутов, которые не принадлежат корпорации Майкрософт. В случае изменения оно будет внося 1 числа месяца, а номер  версии в разделе сведений страницы будет изменяться при каждом обновлении файла.
  
Существует ряд причин избежать использования URL-адресов [и диапазонов IP-адресов Office 365](https://aka.ms/o365endpoints) для создания списков фильтров префиксов. В том числе:
  
- Префиксы IP Office 365 часто меняются.

- URL-адреса и диапазоны IP-адресов Office 365 предназначены для управления списками и инфраструктурой прокси-серверов брандмауэра, а не маршрутией.

- Url-адреса и диапазоны IP-адресов Office 365 не охватывают другие службы Майкрософт, которые могут оказаться в области ваших подключений ExpressRoute.

|**Option**|**Сложность**|**Управление изменениями**|
|:-----|:-----|:-----|
|Принятие всех маршрутов Майкрософт  <br/> |**Низкая:** Клиент использует средства контроля Майкрософт, чтобы убедиться, что все маршруты находятся в надлежащем владельце.  <br/> |Нет  <br/> |
|Фильтрация собственных суперсетей Майкрософт  <br/> |**Средний:** Клиент реализует суммарные списки фильтров префиксов, чтобы разрешить только маршруты, которые принадлежат корпорации Майкрософт.  <br/> |Клиенты должны убедиться, что редко обновляемая информация отражается в фильтрах маршрутов.  <br/> |
|Фильтрация диапазонов IP-адресов Office 365  <br/> [!CAUTION] Not-Recommended |**Высокая:** Клиент фильтрует маршруты на основе определенных ip-префиксов Office 365.  <br/> |Клиенты должны реализовать надежный процесс управления изменениями для ежемесячных обновлений.  <br/> [!CAUTION] Для этого решения требуются значительные изменения. Изменения, не реализованные во времени, скорее всего, привят к отключению службы.   |

Подключение к Office 365 с помощью Azure ExpressRoute основано на объявлениях BGP определенных IP-подсетей, которые представляют сети, в которых развернуты конечные точки Office 365. Из-за глобального характера Office 365 и количества служб, в которые вмеется Office 365, клиентам часто приходится управлять рекламой, которую они принимают в своей сети. Если вас беспокоит количество префиксов, объявленных в вашей среде, функция сообщества [BGP](https://support.office.com/article/Using-BGP-communities-in-ExpressRoute-for-Office-365-scenarios-preview-9ac4d7d4-d9f8-40a8-8c78-2a6d7fe96099) позволяет фильтровать объявления по определенному набору служб Office 365. Эта функция теперь находится в предварительной версии.
  
Независимо от того, как вы управляете рекламными объявлениями маршрута BGP, исходя из Майкрософт, вы не получите особого доступа к службам Office 365 по сравнению с подключением к Office 365 через интернет-канал. Корпорация Майкрософт поддерживает одинаковые уровни безопасности, соответствия требованиям и производительности независимо от типа каналов, которые клиент использует для подключения к Office 365.
  
### <a name="security"></a>Безопасность

Корпорация Майкрософт рекомендует вам поддерживать собственные элементы управления сетью и периметром безопасности для подключений к общедоступным сетям ExpressRoute и пирингу Майкрософт, в том числе подключений к службам Office 365 и из них. Средства безопасности должны быть на месте для сетевых запросов, исходящие из вашей сети в сеть Майкрософт, а также входящие из сети Майкрософт в вашу сеть.
  
#### <a name="outbound-from-customer-to-microsoft"></a>Исходящие исходящие данные от клиента в Корпорацию Майкрософт
  
Когда компьютеры подключаются к Office 365, они подключаются к одному набору конечных точек независимо от того, установлено ли подключение через Интернет или через канал ExpressRoute. Независимо от используемой схемы Майкрософт рекомендует рассматривать службы Office 365 как более надежные, чем универсальные интернет-назначения. Исходящие элементы управления безопасностью должны сосредоточиться на портах и протоколах, чтобы снизить риск и свести к минимуму текущее обслуживание. Необходимые сведения о портах доступны в справочной статье по конечным [точкам Office 365.](https://aka.ms/o365endpoints)
  
Для добавленных элементов управления можно использовать фильтрацию на уровне FQDN в инфраструктуре прокси-сервера, чтобы ограничить или проверить некоторые или все сетевые запросы, предназначенные для Интернета или Office 365. Ведение списка FQDNs по мере выпуска функций и развития предложений Office 365 требует более надежного управления изменениями и отслеживания изменений опубликованных конечных точек [Office 365.](https://aka.ms/o365endpoints)
  
> [!CAUTION]
> Корпорация Майкрософт рекомендует не использовать только префиксы IP для управления исходячей безопасностью в Office 365.

|**Option**|**Сложность**|**Управление изменениями**|
|:-----|:-----|:-----|
|Без ограничений  <br/> |**Низкая:** Клиент разрешает неограниченный исходящие доступ к Корпорации Майкрософт.  <br/> |Нет  <br/> |
|Ограничения портов  <br/> |**Низкая:** Клиент ограничивает исходящие доступы к Майкрософт ожидаемыми портами.  <br/> |Редко.  <br/> |
|Ограничения FQDN  <br/> |**Высокая:** Клиент ограничивает исходящие доступ к Office 365 на основе опубликованных FQDNs.  <br/> |Ежемесячные изменения.  <br/> |

#### <a name="inbound-from-microsoft-to-customer"></a>Входящие данные от Майкрософт к клиенту
  
Существует несколько необязательных сценариев, которые требуют от корпорации Майкрософт инициировать подключения к вашей сети.
  
- ADFS во время проверки пароля для регистрации.

- [Exchange Server гибридных развертывании.](https://technet.microsoft.com/library/jj200581%28v=exchg.150%29.aspx)

- Почта из клиента Exchange Online на локальное хост-сервер.

- Отправка почты SharePoint Online из SharePoint Online на локальное хост-сервер.

- [Федераированный гибридный поиск SharePoint.](https://technet.microsoft.com/library/dn197174.aspx)

- [Гибридные BCS SharePoint.](https://technet.microsoft.com/library/dn197239.aspx )

- [Гибридная федерация Skype для](https://technet.microsoft.com/library/jj205403.aspx) бизнеса и/или [Skype для бизнеса.](https://technet.microsoft.com/library/skype-for-business-online-federation-and-public-im-conectivity.aspx)

- [Skype для бизнеса Cloud Connector.](https://technet.microsoft.com/library/mt605227.aspx )

Корпорация Майкрософт рекомендует принимать эти подключения через интернет-канал, а не по каналу ExpressRoute, чтобы снизить сложность. Если требования соответствия требованиям или производительности определяют, что эти входящие подключения должны приниматься по каналу ExpressRoute, рекомендуется использовать брандмауэр или обратный прокси-сервер для области действия принятых подключений. Вы можете использовать конечные [точки Office 365,](https://aka.ms/o365endpoints) чтобы найти правильные FQDNs и IP-префиксы.
  
### <a name="compliance"></a>Соответствие требованиям

Мы не опираемся на путь маршрутки, который вы используете для каких-либо из наших средств контроля соответствия требованиям. Независимо от того, подключается ли вы к службам Office 365 через ExpressRoute или интернет-канал, наши средства контроля соответствия требованиям не изменятся. Чтобы выбрать оптимальный вариант для соответствия требованиям организации, просмотрите различные уровни сертификации соответствия требованиям и безопасности для Office 365.
