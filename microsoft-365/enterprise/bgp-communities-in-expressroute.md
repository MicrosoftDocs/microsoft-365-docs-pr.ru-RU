# <a name="using-bgp-communities-in-expressroute-for-office-365-scenarios"></a>Использование сообществ BGP в сценариях ExpressRoute для Office 365

Подключение к Office 365 с помощью Azure ExpressRoute основано на рекламе BGP определенных ip-сетей, которые представляют сети, в которых развернуты конечные точки Office 365. Из-за глобального характера Office 365 и количества служб, составляющих Office 365, клиентам часто приходится управлять рекламой, которую они принимают в своей сети. Уменьшение числа подсетей IP; которые называются префиксами IP в оставшейся части этой статьи в соответствии с терминологией управления сетью BGP, служат следующим конечным целям для клиентов:
  
- Управление числом объявленных префиксов **IP-** Клиенты, которые имеют внутреннюю сетевую инфраструктуру или сетевого оператора, поддерживающую только ограниченное число префиксов IP, и клиенты с сетевым носителем, который взимает плату за прием префиксов выше ограниченного числа, захотят оценить общее число префиксов, уже рекламируемых в их сети, и выбрать, какие приложения Office 365 лучше всего подходят для ExpressRoute.

- Управление объемом пропускной способности, требуемой в схеме **Azure ExpressRoute.** Клиенты могут захоть управлять конвертом пропускной способности служб Office 365 по пути ExpressRoute и пути к Интернету. Это позволяет клиентам зарезервировать пропускную способность ExpressRoute для определенных приложений, таких как Skype для бизнеса, и маршрутировать оставшиеся приложения Office 365 по пути к Интернету.

Чтобы помочь клиентам с этими целями, префиксы IP-адресов Office 365, рекламируемые через ExpressRoute, помечены определенными значениями сообщества BGP службы, как показано в примере ниже.
  
> [!NOTE]
> Следует ожидать, что часть сетевого трафика, связанного с другими приложениями, будет включена в значение сообщества. Это ожидаемое поведение для глобального программного обеспечения в качестве службы с общими службами и центрами обработки данных. Это было минимизировано, если это возможно, с помощью этих двух целей, с помощью управления количеством префиксов и/или пропускной способностью.

|**Служба**|**Значение сообщества BGP**|**Примечания**|
|:-----|:-----|:-----|
|Exchange Online\*  <br/> |12076:5010  <br/> |Включает службы Exchange и EOP\*  <br/> |
|SharePoint Online\*  <br/> |12076:5020  <br/> |SharePoint Online  <br/> |
|Skype для бизнеса\*  <br/> |12076:5030  <br/> |Службы Skype для бизнеса & Microsoft Teams  <br/> |
|Другие службы Office 365\*  <br/> |12076:5100  <br/> |Включает Azure Active Directory (сценарии проверки подлинности и синхронизации каталогов), а также службы портала Office 365  <br/> |
|\*Область сценариев служб, включенных в ExpressRoute, задокументирована в статье Конечные точки [Office 365.](./urls-and-ip-address-ranges.md)  <br/> \*\*В будущем могут быть добавлены дополнительные службы и значения сообщества BGP. [См. текущий список BGP-сообществ.](/azure/expressroute/expressroute-routing)  <br/> |

## <a name="what-are-the-most-common-scenarios-for-using-bgp-communities"></a>Каковы наиболее распространенные сценарии использования сообществ BGP?

Клиенты могут использовать сообщества BGP для регулирования групп префиксов IP, которые принимаются их сетью через Azure ExpressRoute, что влияет на общее число префиксов IP и ожидаемый конверт пропускной способности определенных служб Office 365. Важно понимать, что для всех Office 365 потребуется трафик, связанный с Интернетом, независимо от использования сообществ Azure ExpressRoute или BGP. Следующие три сценария являются наиболее распространенным использованием этой функции.
  
### <a name="scenario-1-minimizing-the-number-of-office-365-ip-prefixes"></a>Сценарий 1. Минимизация числа префиксов IP Office 365

Корпорация Contoso — это компания с 50 000 человек, которая в настоящее время использует Office 365 для Exchange Online и SharePoint Online. При просмотре требований ExpressRoute Contoso определяет свои сетевые устройства во многих региональных расположениях, которые не могут обрабатывать размеры таблицы маршрутов свыше 100 дополнительных записей маршрутов. Contoso проанализировала общее число префиксов IP, которые ExpressRoute будет рекламировать для полного набора служб Office 365, и пришла к выводу, что она превышает 100. Чтобы оставаться в 100 дополнительных записях маршрута, Contoso использует ExpressRoute для Office 365 только для сообщества SharePoint Online BGP, 12076:5020, полученного через peering Microsoft ExpressRoute.

|**Тег сообщества BGP, используемый**|**Маршрутивка функциональных возможностей по Azure ExpressRoute**|**Необходимые маршруты Интернета**|
|:-----|:-----|:-----|
|SharePoint  <br/> (12076:5020)  <br/> |SharePoint Online &amp; OneDrive для бизнеса  <br/> | Запросы DNS, CRL, &amp; CDN  <br/>  Все другие службы Office 365, не поддерживаемые специально для Azure ExpressRoute  <br/>  Все остальные облачные службы Майкрософт  <br/>  Портал Office 365, проверка подлинности Office 365, &amp; Office в браузере  <br/>  Exchange Online, Exchange Online Protection и Skype для бизнеса Online  <br/> |

> [!NOTE]
> Для достижения более низких префиксов для каждой службы сохранится минимальное количество совпадений между службами. Это поведение не является отклонением от нормы.
  
### <a name="scenario-2-scoping-expressroute-and-internal-bandwidth-use-to-some-office-365-services"></a>Сценарий 2. Scoping ExpressRoute и использование внутренней пропускной способности для некоторых служб Office 365

Fabrikam Inc, крупное многонациональные предприятия с распределенной неоднородной сетью, является абонентом многих служб Office 365, включая; Exchange Online, SharePoint Online и Skype для бизнеса Online. Внутренняя инфраструктура маршрутов Fabrikam может обрабатывать тысячи префиксов IP в таблицах маршрутиза; Однако Fabrikam хочет только упростить ExpressRoute и внутреннюю пропускную способность для приложений Office 365, наиболее чувствительных к производительности сети, и использовать существующую пропускную способность Интернета для всех остальных приложений Office 365.
  
По этой причине Fabrikam передает пропускную способность Azure ExpressRoute только для сообщества BGP Skype для бизнеса в Интернете, 12076:5030, полученного через одноранговую службу ExpressRoute Microsoft. Оставшийся сетевой трафик, связанный с Office 365, продолжает использовать точки egress Интернета.

|**Тег сообщества BGP, используемый**|**Маршрутивка функциональных возможностей по Azure ExpressRoute**|**Необходимые маршруты Интернета**|
|:-----|:-----|:-----|
|Skype для бизнеса  <br/> (12076:5030)  <br/> |SIP-сигналы Skype, загрузки, голосовой, видео- и настольный обмен  <br/> | Запросы DNS, CRL, &amp; CDN  <br/>  Все другие службы Office 365, не поддерживаемые специально для Azure ExpressRoute  <br/>  Все остальные облачные службы Майкрософт  <br/>  Портал Office 365, проверка подлинности Office 365, &amp; Office в браузере  <br/>  Телеметрия Skype для бизнеса, краткие советы клиентов Skype, подключение к чатам общего звонков  <br/>  Exchange Online, Exchange Online Protection и SharePoint Online  <br/> |

### <a name="scenario-3-scoping-azure-expressroute-for-office-365-services-only"></a>Сценарий 3. Только для служб Office 365 scoping Azure ExpressRoute

Woodgrove Bank является клиентом нескольких облачных служб Майкрософт, включая Office 365. После оценки емкости сети и потребления Woodgrove Bank решает развернуть Azure ExpressRoute в качестве предпочтительного пути для поддерживаемых служб Office 365. Таблицы маршрутов могут поддерживать полный набор префиксов IP Office 365 и цепи Azure ExpressRoute, которые они создали, поддерживают все прогнозируемые потребности в пропускной способности и задержке.
  
Чтобы обеспечить сетевой трафик, связанный с облачными службами Майкрософт, помимо Office 365, Woodgrove Bank использует ExpressRoute для Office 365 для всех префиксов IP с тегами с определенными значениями сообщества BGP Office 365, 12076:5010, 12076:5020, 12076:5030, 12076:5100.

|**Тег сообщества BGP, используемый**|**Маршрутивка функциональных возможностей по Azure ExpressRoute**|**Необходимые маршруты Интернета**|
|:-----|:-----|:-----|
|Exchange, Skype для бизнеса & Microsoft Teams, SharePoint и &amp; другие службы  <br/> (12076:5010, 12076:5020, 12076:5030, 12076:5100)  <br/> |Exchange Online &amp; Exchange Online Protection  <br/> SharePoint Online &amp; OneDrive для бизнеса  <br/> SIP-сигналы Skype, загрузки, голосовой, видео- и настольный обмен  <br/> Портал Office 365, проверка подлинности Office 365, &amp; Office в браузере  <br/> | Запросы DNS, CRL, &amp; CDN  <br/>  Все другие службы Office 365, не поддерживаемые специально для Azure ExpressRoute  <br/>  Все остальные облачные службы Майкрософт  <br/> |

## <a name="key-planning-considerations-to-using-bgp-communities"></a>Основные соображения планирования использования сообществ BGP

Пользователи, которые предпочитают использовать сообщества BGP, чтобы повлиять на то, как рекламируется и распространяется ExpressRoute через клиентскую сеть, должны учитывать следующие соображения:
  
- При использовании сообществ BGP в сетевом дизайне важно обеспечить сохранение симметрии маршрутов. В некоторых случаях при добавлении или удалении сообществ BGP может возникнуть ситуация, при которой нарушена симметричная маршрутивка и необходимо обновить конфигурацию маршрутов, чтобы восстановить симметричную маршрутику.

- Scoping Azure ExpressRoute с помощью значений сообщества BGP — это действие клиента. Корпорация Майкрософт будет размещать рекламу всех префиксов IP, связанных с отношениями одноранговых пользователей, независимо от того, настроены ли они клиентом.

- Azure ExpressRoute не поддерживает действия в сети Майкрософт, основанные на сообществах BGP, назначенных клиентам.

- Префиксы IP, используемые Office 365, помечены только определенными значениями сообщества BGP службы, конкретные сообщества BGP не поддерживаются. Службы Office 365 имеют глобальный характер, фильтрация префиксов в зависимости от расположения клиента или данных в облаке Office 365 не поддерживается. Рекомендуется настроить сеть для координации самого короткого или наиболее предпочтительного сетевого пути из расположения сети пользователя в глобальную сеть Майкрософт независимо от физического расположения IP-адреса запрашиваемой службы Office 365.

- Префиксы IP, включенные в каждое значение сообщества BGP, представляют собой подсети, которые содержат IP-адреса для приложения Office 365, связанного со значением. В некоторых случаях несколько приложений Office 365 имеет IP-адреса в подсети, в результате чего префикс IP существует в нескольких сообществах. Такое поведение ожидается, хотя и редко, из-за фрагментации выделения и не влияет на количество префиксов или цели управления пропускной способностью. Клиентам рекомендуется использовать подход "разрешить то, что необходимо", а не "отказывать в том, что не нужно", когда они используют сообщества BGP для Office 365, чтобы свести к минимуму эффект.

- Использование сообществ BGP не меняет коренных требований к сетевому подключению или конфигурации, необходимых для использования Office 365. Клиенты, которые хотят получить доступ к Office 365, по-прежнему должны иметь доступ к Интернету.

- Scoping Azure ExpressRoute с сообществами BGP влияет только на маршруты, которые ваша внутренняя сеть может видеть во взаимоотношениях с microsoft peering. Возможно, вам потребуется внести дополнительные конфигурации уровня приложений, такие как использование конфигурации PAC или WPAD в сочетании с масштабной маршрутирой.

- Помимо использования сообществ BGP, назначаемой Корпорацией Майкрософт, клиенты могут назначить свои собственные сообщества BGP префиксам IP Office 365, научимся через Azure ExpressRoute влиять на внутреннюю маршрутику. Популярным случаем использования является назначение сообщества BGP на основе расположения всем маршрутам, которые будут изучаться через каждое заданное расположение одноранговых служб ExpressRoute, а затем использовать эти сведения ниже по течению в клиентской сети для координации кратчайших или наиболее предпочтительных сетевых путей в сети Майкрософт. Использование клиентских сообществ BGP с сценариями ExpressRoute для Office 365 не является областью контроля или видимости Майкрософт.
