## <a name="azure-active-directory"></a>Azure Active Directory

Что не меняется:

- Начальный домен клиента (например, ) с ИД клиента (GUID) и настраиваемые домены будут `contoso.onmicrosoft.de` сохраняться после миграции. 

- Запросы на проверку подлинности для ресурсов, которые переносят в службы Office 365, предоставляет служба проверки подлинности Azure служб Office 365 ( `login.microsoftonline.com` ). Во время миграции ресурсы, которые остаются в Office 365 Germany, аутентификация с помощью существующей службы Azure в Германии ( `login.microsoftonline.de` ).

Следует учитывать:

- Для управляемых учетных записей домена после копирования первоначального клиента Azure Active Directory (Azure AD) завершится (что является первым этапом миграции Azure AD в службу Azure AD Office 365), изменения паролей, самостоятельного сброса пароля (SSPR) и сброс паролей администраторами необходимо выполнить с порталов служб Office 365. На этом этапе запросы на обновление паролей из службы Germany не будут успешными, так как клиент Azure AD был перенесен в службы Office 365. Сброс паролей федератора домена не затрагивается, так как они завершались в локальном каталоге.

- Входы в Azure представлены на портале, где пользователь пытается получить доступ. Журналы аудита доступны только из конечной точки служб Office 365 после перехода. Перед завершением миграции необходимо сохранить журналы входа и аудита на портале Microsoft Cloud Deutschland.

- Сброс паролей, смена паролей, сброс паролей администратором управляемых организаций (не использующих службы федерации Active Directory) должны выполняться на портале служб Office 365. Попытки пользователей, которые имеют доступ к порталам Microsoft Cloud Deutschland для сброса паролей, не удастся.

- Запросы субъектов данных (DSR) Общего регламента по защите данных (GDPR) выполняются с портала администрирования Службы Office 365 Azure для будущих запросов. Все устаревшие или не клиентские диагностические данные, которые есть в Microsoft Cloud Deutschland, удаляются не ранее 30 дней.

## <a name="subscriptions--licenses"></a>Подписки & лицензий

- Подписки на Office 365 и Dynamics из Microsoft Cloud Deutschland переносят в немецкий регион с перемещением Azure AD. Затем организация обновляется, чтобы отразить новые подписки на службы Office 365. Во время краткого процесса передачи подписки изменения подписки блокируются.

- По мере перехода клиента на службы Office 365 его подписки и лицензии для Германии стандартизются с новыми предложениями служб Office 365. Соответствующие подписки на службы Office 365 приобретаются для перенаправленных подписок в Германии. Пользователям с лицензиями на службы Office 365 в Германии будут назначены лицензии на службы Office 365. По завершении подписки на устаревшую службу Germany отменяются и удаляются из текущего клиента служб Office 365.

- После переноса отдельных рабочих нагрузок дополнительные функции доступны через службы Office 365 (например, Планировщик Майкрософт и Microsoft Flow) из-за новых подписок на службы Office 365. Если это подходит для вашей организации, клиент или администратор лицензирования может отключить новые планы обслуживания при планировании управления изменениями для внедрения новых служб. Инструкции по отключению планов обслуживания, назначающихся лицензиям пользователей, см. в подсети "Отключение доступа к службам [Microsoft 365](https://docs.microsoft.com/office365/enterprise/powershell/disable-access-to-services-while-assigning-user-licenses)при назначении пользовательских лицензий".

## <a name="exchange-online"></a>Exchange Online

- URL-адреса ресурсов Exchange после миграции переходят с устаревшей конечной точки Germany на конечную точку служб `outlook.office.de` Office 365. `outlook.office365.com` Пользователи могут получать доступ к перенесенным почтовым ящикам по устаревшему URL-адресу до завершения миграции. Клиенты должны как можно скорее перейти на новый URL-адрес после начала миграции Exchange, чтобы не повлиять на выход из среды в Германии. URL-адреса служб Office 365 для служб Outlook становятся доступными только после начала миграции Exchange.

- Почтовые ящики переносят в качестве серверного процесса. Во время перехода пользователи в вашей организации могут быть либо в Microsoft Cloud Deutschland, либо в немецком регионе и являются частью одной организации Exchange (в том же глобальном списке адресов).

- Пользователи веб-Outlook Web App, которые имеют доступ к службе по URL-адресу, где их почтовый ящик не находится, увидят запрос на дополнительную проверку подлинности. Например, если почтовый ящик пользователя находится в службах Office 365, а подключение Outlook Web App пользователя использует устаревшую конечную точку, пользователь сначала пройдет проверку подлинности, а затем — `outlook.office.de` `login.microsoftonline.de` в `login.microsoftonline.com` . После завершения миграции пользователь сможет получить доступ к новому URL-адресу (), и он увидит только один ожидаемый запрос на `https://outlook.office365.com` вход. 

## <a name="office-services"></a>Службы Office

Службы Office Online доступны до `office.de` и во время перехода. После перехода почтовых ящиков пользователей на службы Office 365 пользователям следует начать использовать URL-адреса служб Office 365. По мере переноса последующих рабочих нагрузок в службы Office 365 интерфейс office.com с портала office 365 начнет работать.

- Функции exchange Online Protection (EOP) копируется в новый регион Германии.
- В рамках миграции пользователям Центра безопасности и соответствия требованиям Office 365 необходимо перейти на использование глобальных `https://protection.office.com` URL-адресов.
