# <a name="customer-experience-during-the-migration-to-office-365-services-in-the-new-german-datacenter-regions"></a>Взаимодействие с клиентами в процессе миграции на службы Office 365 в новых регионах центров обработки данных в Германии

Миграция клиентов обеспечивает минимальное влияние на администраторов и пользователей. Однако для каждой рабочей нагрузки следует учитывать ряд моментов. Ознакомьтесь со следующими разделами, чтобы лучше понять процесс миграции для рабочих нагрузок.

## <a name="azure-active-directory"></a>Azure Active Directory

Что не меняется:

- Исходный домен клиента (например, `contoso.onmicrosoft.de` ) с идентификатором клиента (GUID) и пользовательские домены будут сохранены после миграции. 

- Запросы на проверку подлинности для ресурсов, перенесенных в службы Office 365, предоставляются службой проверки подлинности Azure для служб Office 365 ( `login.microsoftonline.com` ). Во время миграции ресурсы, оставшиеся до Office 365 Германии, проходят проверку подлинности существующей службой Azure ( `login.microsoftonline.de` ).

Замечания к заметке:

- Для управляемых учетных записей домена после завершения копирования исходного клиента Azure Active Directory (Azure AD) (это первый этап миграции Azure AD в службу Azure AD службы Office 365), изменения паролей, изменения паролей и очистки паролей администратором необходимо выполнить на порталах служб Office 365. Запросы на обновление паролей от службы Германии не будут успешно выполнены на этом шаге, так как клиент Azure AD был перенесен в службы Office 365. Сброс паролей федеративных доменов не затрагивается, так как они выполняются в локальном каталоге.

- Входы Azure отображаются на портале, где пользователь попытается получить доступ. После перехода журналы аудита доступны только в конечной точке служб Office 365. Перед переносом до завершения миграции необходимо сохранить журналы входа и аудита на портале Microsoft Cloud записей.

- Сброс паролей, изменения паролей, пароль, сброшенный администратором для управляемых организаций (не использующих службы федерации Active Directory), необходимо выполнить через портал служб Office 365. Попытки пользователей, обращающихся к порталам Microsoft Cloud записей для сброса паролей, завершатся неудачей.

- Общие запросы субъектов данных по защите данных (GDPR) (DSR) выполняются на портале администрирования Azure службы Office 365 для будущих запросов. Все устаревшие или неклиентские диагностические данные, резидентные в Microsoft Cloud записей, удаляются в течение или до 30 дней.

## <a name="subscriptions--licenses"></a>Лицензии & подписки

- Подписки на Office 365 и Dynamics из Microsoft Cloud записей переходят в регион на немецком языке с перемещением Azure AD. После этого Организация обновляется в соответствии с новыми подписками на службы Office 365. Во время краткого переноса подписки изменения в подписках блокируются.

- Так как клиент переходит в службы Office 365, его подписки и лицензии, относящиеся к Германии, были стандартизированы с помощью новых служб Office 365. Соответствующие подписки на Office 365 Services приобретены для перенесенных подписок на Германии. Пользователям с лицензиями на Германии будут назначены лицензии на Office 365 Services. После завершения устаревшие подписки на немецком (Германия) отменяются и удаляются из текущего клиента служб Office 365.

- После миграции отдельных рабочих нагрузок становятся доступны дополнительные функции с помощью служб Office 365 (таких как Microsoft Planner и Microsoft Flow) из-за новых подписок на службы Office 365. При необходимости администратор клиента или лицензирования может отключить новые планы обслуживания при планировании управления изменениями для введения новых служб. Рекомендации по отключению планов обслуживания, назначенных лицензиям пользователей, приведены в статье [Отключение доступа к службам Microsoft 365 при назначении пользовательских лицензий](https://docs.microsoft.com/office365/enterprise/powershell/disable-access-to-services-while-assigning-user-licenses).

## <a name="exchange-online"></a>Exchange Online

- URL-адреса ресурсов Exchange переход от устаревшей конечной точки Германии `outlook.office.de` к конечной точке служб Office 365 `outlook.office365.com` после миграции. Пользователи могут получить доступ к перенесенному почтовому ящику с помощью устаревшего URL-адреса, пока миграция не будет завершена. Пользователи должны сразу перейти на новый URL-адрес после начала миграции Exchange, чтобы избежать влияния на выбытие Германии. URL-адреса служб Office 365 для служб Outlook становятся доступны только после начала миграции Exchange.

- Почтовые ящики переносятся в виде внутреннего процесса. Пользователи в вашей организации могут находиться в Microsoft Cloud записей или на немецком регионе во время перехода и являются частью той же организации Exchange (в том же глобальном списке адресов).

- Пользователи Outlook Web App, обращающиеся к службе с помощью URL-адреса, не размещенного в почтовом ящике, будут видеть дополнительный запрос проверки подлинности. Например, если почтовый ящик пользователя находится в службах Office 365, а подключение Outlook Web App пользователя использует конечную точку прежней версии `outlook.office.de` , пользователь сначала проверяет подлинность `login.microsoftonline.de` , а затем — `login.microsoftonline.com` . По завершении миграции пользователь может получить доступ к новому URL-адресу ( `https://outlook.office365.com` ), и он увидит только один, ожидаемый запрос на вход. 

## <a name="office-services"></a>Службы Office

Доступ к службам Office Online осуществляется с помощью `office.de` до и во время перехода. После перехода почтовых ящиков пользователей в службы Office 365 пользователям следует начать использовать URL-адреса служб Office 365. По мере перехода по следующим рабочим нагрузкам в службы Office 365 их интерфейс с портала office.com начнет работать.

## <a name="exchange-online-protection"></a>Exchange Online Protection

- Внутренние функции Exchange Online Protection (EOP) копируются в новый регион Германии.
- Пользователям центра безопасности и соответствия требованиям Office 365 необходимо переходить на использование глобальных URL-адресов в `https://protection.office.com` рамках миграции.

Существующие клиенты Skype для бизнеса Online будут переходить в Microsoft Teams. Для получения дополнительных сведений см [https://aka.ms/SkypeToTeams-Home](https://aka.ms/SkypeToTeams-Home) .
