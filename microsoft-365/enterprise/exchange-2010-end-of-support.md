# <a name="exchange-2010-end-of-support-roadmap"></a>План действий после прекращения поддержки Exchange 2010

*Эта статья относится к Microsoft 365 корпоративный и Office 365 корпоративный.*

Exchange Server 2010 г. поддержка была **13 октября 2020 г.** Если вы еще не начали миграцию с Exchange 2010 на Microsoft 365, Office 365 или Exchange 2016, самое время приступить к планированию.

## <a name="what-does-end-of-support-mean"></a>Что означает *завершение поддержки?*

Большинство продуктов Майкрософт имеют жизненный цикл поддержки, в течение которого они получают новые функции, исправления ошибок, исправления безопасности и так далее. Этот жизненный цикл обычно длится 10 лет с начального выпуска продукта. Завершение этого жизненного цикла называется завершением поддержки продукта. Так как 13 октября 2020 г. поддержка Exchange 2010 перестала поддерживаться, корпорация Майкрософт больше не предоставляет:

- Техническая поддержка для проблем, которые могут возникнуть.
- Исправления ошибок, которые могут повлиять на стабильность и доступность сервера.
- Исправления безопасности для уязвимостей, которые могут сделать сервер уязвимым к нарушению безопасности.
- Обновления часовой пояс.

Установка Exchange 2010 продолжится после этой даты. Но из-за изменений, перечисленных выше, настоятельно рекомендуется как можно скорее перейти с Exchange 2010.

Дополнительные сведения о том, как закончить поддержку, см. в "Ресурсах", которые помогут вам обновить серверы и клиенты [Office 2010.](upgrade-from-office-2010-servers-and-products.md)

## <a name="what-are-my-options"></a>Какие у меня есть варианты?

Это отличное время для изучения параметров и подготовки плана миграции. Вы можете выполнить указанные ниже действия.

- Полностью переходить на Microsoft 365. Перенос почтовых ящиков с помощью cutover, minimal hybrid или full hybrid migration. Затем удалите локальное серверы Exchange и Active Directory.
- Перенос серверов Exchange 2010 в Exchange 2016 на локальном сервере.

> [!IMPORTANT]
> Если ваша организация хочет перенести почтовые ящики в Microsoft 365, но планирует сохранить DirSync или Azure AD Connect для продолжения управления учетными записями пользователей из локальной службы Active Directory, необходимо сохранить по крайней мере один сервер Microsoft Exchange в локальной организации. Если удалить все серверы Exchange, вы не сможете вносить изменения в получателей Exchange в Exchange Online, так как источник полномочий остается в локальной службе Active Directory. В нее необходимо внести изменения. В этом сценарии доступны следующие параметры:
>
>- *Рекомендуется:* Если вы перенесли почтовые ящики в Microsoft 365 и обновили серверы до 13 октября 2020 г., используйте Exchange 2010 для подключения к Microsoft 365 и переноса почтовых ящиков. Далее необходимо перенести Exchange 2010 в Exchange 2016 и списать все оставшиеся серверы Exchange 2010.
>- Если вы не завершили миграцию почтовых ящиков и обновление локального сервера до 13 октября 2020 г., сначала обновите свои серверы Exchange 2010 до Exchange 2016. Затем используйте Exchange 2016 для подключения к Microsoft 365 и переноса почтовых ящиков.

> [!NOTE]
> Это немного сложнее, но вы также можете перенести почтовые ящики в Microsoft 365 при переносе локального сервера Exchange 2010 в Exchange 2016.

Вот три пути, которые можно использовать, чтобы избежать окончания поддержки Exchange Server 2010.

## <a name="migrate-to-microsoft-365"></a>Миграция в Microsoft 365

Перенос электронной почты в Microsoft 365 — это лучший и простой вариант, который поможет вам завершить развертывание Exchange 2010. При переходе на Microsoft 365 вы можете сделать один переход от старой технологии к текущим функциям, в том числе:

- Такие возможности соответствия требованиям, как политики хранения, In-Place хранение для судебного разбирательства, eDiscovery на месте и другие.
- Microsoft Teams.
- Power BI.
- Почта с фокусом.
- MyAnalytics.

Microsoft 365 также в первую очередь получает новые функции и возможности, чтобы ваша организация сразу же начала их использовать. Кроме того, вам не придется беспокоиться о:

- Приобретение и обслуживание оборудования.
- Оплата загрев и хладнокровие серверов.
- Следите за исправлениями безопасности, продуктов и часового пояса.
- Обслуживание хранилища и программного обеспечения для поддержки требований соответствия требованиям.
- Обновление до новой версии Exchange. Вы всегда на последней версии Exchange в Microsoft 365.

### <a name="how-should-i-migrate-to-microsoft-365"></a>Как перейти на Microsoft 365?

В зависимости от организации у вас есть несколько вариантов для получения microsoft 365. Во-первых, необходимо учесть несколько моментов, например:
- Количество мест или почтовых ящиков, которые необходимо переместить.
- Время, в течение времени, в течение чего будет длиться миграция.
- Требуется ли у вас плавная интеграция между локальной установкой и Microsoft 365 во время миграции.
 
В этой таблице показаны варианты миграции и наиболее важные факторы, определяющих, какой метод использовать.

|Параметр миграции|Размер организации|Duration|
|---|---|---|
|Прямая миграция|Менее 150 мест|Неделя или меньше|
|Минимальная гибридная миграция|Менее 150 мест|Несколько недель или меньше|
|Полная гибридная миграция|Более 150 мест|Несколько недель или больше|

В следующих разделах представлен обзор этих методов. Дополнительные сведения см. в под [вопросе "Решение о пути миграции".](https://support.office.com/article/Decide-on-a-migration-path-0d4f2396-9cef-43b8-9bd6-306d01df1e27)

### <a name="cutover-migration"></a>Прямая миграция

В результате миграции все почтовые ящики, группы рассылки, контакты и другие почтовые ящики переносят в Office 365 в установленные дату и время. После этого вы закройте свои серверы Exchange Server и начнете использовать Microsoft 365 исключительно.

Быстрая миграция отлично подходит для небольших организаций, которые не имеют большого количества почтовых ящиков, хотят быстро ться в Microsoft 365 и не хотят иметь дело со сложностью других методов. Но это должно быть завершено через неделю или меньше. Кроме того, пользователям необходимо перенастроить свои профили Outlook. При такой миграции может быть перенесено до 2000 почтовых ящиков, но мы рекомендуем использовать ее не более чем для 150 почтовых ящиков. Если попытаться перенести дополнительные почтовые ящики, у вас может не оказаться времени для переноса всех почтовых ящиков до крайнего срока, и ваш ИТ-персонал может быть перегружен запросами на помощь пользователям в перенастройке Outlook.

Ниже учесть некоторые вопросы, которые следует учитывать при переносе.

- Microsoft 365 потребуется подключиться к серверам Exchange 2010 с помощью outlook Anywhere через TCP-порт 443.
- Все почтовые ящики локальной службы будут перемещены в Microsoft 365.
- Вам потребуется учетная запись локального администратора с доступом на чтение почтовых ящиков пользователей.
- Принятые домены Exchange 2010, которые вы хотите использовать в Microsoft 365, необходимо добавить в качестве проверенных доменов в службу.
- Между началом миграции и началом этапа завершения Microsoft 365 будет периодически синхронизировать Microsoft 365 и почтовые ящики локальной организации. Это позволяет завершить миграцию, не беспокоясь о том, что почта осталась в ваших почтовых ящиках.
- Пользователи получат новые временные пароли для своей учетной записи Microsoft 365. Им потребуется изменить их при первом входе в свои почтовые ящики.
- Вам потребуется лицензия Microsoft 365, включаемая Exchange Online для каждого переносимго почтового ящика пользователя.
- Пользователям потребуется настроить новый профиль Outlook на каждом из своих устройств и снова скачать электронную почту. Количество сообщений электронной почты, которое будет скачивать Outlook, может отличаться. Дополнительные сведения [см. в автономном режиме работы в Outlook.](https://support.microsoft.com/office/f3a1251c-6dd5-4208-aef9-7c8c9522d633)

Дополнительные информацию о миграции можно найти в:

- [Что необходимо знать о миграции электронной почты](https://docs.microsoft.com/Exchange/mailbox-migration/what-to-know-about-a-cutover-migration)
- [Выполнение миграции электронной почты в Office 365](https://docs.microsoft.com/Exchange/mailbox-migration/cutover-migration-to-office-365)

### <a name="minimal-hybrid-migration"></a>Минимальная гибридная миграция

При минимальной гибридной или экспресс-миграции несколько сотен почтовых ящиков перемещаются в Microsoft 365 в течение нескольких недель. Этот метод не поддерживает расширенные функции гибридной миграции, такие как общие сведения календаря о занятости.

Минимальная гибридная миграция отлично подходит для организаций, которые нуждаются в большем времени для переноса почтовых ящиков в Microsoft 365, но по-прежнему планируют завершить миграцию в течение нескольких недель. Некоторые преимущества более сложной полной  гибридной миграции можно получить без особой сложности. Вы можете управлять количеством почтовых ящиков и почтовыми ящиками, которые необходимо перенести в заданный момент времени. Почтовые ящики Microsoft 365 будут созданы с именами пользователей и паролями учетных записей локальной службы. И, в отличие от переключеных миграций, пользователям не нужно воссоздавать свои профили Outlook.

Ниже учесть некоторые вопросы, которые следует учитывать при минимальной гибридной миграции.

- Вам потребуется разовая синхронизация каталогов между вашими серверами Active Directory и Microsoft 365.
- Пользователи смогут войти в свой почтовый ящик Microsoft 365 с тем же именем пользователя и паролем, что и до почтового ящика.
- Вам потребуется лицензия Microsoft 365, включаемая Exchange Online для каждого переносимго почтового ящика пользователя.
- Пользователям не нужно будет настроить новый профиль Outlook на большинстве своих устройств, хотя на некоторых старых телефонах с Android может потребоваться новый профиль. Пользователям не нужно перезагружать свою электронную почту.

Дополнительные сведения см. в теме "Использование минимальной гибридной структуры для быстрого переноса почтовых ящиков [Exchange в Office 365".](https://docs.microsoft.com/Exchange/mailbox-migration/use-minimal-hybrid-to-quickly-migrate)

### <a name="full-hybrid"></a>Полная гибридная версия

При полной гибридной миграции у вас есть несколько сотен (до десятков тысяч) почтовых ящиков, и вы перемещаете некоторые или все почтовые ящики в Microsoft 365. Так как эти миграции обычно являются более длительными, гибридные миграции делают возможным:

- Показывать пользователям локальной сети сведения календаря о занятости для пользователей в Microsoft 365 и наоборот.
- См. единый глобальный список адресов, содержащий получателей как в локальной службе, так и в Microsoft 365.
- Просмотр полных свойств получателей Outlook для всех пользователей независимо от того, где они : локально или в Microsoft 365.
- Защита обмена электронной почтой между локальной службой Exchange Server и Office 365 с помощью TLS и сертификатов.
- Обрабатывайте сообщения, отправленные между локальной службой Exchange Server и Microsoft 365, как внутренние, что позволяет им:
  - Должны правильно оцениваться и обрабатываться агентами транспорта и соответствия требованиям, нацеленными на внутренние сообщения.
  - Обход фильтров нежелательной почты.

Полная гибридная миграция лучше всего для организаций, которые ожидают, что гибридная конфигурация будет сохраняться в течение нескольких месяцев или более. Вы получаете функции, перечисленные ранее в этом разделе, а также синхронизацию службы каталогов, более интегрированные функции соответствия требованиям и возможность перемещения почтовых ящиков в Microsoft 365 и из него с помощью перемещения почтовых ящиков через Интернет. Microsoft 365 становится расширением локальной организации.

Что следует учитывать при полной гибридной миграции:

- Они подходят не для всех организаций. Из-за сложности полной гибридной миграции организации с менее чем сотней почтовых ящиков обычно не видят преимуществ, которые могут служить обоснованием затрат и усилий. В таких случаях рекомендуется учесть, что вместо гибридной миграции следует учесть как учесть, так и минимальную гибридную миграцию.
- Вам необходимо настроить синхронизацию службы каталогов с помощью Azure Active Directory (Azure AD) Connect между вашими серверами Active Directory и Microsoft 365.
- Пользователи смогут войти в свой почтовый ящик Microsoft 365 с одинаковыми именем пользователя и паролем, которые они используют при входе в локальную сеть. (Для этой функции требуется Azure AD Connect с синхронизацией паролей и/или службами федерации Active Directory).
- Вам потребуется лицензия Microsoft 365, включаемая Exchange Online для каждого переносимго почтового ящика пользователя.
- Пользователям не нужно устанавливать новый профиль Outlook на большинстве своих устройств, хотя для некоторых старых телефонов с Android может потребоваться новый профиль. Пользователям не нужно перезагружать свою электронную почту.

> [!IMPORTANT]
> Если ваша организация хочет перенести почтовые ящики в Microsoft 365, но планирует сохранить DirSync или Azure AD Connect для продолжения управления учетными записями пользователей из локальной службы Active Directory, необходимо сохранить по крайней мере один сервер Exchange в локальной организации. Если удалить все серверы Exchange, вы не сможете вносить изменения в получателей Exchange в Exchange Online. Это происходит потому, что источник полномочий остается в локальной службе Active Directory, и в нее необходимо внести изменения.

Если полная гибридная миграция вам кажется правильной, см. следующие полезные ресурсы:

- [Гибридные развертывания Exchange Server](https://docs.microsoft.com/exchange/exchange-hybrid)

- [Мастер гибридной конфигурации: вопросы и ответы](https://docs.microsoft.com/exchange/hybrid-configuration-wizard-faqs)

## <a name="upgrade-to-a-newer-version-of-exchange-server-on-premises"></a>Обновление до более новой версии Exchange Server локальной версии

Мы настоятельно верим, что вы получаете наилучшие значения и пользовательский интерфейс, полностью переходить на Microsoft 365. Но мы понимаем, что некоторым организациям необходимо хранить некоторые серверы Exchange Server в локальной организации. Это может быть из-за нормативных требований, чтобы гарантировать, что данные не хранятся во внешнем центре обработки данных, так как у вас есть уникальные параметры или требования, которые не могут быть выполнены в облаке, или потому, что вам требуется Exchange для управления облачными почтовыми ящиками, так как вы по-прежнему используете Active Directory локально. В любом случае, если вы храните Exchange локально, необходимо обновить среду Exchange 2010 по крайней мере до Exchange 2013 или Exchange 2016.

Для лучшего опыта рекомендуется обновить оставшиеся локальной среды до Exchange 2016. Вам не нужно устанавливать Exchange Server 2013, если вы хотите перейти непосредственно с Exchange Server 2010 на Exchange Server 2016.

Exchange 2016 включает все функции предыдущих выпусков Exchange. Он наиболее точно соответствует возможностям, доступным в Microsoft 365, хотя некоторые функции доступны только в Microsoft 365. Ознакомьтесь с несколькими пропущенными моментами:

|Выпуск Exchange|Возможности|
|---|---|
|**Exchange 2013**|Упрощенная архитектура позволяет сократить количество ролей сервера до трех (почтовый ящик, клиентский доступ, edge Transport)|
||Политики защиты от потери данных (DLP), которые помогают предотвратить утечку конфиденциальной информации|
||Улучшенное Outlook Web App работы|
|**Exchange 2016**|*Функции из Exchange 2013 и ...*|
||Дополнительные упрощенные роли серверов только для почтовых ящиков и edge Transport|
||Улучшенная DLP наряду с интеграцией с SharePoint|
||Улучшенная устойчивость баз данных|
||Совместная работа с документами в Интернете|

|Фактор|Дополнительные сведения|
|---|---|
|Даты окончания поддержки|Как и в Exchange 2010, каждая версия Exchange имеет собственную дату окончания поддержки:<br/><br/>Exchange 2013 — апрель 2023 г.<br/>Exchange 2016 — октябрь 2025 г.<br/><br/>Чем раньше дата окончания поддержки, тем раньше потребуется выполнить другую миграцию. Апрель 2023 г. намного ближе, чем вы думаете!|
|Путь миграции в Exchange 2013 или 2016|Путь миграции из Exchange 2010 в более новую версию такой же, как при выборе Exchange 2013 или Exchange 2016:<br/><br/>Установите Exchange 2013 или 2016 в существующую организацию Exchange 2010.<br/>Перемещение служб и другой инфраструктуры в Exchange 2013 или 2016.<br/>Переместите почтовые ящики и общедоступные папки в Exchange 2013 или 2016 списание оставшихся серверов Exchange 2010.|
|Сосуществование версий|При переходе на Exchange 2013 или Exchange 2016 можно установить в существующую организацию Exchange 2010 версию. Это позволяет установить один или несколько серверов Exchange 2013 или Exchange 2016 и миграцию.|
|Серверное оборудование|Требования к серверу к оборудованию были изменены с Exchange 2010. Убедитесь, что оборудование совместимо. Дополнительные информацию о требованиях к оборудованию для каждой версии можно узнать здесь:<br/><br/>[Требования к системе для Exchange 2016](https://docs.microsoft.com/Exchange/plan-and-deploy/system-requirements?view=exchserver-2016)<br/>[Требования к системе для Exchange 2013](https://docs.microsoft.com/Exchange/exchange-2013-system-requirements-exchange-2013-help)<br/><br/>Благодаря значительному повышению производительности Exchange и увеличению вычислительной мощности и емкости хранилища на новых серверах, скорее всего, потребуется меньше серверов для поддержки одинакового количества почтовых ящиков.|
|Версия операционной системы|Минимальные поддерживаемые версии операционной системы для каждой версии:<br/><br/>Exchange 2016 — Windows Server 2012<br/>Exchange 2013 — Windows Server 2008 R2 SP1<br/><br/>Дополнительные сведения о поддержке операционной системы можно найти в [матрице поддержки Exchange.](https://docs.microsoft.com/exchange/plan-and-deploy/supportability-matrix)|
|Функциональный уровень леса Active Directory|Минимальные поддерживаемые функциональные уровни леса Active Directory для каждой версии:<br/><br/>Exchange 2016 — Windows Server 2008 R2 SP1<br/>Exchange 2013 — Windows Server 2003<br/><br/>Дополнительные сведения о поддержке функционального уровня леса можно найти в матрице поддержки [Exchange.](https://docs.microsoft.com/exchange/plan-and-deploy/supportability-matrix)|
|Версии клиентов Office|Минимальные поддерживаемые версии клиентов Office для каждой версии:<br/><br/>Exchange 2016 — Office 2010 (с последними обновлениями)<br/>Exchange 2013 — Office 2007 с sp3<br/><br/>Дополнительные сведения о поддержке клиентов Office можно найти в [матрице поддержки Exchange.](https://docs.microsoft.com/exchange/plan-and-deploy/supportability-matrix)|||

## <a name="what-if-i-need-help"></a>Что делать, если мне нужна помощь?

При переходе на Microsoft 365 вы можете использовать нашу службу Microsoft FastTrack. FastTrack предоставляет лучшие методики, инструменты и ресурсы, которые повысят вашу миграцию в Microsoft 365. Лучше всего то, что у вас есть инженер службы поддержки, который сможет перейти от планирования и проектирования к переносу последнего почтового ящика. Дополнительные информацию о FastTrack см. [в microsoft FastTrack.](https://fasttrack.microsoft.com/)

Если во время миграции на Microsoft 365 возникают проблемы и вы не используете FastTrack или переход на более новую версию Exchange Server, воспользуйтесь некоторыми ресурсами:

- [Техническое сообщество](https://social.technet.microsoft.com/Forums/office/home?category=exchangeserver)
- [Служба поддержки клиентов](https://support.microsoft.com/gp/support-options-for-business)
