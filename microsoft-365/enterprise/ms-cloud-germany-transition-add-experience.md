# <a name="migration-phases-actions-and-impacts-for-the-migration-from-microsoft-cloud-deutschland-advanced"></a>Действия и последствия миграции из Microsoft Cloud Deutschland (дополнительные действия и последствия миграции) 

В следующих разделах приводится дополнительная информация о впечатлениях клиентов при переходе с Microsoft Cloud в Германии (Microsoft Cloud Deutschland) на службы Office 365 в новом регионе центра обработки данных в Германии.

## <a name="services"></a>Службы

### <a name="azure-ad"></a>Azure AD

| Этапы | Описание | Сфера применения | Влияние |
|:-------|:-----|:-------|:-------|
| Клиент Azure AD в Microsoft Cloud Deutschland, скопирован в службы Office 365. | Azure AD копирует клиент в службы Office 365. Идентификаторы клиентов и пользователей сохраняются. Вызовы служб Azure AD перенаправляются из Microsoft Cloud Deutschland в службы Office 365 и прозрачны для служб. | Все клиенты Office | - Запросы субъектов данных (DSR) общего регламента по защите данных (GDPR) выполняются с портала администрирования Azure для будущих запросов. Все устаревшие или не клиентские диагностические данные, которые есть в Microsoft Cloud Deutschland, удаляются не ранее чем через 30 дней. <br><br> - Клиенты, которые используют федераторские проверки подлинности со службами федерации Active Directory (AD FS), не должны вносить изменения в IS-адреса субъектов, используемые для всех проверки подлинности с локальной службой Active Directory во время миграции. Изменение URIS-адресов пользователей в домене приведет к сбоям проверки подлинности. ЮРИСовщика можно изменить непосредственно в AD FS  или  при преобразовании домена из управляемого в федераированный и наоборот. Мы рекомендуем клиентам не добавлять, удалять и не преобразовывать федераированный домен в перенесенном клиенте Azure AD. URIS-адреса выдавлили можно изменить после полного завершения миграции. <br><br> - Запросы многофакторной проверки подлинности (MFA), которые используют Microsoft Authenticator, отображаются в качестве guID пользователя, когда клиент копируется в службы Office 365. Запросы MFA будут выполняться ожидаемым образом, несмотря на такое поведение.  Учетные записи Microsoft Authenticator, активированные с помощью конечных точек служб Office 365, будут отображать имя основного пользователя (UPN).  Учетные записи, добавленные с помощью конечных точек Microsoft Cloud Deutschland, будут отображать user ObjectID, но будут работать как с конечными точками Microsoft Cloud Deutschland, так и с конечными точками служб Office 365.  |
| Установить authServer локально, указав на глобальную службу STS. | Это гарантирует проверку подлинности запросов пользователей, которые переносят в Microsoft Cloud Deutschland запросы доступности Exchange, нацеленные на гибридную локальной среду, для доступа к локальной службе. Аналогичным образом обеспечивается проверка подлинности запросов из локальной точки в конечные точки служб Office 365. | Клиенты Exchange Online с гибридными (локально) развертываниями | После завершения миграции Azure AD администратор локальной топологии Exchange (гибридной) должен добавить новую конечную точку службы проверки подлинности для служб Office 365. С помощью этой команды из Exchange PowerShell замените на ИД клиента вашей организации (который находится на портале `<TenantID>` Azure в Azure Active **Directory).** <br><br> - `New-AuthServer GlobalMicrosoftSts -AuthMetadataUrl https://accounts.accesscontrol.windows.net/<TenantId>/metadata/json/1` <br> <br> Невыполнеть эту задачу может привести к невозможности предоставления сведений о пользователях почтовых ящиков, которые были перенесены из Microsoft Cloud Deutschland в службы Office 365.  |
| Перенос ресурсов Azure. | Клиенты, которые используют ресурсы Office 365 и Azure (например, сеть, вычисления и хранилище), будут выполнять перенос ресурсов в экземпляр служб Office 365. Эта миграция отвечает за клиентов. Сообщения в Центре сообщений будут сигнализировать о начале. Миграция должна быть завершена до завершения работы организации Azure AD в среде служб Office 365. | Клиенты Azure | Сведения о миграции Azure см. в руководстве по миграции Azure, обзоре руководства по миграции [для Azure в Германии.](https://docs.microsoft.com/azure/germany/germany-migration-main) |

Пример: повторно зарегистрируйте версии приложений коллекции приложений по согласованию с поставщиками и партнерами, если они доступны. <br><br> - Указать все настраиваемые приложения, в которые используют API Graph, `graph.microsoft.de` на `graph.microsoft.com` . При необходимости также необходимо обновить другие API с измененными конечными точками. <br><br> - Измените все сторонние корпоративные приложения на перенаправление на конечные точки по всему миру.  | Все клиенты Office | Обязательное действие. Невыполнение этого может привести к сбою службы или программного клиента. |

После завершения миграции служб Exchange мы отмещаем конфигурацию локальной сети от Microsoft Cloud Deutschland. | Клиенты Exchange Online, которые запускают гибридное развертывание | - Обязательное действие. Перед началом миграции почтовых ящиков Exchange (с уведомлением за 5 или более дней) уведомите клиентов о том, что они должны остановить и удалить все перемещения почтовых ящиков.  В этом случае в запросах на перемещение от них будут отсчеты ошибки. <br><br> - После завершения миграции почтовых ящиков Exchange уведомите клиентов о том, что они могут возобновить входящие и отходящие перемещения. <br> Запуск **test-MigrationServerAvailabiilty**, служивского управления PowerShell, во время миграции Exchange из Microsoft Cloud Deutschland в службы Office 365 может не работать. Однако после завершения миграции она будет работать правильно. <br><br> Если после переноса почтовых ящиков у клиентов могут быть проблемы с учетными данными или авторизацией, пользователи могут повторно входить в учетные данные локального администратора в конечной точке миграции, задав их с помощью панели управления `Set-MigrationEndpoint endpointName -Credential $(Get-Credential)` Exchange (ECP).  |

### <a name="sharepoint-online"></a>SharePoint Online

| Этапы | Описание | Сфера применения | Влияние |
|:-------|:-----|:-------|:-------|
| Повторно опубликуем рабочий процесс SharePoint 2013. | В работе перед миграцией мы снизили количество процессов SharePoint 2013.
