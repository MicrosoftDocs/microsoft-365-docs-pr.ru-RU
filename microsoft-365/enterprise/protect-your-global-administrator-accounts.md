# <a name="protect-your-microsoft-365-global-administrator-accounts"></a>Защита учетных записей глобального администратора Microsoft 365

*Эта статья относится к Microsoft 365 корпоративный и Office 365 корпоративный.*

Нарушения безопасности подписки Microsoft 365, включая сбор информации и фишинговые атаки, обычно делаются путем взлома учетных данных учетной записи глобального администратора Microsoft 365. Безопасность в облаке — это партнерство между вами и Корпорацией Майкрософт:
  
- Облачные службы Майкрософт построены на основе доверия и безопасности. Корпорация Майкрософт предоставляет средства управления безопасностью и возможности для защиты данных и приложений.
    
- Вы владеете своими данными и удостоверениями, а также несете ответственность за их защиту, безопасность своих ресурсов и безопасность облачных компонентов, которые вы контролируете.
    
Корпорация Майкрософт предоставляет возможности для защиты вашей организации, но они эффективны только при их использовании. Если вы не используете их, вы можете быть уязвимыми для атак. Чтобы защитить учетные записи глобальных администраторов, корпорация Майкрософт поможет вам получить подробные инструкции по:
  
1. Создайте выделенные учетные записи глобального администратора Microsoft 365 и используйте их только при необходимости.
    
2. Настройте многофакторную проверку подлинности для выделенных учетных записей глобальных администраторов Microsoft 365 и используйте самый высокий уровень вторичной проверки подлинности.
    
> [!Note]
> Хотя эта статья посвящена учетным записям глобальных администраторов, следует рассмотреть возможность защиты дополнительных учетных записей с широкими разрешениями на доступ к данным в подписке, таких как учетные записи администратора eDiscovery, администратора безопасности или соответствия требованиям. <br > Учетную запись глобального администратора можно создать без добавления лицензий.
  
## <a name="step-1-create-dedicated-microsoft-365-global-administrator-accounts-and-use-them-only-when-necessary"></a>Этап 1. Создание выделенных учетных записей глобального администратора Microsoft 365 и их использование только при необходимости

Существует относительно небольшое количество административных задач, таких как назначение ролей учетным записям пользователей, которые требуют прав глобального администратора. Поэтому вместо использования повседневных учетных записей пользователей, для которых назначена роль глобального администратора, сделайте вот что:
  
1. Определите набор учетных записей пользователей, для которых назначена роль глобального администратора. Это можно сделать в Центре администрирования Microsoft 365 или с помощью следующей команды Azure Active (Azure AD) Directory PowerShell для Graph:
  
  ```powershell
  Get-AzureADDirectoryRole | where { $_.DisplayName -eq "Company Administrator" } | Get-AzureADDirectoryRoleMember | Ft DisplayName
  ```

2. Во sign into your Microsoft 365 subscription with a user account that has been assigned the global admin role.
    
3. Создайте до четырех выделенных учетных записей глобальных администраторов. **Используйте надежные пароли длиной не менее 12 символов.** Дополнительные [сведения см. в](https://support.microsoft.com/help/4026406/microsoft-account-create-a-strong-password) записи "Создание надежных паролей". Храните пароли новых учетных записей в надежном месте. 
    
4. Назначьте роль глобального администратора каждой новой выделенной учетной записи глобального администратора.
    
5. Выход из Microsoft 365.
    
6. Во sign in with one of the new dedicated global administrator user accounts.
    
7. Для каждой существующей учетной записи пользователя, которая была назначена роль глобального администратора с шага 1:
    
  - Удалите роль глобального администратора.
    
  - Назначьте роли администратора учетной записи, соответствующие его должности и ответственности. Дополнительные сведения о различных ролях администраторов в Microsoft 365 см. в [сведениях о ролях администраторов.](https://docs.microsoft.com/office365/admin/add-users/about-admin-roles)
    
8. Выход из Microsoft 365.
    
Результаты должны быть:
  
- Роль глобального администратора имеют только новые специальные учетные записи глобального администратора. Проверьте это с помощью следующей команды PowerShell:
    
  ```powershell
  Get-AzureADDirectoryRole | where { $_.DisplayName -eq "Company Administrator" } | Get-AzureADDirectoryRoleMember | Ft DisplayName
  ```

- Всем остальным повседневным учетным записям пользователей, которые управляют вашей подпиской, назначены роли администраторов, соответствующие должностным обязанностям этих пользователей.
    
С этого момента вы можете войти с помощью выделенных учетных записей глобальных администраторов только для задач, которые требуют прав глобального администратора. Все остальные задачи администрирования Microsoft 365 необходимо назначить другим административным ролям учетным записям пользователей.
  
> [!NOTE]
> Для этого требуются дополнительные действия, чтобы выйти из учетной записи обычного пользователя и войти с помощью выделенной учетной записи глобального администратора. Однако это необходимо делать только время от времени для операций глобального администратора. Подумайте, что для восстановления подписки На Microsoft 365 после нарушения учетной записи глобального администратора требуется намного больше действий.
  
## <a name="step-2-configure-multi-factor-authentication-for-your-dedicated-microsoft-365-global-administrator-accounts"></a>Шаг 2. Настройка многофакторной проверки подлинности для выделенных учетных записей глобального администратора Microsoft 365

Для многофакторной проверки подлинности (MFA) требуются дополнительные сведения помимо имени и пароля учетной записи. Microsoft 365 поддерживает эти дополнительные методы проверки:
  
- Приложение Microsoft Authenticator

- телефонный звонок;
    
- Код проверки, созданный случайным образом, отправленный через текстовое сообщение
    
- смарт-карта (виртуальная или физическая);
    
- биометрическое устройство.
    
>[!Note]
>Для организаций, которые должны соблюдать стандарты Национального института стандартов и технологий (NIST), использование телефонного вызова или дополнительных методов проверки на основе текстовых сообщений ограничено. Щелкните [здесь,](https://pages.nist.gov/800-63-FAQ/#q-b01) чтобы найти подробные сведения.
>

Если вы малый бизнес, использующий учетные записи пользователей, хранимые только в облаке (модель удостоверений только в облаке), настройте MFA для настройки [MFA](https://docs.microsoft.com/office365/admin/security-and-compliance/set-up-multi-factor-authentication) с помощью телефонного вызова или sms-кода проверки сообщений, отправленного на смартфон для каждой выделенной учетной записи глобального администратора.
    
Если вы крупной организации, использующей гибридную модель удостоверений Microsoft 365, у вас будет больше возможностей проверки. Если у вас уже есть инфраструктура безопасности для более строгого дополнительного метода проверки подлинности, настройте [MFA](https://docs.microsoft.com/microsoft-365/admin/security-and-compliance/set-up-multi-factor-authentication) и настройте каждую выделенную учетную запись глобального администратора для соответствующего метода проверки.
  
Если инфраструктура безопасности для нужного более строгого метода проверки не существует и не работает для Microsoft 365 MFA, мы настоятельно рекомендуем настроить выделенные учетные записи глобального администратора с MFA с помощью приложения Microsoft Authenticator, телефонного вызова или кода проверки текстового сообщения, отправленного на смартфон для учетных записей глобальных администраторов в качестве промежуточной меры безопасности. Не оставляйте выделенные учетные записи глобального администратора без дополнительной защиты, предоставляемой MFA.
  
Дополнительные сведения см. в [MFA для Microsoft 365.](https://docs.microsoft.com/microsoft-365/admin/security-and-compliance/multi-factor-authentication-microsoft-365)

## <a name="additional-protections-for-enterprise-organizations"></a>Дополнительная защита корпоративных организаций

Используйте эти дополнительные методы для обеспечения максимальной безопасности учетной записи глобального администратора и используемой конфигурации.
  
### <a name="privileged-access-workstation"></a>Рабочие станции с привилегированным доступом

Чтобы обеспечить как можно более безопасное выполнение задач с высокой привилегией, воспользуйтесь рабочей станцияю с привилегированным доступом (PAW). PAW — это выделенный компьютер, который используется только для выполнения конфиденциальных задач настройки, таких как конфигурация Microsoft 365, для выполнения которую требуется учетная запись глобального администратора. Поскольку этот компьютер не используется ежедневно для просмотра веб-страниц или электронной почты, он лучше защищен от атак и угроз в Интернете.
  
Инструкции по настройкам PAW см. в [https://aka.ms/cyberpaw](https://aka.ms/cyberpaw) .

Чтобы включить Azure PIM для вашего клиента Azure AD и учетных записей администраторов, см. [указания по настройке PIM](https://docs.microsoft.com/azure/active-directory/active-directory-privileged-identity-management-configure).

Чтобы разработать комплексный стратегический план защиты привилегированного доступа от кибератак, см. статью [Защита привилегированного доступа для гибридных и облачных развертываний в Azure AD](https://docs.microsoft.com/azure/active-directory/admin-roles-best-practices).

### <a name="azure-ad-privileged-identity-management"></a>Azure AD Privileged Identity Management

Вместо того чтобы иметь постоянную роль глобального администратора, вы можете использовать Azure AD Privileged Identity Management (PIM) для выполнения по требованию и точного назначения роли глобального администратора, когда это необходимо.
  
Учетные записи глобальных администраторов перейдут от постоянных администраторов к подходящим администраторам. Роль глобального администратора неактивна, пока она не нужна кому-либо. Затем необходимо выполнить активацию, чтобы добавить роль глобального администратора в учетную запись глобального администратора в течение заранее определенного времени. По истечении этого времени PIM удаляет роль глобального администратора из учетной записи глобального администратора.
  
Использование PIM и этот процесс значительно сокращает время, в которое учетные записи глобальных администраторов будут уязвимы для атак и использования злоумышленниками.

PIM доступна в службе Azure Active Directory Premium P2, которая входит в состав Microsoft 365 E5. Кроме того, вы можете приобрести отдельные лицензии Azure Active Directory Premium P2 для ваших учетных записей администраторов.
  
Дополнительные сведения см. в [службе Azure AD Privileged Identity Management.](https://docs.microsoft.com/azure/active-directory/active-directory-privileged-identity-management-configure)
  

### <a name="privileged-access-management"></a>Управление привилегированным доступом

Управление привилегированным доступом включается посредством настройки политик, определяющим своевременный доступ для действий на основе задач в вашем клиенте. Вашей организации может понадобиться защита от нарушений с использованием существующих привилегированных учетных записей администратора с постоянным доступом к конфиденциальным данным или доступом к критическим параметрам конфигурации. Например, можно настроить политику управления привилегированным доступом таким образом, чтобы требовалось явное утверждение для доступа к параметрам почтового ящика организации и их изменения в клиенте.

На этом этапе вы включаете управление привилегированным доступом в вашем клиенте и настраиваете политики привилегированного доступа, которые обеспечивают дополнительную защиту для доступа на основе задач к данным и параметрам конфигурации для вашей организации. Существуют три основных этапа настройки управления привилегированным доступом в вашей организации:

- создание группы утверждающего;
- включение привилегированного доступа;
- создание политик утверждения.

Управление привилегированным доступом позволяет организации работать без постоянных привилегий и обеспечивать уровень защиты от уязвимостей, возникающих из-за такого постоянного административного доступа. Для привилегированного доступа требуются утверждения для выполнения любой задачи с определенной политикой утверждения. Пользователи, которые должны выполнять задачи, включенные в политику утверждения, должны запрашивать и получать утверждение доступа.

Чтобы включить управление привилегированным доступом, см. ["Настройка управления привилегированным доступом".](https://docs.microsoft.com/office365/securitycompliance/privileged-access-management-configuration)

Дополнительные сведения см. в [руководстве по управлению привилегированным доступом.](https://docs.microsoft.com/office365/securitycompliance/privileged-access-management-overview)

### <a name="security-information-and-event-management-siem-software-for-microsoft-365-logging"></a>Программное обеспечение SIEM для ведения журнала Microsoft 365

Программное обеспечение SIEM, выполняемого на сервере, выполняет анализ оповещений и событий безопасности в режиме реального времени, созданных приложениями и сетевым оборудованием. Чтобы разрешить серверу SIEM включать оповещения и события системы безопасности Microsoft 365 в функции анализа и отчетности, интегрируем Azure AD в seIM. См. [введение в интеграцию с журналом Azure.](https://docs.microsoft.com/azure/security/security-azure-log-integration-overview)
