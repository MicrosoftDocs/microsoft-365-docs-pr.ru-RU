# <a name="protect-your-microsoft-365-global-administrator-accounts"></a>Защита учетных записей Microsoft 365 администратора

*Эта статья относится к Microsoft 365 корпоративный и Office 365 корпоративный.*

Нарушения безопасности подписки Microsoft 365, включая сбор информации и фишинговые атаки, как правило, нарушают учетные данные учетной записи Microsoft 365 администратора. Безопасность в облаке — это партнерство между вами и Корпорацией Майкрософт:
  
- Облачные службы Майкрософт построены на основе доверия и безопасности. Корпорация Майкрософт предоставляет вам средства управления безопасностью и возможности, которые помогут защитить данные и приложения.
    
- Вы владеете своими данными и удостоверениями, а также несете ответственность за их защиту, безопасность собственных ресурсов и безопасность подконтрольных вам облачных компонентов.
    
Корпорация Майкрософт предоставляет возможности для защиты организации, но они эффективны только при их использовании. Если вы не используете их, вы можете быть уязвимы для атаки. Чтобы защитить глобальные учетные записи администратора, корпорация Майкрософт поможет вам с подробными инструкциями по:
  
1. Создайте выделенные Microsoft 365 учетные записи администратора и используйте их только при необходимости.
    
2. Настройте многофакторную проверку подлинности для Microsoft 365 учетных записей глобального администратора и используйте сильнейшую форму вторичной проверки подлинности.
    
> [!Note]
> Хотя в этой статье основное внимание уделялось учетным записям глобального администратора, следует также рассмотреть вопрос о том, следует ли также защищать дополнительные учетные записи с широкими разрешениями на доступ к данным в подписке, например администратору eDiscovery или учетным записям администратора безопасности или администратора соответствия требованиям. <br > Глобальная учетная запись администратора может быть создана без добавления лицензий.
  
## <a name="step-1-create-dedicated-microsoft-365-global-administrator-accounts-and-use-them-only-when-necessary"></a>Этап 1. Создание выделенных Microsoft 365 учетных записей администратора и их использование только при необходимости

Существует относительно небольшое количество административных задач, например назначение ролей учетным записям пользователей, которые требуют глобальных привилегий администратора. Поэтому вместо использования повседневных учетных записей пользователей, которые были назначены глобальной роли администратора, делайте следующие действия:
  
1. Определите набор учетных записей пользователей, которые назначены глобальной роли администратора. Это можно сделать в центре администрирования Microsoft 365 или в следующей командной командной команде Azure Active (Azure AD) Directory PowerShell для Graph:
  
  ```powershell
  Get-AzureADDirectoryRole | where { $_.DisplayName -eq "Global Administrator" } | Get-AzureADDirectoryRoleMember | Ft DisplayName
  ```

2. Вопишите Microsoft 365 подписку с учетной записью пользователя, назначенной глобальной роли администратора.
    
3. Создайте не более четырех учетных записей пользователей глобального администратора. **Используйте надежные пароли длиной не менее 12 символов.** Дополнительные [сведения см. в "Создание](https://support.microsoft.com/help/4026406/microsoft-account-create-a-strong-password) надежных паролей". Храните пароли для новых учетных записей в безопасном расположении. 
    
4. Назначение роли глобального администратора каждой из новых учетных записей пользователей глобального администратора.
    
5. Выпишитесь из Microsoft 365.
    
6. Во входе в одну из новых учетных записей пользователей глобального администратора.
    
7. Для каждой существующей учетной записи пользователя, назначенной глобальной роли администратора с шага 1:
    
  - Удаление роли глобального администратора.
    
  - Назначение ролей администратора учетной записи, соответствующих функции и ответственности этого пользователя. Дополнительные сведения о различных ролях администратора в Microsoft 365 см. в [дополнительных сведениях о ролях администратора.](/office365/admin/add-users/about-admin-roles)
    
8. Выпишитесь из Microsoft 365.
    
Результаты должны быть:
  
- Роль глобального администратора имеют только новые специальные учетные записи глобального администратора. Убедитесь в этом с помощью следующей команды PowerShell:
    
  ```powershell
  Get-AzureADDirectoryRole | where { $_.DisplayName -eq "Company Administrator" } | Get-AzureADDirectoryRoleMember | Ft DisplayName
  ```

- Всем остальным повседневным учетным записям пользователей, которые управляют вашей подпиской, назначены роли администраторов, соответствующие должностным обязанностям этих пользователей.
    
С этого момента вы вошел в выделенные учетные записи глобального администратора только для задач, которые требуют глобальных привилегий администратора. Все другие Microsoft 365 администрирования должны быть сделаны путем назначения других ролей администрирования учетным записям пользователей.
  
> [!NOTE]
> Это требует дополнительных действий для регистрации в повседневной учетной записи пользователя и вход с помощью выделенной учетной записи глобального администратора. Но это нужно делать только время от времени для глобальных операций администратора. Подумайте, что восстановление Microsoft 365 подписки после глобального нарушения учетной записи администратора требует гораздо больше действий.
  
## <a name="step-2-configure-multi-factor-authentication-for-your-dedicated-microsoft-365-global-administrator-accounts"></a>Этап 2. Настройка многофакторной проверки подлинности для Microsoft 365 учетных записей глобального администратора

Многофакторная проверка подлинности (MFA) требует дополнительных сведений за пределами имени и пароля учетной записи. Microsoft 365 поддерживает эти дополнительные методы проверки:
  
- Приложение Microsoft Authenticator

- телефонный звонок;
    
- Код проверки, созданный случайным образом, отправленный через текстовое сообщение
    
- смарт-карта (виртуальная или физическая);
    
- биометрическое устройство.
    
>[!Note]
>Для организаций, которые должны придерживаться стандартов Национального института стандартов и технологий (NIST), использование телефонного звонка или дополнительных методов проверки на основе текстовых сообщений ограничено. Щелкните [здесь](https://pages.nist.gov/800-63-FAQ/#q-b01) для получения сведений.
>

Если вы малый бизнес, использующий учетные записи пользователей, хранимые только в облаке (модель удостоверений только для облака), настройте MFA для настройки [MFA](/office365/admin/security-and-compliance/set-up-multi-factor-authentication) с помощью телефонного звонка или кода проверки текстовых сообщений, отправленных на смартфон для каждой отдельной учетной записи глобального администратора.
    
Если вы — крупная организация, использующая гибридную модель удостоверений Microsoft 365, у вас есть дополнительные параметры проверки. Если у вас уже есть инфраструктура безопасности для более сильного метода вторичной проверки подлинности, настройте [MFA](../admin/security-and-compliance/set-up-multi-factor-authentication.md) и настройте каждую выделенную учетную запись глобального администратора для соответствующего метода проверки.
  
Если инфраструктура безопасности для желаемого более сильного метода проверки не на месте и не функционирует для Microsoft 365 MFA, настоятельно рекомендуем настроить выделенные учетные записи глобального администратора в MFA с помощью приложения Microsoft Authenticator, телефонного звонка или кода проверки текстовых сообщений, отправленных на смартфон для учетных записей глобального администратора в качестве промежуточной меры безопасности. Не оставляйте выделенные учетные записи глобального администратора без дополнительной защиты, предоставляемой MFA.
  
Дополнительные сведения см. [в Microsoft 365.](../admin/security-and-compliance/multi-factor-authentication-microsoft-365.md)

## <a name="additional-protections-for-enterprise-organizations"></a>Дополнительные средства защиты для корпоративных организаций

Используйте эти дополнительные методы для обеспечения максимальной безопасности учетной записи глобального администратора и конфигурации, которую вы выполняете с ее помощью.
  
### <a name="privileged-access-workstation"></a>Рабочие станции с привилегированным доступом

Чтобы обеспечить максимальное безопасность выполнения высоко привилегированных задач, используйте рабочие станции с привилегированным доступом (PAW). PAW — это выделенный компьютер, который используется только для конфиденциальных задач конфигурации, например Microsoft 365 конфигурации, которая требует глобальной учетной записи администратора. Поскольку этот компьютер не используется ежедневно для просмотра интернета или электронной почты, он лучше защищен от атак и угроз в Интернете.
  
Инструкции по настройкам PAW [https://aka.ms/cyberpaw](/security/compass/privileged-access-devices) см. в .

Чтобы включить Azure PIM для вашего клиента Azure AD и учетных записей администраторов, см. [указания по настройке PIM](/azure/active-directory/active-directory-privileged-identity-management-configure).

Чтобы разработать комплексный стратегический план защиты привилегированного доступа от кибератак, см. статью [Защита привилегированного доступа для гибридных и облачных развертываний в Azure AD](/azure/active-directory/admin-roles-best-practices).

### <a name="azure-ad-privileged-identity-management"></a>Azure AD Privileged Identity Management

Вместо того чтобы навсегда надолго назадать глобальным учетным записям администратора роль глобального администратора, вы можете использовать Azure AD управление привилегированными пользователями (PIM), чтобы включить назначение роли глобального администратора по требованию, когда это необходимо.
  
Учетные записи глобального администратора перейдут от постоянных администраторов до подходящих администраторов. Роль глобального администратора неактивна до тех пор, пока она не будет нужна кому-либо. Затем завершится процесс активации, чтобы добавить роль глобального администратора в глобальную учетную запись администратора за заранее определенное время. По истечении времени PIM удаляет роль глобального администратора из глобальной учетной записи администратора.
  
Использование PIM и этот процесс значительно сокращает время, которое ваши учетные записи глобального администратора уязвимы для атак и использования вредоносными пользователями.

PIM доступна в службе Azure Active Directory Premium P2, которая входит в состав Microsoft 365 E5. Кроме того, вы можете приобрести отдельные лицензии Azure Active Directory Premium P2 для ваших учетных записей администраторов.
  
Дополнительные сведения см. в [управление привилегированными пользователями Azure AD.](/azure/active-directory/active-directory-privileged-identity-management-configure)
  

### <a name="privileged-access-management"></a>Управление привилегированным доступом

Управление привилегированным доступом включается посредством настройки политик, определяющим своевременный доступ для действий на основе задач в вашем клиенте. Вашей организации может понадобиться защита от нарушений с использованием существующих привилегированных учетных записей администратора с постоянным доступом к конфиденциальным данным или доступом к критическим параметрам конфигурации. Например, можно настроить политику управления привилегированным доступом таким образом, чтобы требовалось явное утверждение для доступа к параметрам почтового ящика организации и их изменения в клиенте.

На этом этапе вы включаете управление привилегированным доступом в вашем клиенте и настраиваете политики привилегированного доступа, которые обеспечивают дополнительную защиту для доступа на основе задач к данным и параметрам конфигурации для вашей организации. Существуют три основных этапа настройки управления привилегированным доступом в вашей организации:

- создание группы утверждающего;
- включение привилегированного доступа;
- создание политик утверждения.

Управление привилегированным доступом позволяет организации работать с нулевыми привилегиями и обеспечивать защиту от уязвимостей, возникающих из-за такого постоянного административного доступа. Привилегированный доступ требует утверждений для выполнения любой задачи, которая имеет связанную политику утверждения. Пользователи, нуждающиеся в выполнении задач, включенных в политику утверждения, должны запрашивать и получать разрешение доступа.

Чтобы включить управление привилегированным доступом, см. в [перенаправлении Настройка управления привилегированным доступом.](/office365/securitycompliance/privileged-access-management-configuration)

Дополнительные сведения см. в [руководстве привилегированным доступом.](/office365/securitycompliance/privileged-access-management-overview)

### <a name="security-information-and-event-management-siem-software-for-microsoft-365-logging"></a>Программное обеспечение для управления сведениями о безопасности и событиями (SIEM) для Microsoft 365 ведения журнала

Чтобы разрешить вашему серверу SIEM включать Microsoft 365 оповещения и события безопасности в функции анализа и отчетности, включите Azure AD в seIM. См. [введение в Интеграция журналов данных Azure.](/azure/security/security-azure-log-integration-overview)
