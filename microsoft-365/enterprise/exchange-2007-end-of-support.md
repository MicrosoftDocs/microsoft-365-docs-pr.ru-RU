# <a name="exchange-2007-end-of-support-roadmap"></a>План действий после прекращения поддержки Exchange 2007;

*Эта статья относится к Microsoft 365 корпоративный и Office 365 корпоративный.*

Exchange Server 2007 г. поддержка была достигнута в апреле 2017 г. Если вы еще не начали миграцию с Exchange 2007 г. Microsoft 365, Office 365 или Exchange 2016 г., настало время приступить к планированию.
  
## <a name="what-does-end-of-support-mean"></a>Что означает *конец поддержки?*

Exchange Server, как и почти все продукты Майкрософт, имеет жизненный цикл поддержки, в ходе которого мы предоставляем новые функции, исправления ошибок, исправления безопасности и так далее. Этот жизненный цикл обычно длится 10 лет с начального выпуска продукта. Конец жизненного цикла называется завершением поддержки продукта. С Exchange 2007 г. 11 апреля 2017 г. корпорация Майкрософт больше не предоставляет:
  
- Техническая поддержка проблем, которые могут возникнуть.
    
- Ошибка устраняет проблемы, которые могут повлиять на стабильность и доступность сервера.
    
- Устранение уязвимостей, которые могут сделать сервер уязвимым для нарушений безопасности.
    
- Обновления часовой зоны.
    
Установка Exchange 2007 года будет продолжать работать после даты окончания поддержки. Но так как новых обновлений и поддержки нет, настоятельно рекомендуется как можно скорее перенести Exchange 2007 г.
  
Дополнительные сведения о Office 2007 серверов, близких к концу поддержки, см. в Office обновления с серверов и продуктов [2007 г.](upgrade-from-office-2007-servers-and-products.md)
  
## <a name="what-are-my-options"></a>Какие у меня варианты?

Вы получаете перечисленные ниже возможности.
  
- Миграция в Microsoft 365 с помощью переноса, поэтапной или гибридной миграции.
    
- Перенос серверов Exchange 2007 г. на более новую версию Exchange на локальном сервере.
    
В следующих разделах подробно изучаются все параметры.
  
### <a name="migrate-to-microsoft-365"></a>Миграция в Microsoft 365

Перенос электронной почты в Microsoft 365 является наилучшим и простым вариантом, который поможет завершить развертывание Exchange 2007 г. При переносе Microsoft 365 можно сделать один переход из 10-летней технологии в современные функции, в том числе:
  
- Возможности соответствия требованиям, такие как политики хранения, In-Place и судебное удержание, хранение электронных данных на месте и другие
    
- Группы Microsoft 365
    
- Сортировка почты
    
- MyAnalytics
    
- API REST для программного доступа к электронной почте, календарям, контактам и так далее
    
Microsoft 365 также получает новые функции и возможности, поэтому вы и ваши пользователи обычно можете начать использовать их сразу. И вам не придется беспокоиться о:
  
- Приобретение и обслуживание оборудования.
    
- Оплата тепла и охлаждения серверов.
    
- Ведение в курсе исправлений безопасности, продуктов и часового пояса.
    
- Ведение хранилища и программного обеспечения в соответствии с требованиями соответствия требованиям.
    
- Обновление до новой версии Exchange. С Microsoft 365 вы всегда в последней версии Exchange.
    
#### <a name="how-should-i-migrate-to-microsoft-365"></a>Как перейти на Microsoft 365?

- Количество мест или почтовых ящиков, которые необходимо переместить.
- Как долго вы хотите, чтобы миграция длилась.
- Нужна ли вам бесшовная интеграция между локальной установкой и Microsoft 365 во время миграции.

#### <a name="cutover-migration"></a>Прямая миграция

При переносе переключельных данных все почтовые ящики, группы рассылки, контакты и так далее переносят все почтовые ящики в Microsoft 365 в предварительно предварительное время. После завершения миграции вы закрываете локальное Exchange и начинаете использовать Microsoft 365 исключительно.
  
Миграция отсечий отлично подходит для небольших организаций, которые не имеют много почтовых ящиков, хотят быстро Microsoft 365 и не хотят иметь дело с некоторыми сложностями других методов. Но он должен быть завершен в неделю или меньше, и это требует от пользователей, чтобы перенастроить свои Outlook профилей. Миграция перереза может обрабатывать до 2000 почтовых ящиков, но мы настоятельно рекомендуем использовать его для переноса не более 150 почтовых ящиков. Если вы попробуете перенести больше, у вас может не оказаться времени для переноса всех почтовых ящиков до истечения срока, и сотрудники службы поддержки ИТ могут быть перегружены запросами на помощь пользователям в перенастройке Outlook.
  
Если вы думаете о переносе переключе-
  
- Microsoft 365 необходимо подключиться к серверам 2007 Exchange 2007 с помощью Outlook по TCP port 443.
    
- Все почтовые ящики на месте будут перемещены в Microsoft 365.
    
- Вам понадобится учетная запись локального администратора, которая имеет доступ к почтовым ящикам пользователей.
    
- Принятые Exchange 2007 г., которые вы хотите использовать в Microsoft 365, должны быть добавлены в качестве проверенных доменов в службе.
    
- Между началом миграции и началом этапа завершения Microsoft 365 периодически синхронизируются почтовые ящики Microsoft 365 и локально. Это позволяет завершить миграцию, не беспокоясь о том, что электронная почта останется в локальном почтовом ящике.
    
- Пользователи получат новые временные пароли для Microsoft 365 учетных записей. При первом входе в почтовый ящик им потребуется изменить пароль.
    
- Вам потребуется лицензия Microsoft 365, которая включает Exchange Online для каждого переносимый почтовый ящик пользователя.
    
- Пользователям потребуется настроить новый профиль Outlook на каждом из своих устройств и скачать электронную почту снова. Количество электронной почты, которую Outlook будет скачивать, может отличаться. Дополнительные сведения см. [в сообщении Change how much mail to keep offline.](https://support.office.com/article/Change-how-much-mail-to-keep-offline-f3a1251c-6dd5-4208-aef9-7c8c9522d633?ui=en-US&amp;rs=en-US&amp;ad=US&amp;fromAR=1)

#### <a name="staged-migration"></a>Поэтапная миграция

При поэтапной миграции у вас есть несколько сотен или нескольких тысяч почтовых ящиков, которые необходимо перенести в Microsoft 365, для завершения миграции потребуется неделю или больше, и не требуется никаких расширенных функций гибридной миграции, таких как общие сведения о календаре free/busy.
  
Этапная миграция отлично подходит для организаций, которые должны перенести свои почтовые ящики в Microsoft 365, но все же планируют завершить миграцию в течение нескольких недель. Вы можете перенести почтовые ящики пакетами. Вы управляете количеством и переносом почтовых ящиков в данный момент времени. Например, можно пакетно пакетить почтовые ящики пользователей в одном отделе, чтобы убедиться, что все они перемещаются одновременно. Или можно оставить почтовые ящики руководителей до последней партии. Как и в переносах отсеяния, пользователям потребуется воссоздать Outlook профилей.
  
Если вы думаете о поэтапной миграции, рассмотрим ниже:
  
- Microsoft 365 необходимо подключиться к серверам 2007 Exchange с помощью Outlook по TCP port 443.
    
- Вам понадобится учетная запись локального администратора, которая имеет доступ к почтовым ящикам пользователей.
    
- Принятые Exchange 2007 г., которые вы планируете использовать в Microsoft 365, должны быть добавлены в качестве проверенных доменов в службе.
    
- Необходимо создать CSV-файл с полным именем и адресом электронной почты каждого почтового ящика, который планируется перенести пакетом. Кроме того, необходимо включить новый пароль для каждого переносимго почтового ящика и отправить его каждому пользователю. Пользователю будет предложено изменить пароль при первом входе в новый Microsoft 365 почтовый ящик.
    
- Между началом пакета миграции и началом этапа завершения Microsoft 365 периодически синхронизируются Microsoft 365 и локального почтового ящика, включенных в пакет. Это позволяет завершить миграцию, не беспокоясь о том, что электронная почта останется в локальном почтовом ящике.
    
- Вам потребуется лицензия Microsoft 365, которая включает Exchange Online для каждого переносимый почтовый ящик пользователя.
    
- Пользователям потребуется настроить новый профиль Outlook на каждом из своих устройств и скачать электронную почту снова. Количество электронной почты, которую Outlook будет скачивать, может отличаться. Дополнительные сведения см. [в сообщении Change how much mail to keep offline.](https://support.office.com/article/Change-how-much-mail-to-keep-offline-f3a1251c-6dd5-4208-aef9-7c8c9522d633?ui=en-US&amp;rs=en-US&amp;ad=US&amp;fromAR=1)

#### <a name="full-hybrid"></a>Полный гибрид

При полной гибридной миграции в вашей организации имеется несколько сотен, до десятков тысяч почтовых ящиков, и вы хотите переместить некоторые или все из них в Microsoft 365. Поскольку эти миграции, как правило, являются более длительными, гибридные миграции делают возможным:
  
- Покажите пользователям на локальном сайте сведения о свободном или загруженном календаре для пользователей в Microsoft 365 и наоборот.
    
- См. единый глобальный адресный список, содержащий получателей как в локальном, так и в Microsoft 365.
    
- Просмотр полных Outlook свойств получателей для всех пользователей независимо от того, являются ли они локальной или Microsoft 365.
    
- Безопасное сообщение электронной почты между Exchange серверами и Microsoft 365 TLS и сертификатами.
    
- Обрабатывайте сообщения, отправленные между Exchange серверами и Microsoft 365 как внутренние, что позволяет:
    
  - Должны правильно оцениваться и обрабатываться агентами транспорта и соответствия требованиям, нацеленными на внутренние сообщения.
    
  - Обход фильтров от нежелательной почты.
    
Полная гибридная миграция лучше всего для организаций, которые рассчитывают оставаться в гибридной конфигурации в течение многих месяцев или более. Вы получите функции, перечисленные ранее в этом разделе, а также синхронизацию каталогов, более интегрированные функции соответствия требованиям, а также возможность перемещения почтовых ящиков в Microsoft 365 и из него с помощью интерактивных ходов почтовых ящиков. Microsoft 365 становится расширением локальной организации.
  
Если вы думаете о полной гибридной миграции, рассмотрим ниже:
  
- Полная гибридная миграция не подходит всем типам организаций. Из-за сложности полных гибридных миграций организации с менее чем несколькими сотнями почтовых ящиков обычно не видят преимуществ, которые оправдывают усилия и затраты, необходимые для их настройка. Если это похоже на организацию, рекомендуется вместо этого рассмотреть вопрос о переносе или поэтапной миграции.
    
- Чтобы выступать в качестве "гибридного сервера Exchange 2013 г., в организации Exchange 2007 г., необходимо развернуть по крайней мере один сервер Exchange 2013 г. Этот сервер будет взаимодействовать с Microsoft 365 от имени Exchange серверов 2007 года.
    
- Microsoft 365 необходимо подключиться к "гибридному серверу" с помощью Outlook по TCP port 443.
    
- Необходимо настроить синхронизацию каталогов с помощью Azure Active Directory (Azure AD) Подключение между локальной серверами Active Directory и Microsoft 365.
    
- Пользователи смогут войти в свой почтовый ящик Microsoft 365 с тем же именем пользователя и паролем, что и при входе в локализованную сеть. (Для этой функции требуется Подключение Azure AD с синхронизацией паролей и/или службами Федерации Active Directory.)
    
- Вам потребуется лицензия Microsoft 365, которая включает Exchange Online для каждого переносимый почтовый ящик пользователя.
    
- Пользователям не нужно устанавливать новый профиль Outlook на большинстве своих устройств, хотя некоторым старым телефонам с Android может потребоваться новый профиль. Пользователям не придется перезагружать электронную почту.

### <a name="migrate-to-a-newer-version-of-exchange-server"></a>Миграция в более новую версию Exchange Server

Мы твердо верим, что вы можете добиться наилучшего качества и пользовательского интерфейса, переехав в Microsoft 365. Но мы также понимаем, что некоторые организации должны хранить электронную почту на месте. Это может быть из-за нормативных требований, чтобы гарантировать, что данные не хранятся в центре обработки данных, расположенном в другой стране, или аналогичные. Если вы решите сохранить электронную почту локально, вы можете перенести среду 2007 Exchange 2007 на Exchange 2010, Exchange 2013 или Exchange 2016.
  
Если вы не можете перейти в Microsoft 365, рекомендуем перейти в Exchange 2016 г. Exchange 2016 включает все функции предыдущих выпусков Exchange. Он также наиболее тесно совпадает с опытом, доступным с Microsoft 365, хотя некоторые функции доступны только в Microsoft 365. Ознакомьтесь с несколькими недостающими вещами:
  
|**Exchange выпуска**|**Компоненты**|
|:-----|:-----|
|Exchange 2010  <br/> | Role-Based управления доступом (разрешения без acLs)  <br/>  Политики почтовых ящиков Outlook Web App  <br/>  Возможность совместной работы и делегирования календарей между организациями  <br/> |
|Exchange 2013  <br/> | *Функции Exchange 2010 и ...*  <br/>  Упрощенная архитектура, снизив число ролей сервера до трех (почтовый ящик, клиентский доступ, edge transport)  <br/>  Политики предотвращения потери данных (DLP), которые помогают предотвратить утечку конфиденциальной информации  <br/>  Улучшенная Outlook Web App  <br/> |
|Exchange 2016  <br/> | *Функции Exchange 2013 и ...*  <br/>  Дополнительные упрощенные роли сервера только для почтовых ящиков и edge transport  <br/>  Улучшенная DLP вместе с интеграцией с SharePoint  <br/>  Повышение устойчивости баз данных  <br/>  Совместная работа с документами в Интернете |
   
#### <a name="which-version-should-i-migrate-to"></a>В какую версию следует перейти?

Мы рекомендуем сначала предположить, что вы будете Exchange 2016. Затем используйте следующую информацию, чтобы подтвердить свое предположение или исключить Exchange 2016 г. Если вы не можете перейти на Exchange 2016 г., сделайте то же самое с Exchange 2013 г. и т. д.
  
|**Фактор**|**Дополнительные сведения**|
|:-----|:-----|
|Окончание дат поддержки  <br/> | Как Exchange 2007 г., каждая версия Exchange имеет свою дату окончания поддержки:  <br/> *Exchange 2010* — январь 2020 г.  <br/> *Exchange 2013* — апрель 2023 г.  <br/> *Exchange 2016* — октябрь 2025 г.  <br/>  Чем раньше закончится поддержка, тем быстрее потребуется выполнить другую миграцию.<br/> |
|Путь миграции Exchange 2010 и 2013 годах.  <br/> |Вот общие этапы миграции в Exchange 2010 или Exchange 2013 г.:  <br/> - Установите Exchange 2010 или 2013 г. в существующую организацию Exchange 2007 г. <br/>- Перемещение служб и другой инфраструктуры Exchange 2010 или 2013.<br/>- Перемещение почтовых ящиков и общедоступных папок в Exchange 2010 или 2013.<br/>- Вывод из эксплуатации Exchange серверов 2007 г. |
|Путь миграции Exchange 2016 г.  <br/> |Вот общие этапы переноса на Exchange 2016 г.:  <br/> - Установите Exchange 2013 г. в существующую организацию Exchange 2007 г.<br/>- Перемещение служб и другой инфраструктуры в Exchange 2013 г.<br/>- Перемещение почтовых ящиков и общедоступных папок в Exchange 2013 г.<br/>- Вывод из эксплуатации Exchange серверов 2007 г.<br/>- Установите Exchange 2016 г. в существующую организацию Exchange 2013 г.<br/>- Перемещение почтовых ящиков, общедоступных папок, служб и другой инфраструктуры Exchange 2016 г. (порядок не имеет значения). Вывод из эксплуатации Exchange серверов 2013 г.<br/><br/> **Примечание:** Миграция с Exchange 2013 г. на Exchange 2016 г. проста. У двух версий практически одинаковые требования к оборудованию, и эти версии очень совместимы. Таким образом, вы можете восстановить сервер, купленный Exchange 2013 г., и установить на Exchange 2016 г. Для перемещения почтовых ящиков в Интернете большинство пользователей даже не заметит, что их почтовый ящик был перемещен с сервера, а затем обратно после того, как вы перестроили его с Exchange 2016.           |
|Сосуществование версий  <br/> | При миграции в ...  <br/> **Exchange 2016 г.:** Exchange 2016 г. не может быть установлена в организации, которая включает сервер 2007 Exchange 2007 г. Сначала необходимо перейти на Exchange 2010 или 2013 годы (мы настоятельно рекомендуем Exchange 2013), удалить все серверы Exchange 2007 г., а затем перейти на Exchange 2016 г.  <br/> **Exchange 2010 или Exchange 2013:** Вы можете установить Exchange 2010 или Exchange 2013 г. в существующую организацию Exchange 2007 г. Это позволяет установить один или несколько серверов Exchange 2010 или 2013 и выполнить миграцию.  <br/> |
|Серверное оборудование  <br/> | Требования к оборудованию сервера изменились с Exchange 2007 г. Убедитесь, что оборудование совместимо. Дополнительные сведения см. в следующих статьях:  <br/> [Exchange 2016 г. Требования к системе](/Exchange/plan-and-deploy/system-requirements) <br/> [Exchange 2013 г. Требования к системе](/exchange/exchange-2013-system-requirements-exchange-2013-help) <br/> [Exchange 2010 г. Требования к системе](/previous-versions/office/exchange-server-2010/aa996719(v=exchg.141)) <br/>  Вы увидите, что значительные улучшения Exchange производительности и увеличение вычислительной мощности и емкости хранилища на новых серверах означают, что для поддержки одинакового числа почтовых ящиков потребуется меньше серверов.  <br/> |
|Версия операционной системы  <br/> | Минимальные поддерживаемые версии операционной системы для каждой версии:  <br/> **Exchange 2016** - Windows Server 2012  <br/> **Exchange 2013** — Windows 2008 R2 SP1  <br/> **Exchange 2010** — Windows 2008 SP2  <br/>  Дополнительные сведения о поддержке операционной системы можно найти [в Exchange Supportability Matrix.](/Exchange/plan-and-deploy/supportability-matrix)  <br/> |
|Функциональный уровень леса Active Directory  <br/> | Минимальные поддерживаемые функциональные уровни леса Active Directory для каждой версии:  <br/> **Exchange 2016** Windows 2008 R2 SP1  <br/> **Exchange 2013** Windows Server 2003  <br/> **Exchange 2010** Windows Server 2003  <br/>  Дополнительные сведения о поддержке функционального уровня леса можно найти [в Exchange Supportability Matrix.](/Exchange/plan-and-deploy/supportability-matrix)  <br/> |
|Office клиентских версий  <br/> | Минимальные поддерживаемые Office для каждой версии:  <br/> **Exchange 2016** — Office 2010 (с последними обновлениями)  <br/> **Exchange 2013** — Office 2007 SP3  <br/> **Exchange 2010** — Office 2003 г.  <br/>  Дополнительные сведения о поддержке Office в [Exchange Supportability Matrix.](/Exchange/plan-and-deploy/supportability-matrix)  <br/> |
