# <a name="exchange-2007-end-of-support-roadmap"></a>План действий после прекращения поддержки Exchange 2007;

*Эта статья относится к Microsoft 365 корпоративный и Office 365 корпоративный.*

Exchange Server 2007 г. поддержка была достигнута в апреле 2017 г. Если вы еще не начали миграцию с Exchange 2007 на Microsoft 365, Office 365 или Exchange 2016, самое время приступить к планированию.
  
## <a name="what-does-end-of-support-mean"></a>Что означает *окончание поддержки?*

Exchange Server, как и почти все продукты Майкрософт, имеет жизненный цикл поддержки, в течение которого мы предоставляем новые функции, исправления ошибок, исправления безопасности и так далее. Этот жизненный цикл обычно длится 10 лет с начального выпуска продукта. Завершение этого жизненного цикла называется завершением поддержки продукта. С момента окончания поддержки Exchange 2007 11 апреля 2017 г. корпорация Майкрософт больше не предоставляет:
  
- Техническая поддержка для проблем, которые могут возникнуть.
    
- Исправления ошибок, которые могут повлиять на стабильность и доступность сервера.
    
- Исправления безопасности для уязвимостей, которые могут сделать сервер уязвимым к нарушению безопасности.
    
- Обновления часовой пояс.
    
Установка Exchange 2007 продолжится после даты окончания поддержки. Но так как новые обновления и поддержка не существуют, настоятельно рекомендуется как можно скорее перейти с Exchange 2007.
  
Дополнительные сведения о серверах Office 2007, близких к концу поддержки, см. в под вопросе "Планирование обновления серверов и продуктов [Office 2007".](upgrade-from-office-2007-servers-and-products.md)
  
## <a name="what-are-my-options"></a>Какие у меня есть варианты?

Вы можете выполнить указанные ниже действия.
  
- Миграция в Microsoft 365 с помощью cutover, staged или hybrid migration.
    
- Перенос серверов Exchange 2007 в более новую версию Exchange на локальном сервере.
    
В следующих разделах каждый вариант более подробно изучается.
  
### <a name="migrate-to-microsoft-365"></a>Миграция в Microsoft 365

Перенос электронной почты в Microsoft 365 — это лучший и простой вариант, который поможет завершить развертывание Exchange 2007. При переходе на Microsoft 365 вы можете сделать один переход с 10-летней технологии на современные функции, в том числе:
  
- Такие возможности соответствия требованиям, как политики хранения, In-Place и хранение для судебного разбирательства, eDiscovery на месте и другие
    
- Группы Microsoft 365
    
- Сортировка почты
    
- MyAnalytics
    
- REST API для программного доступа к электронной почте, календарям, контактам и так далее
    
Microsoft 365 также в первую очередь получает новые функции и возможности, поэтому вы и ваши пользователи можете сразу же начать использовать их. И вам не придется беспокоиться о:
  
- Приобретение и обслуживание оборудования.
    
- Оплата за температуру и хладнокровие серверов.
    
- Постоянное исправление для системы безопасности, продукта и часового пояса.
    
- Поддержка хранилища и программного обеспечения для обеспечения соответствия требованиям.
    
- Обновление до новой версии Exchange. В Microsoft 365 вы всегда можете работать с последней версией Exchange.
    
#### <a name="how-should-i-migrate-to-microsoft-365"></a>Как перейти на Microsoft 365?

У вас есть несколько вариантов миграции. Необходимо учесть несколько моментов, в том числе:

- Количество мест или почтовых ящиков, которые необходимо переместить.
- Время, в течение времени, в течение чего будет длиться миграция.
- Требуется ли у вас плавная интеграция между локальной установкой и Microsoft 365 во время миграции.

В этой таблице показаны варианты миграции и наиболее важные факторы, определяющих, какой метод использовать:
  

|**Параметр миграции**|**Размер организации**|**Duration**|
|:-----|:-----|:-----|
|Прямая миграция  <br/> |Менее 150 мест  <br/> |Неделя или меньше  <br/> |
|Поэтапная миграция  <br/> |Более 150 мест  <br/> |Несколько недель  <br/> |
|Полная гибридная миграция  <br/> |От нескольких сотен до тысяч мест  <br/> |Несколько месяцев или больше  <br/> |
   
В следующих разделах представлен обзор этих методов. Дополнительные подробности см. в [под вопросе "Решение о пути миграции".](https://support.office.com/article/Decide-on-a-migration-path-0d4f2396-9cef-43b8-9bd6-306d01df1e27) 
  
#### <a name="cutover-migration"></a>Прямая миграция

При предварительной миграции все почтовые ящики, группы рассылки, контакты и другие почтовые ящики переносят в Microsoft 365 в предварительно заранее выбираемую дату и время. После завершения миграции вы завершите работу своих серверов Exchange и начнете использовать Microsoft 365 исключительно.
  
Быстрая миграция отлично подходит для небольших организаций, которые не имеют большого количества почтовых ящиков, хотят быстро ться в Microsoft 365 и не хотят решать некоторые сложности других методов. Но это должно быть завершено через неделю или меньше, и пользователям потребуется перенастроить свои профили Outlook. При такой миграции может обрабатываться до 2000 почтовых ящиков, но мы настоятельно рекомендуем использовать ее для миграции не более 150 почтовых ящиков. Если попытаться перенести дополнительные почтовые ящики, у вас может не оказаться времени для переноса всех почтовых ящиков до крайнего срока, и ваш ИТ-персонал может быть перегружен запросами на помощь в перенастройке Outlook.
  
Если вы думаете о переносе, следует учесть некоторые из них.
  
- Microsoft 365 потребуется подключиться к серверам Exchange 2007 с помощью outlook Anywhere через TCP-порт 443.
    
- Все почтовые ящики локальной службы будут перемещены в Microsoft 365.
    
- Вам потребуется учетная запись локального администратора с доступом на чтение почтовых ящиков пользователей.
    
- Принятые домены Exchange 2007, которые вы хотите использовать в Microsoft 365, необходимо добавить в качестве проверенных доменов в службу.
    
- Между началом миграции и началом этапа завершения Microsoft 365 будет периодически синхронизировать Microsoft 365 и почтовые ящики локальной организации. Это позволяет завершить миграцию, не беспокоясь о том, что почта осталась в ваших почтовых ящиках.
    
- Пользователи получат новые временные пароли для своих учетных записей Microsoft 365. Им потребуется изменить пароль при первом входе в свой почтовый ящик.
    
- Вам потребуется лицензия Microsoft 365, включаемая Exchange Online для каждого переносимго почтового ящика пользователя.
    
- Пользователям потребуется настроить новый профиль Outlook на каждом из своих устройств и снова скачать электронную почту. Количество сообщений электронной почты, которое будет скачивать Outlook, может отличаться. Дополнительные сведения см. в сведениях об изменении [того, сколько сообщений необходимо хранить в автономном режиме.](https://support.office.com/article/Change-how-much-mail-to-keep-offline-f3a1251c-6dd5-4208-aef9-7c8c9522d633?ui=en-US&amp;rs=en-US&amp;ad=US&amp;fromAR=1)
    
Дополнительные сведения о cutover migration см. в:
  
- [Что нужно знать о миграции электронной почты](https://support.office.com/article/What-you-need-to-know-about-a-cutover-email-migration-to-Office-365-961978ef-f434-472d-a811-1801733869da)

#### <a name="staged-migration"></a>Поэтапная миграция

При поэтапной миграции у вас есть несколько сотен или нескольких тысяч почтовых ящиков, которые вы хотите перенести в Microsoft 365, на завершение миграции требуется неделя или больше, и вам не нужны расширенные функции гибридной миграции, такие как общие сведения календаря о занятости.
  
Поэтапное переносное решение отлично подходит для организаций, которые нуждаются в большем времени для переноса почтовых ящиков в Microsoft 365, но по-прежнему планируют завершить миграцию в течение нескольких недель. Почтовые ящики можно перенести пакетами. Вы управляете количеством почтовых ящиков и их переносом в заданный момент времени. Например, можно пакетно отправить почтовые ящики пользователей в одном отделе, чтобы убедиться, что все они перемещаются одновременно. Вы также можете оставить почтовые ящики руководителей до последнего пакета. Как и при переключеных миграциях, пользователям потребуется повторно создать свои профили Outlook.
  
Если вы думаете о поэтапной миграции, следует учесть некоторые из них.
  
- Microsoft 365 потребуется подключиться к серверам Exchange 2007 с помощью outlook Anywhere через TCP-порт 443.
    
- Вам потребуется учетная запись локального администратора с доступом на чтение почтовых ящиков пользователей.
    
- Принятые домены Exchange 2007, которые вы планируете использовать в Microsoft 365, необходимо добавить в качестве проверенных доменов в службу.
    
- Необходимо создать CSV-файл с полным именем и адресом электронной почты каждого почтового ящика, который планируется перенести в пакете. Вам также потребуется включить новый пароль для каждого почтового ящика, который вы хотите перенести, и отправить этот пароль каждому пользователю. Пользователю будет предложено изменить пароль при первом входе в новый почтовый ящик Microsoft 365.
    
- Между моментом запуска пакета миграции и началом этапа завершения Microsoft 365 будет периодически синхронизировать Microsoft 365 и почтовые ящики, включенные в пакет. Это позволяет завершить миграцию, не беспокоясь о том, что почта осталась в ваших почтовых ящиках.
    
- Вам потребуется лицензия Microsoft 365, включаемая Exchange Online для каждого переносимго почтового ящика пользователя.
    
- Пользователям потребуется настроить новый профиль Outlook на каждом из своих устройств и снова скачать электронную почту. Количество сообщений электронной почты, которое будет скачивать Outlook, может отличаться. Дополнительные сведения см. в сведениях об изменении [того, сколько сообщений необходимо хранить в автономном режиме.](https://support.office.com/article/Change-how-much-mail-to-keep-offline-f3a1251c-6dd5-4208-aef9-7c8c9522d633?ui=en-US&amp;rs=en-US&amp;ad=US&amp;fromAR=1)

#### <a name="full-hybrid"></a>Полная гибридная версия

При полной гибридной миграции в организации имеется много сотен (до десятков тысяч) почтовых ящиков, и вы хотите переместить некоторые или все из них в Microsoft 365. Так как эти миграции обычно являются более длительными, гибридные миграции делают возможным:
  
- Показывать пользователям локальной сети сведения календаря о занятости для пользователей в Microsoft 365 и наоборот.
    
- См. единый глобальный список адресов, содержащий получателей как в локальной службе, так и в Microsoft 365.
    
- Просмотр полных свойств получателей Outlook для всех пользователей независимо от того, где они : локально или в Microsoft 365.
    
- Защита обмена электронной почтой между локальной службой Exchange Server и Microsoft 365 с помощью TLS и сертификатов.
    
- Обрабатывайте сообщения, отправленные между локальной службой Exchange Server и Microsoft 365, как внутренние, что позволяет им:
    
  - Должны правильно оцениваться и обрабатываться агентами транспорта и соответствия требованиям, нацеленными на внутренние сообщения.
    
  - Обход фильтров нежелательной почты.
    
Полная гибридная миграция лучше всего для организаций, которые ожидают, что гибридная конфигурация будет сохраняться в течение нескольких месяцев или более. Вы получите функции, перечисленные ранее в этом разделе, а также синхронизацию службы каталогов, более интегрированные функции соответствия требованиям и возможность перемещения почтовых ящиков в Microsoft 365 и из него с помощью перемещения почтовых ящиков через Интернет. Microsoft 365 становится расширением локальной организации.
  
Если вы думаете о полной гибридной миграции, следует учесть некоторые из этих проблем.
  
- Полная гибридная миграция подходит не для всех типов организаций. Из-за сложности полной гибридной миграции организации с менее чем сотней почтовых ящиков обычно не видят преимуществ, которые могут служить обоснованием усилий и затрат, необходимых для их организации. Если это похоже на вашу организацию, рекомендуем вместо этого учесть cutover или staged migration.
    
- Вам потребуется развернуть хотя бы один сервер Exchange 2013 в организации Exchange 2007, чтобы выступать в качестве "гибридного сервера". Этот сервер будет взаимодействовать с Microsoft 365 от имени серверов Exchange 2007.
    
- Microsoft 365 потребуется подключиться к "гибридному серверу" с помощью outlook Anywhere через TCP-порт 443.
    
- Вам потребуется настроить синхронизацию службы каталогов с помощью Azure Active Directory (Azure AD) Connect между вашими серверами Active Directory и Microsoft 365.
    
- Пользователи смогут войти в свой почтовый ящик Microsoft 365, используя те же имя пользователя и пароль, что и при входе в локальную сеть. (Для этой функции требуется Azure AD Connect с синхронизацией паролей и/или службами федерации Active Directory.)
    
- Вам потребуется лицензия Microsoft 365, включаемая Exchange Online для каждого переносимго почтового ящика пользователя.
    
- Пользователям не нужно устанавливать новый профиль Outlook на большинстве своих устройств, хотя для некоторых старых телефонов с Android может потребоваться новый профиль. Пользователям не придется перезагружать свою электронную почту.
    
Если полная гибридная миграция вам кажется правильной, см. следующие ресурсы, которые помогут вам с миграцией:

- [Мастер гибридной конфигурации](https://technet.microsoft.com/library/hh529921%28v=exchg.150%29.aspx)
    
- [Мастер гибридной конфигурации: вопросы и ответы](https://technet.microsoft.com/library/mt488940%28v=exchg.150%29.aspx)

### <a name="migrate-to-a-newer-version-of-exchange-server"></a>Переход на более новую версию Exchange Server

Мы настоятельно верим, что вы можете добиться наилучших результатов и обеспечить максимальное качество работы пользователей путем перехода на Microsoft 365. Но мы также понимаем, что некоторым организациям необходимо хранить свою электронную почту в локальной организации. Это может быть из-за нормативных требований, чтобы гарантировать, что данные не хранятся в центре обработки данных, расположенном в другой стране, или аналогично. Если вы решили сохранить электронную почту локально, вы можете перенести среду Exchange 2007 в Exchange 2010, Exchange 2013 или Exchange 2016.
  
Если вы не можете перейти на Microsoft 365, рекомендуем перейти на Exchange 2016. Exchange 2016 включает все функции предыдущих выпусков Exchange. Он также наиболее точно соответствует возможностям, доступным в Microsoft 365, хотя некоторые функции доступны только в Microsoft 365. Ознакомьтесь с несколькими пропущенными моментами:
  
|**Выпуск Exchange**|**Компоненты**|
|:-----|:-----|
|Exchange 2010  <br/> | Role-Based доступа (разрешения без ALS)  <br/>  Политики почтовых ящиков Outlook Web App  <br/>  Возможность обмена данными о занятости и делегирования календарей между организациями  <br/> |
|Exchange 2013  <br/> | *Функции из Exchange 2010 и ...*  <br/>  Упрощенная архитектура, которая снизила количество ролей сервера до трех (почтовый ящик, клиентский доступ, edge Transport)  <br/>  Политики защиты от потери данных (DLP), которые помогают предотвратить утечку конфиденциальной информации  <br/>  Улучшенное Outlook Web App работы  <br/> |
|Exchange 2016  <br/> | *Функции из Exchange 2013 и ...*  <br/>  Дополнительные упрощенные роли серверов только для почтовых ящиков и edge Transport  <br/>  Улучшенная DLP наряду с интеграцией с SharePoint  <br/>  Улучшенная устойчивость баз данных  <br/>  Совместная работа с документами в Интернете |
   
#### <a name="which-version-should-i-migrate-to"></a>В какую версию следует перейти?

Рекомендуем сначала предположить, что вы будете переходить на Exchange 2016. Затем используйте следующую информацию, чтобы подтвердить свое предположения или исключить Exchange 2016. Если вы не можете перейти на Exchange 2016 по какой-либо причине, сделайте то же самое с Exchange 2013 и т. д.
  
|**Фактор**|**Дополнительные сведения**|
|:-----|:-----|
|Даты окончания поддержки  <br/> | Как и в Exchange 2007, каждая версия Exchange имеет собственную дату окончания поддержки:  <br/> *Exchange 2010* – январь 2020 г.  <br/> *Exchange 2013* — апрель 2023 г.  <br/> *Exchange 2016* — октябрь 2025 г.  <br/>  Чем раньше закончится поддержка, тем раньше потребуется выполнить другую миграцию.<br/> |
|Путь миграции в Exchange 2010 и 2013.  <br/> |Вот общие этапы миграции на Exchange 2010 или Exchange 2013:  <br/> - Установите Exchange 2010 или 2013 в существующую организацию Exchange 2007. <br/>- Переместить службы и другую инфраструктуру в Exchange 2010 или 2013.<br/>- Перемещение почтовых ящиков и общедоступных папок в Exchange 2010 или 2013.<br/>- Списание оставшихся серверов Exchange 2007. |
|Путь миграции в Exchange 2016  <br/> |Вот общие этапы миграции на Exchange 2016:  <br/> - Установите Exchange 2013 в существующую организацию Exchange 2007.<br/>- Переместить службы и другую инфраструктуру в Exchange 2013.<br/>- Перемещение почтовых ящиков и общедоступных папок в Exchange 2013.<br/>- Списание оставшихся серверов Exchange 2007.<br/>- Установите Exchange 2016 в существующую организацию Exchange 2013.<br/>- Перемещение почтовых ящиков, общедоступных папок, служб и другой инфраструктуры в Exchange 2016 (порядок не имеет значения). Списание оставшихся серверов Exchange 2013.<br/><br/> **Примечание.** Миграция с Exchange 2013 на Exchange 2016 проста. Две версии имеют практически одинаковые требования к оборудованию, и эти версии очень совместимы. Таким образом, можно перестроить сервер, который вы приобрели для Exchange 2013, и установить на нем Exchange 2016. При перемещении почтовых ящиков по сети большинство пользователей даже не заметит, что их почтовый ящик был перемещен с сервера, а затем снова перестроен с помощью Exchange 2016.           |
|Сосуществование версий  <br/> | При переходе на ...  <br/> **Exchange 2016:** Exchange 2016 нельзя установить в организации, которая включает сервер Exchange 2007. Сначала необходимо перейти на Exchange 2010 или 2013 (настоятельно рекомендуется Exchange 2013), удалить все серверы Exchange 2007, а затем перейти на Exchange 2016.  <br/> **Exchange 2010 или Exchange 2013:** Вы можете установить Exchange 2010 или Exchange 2013 в существующую организацию Exchange 2007. Это позволяет установить один или несколько серверов Exchange 2010 или 2013 и выполнить миграцию.  <br/> |
|Серверное оборудование  <br/> | Требования к серверу к оборудованию были изменены с Exchange 2007. Убедитесь, что оборудование совместимо. Дополнительные сведения см. в следующих статьях:  <br/> [Требования к системе для Exchange 2016](https://technet.microsoft.com/library/aa996719%28v=exchg.160%29.aspx) <br/> [Требования к системе для Exchange 2013](https://technet.microsoft.com/library/aa996719%28v=exchg.150%29.aspx) <br/> [Требования к системе для Exchange 2010](https://technet.microsoft.com/library/aa996719%28v=exchg.141%29.aspx) <br/>  Вы увидите, что значительное повышение производительности Exchange и увеличение вычислительной мощности и емкости хранилища на новых серверах означает, что для поддержки одинакового количества почтовых ящиков потребуется меньше серверов.  <br/> |
|Версия операционной системы  <br/> | Минимальные поддерживаемые версии операционной системы для каждой версии:  <br/> **Exchange 2016** — Windows Server 2012  <br/> **Exchange 2013** — Windows Server 2008 R2 SP1  <br/> **Exchange 2010 —** Windows Server 2008 с sp2  <br/>  Дополнительные сведения о поддержке операционной системы можно найти в [матрице поддержки Exchange.](https://technet.microsoft.com/library/ff728623%28v=exchg.150%29.aspx)  <br/> |
|Функциональный уровень леса Active Directory  <br/> | Минимальные поддерживаемые функциональные уровни леса Active Directory для каждой версии:  <br/> **Exchange 2016** Windows Server 2008 R2 SP1  <br/> **Exchange 2013** Windows Server 2003  <br/> **Exchange 2010** Windows Server 2003  <br/>  Дополнительные сведения о поддержке функционального уровня леса можно найти в [матрице поддержки Exchange.](https://technet.microsoft.com/library/ff728623%28v=exchg.150%29.aspx)  <br/> |
|Версии клиентов Office  <br/> | Минимальные поддерживаемые версии клиентов Office для каждой версии:  <br/> **Exchange 2016** — Office 2010 (с последними обновлениями)  <br/> **Exchange 2013** — Office 2007 с sp3  <br/> **Exchange 2010** — Office 2003  <br/>  Дополнительные сведения о поддержке клиентов Office можно найти в [матрице поддержки Exchange.](https://technet.microsoft.com/library/ff728623%28v=exchg.150%29.aspx)  <br/> |
