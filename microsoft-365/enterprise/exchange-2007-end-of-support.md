# <a name="exchange-2007-end-of-support-roadmap"></a>План действий после прекращения поддержки Exchange 2007;

*Эта статья относится к Microsoft 365 корпоративный и Office 365 корпоративный.*

Сервер Exchange Server 2007 достиг конца поддержки в апреле 2017. Если вы еще не настроили миграцию с Exchange 2007 на Microsoft 365, Office 365 или Exchange 2016, настало время начать планирование.
  
## <a name="what-does-end-of-support-mean"></a>Что означает *Окончание поддержки* ?

На сервере Exchange Server, как и почти все продукты корпорации Майкрософт, предусмотрены жизненные циклы поддержки, в которых мы предоставляем новые функции, исправления ошибок, исправления для системы безопасности и т. д. Этот жизненный цикл обычно длится через 10 лет от первоначального выпуска продукта. Окончание этого жизненного цикла называется окончанием поддержки продукта. Так как Exchange 2007 достиг конца поддержки 11 апреля 2017, корпорация Майкрософт больше не предоставляет:
  
- Техническая поддержка проблем, которые могут возникнуть.
    
- Исправление ошибок для проблем, которые могут повлиять на стабильность и удобство использования сервера.
    
- Исправления для системы безопасности уязвимостей, которые могут сделать сервер уязвимым для нарушений безопасности.
    
- Обновления часового пояса.
    
Установка Exchange 2007 продолжится после даты окончания поддержки. Однако из-за отсутствия новых обновлений или поддержки настоятельно рекомендуется выполнить миграцию из Exchange 2007 как можно скорее.
  
Дополнительные сведения о серверах Office 2007, близких в конце поддержки, приведены в статье [Планирование обновления с серверов и продуктов office 2007](upgrade-from-office-2007-servers-and-products.md).
  
## <a name="what-are-my-options"></a>Что такое параметры?

Варианты действий:
  
- Миграция на Microsoft 365 с помощью прямой, поэтапной или гибридной миграции.
    
- Перенесите серверы Exchange 2007 в более новую версию Exchange на локальных серверах.
    
В следующих разделах каждый из этих вариантов рассматривается более подробно.
  
### <a name="migrate-to-microsoft-365"></a>Миграция в Microsoft 365

Перенос электронной почты в Microsoft 365 — это самый лучший и простой способ отказаться от использования развертывания Exchange 2007. При переходе на Microsoft 365 вы можете сделать один переход с более 10 лет старой технологии на функциональные возможности, включая следующие:
  
- Возможности обеспечения соответствия требованиям, такие как политики хранения, In-Place и хранение для судебного разбирательства, обнаружение электронных данных на месте и т. д.
    
- Группы Microsoft 365
    
- Сортировка почты
    
- MyAnalytics
    
- REST API для программного доступа к электронной почте, календарям, контактам и т. д.
    
Кроме того, Microsoft 365 также получает новые функции и возможности, поэтому вы и ваши пользователи обычно можете начать их использовать сразу же. И вам не придется беспокоиться:
  
- Приобретение и обслуживание оборудования.
    
- Оплата тепла и охлаждение серверов.
    
- Своевременное обновление системы безопасности, продуктов и часовых поясов.
    
- Поддержание хранилища и программного обеспечения для поддержки требований соответствия требованиям.
    
- Обновление до новой версии Exchange. С помощью Microsoft 365 вы всегда используете последнюю версию Exchange.
    
#### <a name="how-should-i-migrate-to-microsoft-365"></a>Как выполнить миграцию в Microsoft 365?

- Срок, в течение которого вы хотите выполнить миграцию.
- Необходимость беспрепятственной интеграции между локальной установкой и Microsoft 365 во время миграции.

#### <a name="cutover-migration"></a>Прямая миграция

В прямой миграции вы переносите все почтовые ящики, группы рассылки, контакты и т. д. в Microsoft 365 по предварительно выбранным дате и времени. После завершения миграции вы завершите работу локальных серверов Exchange и начнете использовать только Microsoft 365.
  
Прямой миграция отлично подходит для небольших организаций, у которых нет большого количества почтовых ящиков, и вы хотите быстро перейти в Microsoft 365 и не хотите справляться с некоторыми сложностями других методов. Но она должна быть выполнена за неделю или меньше, и для нее необходимо изменить конфигурацию профилей Outlook. Прямой миграция может обрабатывать до 2 000 почтовых ящиков, но настоятельно рекомендуется использовать ее для переноса не более 150 почтовых ящиков. Если вы попытаетесь выполнить миграцию, вы можете недолго выполнить перенос всех почтовых ящиков перед крайним сроком, и сотрудники ИТ, поддерживающие ИТ, могут перестать загрузить запросы, чтобы помочь пользователям изменить конфигурацию Outlook.
  
Если вы задумываете, как выполнить миграцию прямой, примите во внимание следующее:
  
- Microsoft 365 вам потребуется подключиться к серверам Exchange 2007 с помощью Outlook Anywhere через TCP-порт 443.
    
- Все локальные почтовые ящики будут перемещены в Microsoft 365.
    
- Вам потребуется локальная учетная запись администратора, имеющая доступ на чтение к почтовым ящикам пользователей.
    
- Обслуживаемые домены Exchange 2007, которые вы хотите использовать в Microsoft 365, необходимо добавить в качестве проверенных доменов в службе.
    
- Между моментом начала миграции и началом фазы завершения Microsoft 365 будет периодически синхронизировать почтовые ящики Microsoft 365 и локальные почтовые ящики. Это позволяет выполнить миграцию, не беспокоясь о том, что в локальных почтовых ящиках остались сообщения.
    
- Пользователи будут получать новые временные пароли для учетных записей Microsoft 365. Им потребуется изменить пароль при первом входе в почтовый ящик.
    
- Вам потребуется лицензия Microsoft 365, включающая Exchange Online для каждого переносимого почтового ящика пользователя.
    
- Пользователям потребуется настроить новый профиль Outlook на каждом устройстве и скачать электронную почту еще раз. Количество сообщений электронной почты, загружаемых Outlook, может различаться. Дополнительную информацию можно узнать [в статье изменение объема почты, который необходимо сохранить в автономном режиме](https://support.office.com/article/Change-how-much-mail-to-keep-offline-f3a1251c-6dd5-4208-aef9-7c8c9522d633?ui=en-US&amp;rs=en-US&amp;ad=US&amp;fromAR=1).
    
Более подробную информацию о прямой миграции можно найти в следующих статьях:

#### <a name="staged-migration"></a>Поэтапная миграция

При поэтапной миграции необходимо выполнить миграцию нескольких сотен или более тысяч почтовых ящиков, которые необходимо перенести в Microsoft 365, и не потребуют дополнительных функций гибридной миграции, таких как общие сведения о доступности в календаре.
  
Поэтапная миграция отлично подходит для организаций, которым требуется больше времени на перенос почтовых ящиков в Microsoft 365, но по-прежнему планируете выполнить миграцию в течение нескольких недель. Вы можете перенести почтовые ящики в пакеты. Вы управляете, сколько и какие почтовые ящики переносятся в данный момент времени. Вы можете выполнять пакетную обработку почтовых ящиков пользователей в одном отделе, например, чтобы убедиться в том, что все они перемещены одновременно. Вы также можете оставлять почтовые ящики руководителей, пока не будет выполнен последний пакет. Как и при прямой миграции, пользователям потребуется воссоздать свои профили Outlook.
  
Если вы планируете выполнить поэтапную миграцию, примите во внимание следующее:
  
- Microsoft 365 вам потребуется подключиться к серверам Exchange 2007 с помощью Outlook Anywhere через TCP-порт 443.
    
- Вам потребуется локальная учетная запись администратора, имеющая доступ на чтение к почтовым ящикам пользователей.
    
- Обслуживаемые домены Exchange 2007, которые планируется использовать в Microsoft 365, необходимо добавить в качестве проверенных доменов в службе.
    
- Вам потребуется создать CSV-файл с полным именем и адресом электронной почты каждого почтового ящика, который планируется перенести в пакет. Кроме того, необходимо включить новый пароль для каждого почтового ящика, который вы переносите, и отправить этот пароль каждому пользователю. Пользователю будет предложено сменить пароль при первом входе в новый почтовый ящик Microsoft 365.
    
- Между моментом запуска пакета миграции и началом фазы завершения Microsoft 365 будет периодически синхронизировать почтовые ящики Microsoft 365 и локальные почтовые ящики, включенные в пакет. Это позволяет выполнить миграцию, не беспокоясь о том, что в локальных почтовых ящиках остались сообщения.
    
- Вам потребуется лицензия Microsoft 365, включающая Exchange Online для каждого переносимого почтового ящика пользователя.
    
- Пользователям потребуется настроить новый профиль Outlook на каждом устройстве и скачать электронную почту еще раз. Количество сообщений электронной почты, загружаемых Outlook, может различаться. Дополнительную информацию можно узнать [в статье изменение объема почты, который необходимо сохранить в автономном режиме](https://support.office.com/article/Change-how-much-mail-to-keep-offline-f3a1251c-6dd5-4208-aef9-7c8c9522d633?ui=en-US&amp;rs=en-US&amp;ad=US&amp;fromAR=1).

#### <a name="full-hybrid"></a>Полное гибридное развертывание

При полной гибридной миграции организация имеет множество сотен, до десятков тысяч почтовых ящиков, и вы хотите переместить их в Microsoft 365. Так как эти миграции обычно являются более длительными, гибридная миграция позволяет выполнять следующие действия:
  
- Показывать локальным пользователям сведения о доступности в календаре для пользователей в Microsoft 365 и наоборот.
    
- Просмотр объединенного глобального списка адресов, который содержит получателей как в локальной среде, так и в Microsoft 365.
    
- Просмотр полных свойств получателей Outlook для всех пользователей независимо от того, являются ли они локальными или в Microsoft 365.
    
- Безопасный обмен сообщениями электронной почты между локальными серверами Exchange и Microsoft 365 с использованием TLS и сертификатов.
    
- Обрабатывать сообщения, отправляемые между локальными серверами Exchange и Microsoft 365 как внутренние, позволяя им:
    
  - Должны быть правильно оценены и обработаны транспортом и агентами соответствия требованиям, нацеленными на внутренние сообщения.
    
  - Обход фильтров защиты от нежелательной почты.
    
Полная Гибридная миграция лучше всего подходит для организаций, которые должны оставаться в гибридной конфигурации в течение многих месяцев или более многих месяцев. Вы получаете функции, описанные в этом разделе, а также синхронизацию службы каталогов, улучшения интегрированных функций соответствия и возможность перемещать почтовые ящики в Microsoft 365 с помощью перемещения почтовых ящиков в сети. Microsoft 365 становится расширением локальной организации.
  
Если вы думаете, как выполнить полную гибридную миграцию, примите во внимание следующее:
  
- Полная Гибридная миграция подходит не для всех типов организаций. Из-за сложности полных гибридных миграций Организации с менее чем несколькими сотнями почтовых ящиков обычно не видят преимущества, необходимые для установки. Если это звучит, как ваша организация, рекомендуется рассмотреть прямой или поэтапную миграцию.
    
- Вам потребуется развернуть по крайней мере один сервер Exchange 2013 в организации Exchange 2007, чтобы он действовал в качестве гибридного сервера. Этот сервер будет обмениваться данными с Microsoft 365 от имени серверов Exchange Server 2007.
    
- Microsoft 365 вам потребуется подключиться к "гибридному серверу" с помощью Outlook Anywhere через TCP-порт 443.
    
- Вам потребуется настроить синхронизацию службы каталогов с помощью Azure Active Directory (Azure AD) для подключения между локальными серверами Active Directory и Microsoft 365.
    
- Пользователи смогут входить в свой почтовый ящик Microsoft 365, используя те же имя пользователя и пароль, что и при входе в локальную сеть. (Для этой функции требуется Azure AD Connect с синхронизацией паролей и/или службами федерации Active Directory.)
    
- Вам потребуется лицензия Microsoft 365, включающая Exchange Online для каждого переносимого почтового ящика пользователя.
    
- Пользователям не нужно настраивать новый профиль Outlook на большинстве устройств, хотя для некоторых старых телефонов Android может потребоваться новый профиль. Пользователям не придется снова загружать свои сообщения.
    
Если вы хотите полностью выполнить гибридную миграцию, ознакомьтесь со следующими материалами, которые помогут вам выполнить миграцию:

### <a name="migrate-to-a-newer-version-of-exchange-server"></a>Переход на более новую версию Exchange Server

Мы настоятельно уверены, что вы можете добиться наилучшего качества и удобства работы пользователей, переполнив их на Microsoft 365. Кроме того, мы также понимаем, что в некоторых организациях необходимо хранить локальную электронную почту. Это может быть вызвано соблюдением нормативных требований, чтобы гарантировать, что данные не хранятся в центре обработки данных, расположенном в другой стране, или аналогично. Если вы решили сохранить локальную электронную почту, вы можете перенести среду Exchange 2007 в Exchange 2010, Exchange 2013 или Exchange 2016.
  
Если вы не можете выполнить миграцию в Microsoft 365, рекомендуем выполнить миграцию в Exchange 2016. Exchange 2016 включает все функции предыдущих выпусков Exchange. Кроме того, он наиболее полно соответствует интерфейсу, доступному в Microsoft 365, хотя некоторые функции доступны только в Microsoft 365. Ознакомьтесь только с некоторыми из отсутствующих вещей.
  
|**Выпуск Exchange**|**Компоненты**|
|:-----|:-----|
|Exchange 2010  <br/> | Role-Based управления доступом (разрешения без списков ACL)  <br/>  Политики почтовых ящиков Outlook Web App  <br/>  Возможность совместного использования календарей сведений о занятости и делегировании между организациями  <br/> |
|Exchange 2013  <br/> | *Функции из Exchange 2010 и...*  <br/>  Упрощенная архитектура, которая сокращает количество ролей сервера до трех (почтовый ящик, клиентский доступ, пограничный транспортный сервер)  <br/>  Политики защиты от потери данных (DLP), которые помогают предотвратить утечку конфиденциальной информации  <br/>  Улучшенное взаимодействие с Outlook Web App  <br/> |
|Exchange 2016  <br/> | *Функции из Exchange 2013 и...*  <br/>  Более простые роли сервера для почтового ящика и пограничного транспортного сервера  <br/>  Улучшенная защита от потери данных и интеграция с SharePoint  <br/>  Улучшенная устойчивость базы данных  <br/>  Совместная работа с документами в Интернете |
   
#### <a name="which-version-should-i-migrate-to"></a>В какой версии следует выполнить миграцию?

Рекомендуем сначала выполнить миграцию в Exchange 2016. Затем воспользуйтесь приведенными ниже сведениями, чтобы подтвердить ваше предположение или отправиль к Exchange 2016. Если вы не можете выполнить миграцию в Exchange 2016 по какой либо причине, выполните тот же процесс с Exchange 2013 и т. д.
  
|**Фактор**|**Дополнительные сведения**|
|:-----|:-----|
|Дата окончания поддержки  <br/> | Как и в Exchange 2007, каждая версия Exchange имеет собственную дату окончания срока поддержки:  <br/> *Exchange 2010* — Январь 2020  <br/> *Exchange 2013* – Апрель 2023  <br/> *Exchange 2016* – Октябрь 2025  <br/>  Чем выше поддержка, тем быстрее потребуется выполнить еще одну миграцию.<br/> |
|Путь миграции к Exchange 2010 и 2013.  <br/> |Ниже приведены общие этапы перехода на Exchange 2010 или Exchange 2013:  <br/> — Установите Exchange 2010 или 2013 в существующую организацию Exchange 2007. <br/>— Перемещение служб и другой инфраструктуры в Exchange 2010 или 2013.<br/>Переместите почтовые ящики и общедоступные папки в Exchange 2010 или 2013.<br/>— Оставшиеся серверы Exchange 2007. |
|Путь миграции к Exchange 2016  <br/> |Ниже приведены общие этапы перехода на Exchange 2016.  <br/> — Установите Exchange 2013 в существующую организацию Exchange 2007.<br/>— Перемещение служб и другой инфраструктуры в Exchange 2013.<br/>Переместите почтовые ящики и общедоступные папки в Exchange 2013.<br/>— Оставшиеся серверы Exchange 2007.<br/>— Установите Exchange 2016 в существующую организацию Exchange 2013.<br/>— Перемещение почтовых ящиков, общедоступных папок, служб и других инфраструктур в Exchange 2016 (порядок не имеет значения). Списание оставшихся серверов Exchange 2013.<br/><br/> **Примечание:** Миграция с Exchange 2013 на Exchange 2016 выполняется очень просто. Две версии имеют почти те же требования к оборудованию, и эти версии совместимы. Итак, вы можете перестроить сервер, который вы приобрели для Exchange 2013, и установить Exchange 2016. Для перемещения почтовых ящиков в сети большинство пользователей даже не завнимание, что их почтовые ящики были перемещены с сервера и затем обратно после его перестроения с помощью Exchange 2016.           |
|Совместное использование версий  <br/> | При переходе на...  <br/> **Exchange 2016:** Exchange 2016 невозможно установить в Организации, которая включает сервер Exchange Server 2007. Сначала необходимо выполнить миграцию в Exchange 2010 или 2013 (настоятельно рекомендуем Exchange 2013), удалить все серверы Exchange 2007, а затем перенести их в Exchange 2016.  <br/> **Exchange 2010 или exchange 2013:** Вы можете установить Exchange 2010 или Exchange 2013 в существующую организацию Exchange 2007. Это позволяет установить один или несколько серверов Exchange 2010 или 2013 и выполнить миграцию.  <br/> |
|Серверное оборудование  <br/> | Требования к оборудованию серверов изменились по сравнению с Exchange 2007. Убедитесь, что оборудование совместимо. Дополнительные сведения см. в следующих статьях:  <br/> [Требования к системе для Exchange 2016](https://technet.microsoft.com/library/aa996719%28v=exchg.160%29.aspx) <br/> [Требования к системе для Exchange 2013](https://technet.microsoft.com/library/aa996719%28v=exchg.150%29.aspx) <br/> [Требования к системе для Exchange 2010](https://technet.microsoft.com/library/aa996719%28v=exchg.141%29.aspx) <br/>  Вы обнаружите, что значительные улучшения в производительности Exchange, а также повышенный уровень энергопотребления и емкости хранилища в новых серверах, скорее всего, потребуется меньше серверов для поддержки одинакового количества почтовых ящиков.  <br/> |
|Версия операционной системы  <br/> | Минимальная поддерживаемая версия операционной системы для каждой версии:  <br/> **Exchange 2016** — Windows Server 2012  <br/> **Exchange 2013** — Windows Server 2008 R2 с пакетом обновления 1 (SP1)  <br/> **Exchange 2010** — Windows Server 2008 с пакетом обновления 2 (SP2)  <br/>  Дополнительные сведения о поддержке операционной системы в [матрице поддержки Exchange](https://technet.microsoft.com/library/ff728623%28v=exchg.150%29.aspx).  <br/> |
|Функциональный уровень леса Active Directory  <br/> | Минимальный поддерживаемый режим работы леса Active Directory для каждой версии:  <br/> **Exchange 2016** Windows Server 2008 R2 с пакетом обновления 1  <br/> **Exchange 2013** Windows Server 2003  <br/> **Exchange 2010** Windows Server 2003  <br/>  Дополнительные сведения о поддержке режима работы леса содержатся в [матрице поддержки Exchange](https://technet.microsoft.com/library/ff728623%28v=exchg.150%29.aspx).  <br/> |
|Версии клиента Office  <br/> | Ниже перечислены минимальные поддерживаемые версии клиентов Office для каждой версии.  <br/> **Exchange 2016** — Office 2010 (с последними обновлениями)  <br/> **Exchange 2013** — Office 2007 SP3  <br/> **Exchange 2010** — Office 2003  <br/>  Дополнительные сведения о поддержке клиентов Office в [матрице поддержки Exchange](https://technet.microsoft.com/library/ff728623%28v=exchg.150%29.aspx).  <br/> |
