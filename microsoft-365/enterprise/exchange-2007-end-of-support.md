# <a name="exchange-2007-end-of-support-roadmap"></a>План действий после прекращения поддержки Exchange 2007;

*Эта статья относится к Microsoft 365 корпоративный и Office 365 корпоративный.*

Exchange Server 2007 г. поддержка была достигнута в апреле 2017 г. Если вы еще не начали миграцию из Exchange 2007 в Microsoft 365, Office 365 или Exchange 2016, пришло время приступить к планированию.
  
## <a name="what-does-end-of-support-mean"></a>Что означает *конец поддержки?*

Exchange Server, как и почти все продукты Майкрософт, имеет жизненный цикл поддержки, в ходе которого мы предоставляем новые функции, исправления ошибок, исправления безопасности и так далее. Этот жизненный цикл обычно длится 10 лет с начального выпуска продукта. Конец жизненного цикла называется завершением поддержки продукта. С тех пор, как 11 апреля 2017 г. Exchange 2007 г. перестала поддерживаться, Корпорация Майкрософт больше не предоставляет:
  
- Техническая поддержка проблем, которые могут возникнуть.
    
- Ошибка устраняет проблемы, которые могут повлиять на стабильность и доступность сервера.
    
- Устранение уязвимостей, которые могут сделать сервер уязвимым для нарушений безопасности.
    
- Обновления часовой зоны.
    
Установка Exchange 2007 продолжится после даты окончания службы поддержки. Но поскольку новых обновлений и поддержки нет, настоятельно рекомендуется как можно скорее перейти из Exchange 2007.
  
Дополнительные сведения о серверах Office 2007, близких к концу службы поддержки, см. в руб. Планирование обновления с серверов и продуктов [Office 2007.](upgrade-from-office-2007-servers-and-products.md)
  
## <a name="what-are-my-options"></a>Какие у меня варианты?

Вы можете выполнить указанные ниже действия.
  
- Миграция в Microsoft 365 с помощью переноса, поэтапной или гибридной миграции.
    
- Перенос серверов Exchange 2007 в более новую версию Exchange на локальном сервере.
    
В следующих разделах подробно изучаются все параметры.
  
### <a name="migrate-to-microsoft-365"></a>Миграция в Microsoft 365

Перенос электронной почты в Microsoft 365 — это самый простой и простой вариант, который поможет завершить развертывание Exchange 2007. При переходе на Microsoft 365 можно сделать один переход с 10-летней технологии на современные функции, в том числе:
  
- Возможности соответствия требованиям, такие как политики хранения, In-Place и судебное удержание, хранение электронных данных на месте и другие
    
- Группы Microsoft 365
    
- Сортировка почты
    
- MyAnalytics
    
- API REST для программного доступа к электронной почте, календарям, контактам и так далее
    
Сначала Microsoft 365 также получает новые функции и возможности, поэтому вы и пользователи обычно можете сразу же начать их использовать. И вам не придется беспокоиться о:
  
- Приобретение и обслуживание оборудования.
    
- Оплата тепла и охлаждения серверов.
    
- Ведение в курсе исправлений безопасности, продуктов и часового пояса.
    
- Ведение хранилища и программного обеспечения в соответствии с требованиями соответствия требованиям.
    
- Обновление до новой версии Exchange. С Microsoft 365 вы всегда будете в последней версии Exchange.

- Как долго вы хотите, чтобы миграция длилась.
- Нужна ли вам бесшовная интеграция между локальной установкой и Microsoft 365 во время миграции.

#### <a name="cutover-migration"></a>Прямая миграция

При переносе переключельных данных все почтовые ящики, группы рассылки, контакты и так далее в Microsoft 365 переносились в предварительно предварительно заранее. После завершения миграции вы закрываете локальное серверы Exchange и начинаете использовать microsoft 365 исключительно.
  
Миграция отсечивок отлично подходит для небольших организаций, которые не имеют большого количества почтовых ящиков, хотят быстро добраться до Microsoft 365 и не хотят иметь дело с некоторыми сложностями других методов. Но она должна быть завершена в неделю или меньше, и она требует от пользователей перенастроить свои профили Outlook. Миграция перереза может обрабатывать до 2000 почтовых ящиков, но мы настоятельно рекомендуем использовать его для переноса не более 150 почтовых ящиков. Если вы попробуете перенести больше, у вас может не оказаться времени для переноса всех почтовых ящиков до истечения срока, и сотрудники службы поддержки ИТ могут быть перегружены запросами на помощь пользователям в перенастройке Outlook.
  
Если вы думаете о переносе переключе-
  
- Microsoft 365 необходимо подключиться к серверам Exchange 2007 с помощью Outlook Anywhere over TCP port 443.
    
- Все почтовые ящики будут перемещены в Microsoft 365.
    
- Вам понадобится учетная запись локального администратора, которая имеет доступ к почтовым ящикам пользователей.
    
- Принятые домены Exchange 2007, которые необходимо использовать в Microsoft 365, должны быть добавлены в качестве проверенных доменов в службе.
    
- Между началом миграции и началом этапа завершения Microsoft 365 периодически синхронизирует почтовые ящики Microsoft 365 и локально. Это позволяет завершить миграцию, не беспокоясь о том, что электронная почта останется в локальном почтовом ящике.
    
- Пользователи получат новые временные пароли для учетных записей Microsoft 365. При первом входе в почтовый ящик им потребуется изменить пароль.
    
- Вам потребуется лицензия Microsoft 365, которая включает Exchange Online для каждого переносимый почтовый ящик пользователя.
    
- Пользователям необходимо настроить новый профиль Outlook на каждом из своих устройств и снова скачать электронную почту. Количество электронной почты, которую скачивает Outlook, может отличаться. Дополнительные сведения см. #### <a name="staged-migration"></a>Поэтапная миграция

При поэтапной миграции у вас есть несколько сотен или нескольких тысяч почтовых ящиков, которые необходимо перенести в Microsoft 365, для завершения миграции потребуется неделю или больше, и не требуется никаких расширенных функций гибридной миграции, таких как общие сведения о календаре free/busy.
  
Этапная миграция отлично подходит для организаций, которые должны перенести свои почтовые ящики в Microsoft 365, но все же планируют завершить миграцию в течение нескольких недель. Вы можете перенести почтовые ящики пакетами. Вы управляете количеством и переносом почтовых ящиков в данный момент времени. Например, можно пакетно пакетить почтовые ящики пользователей в одном отделе, чтобы убедиться, что все они перемещаются одновременно. Или можно оставить почтовые ящики руководителей до последней партии. Как и в переносах отсеяния, пользователям потребуется воссоздать профили Outlook.
  
Если вы думаете о поэтапной миграции, рассмотрим ниже:
  
- Microsoft 365 необходимо подключиться к серверам Exchange 2007 с помощью Outlook Anywhere over TCP port 443.
    
- Вам понадобится учетная запись локального администратора, которая имеет доступ к почтовым ящикам пользователей.
    
- Принятые домены Exchange 2007, которые планируется использовать в Microsoft 365, должны быть добавлены в качестве проверенных доменов в службе.
    
- Необходимо создать CSV-файл с полным именем и адресом электронной почты каждого почтового ящика, который планируется перенести пакетом. Кроме того, необходимо включить новый пароль для каждого переносимго почтового ящика и отправить его каждому пользователю. Пользователю будет предложено изменить пароль при первом входе в новый почтовый ящик Microsoft 365.
    
- Между началом пакета миграции и началом этапа завершения Microsoft 365 будет периодически синхронизировать почтовые ящики Microsoft 365 и локально, включенные в пакет. Это позволяет завершить миграцию, не беспокоясь о том, что электронная почта останется в локальном почтовом ящике.
    
- Вам потребуется лицензия Microsoft 365, которая включает Exchange Online для каждого переносимый почтовый ящик пользователя.
    
- Пользователям необходимо настроить новый профиль Outlook на каждом из своих устройств и снова скачать электронную почту. Количество электронной почты, которую скачивает Outlook, может отличаться. Дополнительные сведения см. [в сообщении Change how much mail to keep offline.](https://support.office.com/article/Change-how-much-mail-to-keep-offline-f3a1251c-6dd5-4208-aef9-7c8c9522d633?ui=en-US&amp;rs=en-US&amp;ad=US&amp;fromAR=1)

#### <a name="full-hybrid"></a>Полный гибрид

При полной гибридной миграции в вашей организации имеется несколько сотен, до десятков тысяч почтовых ящиков, и вы хотите переместить некоторые или все из них в Microsoft 365. Поскольку эти миграции, как правило, являются более длительными, гибридные миграции делают возможным:
  
- Покажите пользователям локального доступа сведения о календаре для пользователей в Microsoft 365 и наоборот.
    
- См. единый глобальный адресный список, содержащий получателей как в локальной, так и в Microsoft 365.
    
- Просмотр полных свойств получателей Outlook для всех пользователей, независимо от того, являются ли они локальной или в Microsoft 365.
    
- Безопасное сообщение электронной почты между локально серверами Exchange и Microsoft 365 с помощью TLS и сертификатов.
    
- Обрабатывайте сообщения, отправленные между внутренними серверами Exchange и Microsoft 365, как внутренние, что позволяет:
    
  - Должны правильно оцениваться и обрабатываться агентами транспорта и соответствия требованиям, нацеленными на внутренние сообщения.
    
  - Обход фильтров от нежелательной почты.
    
Полная гибридная миграция лучше всего для организаций, которые рассчитывают оставаться в гибридной конфигурации в течение многих месяцев или более. Вы получите функции, перечисленные ранее в этом разделе, а также синхронизацию каталогов, более интегрированные функции соответствия требованиям, а также возможность перемещения почтовых ящиков в Microsoft 365 и из него с помощью интерактивных ходов почтовых ящиков. Microsoft 365 становится расширением локальной организации.
  
Если вы думаете о полной гибридной миграции, рассмотрим ниже:
  
- Полная гибридная миграция не подходит всем типам организаций. Из-за сложности полных гибридных миграций организации с менее чем несколькими сотнями почтовых ящиков обычно не видят преимуществ, которые оправдывают усилия и затраты, необходимые для их настройка. Если это похоже на организацию, рекомендуется вместо этого рассмотреть вопрос о переносе или поэтапной миграции.
    
- Чтобы выступать в качестве "гибридного сервера", необходимо развернуть хотя бы один сервер Exchange 2013 в организации Exchange 2007. Этот сервер будет взаимодействовать с Microsoft 365 от имени серверов Exchange 2007.
    
- Microsoft 365 необходимо подключиться к "гибридному серверу" с помощью Outlook Anywhere over TCP port 443.
    
- Необходимо настроить синхронизацию каталогов с помощью Azure Active Directory (Azure AD) Connect между локальной серверами Active Directory и Microsoft 365.
    
- Пользователи смогут войти в свой почтовый ящик Microsoft 365 с тем же именем пользователя и паролем, что и при входе в локализованную сеть. (Эта функция требует подключения Azure AD с синхронизацией паролей и/или службами Федерации Active Directory.)
    
- Вам потребуется лицензия Microsoft 365, которая включает Exchange Online для каждого переносимый почтовый ящик пользователя.
    
- Пользователям не нужно настроить новый профиль Outlook на большинстве своих устройств, хотя некоторым старым телефонам с Android может потребоваться новый профиль. Пользователям не придется перезагружать электронную почту.
    
Если полная гибридная миграция звучит правильно, см. в следующих ресурсах, которые помогут с миграцией:

### <a name="migrate-to-a-newer-version-of-exchange-server"></a>Миграция в более новую версию Exchange Server

Мы твердо верим, что вы можете добиться наилучшего качества и пользовательского интерфейса, переехав в Microsoft 365. Но мы также понимаем, что некоторые организации должны хранить электронную почту на месте. Это может быть из-за нормативных требований, чтобы гарантировать, что данные не хранятся в центре обработки данных, расположенном в другой стране, или аналогичные. Если вы хотите сохранить электронную почту локально, вы можете перенести среду Exchange 2007 в Exchange 2010, Exchange 2013 или Exchange 2016.
  
Если вы не можете перейти в Microsoft 365, рекомендуется перейти на Exchange 2016. Exchange 2016 включает все функции предыдущих выпусков Exchange. Он также наиболее тесно совпадает с опытом, доступным в Microsoft 365, хотя некоторые функции доступны только в Microsoft 365. Ознакомьтесь с несколькими недостающими вещами:
  
|**Выпуск Exchange**|**Компоненты**|
|:-----|:-----|
|Exchange 2010  <br/> | Role-Based управления доступом (разрешения без acLs)  <br/>  Политики почтовых ящиков Outlook Web App  <br/>  Возможность совместной работы и делегирования календарей между организациями  <br/> |
|Exchange 2013  <br/> | *Функции от Exchange 2010 и ...*  <br/>  Упрощенная архитектура, снизив число ролей сервера до трех (почтовый ящик, клиентский доступ, edge transport)  <br/>  Политики предотвращения потери данных (DLP), которые помогают предотвратить утечку конфиденциальной информации  <br/>  Улучшенная Outlook Web App  <br/> |
|Exchange 2016  <br/> | *Функции от Exchange 2013 и ...*  <br/>  Дополнительные упрощенные роли сервера только для почтовых ящиков и edge transport  <br/>  Улучшенная DLP вместе с интеграцией с SharePoint  <br/>  Повышение устойчивости баз данных  <br/>  Совместная работа с документами в Интернете |
   
#### <a name="which-version-should-i-migrate-to"></a>В какую версию следует перейти?

Сначала рекомендуется перейти на Exchange 2016. Затем используйте следующую информацию, чтобы подтвердить свое предположение или исключить Exchange 2016. Если вы по каким-либо причинам не можете перейти на Exchange 2016, сделайте то же самое с Exchange 2013 и так далее.
  
|**Фактор**|**Дополнительные сведения**|
|:-----|:-----|
|Окончание дат поддержки  <br/> | Как и Exchange 2007, каждая версия Exchange имеет свою дату окончания поддержки:  <br/> *Exchange 2010* — январь 2020 г.  <br/> *Exchange 2013* — апрель 2023 г.  <br/> *Exchange 2016* — октябрь 2025 г.  <br/>  Чем раньше закончится поддержка, тем быстрее потребуется выполнить другую миграцию.<br/> |
|Путь миграции в Exchange 2010 и 2013.  <br/> |Вот общие этапы перехода на Exchange 2010 или Exchange 2013:  <br/> - Установите Exchange 2010 или 2013 в существующую организацию Exchange 2007. <br/>- Перемещение служб и другой инфраструктуры в Exchange 2010 или 2013.<br/>- Перемещение почтовых ящиков и общедоступных папок в Exchange 2010 или 2013.<br/>- Вывод из эксплуатации оставшихся серверов Exchange 2007. |
|Путь миграции к Exchange 2016  <br/> |Вот общие этапы перехода на Exchange 2016:  <br/> - Установите Exchange 2013 в существующую организацию Exchange 2007.<br/>- Перемещение служб и другой инфраструктуры в Exchange 2013.<br/>- Перемещение почтовых ящиков и общедоступных папок в Exchange 2013.<br/>- Вывод из эксплуатации оставшихся серверов Exchange 2007.<br/>- Установите Exchange 2016 в существующую организацию Exchange 2013.<br/>- Перемещение почтовых ящиков, общедоступных папок, служб и другой инфраструктуры в Exchange 2016 (порядок не имеет значения). Вывод из эксплуатации оставшихся серверов Exchange 2013.<br/><br/> **Примечание:** Миграция с Exchange 2013 на Exchange 2016 проста. У двух версий практически одинаковые требования к оборудованию, и эти версии очень совместимы. Таким образом, можно восстановить сервер, купленный для Exchange 2013, и установить на нем Exchange 2016. Для перемещения почтовых ящиков в Интернете большинство пользователей даже не заметит, что их почтовый ящик был перемещен с сервера, а затем обратно после того, как вы перестроили его с Помощью Exchange 2016.           |
|Сосуществование версий  <br/> | При миграции в ...  <br/> **Exchange 2016:** Exchange 2016 не может быть установлен в организации, которая включает сервер Exchange 2007. Сначала необходимо перейти на Exchange 2010 или 2013 (мы настоятельно рекомендуем Exchange 2013), удалить все серверы Exchange 2007, а затем перейти на Exchange 2016.  <br/> **Exchange 2010 или Exchange 2013:** Вы можете установить Exchange 2010 или Exchange 2013 в существующую организацию Exchange 2007. Это позволяет установить один или несколько серверов Exchange 2010 или 2013 и выполнить миграцию.  <br/> |
|Серверное оборудование  <br/> | Требования к оборудованию сервера изменились с Exchange 2007. Убедитесь, что оборудование совместимо. Дополнительные сведения см. в следующих статьях:  <br/> [Требования к системе Exchange 2016](/Exchange/plan-and-deploy/system-requirements) <br/> [Требования к системе Exchange 2013](/exchange/exchange-2013-system-requirements-exchange-2013-help) <br/> [Требования к системе Exchange 2010](/previous-versions/office/exchange-server-2010/aa996719(v=exchg.141)) <br/>  Вы увидите, что значительные улучшения производительности Exchange и увеличение вычислительной мощности и емкости хранилища на новых серверах означают, что для поддержки одинакового числа почтовых ящиков потребуется меньше серверов.  <br/> |
|Версия операционной системы  <br/> | Минимальные поддерживаемые версии операционной системы для каждой версии:  <br/> **Exchange 2016** — Windows Server 2012  <br/> **Exchange 2013** — Windows Server 2008 R2 SP1  <br/> **Exchange 2010** — Windows Server 2008 SP2  <br/>  Дополнительные сведения о поддержке операционной системы можно найти в [Матрице поддержки Exchange.](/Exchange/plan-and-deploy/supportability-matrix)  <br/> |
|Функциональный уровень леса Active Directory  <br/> | Минимальные поддерживаемые функциональные уровни леса Active Directory для каждой версии:  <br/> **Exchange 2016** Windows Server 2008 R2 SP1  <br/> **Exchange 2013** Windows Server 2003  <br/> **Exchange 2010** Windows Server 2003  <br/>  Дополнительные сведения о поддержке функционального уровня леса можно найти в [Матрице поддержки Exchange.](/Exchange/plan-and-deploy/supportability-matrix)  <br/> |
|Клиентские версии Office  <br/> | Минимальные поддерживаемые клиентские версии Office для каждой версии:  <br/> **Exchange 2016** — Office 2010 (с последними обновлениями)  <br/> **Exchange 2013** — Office 2007 SP3  <br/> **Exchange 2010** — Office 2003  <br/>  Дополнительные сведения о поддержке клиентов Office можно найти в [Матрице поддержки Exchange.](/Exchange/plan-and-deploy/supportability-matrix)  <br/> |
