# <a name="onboard-windows-10-multi-session-devices-in-windows-virtual-desktop"></a>Подключение многосеансовых устройств Windows 10 на виртуальном рабочем столе Windows 
6 минут для чтения 

Применимо к: 
- Windows 10 нескольких сеансов, работающих на Windows виртуальный рабочий стол (WVD) 

Microsoft Defender для конечной точки поддерживает мониторинг сеансов виртуального и Windows рабочего стола. В зависимости от потребностей организации может потребоваться реализовать сеансы виртуального или Windows для виртуальных настольных компьютеров, чтобы помочь сотрудникам получать доступ к корпоративным данным и приложениям с неуголовного устройства, удаленного расположения или аналогичного сценария. С помощью Microsoft Defender для конечной точки вы можете отслеживать эти виртуальные машины для аномальной активности.

 ## <a name="before-you-begin"></a>Прежде чем начать
Ознакомьтесь с [соображениями для нестандартных VDI](/microsoft-365/security/defender-endpoint/configure-endpoints-vdi#onboard-non-persistent-virtual-desktop-infrastructure-vdi-devices-1). Хотя [Windows Virtual Desktop](/azure/virtual-desktop/overview) не предоставляет параметры сохраняемости, он предоставляет способы использования золотого изображения Windows, которое можно использовать для обеспечения новых хостов и передислокационных машин. Это повышает волатильность в среде и таким образом влияет на то, какие записи создаются и поддерживаются на портале Microsoft Defender для конечных точек, что потенциально снижает видимость для аналитиков безопасности.

> [!NOTE]
> В зависимости от выбора метода бортовой работы устройства могут отображаться на портале Microsoft Defender для конечных точек так же: 
> - Одна запись для каждого виртуального рабочего стола 
> - Несколько записей для каждого виртуального рабочего стола 

Корпорация Майкрософт рекомендует использовать Windows виртуальный рабочий стол в качестве одной записи на виртуальном рабочем столе. Это гарантирует, что опыт исследования на портале Конечной точки Защитника Майкрософт находится в контексте одного устройства на основе имени машины. Организации, которые часто удаляют и переназначяют хосты WVD, должны активно использовать этот метод, поскольку он не позволяет создавать несколько объектов для одной и той же машины на портале Microsoft Defender for Endpoint. Это может привести к путанице при расследовании инцидентов. Для тестовых или нестабиленных сред можно выбрать другой выбор. 

Корпорация Майкрософт рекомендует добавить сценарий onboarding Microsoft Defender для конечной точки в золотое изображение WVD. Таким образом, вы можете быть уверены, что этот сценарий для загрузки выполняется сразу же при первой загрузке. Он выполняется в качестве скрипта запуска при первой загрузке на всех машинах WVD, которые являются предварительными из золотого изображения WVD. Однако, если вы используете одно из изображений галереи без изменений, поместите сценарий в общее расположение и позвоните ему из локальной или групповой политики домена. 

> [!NOTE]
> Размещение и конфигурация сценария запуска VDI на золотом изображении WVD настраивает его как сценарий запуска, который запускается при запуске WVD. Не рекомендуется использовать фактическое золотое изображение WVD. Другим фактором является метод, используемый для запуска скрипта. Он должен запуститься как можно раньше в процессе запуска и подготовка, чтобы сократить время между машиной, доступной для получения сеансов, и устройством на борту службы. Ниже приведены сценарии 1 & 2 с учетом этого.

### <a name="scenarios"></a>Сценарии
Существует несколько способов на борту хост-машины WVD:

- Запустите сценарий в золотом изображении (или из общего расположения) во время запуска.
- Используйте средство управления для запуска скрипта.

#### <a name="scenario-1-using-local-group-policy"></a>*Сценарий 1. Использование локальной групповой политики*
Этот сценарий требует размещения скрипта в золотом изображении и использования локальной групповой политики для запуска на ранних стадиях загрузки.

Используйте инструкции на устройствах VDI инфраструктуры виртуальной инфраструктуры виртуальных [настольных компьютеров.](configure-endpoints-vdi.md#onboard-non-persistent-virtual-desktop-infrastructure-vdi-devices-1)

Следуйте инструкциям для одной записи для каждого устройства.

#### <a name="scenario-2-using-domain-group-policy"></a>*Сценарий 2. Использование политики группы доменов*
Этот сценарий использует централизованный сценарий и запускает его с помощью групповой политики на основе домена. Вы также можете разместить сценарий в золотом изображении и запустить его таким же образом.

**Скачайте WindowsDefenderATPOnboardingPackage.zip из центра Защитник Windows безопасности**

1. Откройте файл конфигурации VDI .zip (WindowsDefenderATPOnboardingPackage.zip)  

    1. В области Центр безопасности в Microsoft Defender навигации **выберите** Параметры  >  **onboarding**. 
    1. Выберите Windows 10 в качестве операционной системы. 
    1. В поле **Метод развертывания** выберите скрипты бортового VDI для неохраняющих конечных точек. 
    1. Нажмите **кнопку Загрузка** пакета и сохраните .zip файл. 

2. Извлечение содержимого файла .zip в общее, только для чтения расположение, которое можно получить на устройстве. У вас должна быть папка **OptionalParamsPolicy** и файлы **WindowsDefenderATPOnboardingScript.cmd** **иOnboard-NonPersistentMachine.ps1.**

**При запуске виртуальной машины используйте консоль управления групповой политикой для запуска сценария**

1. Откройте консоль управления групповой политикой (GPMC), щелкните правой кнопкой мыши объект групповой политики (GPO), который необходимо настроить, и нажмите **кнопку Изменить**.

2. В редакторе управления групповой политикой перейдите **к** настройкам панели \> **управления настройками** конфигурации \> компьютера. 

3. Щелкните правой кнопкой мыши **Запланированные задачи,** нажмите **кнопку New** и нажмите кнопку **Немедленная** задача (по крайней мере Windows 7). 

4. В открываемом окне Задача перейдите на **вкладку General.** В **статье Параметры безопасности** нажмите **кнопку Изменить пользователя или группу** и введите SYSTEM. Щелкните **Check Names** и нажмите кнопку ОК. NT AUTHORITY\SYSTEM отображается в качестве учетной записи пользователя, как будет работать задача. 

5. Выберите **Выполнить, вошел** ли пользователь или нет, и проверьте поле Выполнить с **самыми** высокими привилегиями. 

6. Перейдите на вкладку **Действия** и нажмите **кнопку New**. **Убедитесь, что программа Start** выбрана в поле Действия. Введите следующее: 

   `Action = "Start a program"`

   `Program/Script = C:\WINDOWS\system32\WindowsPowerShell\v1.0\powershell.exe`

   `Add Arguments (optional) = -ExecutionPolicy Bypass -command "& \\Path\To\Onboard-NonPersistentMachine.ps1"`

   Затем выберите **ОК** и закрой все открытые окна GPMC.

#### <a name="scenario-3-onboarding-using-management-tools"></a>*Сценарий 3. Использование средств управления*

Если вы планируете управлять своими машинами с помощью средства управления, можно использовать бортовые устройства с Microsoft Endpoint Configuration Manager.

Дополнительные сведения см. в [Windows 10 устройств с помощью Configuration Manager.](configure-endpoints-sccm.md)

> [!WARNING]
> Если вы планируете использовать правила уменьшения [поверхности](attack-surface-reduction.md)атаки, обратите внимание, что правило "Блокируйте создание процессов, происходящих из команд[PSExec](attack-surface-reduction.md#block-process-creations-originating-from-psexec-and-wmi-commands)и WMI", поскольку это правило несовместимо с управлением с помощью Microsoft Endpoint Configuration Manager. Правило блокирует команды WMI, которые клиент Configuration Manager использует для правильной работы. 

> [!TIP]
> После работы на устройстве можно выполнить тест обнаружения, чтобы убедиться, что устройство правильно вошел в службу. Дополнительные сведения см. в таблице Выполнить тест обнаружения на недавно созданном [устройстве Microsoft Defender для конечных точек.](run-detection-test.md)

В рамках вашей работы в центре безопасности Майкрософт может потребоваться установить метку машины, чтобы более легко различать машины WVD. Дополнительные сведения см. в [добавлении тегов устройств, установив ключевое значение реестра.](machine-tags.md#add-device-tags-by-setting-a-registry-key-value) 

#### <a name="other-recommended-configuration-settings"></a>Другие рекомендуемые параметры конфигурации 

При создании золотого изображения может потребоваться также настроить начальные параметры защиты. Дополнительные сведения см. [в других рекомендуемых параметрах конфигурации.](configure-endpoints-gp.md#other-recommended-configuration-settings) 

Кроме того, если вы используете профили пользователей FSlogix, рекомендуем исключить из защиты всегда следующие файлы: 

**Исключение файлов:** 

`%ProgramFiles%\FSLogix\Apps\frxdrv.sys`

`%ProgramFiles%\FSLogix\Apps\frxdrvvt.sys`

`%ProgramFiles%\FSLogix\Apps\frxccd.sys`

`%TEMP%\*.VHD`

`%TEMP%\*.VHDX`

`%Windir%\TEMP\*.VHD`

`%Windir%\TEMP\*.VHDX`

`\\storageaccount.file.core.windows.net\share\*\*.VHD`

`\\storageaccount.file.core.windows.net\share\*\*.VHDX`

**Исключить процессы:**

`%ProgramFiles%\FSLogix\Apps\frxccd.exe`

`%ProgramFiles%\FSLogix\Apps\frxccds.exe`

`%ProgramFiles%\FSLogix\Apps\frxsvc.exe`
