# <a name="the-email-entity-page"></a>Страница сущности "Электронная почта"

**В этой статье:**
- [Достижение страницы сущности электронной почты](#reach-the-email-entity-page)
- [Чтение страницы сущности электронной почты](#read-the-email-entity-page)
- [Использование вкладок страниц сущности электронной почты](#use-email-entity-page-tabs)
- [Новая страница сущности электронной почты](#new-to-the-email-entity-page)

Администраторы Microsoft Defender для Office 365 E5 и Defender для Office P1 и P2 имеют 360-градусное представление электронной почты с помощью страницы сущности **электронной почты**. Эта страница электронной почты была создана для улучшения сведений, доставленных на вылете "сведения электронной почты" [обозревателя угроз.](threat-explorer-views.md)

## <a name="reach-the-email-entity-page"></a>Достижение страницы сущности электронной почты

Страница сущности электронной почты доступна на портале Microsoft 365 Defender <https://security.microsoft.com> () в **обозревателе &** \> **электронной почты.** Или, чтобы перейти непосредственно на **страницу Explorer,** используйте <https://security.microsoft.com/threatexplorer> .

В **Explorer** выберите объект электронной почты, которое вы изучаете. Золотая слитка будет отображаться в верхней части вылетной почты для этой почты. Это приглашение на новую страницу считывания "Попробуйте нашу новую страницу сущности электронной почты с обогащенными данными...". Выберите для просмотра новой страницы.

:::image type="content" source="../../media/email-entities-1-navigation-to-ee.png" alt-text="Вы увидите золотой баннер со словами *Try out our new email entity page with enriched data* to navigate to the new experience.":::

:::image type="content" source="../../media/email-entities-2-eep.png" alt-text="На этом рисунке страницы сущности электронной почты основное внимание уделяется заголовкам, которые вы увидите. Обратите внимание, что заглавная почта отображается здесь.":::

> [!NOTE]
> Разрешения, необходимые для просмотра и использования этой страницы, такие же, как и для **просмотра Explorer.** Администратор должен быть членом глобального администратора или глобального читателя или администратора безопасности или читателя безопасности. Дополнительные сведения см. в статье [Разрешения на портале Microsoft 365 Defender](permissions-microsoft-365-security-center.md).

1. Наиболее необходимые поля находятся на левой стороне вылета. Эти сведения являются "липкими", что означает, что они закреплены слева независимо от того, на какой вкладке вы перемещаете в остальной части вылета.

2. В правом верхнем углу находятся действия, которые можно принять по электронной почте. Любые действия, которые можно принять через **Explorer,** также будут доступны на странице сущности электронной почты.

    :::image type="content" source="../../media/email-entities-5-preview.png" alt-text="График страницы сущности электронной почты с выделенной стороной *right* на этот раз. 3. Более глубокий анализ можно сделать путем сортировки по остальной части страницы. Проверьте сведения об обнаружении электронной почты, состоянии проверки подлинности электронной почты и загонах. Эту область следует искать в индивидуальном порядке, но сведения в этих вкладок доступны для любой электронной почты.

### <a name="use-email-entity-page-tabs"></a>Использование вкладок страниц сущности электронной почты

Вкладки на верхней части страницы сущности позволяют эффективно исследовать электронную почту.

1. **Хронология.** В представлении временной шкалы для электронной почты (в временной шкале **Explorer)** показана исходная доставка для событий после доставки, которые происходят по электронной почте. Для сообщений электронной почты, которые не имеют действий после доставки, в представлении показана оригинальная строка доставки в представлении временной шкалы. События, такие как: автоматическая очистка нулевого часа (ZAP), Исправление, URL-адреса и т. д. Из таких источников, как: система, администратор и пользователь, покажитесь здесь, в порядке, в котором они произошли.
2. **Анализ.** Анализ показывает поля, которые помогают администраторам подробно анализировать электронную почту. В случаях, когда администраторам необходимо больше понимать сведения об обнаружении, отправителье/получателе и сведениях о проверке подлинности электронной почты, они должны использовать вкладку Analysis. Ссылки для вложений и URL-адресов также находятся на этой странице в статье "Связанные сущности". Здесь про номер вложения и выявленные угрозы, и щелкнув, вы сразу же перейдите на страницы вложения и URL-адрес. Эта вкладка также имеет параметр View header, чтобы *показать заглавную* почту . Администраторы могут сравнивать любые сведения из загона электронной почты, рядом с информацией на главной панели, для ясности.
3. **Вложения.** При этом рассматриваются вложения, найденные в электронной почте, и другие сведения, найденные на вложениях. Количество показанных вложений в настоящее время ограничено 10. Обратите внимание, что сведения о детонации вложений, которые признаны вредоносными, также показаны здесь.
4. **URL-адреса.** На этой вкладке перечислены URL-адреса, найденные в электронной почте, с другими сведениями о URL-адресах. Количество URL-адресов ограничено 10 прямо сейчас, но эти 10 имеют приоритет, чтобы сначала показать *вредоносные URL-адреса.* Приоритизация экономит время и работу над догадками. Url-адреса, которые были признаны вредоносными и взорванными, также будут показаны здесь.
5. **Похожие сообщения** электронной почты. На этой вкладке перечислены все электронные письма, похожие на код сетевого сообщения *+ комбинация* получателей, специфична для этого сообщения. Сходство основано только на *теле сообщения.* Определения, сделанные по почте, чтобы классифицировать их как "аналогичные", не включают рассмотрение *вложений*.

## <a name="new-to-the-email-entity-page"></a>Новая страница сущности электронной почты

Есть новые возможности, которые приходят с этой страницей сущности электронной почты. Вот список.

### <a name="email-preview-for-cloud-mailboxes"></a>Предварительный просмотр электронной почты для облачных почтовых ящиков

Администраторы могут просматривать сообщения электронной  почты в облачных почтовых ящиках, если они по-прежнему присутствуют в облаке. В случае мягкого удаления (администратором или пользователем) или ZAP (для карантина) сообщения электронной почты больше не присутствуют в расположении Cloud. В этом случае администраторы не смогут просматривать эти конкретные сообщения. Сообщения электронной почты, которые были отброшены или где доставка не удалась, никогда не были отправлены в почтовый ящик. В результате администраторы также не смогут просматривать эти сообщения.

> [!WARNING]
> Просмотр сообщений электронной почты требует особой роли под названием **Preview**. Эту роль можно добавить на портале Microsoft 365 Defender, как описано в & роли совместной работы на [Microsoft 365 Defender портале](permissions-microsoft-365-security-center.md#email--collaboration-roles-in-the-microsoft-365-defender-portal). Возможно, вам потребуется создать новую группу ролей **&**  электронной почты и добавить  роль Preview в эту новую группу ролей или добавить роль Preview в группу ролей, которая позволяет администраторам в организации работать в **Explorer**.

### <a name="detonation-details"></a>Сведения о детонации

Эти сведения являются специфическими для вложений электронной почты и URL-адресов. Пользователи могут увидеть эти сведения, переехав в Explorer и применив набор фильтра технологий обнаружения для детонации файлов или детонации URL-адреса.  Сообщения электронной почты, фильтруемой для детонации файлов, будут содержать вредоносный файл с сведениями о детонации, а те, которые фильтруются для URL-адресов, содержат вредоносный URL-адрес и сведения о детонации.

Пользователи увидят обогащенные сведения о детонации известных вредоносных вложений или URL-адресов, найденных в их сообщениях электронной почты, которые были взорваны для конкретного клиента. Он будет состоять из цепи детонации, сводки детонации, скриншота и наблюдаемого поведения, чтобы помочь клиентам понять, почему вложение или URL-адрес были признаны вредоносными и детонировали.

1. *Цепочка детонации*. Детонация одного файла или URL-адреса может вызвать несколько детонаций. Цепочка детонации отслеживает путь детонации, включая исходный вредоносный файл или URL-адрес, который вызвал вердикт, и все другие файлы или URL-адреса, вызванные детонацией. Эти URL-адреса или присоединенные файлы могут не присутствовать непосредственно в электронной почте, но этот анализ имеет важное значение для определения того, почему файл или URL-адрес были признаны вредоносными.  

    > [!NOTE]
    > Это может показать только элемент верхнего уровня, если ни одно из связанных с этим сущностями не было обнаружено проблем или не было взорвано.

1. *В* сводке по детонации приводится базовая сводка для детонации, например время *анализа,* время, когда произошла детонация, ОС и приложение, операционная система и приложение, в которых произошла детонация, размер файла и причина вердикта.
1. *На скриншотах* покажут снимки экрана, захваченные во время детонации. Во время детонации может быть несколько скриншотов. Скриншоты не запечатлены для
    - Файлы типа контейнера, такие как .zip или .rar.
    - Если URL-адрес открывается в ссылку, которая непосредственно загружает файл. Однако вы увидите скачаный файл в цепочке детонации.
1. *Сведения* о поведении — это экспорт, который отображает сведения о поведении, такие как точные события, которые произошли во время детонации, и наблюдаемые данные, содержащие URL-адреса, IPs, домены и файлы, найденные во время детонации (и могут быть проблематичными или доброкачественными). Следует помнить, что не может быть никаких сведений о поведении для:
    - Контейнерные файлы, .zip или .rar, в котором находятся другие файлы.

:::image type="content" source="../../media/email-entities-6-detonation-page.png" alt-text="Снимок экрана сводки детонации, показывающая цепочку, сводку, сведения о детонации и снимок экрана под заголовком *Deep Analysis*.":::

### <a name="other-innovations"></a>Другие инновации

*Теги.* Это теги, применяемые к пользователям. Если пользователь является получателем, администраторы увидят тег *получателя.* Кроме того, если пользователь является отправитель, тег *отправитель.* Это будет отображаться в левой стороне страницы сущностями  электронной почты (в части, описанной как липкой и, таким образом, закрепленной на странице).

*Последнее расположение доставки:* последнее расположение доставки — это расположение, где после системных действий, таких как ZAP или действия администратора, например Move to Deleted Items, завершите. Последнее расположение доставки не предназначено для информирования администраторов о текущем расположении *сообщения.* Например, если пользователь удаляет сообщение или перемещает его в архив, расположение доставки не обновляется. Однако, если было принято системное действие и обновлено расположение (например, zaP, в результате чего сообщение электронной почты переходит на карантин), это обновит последнее расположение доставки в карантин.

*Сведения по* электронной почте. Сведения, необходимые для более глубокого понимания электронной почты, доступной на *вкладке Analysis.*

- Exchange правила транспорта (также известные как правила потока почты или *etRs)*: Эти правила применяются к сообщению на транспортном уровне и имеют приоритет над вердиктами о фишинге и спаме. Они могут быть созданы и изменены только в центре администрирования Exchange, но если какой-либо ETR применяется к сообщению, имя ETR и GUID будут показаны здесь. Ценная информация для целей отслеживания.

- *Переопределения* системы. Это средство исключения расположения доставки, предназначенного для сообщения, путем переопределения расположения доставки, предоставленного системой (в зависимости от технологии обнаружения и угрозы).

- *Правило нежелательного почтового* ящика: "Нежелательное" скрыто правило "Входящие", включенное по умолчанию в каждом почтовом ящике.
  - Когда правило нежелательной почты включено в почтовом ящике, Exchange Online Protection EOP может перемещать сообщения в Нежелательное по некоторым критериям. Этот шаг может основываться на действии фильтрации нежелательной почты для перемещения сообщения в папку нежелательной почты *или* в списке заблокированных отправителей в почтовом ящике. Отключение правила нежелательной почты предотвращает доставку сообщений в  папку нежелательной почты на основе списка Сейф отправителей в почтовом ящике.
  - Если правило нежелательной  почты отключено в почтовом ящике, EOP не может перемещать сообщения в папку нежелательной почты на основе действия по фильтрации нежелательной почты *Переместить* сообщение в папку нежелательной почты или коллекцию безопасных списков на почтовом ящике.

- *Объемный уровень жалоб (BCL)*: объемный уровень жалобы (BCL) сообщения. Более высокий BCL указывает на то, что массовое почтовое сообщение с большей вероятностью вызывает жалобы (естественный результат, если сообщение электронной почты, скорее всего, является нежелательной почтой).

- *Уровень доверия нежелательной почты (SCL)*— уровень доверия к нежелательной почте (SCL) сообщения. Чем больше значение, тем вероятнее, что сообщение окажется спамом.

- *Доменное имя.* Является ли отправитель доменным именем.

- *Владелец домена:* указывает владельца отправляемого домена.

- *Расположение домена.* Указывает расположение отправляемого домена.

- *Дата создания домена:* указывает дату создания отправляемого домена. Недавно созданный домен — это то, к чему можно относиться с осторожностью, если другие сигналы указывают на подозрительное поведение.

*Проверка подлинности* электронной почты. Методы проверки подлинности электронной почты, используемые Microsoft 365, включают SPF, DKIM и DMARC.

- Структура политики отправитель **(SPF):** Описывает результаты проверки SPF для сообщения. Возможные значения:
  - Pass (IP-адрес): проверка SPF для переданного сообщения и включает IP-адрес отправитель. Клиент уполномочен отправлять или пересылать сообщение электронной почты от имени домена отправителя.
  - Сбой (IP-адрес): проверка SPF для сообщения не удалась и включает IP-адрес отправитель. This is sometimes called hard fail.
  - Softfail (причина): запись SPF назначила хост как не разрешенный для отправки, но находится в процессе перехода.
  - Нейтральный. В записи SPF явно говорится, что она не утверждает, разрешен ли отправлять IP-адрес.
  - Нет. Домен не имеет записи SPF, или запись SPF не оценивает результат.
  - Температор. Произошла временная ошибка. Например, ошибка DNS. В дальнейшем проверка может быть пройдена.
  - Permerror. Произошла постоянная ошибка. Например, запись SPF для домена имеет неправильный формат.

- DomainKeys Identified Mail **(DKIM):**
  - Pass. Указывает проверку DKIM для переданного сообщения.
  - Fail (причина): указывает, что проверка DKIM на сообщение сбой и почему. Например, когда сообщение не подписано или подпись не проверена.
  - Нет. Указывает, что сообщение не было подписано. Это может быть связано с тем, что для домена задана запись DKIM, которая не приводит к результату.

- Проверка подлинности, отчетности и соответствия на основе доменных сообщений **(DMARC):**
  - Pass. Указывает проверку DMARC для переданного сообщения.
  - Fail. Указывает на сбой проверки DMARC для сбой сообщения.
  - Bestguesspass: указывает на отсутствие записи DMARC TXT для домена, но если бы она существовала, проверка DMARC для сообщения прошла бы.
  - Нет. Указывает на отсутствие записи DMARC TXT для домена отправки в DNS.

*Композитная* проверка подлинности. Это значение используется Microsoft 365 для объединения проверки подлинности электронной почты, например SPF, DKIM и DMARC, чтобы определить подлинность сообщения. В качестве основы для оценки используется домен *"От:* почта".
