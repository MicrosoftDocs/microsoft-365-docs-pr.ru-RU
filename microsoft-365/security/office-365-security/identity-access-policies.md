# <a name="common-identity-and-device-access-policies"></a>Основные политики доступа для удостоверений и устройств

**Относится к**
- [Exchange Online Protection](https://go.microsoft.com/fwlink/?linkid=2148611)
- [Microsoft Defender для Office 365 (план 1) и план 2](https://go.microsoft.com/fwlink/?linkid=2148715)
- Azure

В этой статье описываются распространенные рекомендуемые политики для защиты доступа к облачным службам Microsoft 365, включая локальное приложение, опубликованное с помощью прокси приложения Azure Active Directory (Azure AD).

В этом руководстве обсуждается развертывание рекомендуемых политик в новой среде. Настройка этих политик в отдельной лабораторной среде позволяет понять и оценить рекомендуемые политики перед промежуточной установкой выпуска в предпроизводильных и производственных средах. Ваша новая среда может быть облачной или гибридной, чтобы отразить ваши потребности в оценке.

## <a name="assigning-policies-to-groups-and-users"></a>Назначение политик группам и пользователям

Перед настройкой политик определите группы Azure AD, которые вы используете для каждого уровня защиты. Обычно базовая защита применяется ко всем в организации. Для пользователя, включенного в базовую и конфиденциальную защиту, будут применены все базовые политики, а также политики конфиденциальной информации. Защита является накопительной, и применяются самые строгие политики.

Рекомендуется создать группу Azure AD для исключения условного доступа. Добавьте эту группу во все политики условного доступа в параметре **"Исключить"** параметра **"Пользователи** и группы" в разделе **"Назначения".** Это дает способ предоставления доступа пользователю при устранении неполадок доступа. Это рекомендуется только в качестве временного решения. Отслеживайте изменения в этой группе и убедитесь, что группа исключений используется только по назначению.

Ниже приводится пример назначения групп и исключений для требования многофаксной многофаксной многофазыковой поддержки.

![Примеры назначений групп и исключений для политик MFA](../../media/microsoft-365-policies-configurations/identity-access-policies-assignment.png)

Вот результаты:

- Все пользователи должны использовать MFA, если риск для входов средний или высокий.

- Члены группы "Исполнительный персонал" должны использовать MFA при низком, среднем или высоком риске при входе.

  В этом случае члены группы "Исполнительный персонал" соответствуют базовым и конфиденциальным политикам условного доступа. Элементы управления доступом для обеих политик объединяются, что в данном случае эквивалентно конфиденциальной политике условного доступа.

- Для использования MFA всегда требуются члены группы Top Secret Project X.

  В этом случае члены группы "Проект X верхнего секрета" соответствуют базовым и строго регулируемым политикам условного доступа. Элементы управления доступом для обеих политик объединены. Поскольку контроль доступа для строго регулируемой политики условного доступа является более строгим, он используется.

Будьте осторожны при применении более высоких уровней защиты к группам и пользователям. Например, члены группы "Проект X с top secret" должны будут использовать MFA при каждом входе, даже если они не работают над строго регулируемым содержимым для Project X.

Все группы Azure AD, созданные в рамках этих рекомендаций, должны быть созданы как группы Microsoft 365. Это важно для развертывания меток конфиденциальности при защите документов в Microsoft Teams и SharePoint.

![Снимок экрана для создания групп Microsoft 365](../../media/microsoft-365-policies-configurations/identity-device-AAD-groups.png)

## <a name="require-mfa-based-on-sign-in-risk"></a>Требовать многофаксную оценку на основе риска при входе

Пользователи должны зарегистрироваться для получения MFA, прежде чем требовать его использования. Если у вас есть microsoft 365 E5, Microsoft 365 E3 с надстройка Identity & Threat Protection, Office 365 с EMS E5 или отдельные лицензии Azure AD Premium P2, вы можете использовать политику регистрации MFA с защитой идентификации Azure AD, чтобы требовать регистрацию пользователей для многофаксальной идентификации. Необходимые [условия](identity-access-prerequisites.md) включают регистрацию всех пользователей с помощью MFA.

После регистрации пользователей вы можете потребовать многофаксальный доступ для регистрации с помощью новой политики условного доступа.

1. Войдите на [портал Azure](https://portal.azure.com) со своими учетными данными.
2. В списке служб Azure выберите **Azure Active Directory.**
3. В **списке "Управление"** **выберите**"Безопасность" и выберите **"Условный доступ".**
4. Выберите **новую политику** и введите имя новой политики.

## <a name="apply-app-data-protection-policies"></a>Применение политик защиты данных APP

Политики защиты приложений (APP) определяют, какие приложения разрешены и какие действия они могут делать с данными вашей организации. Доступные в APP варианты позволяют организациям адаптировать защиту к конкретным потребностям. Для некоторых может быть не очевидно, какие параметры политики необходимы для реализации полного сценария. Чтобы помочь организациям приоритизировать защиту конечных точек мобильных клиентов, корпорация Майкрософт представила таксономию для платформы защиты данных APP для управления мобильными приложениями для iOS и Android.

Структура защиты данных APP имеет три разных уровня конфигурации, каждый из которых построен на предыдущем уровне:

- **Базовая защита корпоративных** данных (уровень 1) обеспечивает защиту приложений с помощью ПИН-кода и шифрования, а также выполняет выборочное выполнение операций стираемой данных. Для устройств с Android этот уровень проверяет подтверждение устройства с Android. Это конфигурация начального уровня, которая обеспечивает аналогичный контроль защиты данных в политиках почтовых ящиков Exchange Online и представляет ИТ-пользователей и пользователей app.
- **Расширенная защита корпоративных данных** (уровень 2) включает механизмы предотвращения утечки данных APP и минимальные требования к ОС. Такая конфигурация применима для большинства мобильных пользователей, которые имеют доступ к данным о работе или учебном замещающих устройствах.
- **Высокий уровень защиты корпоративных** данных (уровень 3) представляет расширенные механизмы защиты данных, улучшенную конфигурацию ПИН-кодов и app Mobile Threat Defense. Эта конфигурация является желательной для пользователей, которые имеют доступ к данным высокого риска.

Чтобы просмотреть конкретные рекомендации для каждого уровня конфигурации и минимальных приложений, которые необходимо защитить, просмотрите структуру защиты данных с помощью [политик защиты приложений.](https://docs.microsoft.com/mem/intune/apps/app-protection-framework)

Используя принципы, изложенные в конфигурациях доступа к удостоверениям и [устройствам,](microsoft-365-policies-configurations.md)базовый уровень защиты и уровень защиты конфиденциальных данных тесно соотнося с корпоративными расширенными настройками защиты данных уровня 2. Уровень защиты строго регулируемых данных сопоошел с корпоративными высокими настройками защиты данных уровня 3.

|Уровень защиты|Политика защиты приложений|Дополнительные сведения|
|---|---|---|
|Базовый уровень|[Расширенная защита данных уровня 2](https://docs.microsoft.com/mem/intune/apps/app-protection-framework#level-2-enterprise-enhanced-data-protection)|Параметры политики, примененные на уровне 2, включают все параметры политики, рекомендуемые для уровня 1, и только добавляют или обновляют следующие параметры политики, чтобы реализовать больше элементов управления и более сложную конфигурацию, чем уровень 1.|

|Строго регулируемый уровень|[Высокий уровень защиты корпоративных данных уровня 3](https://docs.microsoft.com/mem/intune/apps/app-protection-framework#level-3-enterprise-high-data-protection)|Параметры политики, примененные на уровне 3, включают все параметры политики, рекомендуемые для уровней 1 и 2, и только добавляют или обновляют следующие параметры политики, чтобы реализовать больше элементов управления и более сложную конфигурацию, чем уровень 2.|
|

1. Вручную создайте политики, следуя шагам в руководстве по созданию и развертыванию политик защиты приложений [с помощью Microsoft Intune.](https://docs.microsoft.com/mem/intune/apps/app-protection-policies)

## <a name="require-approved-apps-and-app-protection"></a>Требовать утвержденные приложения и защиту приложений

Чтобы применить политики защиты ПРИЛОЖЕНИЙ, примененные в Intune, необходимо создать политику условного доступа, которая требует утверждения клиентских приложений и условий, установленных в политиках защиты ПРИЛОЖЕНИЙ.

Для принудительного соблюдения политик защиты приложений требуется набор политик, описанных в политике "Требовать защиту приложений" для доступа к облачным приложениям с [условным доступом.](https://docs.microsoft.com/azure/active-directory/conditional-access/app-protection-based-conditional-access) Эти политики включены в этот рекомендуемый набор политик конфигурации удостоверений и доступа.

Чтобы создать политику условного доступа, требуемую для утвержденных приложений и защиты приложений, следуйте шагу 1 "Шаг 1. Настройка политики условного доступа Azure AD для Microsoft 365" в сценарии [1.    > [!NOTE]
   > Эта политика гарантирует, что мобильные пользователи смогут получать доступ ко всем конечным точкам Office с помощью соответствующих приложений.

Если вы включите мобильный доступ к Exchange Online, реализуем блокировку клиентов [ActiveSync,](secure-email-recommended-policies.md#block-activesync-clients)Exchange ActiveSync клиентов, которые могут подключаться к Exchange Online с помощью базовой проверки подлинности. Эта политика не изображена на рисунке в верхней части этой статьи. Он описан и изображен в рекомендациях [политики для защиты электронной почты.](secure-email-recommended-policies.md)

Чтобы создать политику условного доступа, для которой требуется Edge для iOS и Android, выполните "Шаг 2. Настройка политики условного доступа Azure AD для Microsoft 365" в сценарии [2.](https://docs.microsoft.com/azure/active-directory/conditional-access/app-protection-based-conditional-access#scenario-2-browser-apps-require-approved-apps-with-app-protection-policies)Для браузерных приложений требуются утвержденные приложения с политиками защиты приложений, что позволяет Edge для iOS и Android, но блокирует подключение других веб-браузеров мобильных устройств к конечным точкам Microsoft 365.

 Эти политики используют элементы управления предоставлением ["Требовать утвержденное клиентские приложения"](https://docs.microsoft.com/azure/active-directory/conditional-access/concept-conditional-access-grant#require-approved-client-app) и ["Требовать политику защиты приложений".](https://docs.microsoft.com/azure/active-directory/conditional-access/concept-conditional-access-grant#require-app-protection-policy)

Наконец, блокировка устаревшей проверки подлинности для других клиентских приложений на устройствах с iOS и Android гарантирует, что эти клиенты не смогут обойти политики условного доступа. Если вы следуете указаниям из этой статьи, вы уже настроили заблокированные клиенты, которые не поддерживают [современную проверку подлинности.](#block-clients-that-dont-support-multi-factor)

## <a name="define-device-compliance-policies"></a>Определение политик соответствия устройств требованиям

Политики соответствия устройств требованиям определяют требования, которые должны соответствовать устройствам, чтобы их можно было определить как совместимые. Политики соответствия устройств Intune создаются в Центре администрирования Microsoft Endpoint Manager.

Необходимо создать политику для каждого компьютера, телефона или планшетной платформы:

- Администратор устройств с Android
- Android Enterprise
- iOS/iPadOS
- macOS
- Windows 8.1 и более поздних
- Windows 10 и более поздние версии

Чтобы создать политики соответствия устройств требованиям, войдите в Центр администрирования [Microsoft Endpoint Manager](https://endpoint.microsoft.com) с учетными данными администратора и перейдите к политикам соответствия  \>  \> **устройств.** Выберите **"Создать политику"**.

Чтобы развертывать политики соответствия устройств требованиям, они должны быть назначены группам пользователей. Вы назначаете политику после ее создания и сохранения. В Центре администрирования выберите политику, а затем **выберите "Назначения".** После выбора групп, которые вы хотите получить  политику, выберите "Сохранить", чтобы сохранить это назначение группы, и разверните политику.

Пошаговая инструкция по созданию политик соответствия требованиям в Intune см. в документации по Intune по созданию политики соответствия требованиям в [Microsoft Intune.](https://docs.microsoft.com/mem/intune/protect/create-compliance-policy)

## <a name="require-compliant-pcs-but-not-compliant-phones-and-tablets"></a>Требовать совместимые компьютеры (но не совместимые телефоны и планшеты)

Перед добавлением политики, требующей компьютеров, удовлетворяющим требованиям, необходимо зарегистрировать устройства для управления в Intune. Перед регистрацией устройств в Intune рекомендуется использовать многофакторную проверку подлинности, чтобы обеспечить владение устройством пользователя.

Чтобы требовать компьютеры, совместимые с этой возможностью, с:

1. Войдите на [портал Azure](https://portal.azure.com) со своими учетными данными.
2. В списке служб Azure выберите **Azure Active Directory.**
3. В **списке "Управление"** выберите **"Безопасность"** и выберите **"Условный доступ".**
4. Выберите **новую политику** и введите имя новой политики.

5. В **области "Назначения"** выберите **"Пользователи и группы"** и включив пользователей, к которым будет применяться политика. Также исключите группу исключений условного доступа.

6. В **области "Назначения"** выберите **облачные приложения или действия.**

7. For **Include**, choose **Select apps > Select**, and then select the desired apps from the Cloud **apps** list. Например, выберите Exchange Online. Choose **Select** when done.

8. Чтобы требовать совместимые компьютеры (но не совместимые телефоны и планшеты), в области "Назначения" выберите "Условия **> устройства".**  Выберите **"Да"** **для настройки.** Choose  **Select device platforms,** select **Windows** and **macOS**, and then choose **Done**.

9. В **области элементов управления Access** выберите **"Предоставить".**

10. Choose **Grant access** and then check Require device to be marked as **compliant**. Для нескольких элементов управления выберите **"Требовать все выбранные элементы управления".** По завершению выберите **"Выбрать".**

11. Выберите **"Включить"** **для политики "Включить",** а затем выберите **"Создать".**

> [!NOTE]
> Перед включением этой политики убедитесь, что устройство соответствует требованиям. В противном случае вы получите блокировку и не сможете изменить эту политику, пока ваша учетная запись пользователя не будет добавлена в группу исключений условного доступа.
