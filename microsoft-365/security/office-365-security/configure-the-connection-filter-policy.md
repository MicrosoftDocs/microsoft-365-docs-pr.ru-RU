# <a name="configure-connection-filtering"></a>Настройка фильтрации подключений

[!INCLUDE [Microsoft 365 Defender rebranding](../includes/microsoft-defender-for-office.md)]


Если вы клиент Microsoft 365 с почтовыми ящиками в Exchange Online или клиентом отдельной службы Exchange Online Protection (EOP) без почтовых ящиков Exchange Online, вы используете фильтрацию подключений в EOP (в частности, политику фильтрации подключений по умолчанию), чтобы определить хорошие или ненадежные исходные почтовые серверы по их IP-адресам. Основные компоненты политики фильтрации подключений по умолчанию:

- **Список ip-адресов:** пропустить фильтрацию нежелательной почты для всех входящих сообщений с исходных почтовых серверов, которые указаны по IP-адресу или диапазону IP-адресов. Сценарии, в которых фильтрация нежелательной почты может [](#scenarios-where-messages-from-sources-in-the-ip-allow-list-are-still-filtered) по-прежнему происходить для сообщений из этих источников, см. в разделе "Сценарии, в которых сообщения из источников в списке ip-адресов по-прежнему фильтруются" далее в этой статье. Дополнительные сведения о том, как список ip-адресов должен [](create-safe-sender-lists-in-office-365.md)соответствовать общей стратегии надежных отправителей, см. в этой теме.

- **Список заблокированных** IP-адресов: блокировать все входящие сообщения с исходных серверов электронной почты, которые указаны по IP-адресу или диапазону IP-адресов. Входящие сообщения отклоняется, не помечаются как нежелательные, и дополнительная фильтрация не происходит. Дополнительные сведения о том, как список заблокированных IP-адресов [](create-block-sender-lists-in-office-365.md)должен вписаться в общую стратегию заблокированных отправителей, см. в этой теме.

- **Безопасный** список: список *надежных устройств* — это динамический список разрешаний в центр обработки данных Майкрософт, для настройки клиента не требуется. Корпорация Майкрософт определяет эти надежные источники электронной почты из подписок на различные сторонние списки. Вы включаете или отключате использование списка надежных надежных безопасную безопасную безопасную безопасную безопасную безопасную безопасную безопасную вы не можете настроить исходные почтовые серверы в списке надежных сообщений. Фильтрация нежелательной почты пропускается для входящих сообщений с почтовых серверов в списке надежных сообщений.

В этом разделе описывается настройка политики фильтрации подключений по умолчанию в Центре безопасности & и соответствия требованиям или в PowerShell (Exchange Online PowerShell для организаций Microsoft 365 с почтовыми ящиками в Exchange Online; автономный EOP PowerShell для организаций без почтовых ящиков Exchange Online). Дополнительные сведения о том, как EOP использует фильтрацию подключений, являются частью общих параметров защиты от нежелательной почты в организации, см. в этой [теме.](anti-spam-protection.md)

> [!NOTE]
> Список ip-адресов, список надежных и заблокированных IP-адресов — это часть общей стратегии, позволяющей разрешить или заблокировать электронную почту в организации. Дополнительные сведения см. в дополнительных сведениях о создании списков надежных [отправников](create-safe-sender-lists-in-office-365.md) и [создании списков заблокированных отправитель.](create-block-sender-lists-in-office-365.md)

## <a name="use-exchange-online-powershell-or-standalone-eop-powershell-to-modify-the-default-connection-filter-policy"></a>Изменение политики фильтрации подключений по умолчанию с помощью Exchange Online PowerShell или автономных EOP PowerShell

Используйте указанный ниже синтаксис.

```powershell
Set-HostedConnectionFilterPolicy -Identity Default [-AdminDisplayName <"Optional Comment">] [-EnableSafeList <$true | $false>] [-IPAllowList <IPAddressOrRange1,IPAddressOrRange2...>] [-IPBlockList <IPAddressOrRange1,IPAddressOrRange2...>]
```

**Примечания**:

- Допустимые значения IP-адреса или диапазона адресов:

  - Один IP-адрес: например, 192.168.1.1.

  - Диапазон IP-адресов: например, 192.168.0.1-192.168.0.254.

  - IP-адрес CIDR: например, 192.168.0.1/25. Допустимые значения сетевой маски: от /24 до /32.

- Чтобы *переписать все* существующие записи с помощью задаваемой вами значения, используйте следующий синтаксис: `IPAddressOrRange1,IPAddressOrRange2,...,IPAddressOrRangeN` .

- Чтобы *добавить или удалить* IP-адреса или диапазоны адресов, не затрагивая другие существующие записи, используйте следующий синтаксис: `@{Add="IPAddressOrRange1","IPAddressOrRange2",...,"IPAddressOrRangeN";Remove="IPAddressOrRange3","IPAddressOrRange4",...,"IPAddressOrRangeN"}`

- Чтобы очистить список ip-адресов или список заблокированных IP-адресов, используйте `$null` значение.

В этом примере настраивается список ip-адресов и список заблокированных IP-адресов с указанными IP-адресами и диапазонами адресов.

```powershell
Set-HostedConnectionFilterPolicy -Identity Default -IPAllowList 192.168.1.10,192.168.1.23 -IPBlockList 10.10.10.0/25,172.17.17.0/24
```

В этом примере из списка ip-адресов добавляются и удаляются указанные IP-адреса и диапазоны адресов.

```powershell
Set-HostedConnectionFilterPolicy -Identity Default -IPAllowList @{Add="192.168.2.10","192.169.3.0/24","192.168.4.1-192.168.4.5";Remove="192.168.1.10"}
```

Подробные сведения о синтаксисах и параметрах см. в описании [Set-HostedConnectionFilterPolicy.](https://docs.microsoft.com/powershell/module/exchange/set-hostedconnectionfilterpolicy)

## <a name="additional-considerations-for-the-ip-allow-list"></a>Дополнительные соображения по списку ip-адресов

В следующих разделах идентифицировать дополнительные элементы, которые необходимо знать при настройке списка ip-адресов.

### <a name="skip-spam-filtering-for-a-cidr-ip-outside-of-the-available-range"></a>Пропуск фильтрации нежелательной почты для IP-адреса CIDR за пределами доступного диапазона

Как описано выше в этой статье, в списке ip-адресов можно использовать только IP-адрес CIDR с маской сети /24-32. Чтобы пропустить фильтрацию нежелательной почты для сообщений с исходных почтовых серверов в диапазоне от /1 до /23, необходимо использовать правила потока почты Exchange (также известные как правила транспорта). Но мы не рекомендуем делать это, если это возможно, так как сообщения будут блокироваться, если IP-адрес в диапазоне IP-адресов CIDR от /1 до /23 отображается в любом из проприетарных или сторонних списков блокирований Майкрософт.

Теперь, когда вы полностью знаете о потенциальных проблемах, вы можете создать правило потока почты со следующими настройками (как минимум), чтобы сообщения с этих IP-адресов пропускали фильтрацию нежелательной почты:

- Условие **правила:** правило применяется, если IP-адрес отправитель находится в любом из этих диапазонов или точно соответствует (введите IP-адрес CIDR с сетевой маской \>  \>  \> /1–/23).

- Действие правила: **измените свойства сообщения,** задайте уровень достоверности нежелательной почты \> **(SCL)** \> **обход фильтрации нежелательной почты.**

Вы можете проверить правило, проверить его, активировать его в течение определенного периода времени и выбрать другие возможности. Мы рекомендуем протестировать правило в течение определенного времени перед его применением. Дополнительные сведения см. в под [управлением правилами потока почты в Exchange Online.](https://docs.microsoft.com/Exchange/security-and-compliance/mail-flow-rules/manage-mail-flow-rules)

### <a name="skip-spam-filtering-on-selective-email-domains-from-the-same-source"></a>Пропуск фильтрации нежелательной почты для выборочного домена электронной почты из одного источника

Как правило, добавление IP-адреса или диапазона адресов в список разрешить IP-адресов означает, что вы доверяете всем входящие сообщения из этого источника электронной почты. Но что делать, если этот источник отправляет электронную почту из нескольких доменов и вы хотите пропустить фильтрацию нежелательной почты для некоторых из этих доменов, но не для других? Для этого нельзя использовать только список ip-адресов, но этот список можно использовать в сочетании с правилом потока почты.

Например, исходный почтовый сервер 192.168.1.25 отправляет электронную почту из доменов contoso.com, fabrikam.com и tailspintoys.com, но нужно пропустить фильтрацию нежелательной почты только для сообщений от отправителей в fabrikam.com. Для этого сделайте следующее:

1. Добавьте 192.168.1.25 в список ip-адресов.

2. Настройте правило потока почты со следующими настройками (как минимум):

   - Условие **правила:** правило применяется, если IP-адрес отправитель находится в любом из этих диапазонов или точно соответствует \>  \>  192.168.1.25 (тот же IP-адрес или диапазон адресов, который вы добавили в список ip-адресов на предыдущем \> шаге).

   - Действие правила: **измените свойства сообщения,** чтобы установить уровень достоверности нежелательной почты \> **(SCL)** \> **0**.

   - Исключение правила: **домен отправитель** fabrikam.com (только домен или домены, которые нужно пропустить фильтрацию \>  \> нежелательной почты).

### <a name="scenarios-where-messages-from-sources-in-the-ip-allow-list-are-still-filtered"></a>Сценарии, в которых сообщения из источников в списке ip-адресов по-прежнему фильтруются

Сообщения с почтового сервера в списке ip-адресов по-прежнему подвергаются фильтрации нежелательной почты в следующих сценариях:

- IP-адрес в списке ip-адресов также настраивается в локальном входящий  соединителю на основе IP-адреса в любом клиенте Microsoft 365 (назовем его клиентом **А),** а клиент A и сервер EOP, которые сначала столкнулись с сообщением, должны быть в одном лесу *Active* Directory в центрах обработки данных Майкрософт. В этом сценарии **IPV:CAL**  добавляется в заглавные знаки сообщения для борьбы с нежелательной почтой (указывая, что сообщение обойдено фильтрацией нежелательной почты), но это сообщение [по-прежнему](anti-spam-message-headers.md) подвергается фильтрации нежелательной почты.

- Клиент, который содержит список ip-адресов и сервер EOP, который впервые столкнулся с сообщением, находятся в разных лесах *Active* Directory в центрах обработки данных Майкрософт. В этом сценарии **IPV:CAL** *не* добавляется в заглавные папки сообщений, поэтому к сообщению по-прежнему будет фильтроваться спам.
